## 미반영 (9건) 사유

### 1. blocked reason 요약 카드 — "이 고객에게 왜 못 보내는지" 표시

GPT 피드백이 원한 것: 알림톡 발송 전에 "128명 발송 가능 / 23명 야간시간 차단 / 11명 수신동의 없음 / 4명 최근구매 제외" 같은 요약을 보여주는 것.

**못 한 이유**: 이 판단을 하려면 "이 고객이 수신 동의를 했는가?", "지금이 야간(quiet hours)인가?", "최근에 메시지를 보낸 적이 있는가?" 등을 체크하는 로직이 서버에 있어야 한다. Revenue 프로젝트(별도 코드베이스)에는 이 로직(`crm_contact_policy_service`)이 있지만, 지금 우리 SEO 프로젝트 서버에는 없다. 서버에 로직이 없으니 화면에 보여줄 데이터를 만들 수 없다.

**해결 방법 — 선택지 2개 비교 후 판단:**

| 방법 | 장점 | 단점 |
|------|------|------|
| **A. Revenue API를 호출하는 proxy** | 로직이 한 곳에만 존재하여 규칙 변경 시 한 번만 수정하면 됨 | Revenue 서버가 꺼져 있으면 동작 안 함. 현재 Revenue 백엔드(Python FastAPI, 포트 8000)가 상시 실행되지 않고 있음 |
| **B. SEO 백엔드에 TypeScript로 이식** | SEO 서버만으로 독립 동작. Revenue 서버 의존 없음 | 같은 규칙이 Python과 TypeScript 두 곳에 존재하게 됨. 규칙 변경 시 양쪽 다 수정해야 함 |

**실제 코드를 조사한 결과:**
- Revenue의 `crm_contact_policy_service.py`는 266줄의 순수 로직이다. DB를 직접 조회하지 않고, 호출하는 쪽이 고객 정보(수신 동의, 최근 메시지 수, 마지막 구매일 등)를 딕셔너리로 넘기면 "보낼 수 있다/없다 + 차단 사유"를 반환한다.
- Revenue 서버는 Python FastAPI(포트 8000)이고, 현재 상시 실행되지 않고 있다.
- SEO 백엔드와 Revenue 백엔드는 **같은 PostgreSQL DB를 공유**하고 있다.

**판단: B(SEO에 이식)가 더 낫다.** 이유:
1. Revenue 서버가 상시 떠 있지 않으므로 proxy 방식은 불안정하다.
2. 로직이 266줄 순수 함수여서 TypeScript로 옮기기 어렵지 않다.
3. 양쪽이 같은 DB를 쓰고 있으므로, SEO 백엔드에서 고객 정보를 직접 조회하여 정책 판단 함수에 넣을 수 있다.
4. 규칙이 자주 바뀌는 것이 아니라 (야간시간 21~8시, 24시간 쿨다운, 7일 최대 3건 등 고정 상수) 두 곳에 있어도 실질적 동기화 부담이 크지 않다.

### 2. carry-over (탭 간 대상 전달) — "후속관리에서 고른 고객을 알림톡 탭으로 넘기기"

GPT 피드백이 원한 것: 후속관리 탭에서 "이 고객에게 알림톡 보내기" 버튼을 누르면, 알림톡 탭으로 이동하면서 그 고객이 수신자로 자동 입력되는 것.

**못 한 이유**: 현재 각 탭은 독립적으로 동작하고, 탭 사이에 데이터를 넘기는 구조가 없다. "후속관리 탭에서 선택한 고객 ID"를 "알림톡 탭"이 받아서 쓰려면, 공유 저장소(React Context 등)나 URL에 고객 ID를 붙이는 설계가 필요하다.

**해결 방법 — 선택지 2개 비교 후 판단:**

| 방법 | 장점 | 단점 |
|------|------|------|
| **A. URL query로 전달** (`/crm?tab=messaging&phone=010...`) | 구현 간단. 이미 `useSearchParams`로 탭 URL 동기화가 되어 있어 자연스럽게 확장 가능. 공유 링크로도 전달 가능 | URL에 개인 전화번호가 노출됨. 여러 명 선택 시 URL이 길어짐 |
| **B. React Context (페이지 레벨 공유 상태)** | 전화번호가 URL에 노출되지 않음. 대량 선택도 가능 | 페이지 새로고침 시 상태 유실. 공유 링크 불가 |

**판단: A(URL query)가 더 낫다.** 이유:
1. 지금 CRM은 내부 운영 도구이므로 URL에 전화번호가 노출되어도 보안 문제가 되지 않는다.
2. 이미 `useSearchParams`가 탭 전환에 쓰이고 있어서, `&phone=` 파라미터를 추가하는 것은 코드 변경이 최소화된다.
3. 새로고침해도 대상이 유지되고, 다른 사람에게 링크로 공유할 수 있다.
4. 대량 선택이 필요해지면 그때 Context로 확장하면 된다. 지금은 1명 테스트가 우선이므로 URL로 충분하다.

### 3. 실험 카드 운영 메타 — "이 실험이 지금 돌고 있는 건지, 멈춰 있는 건지"

GPT 피드백이 원한 것: 실험 카드에 "상태: 진행 중 / 채널: 알리고 / 마지막 동기화: 3시간 전 / 발송 로그: 128건" 같은 정보를 보여주는 것.

**못 한 이유**: 현재 백엔드 API(`/api/crm-local/experiments`)가 실험의 기본 정보(이름, 키, 가설)만 반환하고, 채널/마지막 동기화 시각/발송 로그 건수를 포함하지 않는다. 화면에 보여주려면 먼저 서버가 이 데이터를 계산해서 내려줘야 한다.

**해결 방법**: 백엔드 `GET /api/crm-local/experiments` API를 확장하여 각 실험 객체에 `status`(draft/running/completed), `channel`(channeltalk/aligo/manual), `last_sync_at`, `message_count`, `assignment_count` 필드를 추가 반환한다. 로컬 SQLite의 `crm_experiments`, `crm_assignment_log`, `crm_message_log` 테이블에서 집계하면 되고, DB 스키마 변경 없이 쿼리 확장만으로 가능하다. Codex에게 백엔드 확장을 위임하고, 완료 후 프론트 카드에 표시한다.

### 4. variant alias — "t2h_24h이 뭔 말이야"

GPT 피드백이 원한 것: 실험 결과 차트에서 `t2h_24h` 대신 `2시간 후 / 24시간` 같은 한글 이름도 함께 보여주는 것.

**못 한 이유**: variant 이름(`t2h_24h`)은 실험을 만들 때 운영자가 직접 입력한 값이다. 코드에서 이걸 자동으로 한글로 번역하면 오히려 혼란이 생긴다 (예: `control`을 "대조군"으로 바꾸면 원래 의도와 다를 수 있음).

**해결 방법**: 실험 생성 폼에 "variant 표시 이름(한글)" 입력 필드를 추가한다. 예를 들어 variant_key `t2h_24h`를 만들 때 alias에 "2시간 후 발송 / 24시간 윈도우"를 입력하게 한다. 백엔드 `crm_experiments` 테이블에 `variant_aliases` JSON 컬럼을 추가하면 스키마 변경 최소화로 가능하다. 이미 만들어진 실험은 수정 API(`PATCH /api/crm-local/experiments/:key`)로 alias를 후에 넣을 수 있게 한다.

### 5. 차트 통계 경고 — "이 결과를 믿어도 되는 건지"

GPT 피드백이 원한 것: 실험 결과 차트 위에 "표본 부족 — 아직 판단하기 이름" 또는 "유의성 없음" 같은 경고를 붙이는 것.

**못 한 이유**: "이 결과가 통계적으로 의미 있는지"를 판단하려면 p-value 계산이나 최소 표본 크기 검증을 서버에서 해야 한다. 현재 백엔드에 이 로직이 없고, 프론트에서 임의로 "표본 100명 미만이면 경고" 같은 규칙을 넣으면 부정확할 수 있다.

**해결 방법 — 선택지 2개 비교 후 판단:**

| 방법 | 장점 | 단점 |
|------|------|------|
| **A. 프론트에서 단순 규칙 적용** (예: variant별 30명 미만이면 "표본 부족" 경고) | 백엔드 변경 없이 즉시 가능 | 30명이라는 기준이 통계적으로 엄밀하지 않음 |
| **B. 백엔드에서 p-value/신뢰구간 계산** | 통계적으로 정확. `significant/not_significant/insufficient` 판정 가능 | 백엔드에 통계 라이브러리(z-test 등) 구현 필요 |

**판단: A(프론트 단순 규칙)를 먼저 하고, B는 나중에 추가하는 것이 맞다.** 이유:
1. 지금 가장 위험한 것은 운영자가 "표본 12명의 결과를 실제 성과로 오해하는 것"이다. 이걸 막으려면 "정확한 p-value"보다 "최소한의 경고"가 먼저다.
2. 프론트에서 `variant별 배정 30명 미만 → "표본 부족, 참고용"`, `발송 0건 → "실험 미실행"` 이 2가지 규칙만 넣어도 오해 80%를 막을 수 있다.
3. B는 실험 규모가 커지고 실제 의사결정에 쓰기 시작할 때 추가하면 된다.

### 6. 상단 pill 재설계 — "SQLite 우선, Revenue bridge 참고용" 같은 기술 문구를 운영자용으로"

GPT 피드백이 원한 것: CRM 페이지 상단의 회색 pill들("상담 액션 즉시 실행", "실험 데이터는 로컬 SQLite 우선" 등)이 개발자 언어로 되어 있으니, "오늘의 상태", "오늘의 blocker", "바로 할 행동" 같은 운영자 언어로 바꾸라는 것.

**못 한 이유**: pill 내용을 바꾸려면 "오늘 상태"를 서버에서 실시간으로 판단해야 한다 (예: "오늘 후속 연락 대상 12명", "발송 가능 템플릿 3개"). 현재 이런 요약 API가 없다. pill 텍스트를 단순히 한글로 바꾸는 것은 가능하지만, 동적 데이터가 들어가야 의미 있으므로 API 확장과 함께 진행해야 한다.

**해결 방법**: 2단계로 진행하는 것이 현실적이다.
1. **즉시 가능**: pill 텍스트를 기술 용어에서 운영 용어로 정적 교체. "실험 데이터는 로컬 SQLite 우선" → "실험 결과는 로컬 검증 모드 (실발송 전)". 코드 변경 5분.
2. **API 추가 후**: `GET /api/crm/today-summary`를 만들어서 "오늘 후속 연락 12명 / 발송 가능 템플릿 3개 / 실험 blocker 1건" 같은 동적 데이터를 반환. pill을 이 데이터 기반 카드로 교체. 이것은 기존 consultation/candidates API에 `today=true` 필터를 추가하는 것으로 시작할 수 있다.

### 7. 색 의미 단순화 — "파랑/초록/주황/빨강이 뒤섞여서 뭐가 중요한지 모르겠다"

GPT 피드백이 원한 것: 현재 민트(탭), 보라(step), 파랑(outline), 초록(버튼)이 동시에 나와서 시각적 의미가 흐리니, 파랑=행동, 초록=성공, 주황=경고, 빨강=실패, 회색=참고로 통일하라는 것.

**못 한 이유**: 색 체계를 바꾸려면 CSS 변수(디자인 토큰)를 전체적으로 재정의하고, 모든 컴포넌트에 적용해야 한다. 기능 개발 중에 시각 리팩토링을 동시에 하면 충돌이 생기므로, 기능 완성 후 한 번에 진행하는 것이 안전하다.

**해결 방법**: 기능 개발이 안정화된 후 한 번에 진행한다. 구체적으로:
1. `frontend/docs/design-tokens.md`에 이미 디자인 토큰 문서가 있다. 여기에 색 의미 규칙을 확정한다.
2. `--color-primary`(파랑, 주요 행동), `--color-success`(초록, 정상/성공), `--color-warning`(주황, 경고), `--color-error`(빨강, 실패/차단), `--color-muted`(회색, 참고)로 5가지 의미를 고정한다.
3. CRM 페이지의 모든 인라인 색상(`#6366f1`, `#10b981` 등)을 CSS 변수로 교체한다.
4. 지금은 기능 추가가 계속 되고 있으므로, 색 리팩토링을 동시에 하면 merge 충돌이 잦아진다. Phase 3 기능이 안정화된 후(예: P3-S4 80%+) 진행하는 것이 안전하다.

### 8. "오늘 해야 할 일" 나머지 3탭 — "이 탭 열면 오늘 뭘 해야 하는지 바로 보여야 한다"

GPT 피드백이 원한 것: 각 탭 맨 위에 "오늘 바로 연락할 대상 12명" (후속관리), "오늘 막힌 핵심 실험 1개" (실험), "오늘 테스트 가능한 템플릿 3개" (알림톡) 같은 카드를 보여주는 것.

**못 한 이유**: "오늘 기준" 데이터를 만들려면 서버에서 날짜 기반 필터링과 집계를 해야 한다. 현재 API는 기간 전체의 데이터만 반환하고, "오늘 할 일"만 뽑아서 요약하는 로직이 없다. 결제 귀속 탭에서는 live 0건이라는 단순 조건으로 blocker를 표시할 수 있었지만, 다른 탭은 더 복잡한 판단이 필요하다.

**해결 방법**: 각 탭별로 기존 API에 "오늘" 필터를 추가하는 것으로 시작한다.
- 후속관리: `GET /api/consultation/candidates?scenario=completed_followup`의 결과에서 상담일이 14일 이내인 건수만 세면 "오늘 연락할 대상 N명"을 표시할 수 있다. 새 API 없이 프론트에서 필터링 가능.
- 실험: `localData.stats.messages === 0`이면 "발송 0건 — 실험 미실행". 이미 구현됨.
- 알림톡: `templates.length`(사용 가능 템플릿 수) + `whitelist.length`(발송 가능 대상 수). 이미 화면에 있는 데이터로 "오늘 테스트 가능: 템플릿 N개, 대상 M명" 카드를 바로 만들 수 있다. 새 API 불필요.
- 결제 귀속: 이미 blocker headline 구현됨.

**결론**: 새 API 없이 기존 데이터만으로 4탭 모두 "오늘 할 일" 카드를 만들 수 있다. 프론트 코드 추가만으로 가능.

### 9. 탭 간 상태 전달 — "후속관리에서 실험으로, 실험에서 알림톡으로 연결"

carry-over(#2)와 같은 문제이며 해결 방법도 동일하다. URL query 기반으로 `?tab=messaging&phone=010...`을 전달하는 방식을 먼저 구현하고, 대량 선택이 필요해지면 React Context로 확장한다. #2에서 상세 비교표 참조.
