첨부 화면은 **실제로 2개 탭**으로 보입니다.
하나는 **알림톡 발송**, 다른 하나는 **실험 운영**이고, 각각 2장씩 거의 같은 화면이 중복 첨부된 상태입니다.

핵심부터 말하면, **우리 화면은 벤치마크 CRM보다 구조는 더 건전합니다.**
이유는 단순합니다.

* 우리는 `/crm 허브` 안에서 **후속 관리 / 실험 운영 / 알림톡 발송 / 결제 귀속**을 한 흐름으로 보게 설계돼 있습니다. 이 방향은 맞습니다. 
* 반면 벤치마크 CRM은 화면 수는 많지만, 실제 운영자가 “오늘 누구에게 무엇을 보낼지”까지 바로 이어지는 흐름은 약했습니다.

다만 지금 우리 화면은 아직 **운영용 위저드**와 **결과 해석용 대시보드**가 반쯤 섞여 있습니다.
그래서 보기엔 깔끔한데, 실제로는 운영자가 빨리 판단하기 어렵습니다.

## 한 줄 비교

* **벤치마크 CRM 강점**: 캠페인 인벤토리, 운영 모드/결과 모드 분리, 매트릭스형 시야
* **우리 CRM 강점**: 실험 장부 관점, 안전장치 관점, 허브형 구조
* **우리 CRM 약점**: 아직 `누구에게 → 무엇을 → 보낼 수 있나 → 결과가 어땠나`가 한 번에 이어지지 않음

## 1) 알림톡 발송 화면 피드백

이 화면은 벤치마크보다 **운영 안전성**은 더 좋습니다.
상단에

* 연동 상태
* 승인 템플릿 수
* 잔여 포인트
* 화이트리스트

를 보여주는 건 맞습니다.
또 `수동 테스트 / 후속 관리 대상 / 실험 대상`으로 대상을 나누려는 방향도 좋습니다.
이건 Phase 3 문서가 지향하는 `대상 선택 → 템플릿 선택 → 테스트 발송 → 결과 확인` 흐름과도 맞습니다.

그런데 지금 화면은 **실제로는 4단계 위저드가 아니라 1.5단계 화면**처럼 보입니다.

### 가장 큰 문제 5가지

**첫째, 단계 구조가 시각적으로 끊깁니다.**
상단 step bar는 `1. 대상 선택 → 2. 템플릿 → 3. 미리보기 → 4. 발송`인데,
실제 본문은 왼쪽에 `1. 대상 선택`, 오른쪽에 바로 `3. 미리보기`가 보입니다.

즉 사용자는 바로 이렇게 느낍니다.

* “템플릿 선택은 어디 있지?”
* “왜 미리보기가 벌써 비어 있지?”
* “다음 버튼을 누르면 뭐가 바뀌지?”

이건 위저드 UX에서 제일 안 좋은 패턴입니다.

**둘째, 오른쪽 미리보기 패널이 너무 빨리 비어 있습니다.**
아직 템플릿도 안 골랐는데 큰 빈 박스가 먼저 보입니다.
이건 정보 밀도를 떨어뜨리고, “아직 아무것도 안 됨” 느낌만 강하게 줍니다.

**셋째, 최근 발송 이력 테이블이 너무 크고 너무 빨리 나옵니다.**
지금 화면의 절반 이상이 이력 테이블입니다.
그런데 현재 운영자가 먼저 해야 할 일은 `과거 이력 보기`가 아니라 `이번 발송 준비`입니다.

**넷째, 상태 컬럼이 사실상 읽히지 않습니다.**
노란 점만 보이는데, 이건 상태가 아닙니다.
최소한

* 성공
* 테스트 성공
* exact-match 실패
* provider 실패
* 보류
  같은 텍스트 상태가 필요합니다.

**다섯째, 테스트 모드와 실발송 모드의 차이가 화면에서 충분히 강하게 안 보입니다.**
하단 안내 문구는 좋지만, 이건 배너 수준으로 더 위에 보여야 합니다.

## 2) 알림톡 발송 화면 개선안

### 바로 고쳐야 할 것

**A. 레이아웃을 4단계 위저드로 진짜 바꾸세요.**

지금 구조:

* 좌측 대상 선택
* 우측 큰 빈 미리보기
* 아래 거대한 이력

추천 구조:

* **1열:** 대상 선택
* **2열:** 템플릿 선택
* **3열:** exact-match 미리보기 + 발송

즉 한 화면 안에서
`누구에게` → `무엇을` → `어떻게 보일지`
가 동시에 보여야 합니다.

**B. 템플릿 선택을 화면 중앙의 핵심으로 올리세요.**
문서상 P3-S4에서 아직 남은 핵심이 `대상자 세그먼트 선택`, `템플릿 변수 치환`, `발송 결과 통계`, `배치 발송`, `UX 재설계`인데, 현재 화면은 그중 템플릿 선택과 변수 치환이 너무 약하게 보입니다. 

**C. 미리보기는 “빈 화면”이 아니라 “선택 요약 카드”부터 보여야 합니다.**
예를 들면:

* 선택 대상: 1명
* 발송 가능: 1명
* 차단: 0명
* 선택 템플릿: 없음
* 다음 단계: 템플릿 선택

이렇게 보여야 오른쪽 패널이 살아납니다.

**D. 최근 발송 이력은 접기 기본값으로 바꾸세요.**
기본은 5건만 보여주고, `전체 보기`를 누르면 테이블 확장하는 구조가 낫습니다.

**E. 상태 표시를 점 아이콘 중심에서 텍스트 중심으로 바꾸세요.**
예:

* 테스트 성공
* 발송 성공
* 대기
* 실패: 템플릿 불일치
* 실패: 차단 정책

### 다음 단계에서 꼭 넣어야 할 것

**1. exact-match 미리보기**
이건 optional이 아닙니다.
Phase 3 문서 기준으로 Aligo의 현재 핵심 blocker는 이미 `IP`가 아니라 **승인 템플릿과 실제 렌더링의 exact-match 불일치**입니다. 
그래서 화면에 꼭 있어야 하는 건:

* 승인 템플릿 원문
* 변수 치환 후 본문
* 버튼
* provider payload
* pass / fail

**2. blocked reason 요약 카드**
문서상 hard gate는 이미 `consent`, `claim review`, `quiet hours`, `cooldown`, `frequency cap`, `suppression`, `fallback`까지 코드화 방향이 잡혀 있습니다.
그러면 UI도 운영자가 바로 보게 해야 합니다.

* 발송 가능 128명
* quiet hours 차단 23명
* consent 없음 11명
* suppression 4명
  이렇게 보여야 합니다.

**3. 후속 관리/실험 탭에서 대상 carry-over**
지금은 알림톡 탭이 독립 툴처럼 보입니다.
좋은 UX는

* 후속 관리에서 고객 선택
* 알림톡 탭으로 바로 넘김
* 선택 대상 유지
  입니다.

지금 Phase 3 문서도 `대상 선택 → 템플릿 선택 → 발송 흐름`을 분명히 두고 있으니, 이 carry-over는 꼭 필요합니다.

## 3) 실험 운영 화면 피드백

이 화면은 벤치마크보다 **실험 철학은 훨씬 낫습니다.**
벤치마크는 캠페인 성과는 잘 보였지만, 인과적으로는 약했습니다.
반면 우리 화면은

* control
* variant
* 순매출
* 구매율 차이
  를 중심으로 보는 구조라서, 방향 자체는 더 맞습니다.
  이건 Phase 1 문서와도 정확히 맞습니다. Phase 1의 목적이 원래 `누가 메시지를 받았고, 실제로 샀는지`를 같은 장부로 보는 것이기 때문입니다.

그런데 지금 화면은 **운영자가 “이 실험이 실제로 라이브인지, 아직 로컬 검증인지”를 헷갈릴 수 있습니다.**

### 가장 큰 문제 6가지

**첫째, `발송 0건`인데 실험 성과 카드가 너무 진짜처럼 보입니다.**
이게 가장 중요합니다.

상단에

* 실험 수 3
* 배정 509
* 전환 264
* 발송 0

이렇게 보이는데, 아래는 바로

* 구매율 차이
* 인당 순매출 차이
* TREATMENT 순매출
* CONTROL 순매출
  이 나옵니다.

운영자는 이걸 보면 쉽게
“실험이 실제로 돌았고 treatment가 졌네”
라고 오해합니다.

그런데 문서상 현재 실험 장부는 **로컬 SQLite shadow 검증**이 핵심이고, 메시지 로그/발송 로그와 직접 연결은 아직 미완료 상태입니다. Phase 3 문서도 `발송 로그를 Phase 1 실험 원장과 직접 조인하는 운영 연결은 아직 없다`고 적고 있습니다.

즉 지금 화면은 **결과처럼 보이지만, 일부는 아직 plumbing 검증값**입니다.
이건 반드시 분리해서 보여줘야 합니다.

**둘째, variant 이름이 사람 언어가 아닙니다.**
`t2h_24h`, `t3w_3d` 같은 이름은 개발자에게는 괜찮지만 운영자에겐 느립니다.

**셋째, 실험 카드가 “선택 리스트”는 되는데 “운영 리스트”는 아닙니다.**
지금 카드에 보이는 정보는 제목, draft 정도인데,
운영자는 여기서

* 상태
* 채널
* holdout 여부
* last sync
* message log 유무
* owner
  를 봐야 합니다.

**넷째, 차트와 표는 있는데 “판정 상태”가 없습니다.**
표본 부족인지, 초반 관찰치인지, 실제 uplift 판정 가능한지 안 보입니다.

**다섯째, 최근 배정 500건 테이블이 너무 크고 효용은 낮습니다.**
지금 위치에서는 과합니다.
이건 기본 보기보다 “상세 보기”나 다운로드용이 낫습니다.

**여섯째, 오른쪽 여백이 너무 큽니다.**
정보 밀도가 어색하게 낮고, 화면을 많이 비워둔 느낌입니다.

## 5) 공통 UX 문제

### 1. 운영자용 화면과 개발자 진단이 아직 섞여 있습니다

상단 회색 pill들:

* 상담 액션 즉시 실행
* 실험 데이터는 로컬 SQLite 우선
* Revenue bridge는 참고용
* PG 귀속은 live receiver row가 핵심

이건 내용 자체는 맞습니다.
그런데 운영자에겐 너무 기술적입니다.

추천은 이겁니다.

* 운영자용 상단: `오늘의 상태`, `오늘의 blocker`, `바로 할 행동`
* 개발자용 진단: 접기 가능한 `기술 진단` 패널

즉 **운영자 화면과 개발자 패널을 층위로 분리**해야 합니다.

### 2. 색 의미가 너무 많습니다

현재 화면에는

* 민트 계열 탭 활성
* 보라 step 활성
* 파란 outline
* 초록 버튼
  이 동시에 나옵니다.

이건 예쁘진 않지만, 의미 체계는 흐립니다.
권장 의미는 이렇게 단순화하는 게 낫습니다.

* 파랑: 주요 행동
* 초록: 정상/성공
* 주황: 경고/주의
* 빨강: 실패/차단
* 회색: 참고/보조

### 3. “오늘 해야 할 일”이 안 보입니다

지금 두 화면 모두 정보는 있는데 액션이 약합니다.
벤치마크 CRM도 같은 문제였는데, 우리는 여기서 더 잘해야 합니다.

각 탭 맨 위에 이 카드 하나씩 있어야 합니다.

* 후속 관리: 오늘 바로 연락할 대상 수
* 실험 운영: 오늘 막힌 핵심 실험 1개
* 알림톡 발송: 오늘 테스트 가능한 템플릿/대상 수
* 결제 귀속: today blocker

### 4. 탭 간 상태 전달이 부족합니다

후속 관리에서 골라서 알림톡으로 넘기고,
실험에서 골라서 발송 대상으로 넘기고,
발송 결과가 다시 실험 장부로 이어져야 합니다.

문서도 결국 이 방향을 말하고 있습니다. `세그먼트 선택 -> 채널 실행 -> 발송 로그 -> 매출 연결`이 Phase 3의 핵심이기 때문입니다.

## 6) 우선순위로 정리한 개선안

### 이번 주 바로 할 것

1. **알림톡 발송 화면**

   * step2 템플릿 선택을 본문에 노출
   * 미리보기 빈 박스를 “선택 요약 카드”로 교체
   * 최근 발송 이력 기본 접기
   * 상태 컬럼 텍스트화

## 최종 판단

냉정하게 말하면, **지금 우리 화면은 벤치마크보다 “덜 화려하지만 더 옳은 방향”입니다.**
특히 허브형 구조와 실험 장부 관점은 더 낫습니다.
다만 지금은

* 알림톡 화면은 **운영 위저드가 덜 닫혔고**
* 실험 화면은 **검증 화면과 실제 성과 화면이 섞여 있어서**
  오해 가능성이 있습니다.

그래서 다음 단계는 화면을 더 늘리는 게 아니라,

**“무엇이 운영용이고, 무엇이 검증용인지 분리”**
그리고
**“누구에게 → 무엇을 → 보낼 수 있나 → 결과가 어땠나”를 한 흐름으로 묶는 것**입니다.

다음엔 **템플릿 선택 직후 화면**과 **발송 결과 상세 팝업** 캡처를 보여주면, 알림톡 UX는 훨씬 더 정확하게 짚어드릴 수 있습니다.
