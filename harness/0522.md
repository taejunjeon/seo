## 이번에 가능해진 것

이 문서는 “우리가 에이전트를 더 과감하게 굴릴 수 있는가”를 감정이 아니라 운영 규칙으로 나눈다.

결론부터 말하면, **가능하다. 다만 우리 프로젝트는 일반 소프트웨어 프로젝트보다 Red Lane이 많아서 ‘전부 무승인 자동화’는 위험하다.** 대신 Green Lane은 지금보다 훨씬 더 자동으로 진행할 수 있고, Yellow Lane은 한 번 승인받은 뒤 중간 컨펌 없이 끝까지 가는 방식으로 바꿀 수 있다.

## 전사 내용의 사실 여부와 적용 가능성

전사 내용에 나온 “OpenAI 내부에서 실제로 9개월간 에디터 사용 금지” 같은 구체 사실은 이 문서에서 직접 검증할 수 없다. 따라서 내부 사례 자체를 사실로 단정하지 않는다.

다만 운영 철학은 현재 모델·에이전트 환경과 잘 맞는다.

- 사람이 직접 코드를 쓰는 시간보다, 목표·제약·검증 기준을 잘 쓰는 시간이 더 중요해졌다.
- 코드는 최종 자산이라기보다, 목표를 달성하기 위해 계속 교체되는 실행 산출물에 가깝다.
- 오래 남는 자산은 프롬프트, 하네스, 데이터 계약, 테스트, runbook, 승인 기준이다.
- 사람이 매번 “계속해”라고 눌러야 한다면 하네스가 일을 충분히 맡기지 못한 것이다.

우리 프로젝트에도 이 방향은 맞다. 단, 범위를 나눠야 한다.

## 우리 프로젝트가 일반 프로젝트와 다른 점

SEO 프로젝트는 단순 웹앱 개발이 아니다. 코드 한 줄이 아래 영역에 바로 영향을 줄 수 있다.

- Meta/Google/TikTok/Naver 같은 외부 광고 플랫폼 전송
- 실제 구매 이벤트와 광고 학습 신호
- ROAS 해석과 예산 판단
- GTM Production publish
- 고객 식별자, 이메일, 전화번호, 외부 ID 같은 개인정보성 데이터
- 운영DB, VM Cloud 원장, GA4 BigQuery, Ads API 사이의 정합성

그래서 이 프로젝트에서 “컨펌을 줄이자”는 말은 “모든 승인 없이 밀자”가 아니다.

정확한 방향은 다음이다.

1. **Green Lane 컨펌은 줄인다.**
   문서 작성, read-only audit, dry-run, no-send preview, 로컬 테스트, 승인안 작성, scoped commit 같은 작업은 Codex가 묻지 않고 진행해야 한다.

2. **Yellow Lane 컨펌은 묶는다.**
   VM Cloud backend deploy, frontend deploy, controlled smoke처럼 운영 영향은 있지만 rollback이 준비된 작업은 “작업 묶음 단위 1회 승인”으로 끝내야 한다. 중간에 사소한 파일 복사·빌드·재시작마다 다시 묻지 않는다.

3. **Red Lane 컨펌은 유지한다.**
   GTM publish, Meta/Google 실제 send, 대량 backfill, 운영DB write/import, permanent env ON, 개인정보 전송 범위 확대는 반드시 멈춰야 한다. 여기까지 자동화하면 빠른 것이 아니라 위험한 것이다.

## 우리가 느린 이유

일부는 컨펌이 많아서 느린 것이 맞다.

특히 아래 작업은 이미 자동으로 가도 된다.

- “조사해봐”라고 하면 read-only audit + 원인 분류 + 문서 생성까지 진행
- “설계해봐”라고 하면 API contract + 프론트 문구 + 승인안까지 진행
- “검토해봐”라고 하면 기존 문서와 코드 근거를 찾아 판단률과 next action까지 작성
- “배포 승인안 만들어”라고 하면 rollback, post-check, 성공 기준까지 포함
- “no-send preview”는 실제 전송 없이 payload 후보만 만들고 문서화

반대로 느릴 수밖에 없는 부분도 있다.

- Meta Events Manager, GTM, Slack, Ads Manager처럼 브라우저 로그인/2FA/권한이 필요한 화면
- 실제 광고 플랫폼 전송이 걸린 변경
- 고객 식별자 보강처럼 개인정보와 매칭 품질이 동시에 걸린 변경
- 결제 완료/가상계좌/환불처럼 돈과 연결된 판정

따라서 문제는 “컨펌이 많다”가 아니라, **컨펌이 필요한 일과 필요 없는 일을 충분히 구분하지 못하면 느려진다**이다.

## 우리 프로젝트 적용안

### 1. Green Lane은 컨티뉴 0번을 목표로 한다

예를 들어 TJ님이 “Meta UTM/source 미매핑 bucket audit 진행해”라고 하면 Codex는 다음을 한 번에 해야 한다.

- 관련 문서와 코드 탐색
- VM Cloud/로컬/GA4 read-only 조회
- 원인 bucket 분류
- 프론트/백엔드 수정 필요 여부 판단
- 문서 작성
- 가능하면 로컬 패치와 테스트
- 배포가 필요하면 승인안 작성

중간에 “계속할까요?”를 묻는 것은 실패로 본다.

### 2. Yellow Lane은 사전 승인 패킷을 표준화한다

VM Cloud 배포처럼 반복되는 일은 매번 새로 설명하지 않고 표준 패킷으로 처리한다.

포함해야 할 항목:

- 무엇을 바꾸는지
- 왜 바꾸는지
- 바꾸는 파일
- pre-snapshot
- backup 경로
- build/typecheck
- restart
- post-check API
- rollback 명령
- 성공 기준
- 실패 시 멈출 조건

승인이 한 번 나면, Codex는 중간 단계별 컨펌 없이 post-check까지 끝낸다.

### 3. Red Lane은 줄이지 말고 더 명확하게 만든다

아래는 계속 explicit approval이 필요하다.

- GTM Production publish
- Meta CAPI 운영 send 또는 대량 backfill
- Google Ads 전환 import 운영 적용
- TikTok/Naver send
- 운영DB write/import
- VM Cloud schema migration
- 고객 식별자 전송 범위 확대
- permanent env ON

컨펌을 없앨 영역이 아니라, “왜 지금 Red인지”를 더 쉽게 보여줘야 한다.

### 4. 코드보다 오래 남는 자산을 만든다

우리 프로젝트에서 진짜 자산은 아래다.

- `capivm/!capiplan.md`: Meta CAPI 운영 정본 계획
- `project/!indicatoragent.md`: 구매 전 선행지표 에이전트 정본 계획
- `data/!data_inventory.md`: 데이터 위치와 source of truth
- `harness/common/*`: Green/Yellow/Red Lane 운영 규칙
- no-send preview script
- post-check script
- rollback runbook
- 프론트엔드 보고서 원칙
- API data contract

코드는 이 자산을 실행하는 부산물이다. 그래서 코드가 늘어나는 것보다, 반복 실수를 막는 하네스와 검증 스크립트가 늘어나는 것이 더 중요하다.

### 5. 금요일 청소의 날을 실제 운영으로 도입한다

매주 1회 30~60분만 잡아도 효과가 크다.

청소 대상:

- 이번 주에 TJ님이 여러 번 같은 질문을 한 지점
- Codex가 자주 멈춘 지점
- VM 배포 승인안이 매번 새로 작성된 지점
- 반복 SQL/BigQuery 조회
- 반복 raw scan/JSON validation
- 반복 Slack/Telegram/Obsidian 보고
- “이게 무슨 뜻이야?”가 나온 기술 용어

청소 결과:

- 하네스 규칙 추가
- runbook 추가
- API contract 보강
- 프론트 문구 쉬운 말로 수정
- 반복 쿼리 script화
- 배포 패킷 템플릿화

## 자율화 가능성 점수

### Green Lane

가능성: 90~95%.

문서, 분석, read-only audit, dry-run, local patch, 테스트, 승인안 작성은 지금보다 훨씬 더 자동화할 수 있다.

### Yellow Lane

가능성: 70~85%.

VM Cloud 배포, frontend 배포, controlled smoke는 표준 패킷과 rollback이 있으면 한 번 승인 후 자율 실행 가능하다. 다만 접근권한, 네트워크, 2FA, 외부 UI 상태가 막을 수 있다.

### Red Lane

가능성: 20~40%.

실제 광고 플랫폼 전송, GTM publish, 운영DB write, permanent env ON은 자동화 대상이 아니라 승인·기록·rollback을 강화할 대상이다.

## 현재 우리 프로젝트의 자동화 성숙도

현재는 65~75% 수준으로 본다.

잘 되고 있는 것:

- Green Lane read-only 분석과 문서화는 꽤 빠르다.
- VM Cloud 배포 패킷과 rollback 관성이 생겼다.
- CAPI Purchase guard, value guard, missing queue, canary 같은 운영 개념이 생겼다.
- 프론트 보고서에 쉬운 말 원칙이 들어가기 시작했다.

부족한 것:

- Yellow Lane 표준 승인 패킷이 아직 문서화만 있고 완전히 재사용되지는 않는다.
- 같은 질문이 여러 번 나오면 하네스/스크립트로 승격하는 루틴이 약하다.
- “컨펌이 필요한가?” 판단을 매번 대화에서 다시 한다.
- 완료 후 다음 할 일을 실행 큐로 고정하는 구조가 약하다.
- 외부 UI 확인이 필요한 경우 자동 브라우저 검증과 TJ님 확인 역할이 섞인다.

30일 안에 85% 수준까지 올릴 수 있다.

## 최종 판단

전사 내용의 극단적 표현을 그대로 따라가면 위험하다. 하지만 핵심 철학은 맞다.

우리 프로젝트에서는 이렇게 적용하는 것이 현실적이다.

- **Green Lane은 컨티뉴 0번에 가깝게 만든다.**
- **Yellow Lane은 1회 승인 후 끝까지 간다.**
- **Red Lane은 절대 자동화하지 않는다.**
- **코드보다 하네스, 데이터 계약, 검증 스크립트, runbook을 자산으로 본다.**
- **매주 반복된 수동 확인을 자동화 자산으로 바꾼다.**

즉, 우리 프로젝트가 특별해서 자율화가 불가능한 것이 아니다.
특별하기 때문에 **무조건 자동화가 아니라, 위험별로 나눈 자동화**가 필요하다.
