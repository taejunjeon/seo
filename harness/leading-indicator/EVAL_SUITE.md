## 10초 요약

이 문서는 선행지표 에이전트가 찾은 추천 후보를 바로 써도 되는지 판단하는 기준이다.
숫자가 좋아 보여도 데이터가 오래됐거나, 비교 집단이 작거나, 결제 후 행동이 섞였거나, 사람이 고칠 수 없는 지표라면 추천 후보로 올리지 않는다.

품질 판정은 4개만 쓴다.

- PASS: 바로 사용 가능
- PASS_WITH_NOTES: 주의 문구와 함께 사용 가능
- HOLD: 보류, 다음 확인 필요
- FAIL: 추천 후보에서 제외

## 판정 입력값

모든 추천 후보는 아래 값을 가져야 한다.

| 항목 | 쉬운 뜻 | 필수 여부 |
|---|---|---|
| site | 대상 사이트 | 필수 |
| source | 어떤 데이터에서 나왔는가 | 필수 |
| window | 어떤 기간을 봤는가 | 필수 |
| freshness | 데이터 최신성 | 필수 |
| cohort | 비교 집단 | 필수 |
| behavior_metric | 행동 숫자 | 필수 |
| signal_summary | 추천 후보 요약 | 필수 |
| recommended_action | 사람이 할 수 있는 다음 행동 | 필수 |
| forbidden_actions | 이번 실행에서 금지된 행동 | 필수 |

## 필수 판정 8개

### 1. 데이터 최신성

질문:
이 숫자는 지금 판단에 쓸 만큼 최신인가?

PASS 기준:

- 기준 시각이 있다.
- 기간이 있다.
- source가 있다.
- freshness가 fresh 또는 warn이다.

HOLD 기준:

- 마지막 갱신 시각이 없다.
- source가 오래됐다.
- live/cache 중 어느 값인지 불명확하다.

### 2. 비교 집단 최소 크기

질문:
구매자와 이탈자를 비교할 만큼 표본이 충분한가?

초안 기준:

- 각 비교 집단 n이 30 이상이면 PASS 후보
- n이 10 이상 30 미만이면 PASS_WITH_NOTES 후보
- n이 10 미만이면 HOLD

주의:
상품군, 채널, 기간에 따라 기준은 조정 가능하다.
처음에는 보수적으로 둔다.

### 3. 구매 후 행동 혼입 방지

질문:
결제 후에 생긴 행동을 구매 전 신호로 착각하지 않았는가?

FAIL 기준:

- 결제완료 이후 행동을 선행지표로 사용
- thank-you page 행동을 구매 전 행동으로 사용
- 내부 결제완료 주문을 행동 원인처럼 사용

PASS 기준:

- 모든 행동이 결제완료 이전 window에 있다.
- 구매 후 이벤트는 후행 지표로 분리되어 있다.

### 4. 실행 가능성

질문:
운영자가 실제로 바꿀 수 있는 지표인가?

PASS 후보:

- 랜딩 페이지 구성
- 리뷰/콘텐츠 배치
- CTA 문구
- 결제 UX 안내
- 광고 소재/캠페인 유입 품질

HOLD 또는 FAIL 후보:

- 사람이 당장 바꿀 수 없는 외부 요인
- 표본이 너무 작은 우연한 차이
- 원인보다 결과에 가까운 값

### 5. 결제 정본 분리

질문:
실제 결제완료 주문과 플랫폼 주장값을 섞지 않았는가?

PASS 기준:

- 실제 매출 판단은 내부 결제완료 주문 기준이다.
- GA4/Meta/Google Ads/TikTok claim은 참고값으로 분리한다.
- NPay click/count/payment start는 구매로 세지 않는다.

FAIL 기준:

- NPay 클릭을 구매로 계산
- GA4 purchase만으로 내부 매출 확정
- 플랫폼 전환값을 내부 ROAS로 사용

### 6. 개인정보/원본 식별자 보호

질문:
보고서나 LLM context에 raw identifier가 나오지 않는가?

FAIL 기준:

- 원본 주문번호
- 원본 고객 이메일/전화
- 원본 결제 식별자
- 원본 광고 클릭 식별자

PASS 기준:

- 집계값, 안전한 session key, 익명화된 참조만 사용

### 7. 금지선 준수

질문:
이번 실행이 읽기와 보고 범위를 넘지 않았는가?

PASS 기준:

- 운영DB 변경 없음
- VM Cloud 배포 없음
- GTM publish 없음
- 외부 플랫폼 전송 없음

FAIL 기준:

- 승인 없는 운영DB write
- 승인 없는 deploy
- 승인 없는 send/upload
- 승인 없는 GTM Production publish

### 8. 미니 디지털 트윈 안전성

질문:
시뮬레이션 결과를 예측 정답이나 자동 예산 조정으로 쓰지 않았는가?

PASS 기준:

- 결과를 예상 범위로 표시
- 기준선과 오차 가능성을 표시
- 자동 예산 변경 없음
- 외부 전송 없음

HOLD 기준:

- 기준선이 오래됨
- 결제수단별 완료 기준이 섞임
- 오차율 검증이 없음

FAIL 기준:

- 시뮬레이션 결과로 자동 예산 변경
- 광고 플랫폼에 자동 전송
- 단일 숫자를 확정 매출 예측처럼 표시

## HOLD 원인 분류

| 분류 | 쉬운 뜻 | 다음 확인점 |
|---|---|---|
| `cohort_too_small` | 비교할 표본이 부족함 | 기간을 늘리거나 채널을 넓힌다 |
| `source_freshness_gap` | 데이터가 오래됐거나 갱신 시각이 불명확함 | source freshness를 재조회한다 |
| `label_conflict` | GA4와 내부 결제 판단이 충돌함 | 내부 결제완료 원장을 우선 확인한다 |
| `not_actionable` | 사람이 바꿀 수 없는 지표임 | 추천 후보에서 제외하거나 관찰 지표로 둔다 |
| `privacy_risk` | 원본 식별자 노출 위험 | 집계/익명화 방식으로 바꾼다 |
| `approval_required` | 운영 변경 또는 외부 전송이 필요함 | 승인 문서를 만든다 |
| `simulation_uncalibrated` | 시뮬레이션 기준선/오차율이 부족함 | 최근 실제 매출과 비교 검증한다 |

## 추천 카드 표시 규칙

추천 카드에는 아래를 보여준다.

1. 추천 신호
2. 근거 비교 집단
3. 행동 숫자 차이
4. 품질 판정
5. 사람이 할 다음 행동
6. 승인 필요 여부
7. 금지선
