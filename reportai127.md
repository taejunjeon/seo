## 0) 결론 요약 (TL;DR)

- `/report`는 **검사결과 확인(회원 전용) 성격의 페이지일 가능성이 매우 높음**.  
  (근거: `/report` 접근 시 로그인으로 리다이렉트되며, 로그인 페이지 내 “검사결과 확인” 링크가 `/reportPC`로 존재)
- 따라서 “/report의 AI 유입 비중 18%”는 **‘신규 유입(획득)’** 이라기보다 **‘기존 고객의 네비게이션/사후 이용’** 을 AI가 도와준 신호일 확률이 큼.
- “AI 유입”이라는 말을 그대로 쓰면 오해가 생김:
  - 기술적으로는 **AI referrer/UTM 기반 세션**이 맞을 수 있지만,
  - 사업적으로 “AI가 신규 고객을 데려왔다/전환을 만들었다”는 의미로는 **부적절**할 수 있음.
- 즉시 권장: 대시보드에서 **AI 유입을 2개로 분리**해서 보여주기
  1) **AI 유입(Discovery/마케팅)**: 공개 콘텐츠/랜딩 중심(예: `/healthinfo`, `/what_biohacking`, 상품/칼럼 등)  
  2) **AI 유입(고객지원/결과조회 네비)**: 회원/결과조회 페이지(예: `/report`, `/login`, 마이페이지 등)

## 1) “/report는 검사 결과 조회 페이지가 맞나?” (팩트 체크)

### 1-1. URL 동작

- `https://biocom.kr/report`로 접근하면 최종적으로 **로그인 페이지로 이동**됨:
  - 예: `/login?back_url=L3JlcG9ydA==` (back_url이 `/report`로 인코딩)

→ 즉, `/report`는 공개 콘텐츠가 아니라 **로그인이 필요한 회원 전용 진입점**에 가까움.

### 1-2. 로그인 페이지 내 버튼/링크 근거

로그인 페이지 HTML 내에 “검사결과 확인” 버튼이 존재하며, 링크가 `https://www.biocom.kr/reportPC`로 노출됨.

→ `/report`는 “검사결과 확인” 흐름의 일부로 보는 것이 자연스럽소.

## 2) 그런데 왜 “AI 유입”으로 잡히나? (정의/측정 로직 관점)

현재 대시보드의 AI 유입은 대체로 다음과 같은 의미이오:

- GA4에서 **세션 레벨(sessionSource / sessionManualSource)** 기준으로
  - referrer가 ChatGPT/Perplexity/Gemini/Claude 등 AI 도메인인 세션
  - (보충) ChatGPT가 붙이는 `utm_source=chatgpt.com` 계열이 잡히는 세션

즉 “AI 유입”은 **‘이 세션의 유입 경로(세션의 출처)’** 를 말하며,
“이 사용자가 최초로 유입된 획득 채널(First touch)”이나 “구매에 기여한 채널(Attribution)”과는 개념이 다릅니다.

## 3) /report를 “AI 유입”이라 불러도 되는가? (해석 레이어 분리)

### 3-1. 기술적으로(Strict) — “AI 유입”이라 부를 수 있음

- 사용자가 ChatGPT에서 “바이오컴 검사 결과 어디서 보나요?” 같은 질문 → 링크 클릭 → `/report` 진입
- 이 경우 **세션 출처가 chatgpt.com(referral)** 로 잡히는 것이 자연스럽고,
  “AI가 해당 세션의 유입을 만들었다”는 말은 사실일 수 있소.

### 3-2. 사업적으로(Business) — “AI가 신규 유입/전환을 만들었다”로 해석하면 위험

TJ님이 언급하신 것처럼:
- 검사 서비스는 구매 → 채취 → 검체 반송 → 결과 수령까지 **시간 지연(수일~수주)** 이 발생
- `/report`는 결과 조회 성격이면, 방문자는 **이미 구매를 끝낸 고객**일 확률이 높소

따라서 `/report`의 세션을 “AI 유입”으로 포함시키면 이런 오해가 생깁니다.
- (오해 1) AI가 신규 고객 획득을 크게 만들고 있다
- (오해 2) AI 유입 전환율이 매우 높다/낮다 (후행 행동이 섞여 왜곡)
- (오해 3) 평균 체류시간이 길다 = 마케팅 품질이 좋다  
  → 실제로는 “결과 읽기/탭 유지” 같은 **서비스 이용 시간**일 수 있음

결론: `/report`는 **“AI 유입(네비게이션/고객지원)”** 으로 따로 분리해 표현하는 게 더 정확하오.

## 4) “구매 후 3일” 같은 시간 컷을 둬야 하나?

### 4-1. 이상적인 방식(추천) — “구매 시점 ↔ 결과 조회 시점”을 사용자 단위로 연결

정말로 “구매 후 N일 내 AI 세션만 보자”를 하려면,
구매 이벤트와 `/report` 방문 이벤트가 **같은 사용자(user_id) 기준으로 연결**되어야 합니다.

가능한 구현 경로(선택지):
- GA4에 `user_id`가 일관되게 들어가고 있다면: `user_id` 기반 분석(단, Data API만으로는 제약이 있을 수 있음)
- GA4 → BigQuery export를 쓰는 경우: 이벤트 테이블에서 `purchase_time`과 `report_view_time` 차이를 계산 가능
- 내부 주문/검사 시스템(DB/CRM)과 연동: “구매일/검체 수거일/결과 발행일”을 함께 붙이면 가장 정확

### 4-2. 현실적인 대안(지금 당장 가능한 것)

구매-결과의 시간차를 “정확히” 자르기 어려운 상황이라면, 다음 3개가 즉시 효과가 있소.

1) **페이지 타입으로 분리(권장)**
- `/report`, `/login` 같은 회원/사후 이용 페이지를 “AI 유입(고객지원/네비)”로 분리
- 공개 콘텐츠/상품/칼럼만 “AI 유입(Discovery)”로 분리

2) **`referralOnly=1`로 ‘진짜 referral’만 보기**
- UTM이 남아있는 북마크/직접입력 등은 sessionMedium이 referral이 아닐 수 있어 걸러짐
- “AI 유입이 과대추정 되었는지”를 빠르게 점검하는 용도로 좋소

3) **평균(mean) 대신 중앙값/절사평균(robust) 지표 병행**
- `/report`처럼 체류시간이 매우 긴 페이지가 섞이면 평균이 쉽게 왜곡됨
- “AI 유입 품질”을 말할 때는 median(중앙값) 또는 상위 5% 절사평균을 같이 보는 편이 안전하오

## 5) 이 케이스에서 “AI 유입”을 어떻게 정의해 보여줄지(권장안)

### 5-1. KPI/리포트용(마케팅 관점) 정의

**AI 유입(Discovery)** = “AI에서 넘어와서 공개 콘텐츠/상품 페이지로 랜딩한 세션”

- 예시 포함: `/healthinfo`, `/what_biohacking`, `/biocomset_store`, `/shop_view` 등
- 예시 제외: `/report`, `/login`, `/membership`, 장바구니/주문조회 등(고객 사후 이용)

→ 이 정의는 “AI가 성장(획득)에 기여하는가?” 질문에 더 잘 맞소.

### 5-2. 운영/CS 관점 정의

**AI 유입(Navigation/Support)** = “AI를 ‘검색창/안내원’처럼 써서 로그인/결과 페이지로 들어오는 세션”

→ 이 정의는 “고객이 AI에 무엇을 물어보는가?”, “고객지원 문서/FAQ를 보강해야 하는가?”에 유용하오.

- 다만 현재 GA4/대시보드만으로는 “구매 후 N일”을 정확히 계산하기 어려울 수 있어,
  당장에는 **페이지 타입 분리 + referralOnly 비교 + 신규/재방문 비중 확인**이 현실적이오.

## 7) 즉시 실행 체크리스트(데이터로 확인)

1) `/report`가 AI 유입을 얼마나 왜곡하는지 확인
- AI 랜딩페이지 테이블에서 `/report` 포함/제외 시 평균 체류시간, PV/세션이 얼마나 바뀌는지 비교

2) `referralOnly=1`로 재조회해 변화 확인
- `/api/ga4/ai-traffic?referralOnly=1&...`로 조회했을 때
  - `/report` 세션이 크게 줄면: “UTM 잔존/북마크” 성격이 강할 가능성 ↑
  - 줄지 않으면: “실제 chatgpt.com 등에서 직접 넘어오는 네비” 성격 가능성 ↑

3) `/report`의 신규 사용자 비중(신규/재방문)을 확인
- 가능하면 GA4에서 `/report` 랜딩의 `newUsers`가 거의 0에 가까운지 확인  
  (가까우면 “획득”이 아니라 “기존 고객 사후 이용” 해석이 더 타당)
