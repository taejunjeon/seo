## 2) 최신 문서 자체의 품질 피드백

### A. `roadmap0327.md`

이 문서는 이번 버전에서 **확실히 좋아졌습니다.**

좋아진 이유는 명확합니다.

* 각 Phase마다 `왜`가 있고
* 그 아래에 **`사용자 베네핏`이 페르소나별로 분리**돼 있고
* 담당이 `Codex / Claude Code`로 갈라져 있고
* 진행률도 `우리 기준 / 운영 기준`으로 나뉘어 있습니다. 

이건 CEO, 운영팀, 개발팀이 같은 문서를 읽을 수 있게 만드는 데 꽤 효과적입니다.

다만 문제도 있습니다.

**문서 안에 아직 “오래된 설명”이 일부 남아 있습니다.**
예를 들어 최신 Phase 3 섹션은 이미 Aligo가 `exact-match` 단계까지 왔다고 말하는데, 문서 앞쪽 `1. 한 줄 결론` 영역에는 아직 **“알리고 발송에 추가 필요한 키 3개가 있어야 된다”**는 예전 문장이 남아 있습니다. 최신 상태에서는 이 표현이 틀리거나 최소한 불완전합니다. 지금은 키 부족보다 렌더링 일치가 더 큰 blocker이기 때문입니다. 

또 하나, **우선순위 층위가 섞여 있습니다.**
문서 앞에서는 “현재 가장 파급력 큰 다음 액션”을 `P1-S1A live row 연결`로 적어 놓고, 뒤 `5-A-2 다음 2주 Codex 실행 순서`에서는 `P3-S1`을 먼저 둡니다. 둘 다 틀린 말은 아니지만, **회사 전체 우선순위**와 **Codex 개인 우선순위**를 같은 종류의 문장처럼 써서 읽는 사람이 헷갈릴 수 있습니다. 

즉, roadmap은 **좋아졌지만 아직 “문서 전체가 완전히 한 목소리”는 아닙니다.**

### B. `Phase1.md`

이 문서는 **상단은 좋아졌고, 본문은 아직 덜 정리됐습니다.**

좋아진 점:

* 아예 맨 위에 `Source Of Truth`를 선언했고
* owner, next action, blocked by를 적었습니다.
* `live / replay / smoke` 구분도 상단 상태 설명에 반영했습니다. 

그런데 본문 아래 `P1-S1A 결제 블랙박스` 섹션으로 내려가면, 아직 **예전 숫자**가 남아 있습니다.

상단 현재 상태는

* `live 0 / replay 5 / smoke 2`
* replay 기준 `5/5` 조인
  이라고 말합니다. 그런데 아래 `P1-S1A` 세부 설명은 아직
* recent sample 토스 승인 `100건`
* payment success entry `1건`
* matched toss row `0건`
  이라는 **예전 상태**를 그대로 들고 있습니다. 이건 한 문서 안의 자기충돌입니다. 

이건 꽤 심각합니다. 이유는 단순합니다.
**Phase 1은 “숫자를 믿어도 되는 상태”를 만드는 문서인데, 문서 자체 숫자가 섞여 있으면 그 철학을 스스로 깨는 셈**이기 때문입니다. 

즉, `Phase1.md`는 지금 **문서 구조는 좋아졌지만, 본문 숫자 청소가 덜 끝난 상태**입니다.

### C. `phase3.md`

셋 중에서는 **이 문서가 가장 정리 잘 됐습니다.**

좋은 이유:

* 맨 위 요약표가 바로 source of truth 역할을 합니다.
* `Codex = 백엔드`, `Claude Code = 프론트` 분리가 분명합니다.
* Aligo는 `키 없음` 단계가 아니라 **`exact-match renderer` 문제**라는 점이 명확합니다.
* `첫 operational live`와 `첫 gold-standard causal test`를 분리한 것도 좋습니다. 

즉, 이 문서는 지금 **누가 무엇을 왜 해야 하는지**가 가장 잘 보입니다.

단점은 하나입니다.
**운영팀 관점의 “하루 업무 흐름” 설명은 아직 약합니다.**
개발자와 PM이 읽기엔 좋지만, 운영팀이 읽었을 때 “그래서 내가 오늘 CRM 화면에서 뭘 누르고, 무엇을 확인하면 되는가”까지는 덜 내려옵니다. 

## 3) ‘사용자 페르소나 베네핏 관점’ 평가

핵심만 말하면 이렇습니다.

### roadmap

**7.5 - 8점 수준입니다.**
이유는 각 Phase에 `왜`와 `사용자 베네핏`이 명시돼 있어서, 적어도 “이게 누구한테 왜 필요한가”는 보입니다. CEO, 운영팀, 마케터, 개발팀, AI/데이터 운영팀 같은 페르소나도 꽤 잘 분리돼 있습니다. 

하지만 아직 완성형은 아닙니다.
왜냐하면 지금의 베네핏은 종종 **“좋아질 것 같다” 수준**에 머무르고, **“이 Phase가 닫히면 이 사람이 내일 무엇을 할 수 있게 되는가”**까지는 덜 내려오거든요. 예를 들어 대표 베네핏은 잘 쓰여 있는데, 운영팀 베네핏은 어떤 화면과 어떤 버튼을 기준으로 행동이 바뀌는지까지 들어가면 훨씬 강해집니다. 

### phase 문서들

**5.5 - 7점 수준입니다.**
이 문서들은 기본적으로 “실행 메모”라서 기술 중심이어도 됩니다. 문제는, 지금은 phase 문서가 꽤 자주 **대표/운영팀/개발팀이 함께 보는 공유 문서** 역할도 하고 있다는 점입니다. 그런데 그 기준으로 보면 아직 **개발자 친화적이고, 페르소나 베네핏은 보조적**입니다.  

특히 Phase 1은

* 무엇이 막혔는지
* 어떤 endpoint가 있는지
* 어떤 수치가 있는지
  는 잘 보이지만,
* 대표가 이걸 읽고 어떤 결정을 내릴 수 있는지
* 운영팀이 아침 daily review에서 뭘 확인할 수 있는지
  는 약합니다. 

Phase 3는 그나마 낫습니다.
왜냐하면 “실행 채널을 붙인다”, “운영자가 발송 준비와 결과 확인을 한다”까지는 써 있기 때문입니다. 그래도 여전히 **운영자 하루 흐름**까지는 한 단계 더 구체화할 수 있습니다. 

## 4) 어떻게 고치면 ‘페르소나 베네핏’이 훨씬 좋아지나

제가 추천하는 건 단순합니다.

### 1. 각 문서 맨 위에 `페르소나 베네핏 카드` 4줄만 고정

형식은 이걸 추천합니다.

* **대표(TJ):** 이 phase가 닫히면 어떤 경영 결정을 숫자로 할 수 있는가
* **운영팀:** 오늘 어떤 화면에서 어떤 행동을 할 수 있는가
* **개발팀:** 어떤 배선/테스트/배포 기준이 고정되는가
* **아직 안 되는 것:** 이 phase가 닫혀도 아직 못 하는 것은 무엇인가

이 4줄만 있으면 문서가 훨씬 좋아집니다.

### 2. “좋아진다”가 아니라 “행동이 바뀐다”로 써야 합니다

예를 들어:

나쁜 표현:

* 운영팀이 더 효율적으로 일할 수 있다

좋은 표현:

* 운영팀은 `/crm 결제 귀속` 탭에서 `live/replay/smoke`를 분리해서 보고, 오늘 개발팀에 요청할 blocker를 5분 안에 고를 수 있다

나쁜 표현:

* 대표가 더 정확한 판단을 할 수 있다

좋은 표현:

* 대표는 `holdout 대비 treatment 순매출`을 보고 상담사 추가 채용 여부를 감이 아니라 숫자로 판단할 수 있다

### 3. Phase 문서는 “기술 문서”가 아니라 “행동 문서”처럼 시작해야 합니다

특히 Phase 1, Phase 3는 아래 한 줄이 꼭 있어야 합니다.

* **이 문서의 주 독자:** 대표 / 운영팀 / 개발팀 중 누구인가
* **이 문서를 읽고 바로 할 행동:** 무엇인가

지금은 이게 약해서, 다 같이 읽긴 하지만 모두에게 완전히 맞진 않습니다.

### 4. 회사 우선순위와 구현자 우선순위를 분리해 적어야 합니다

현재 roadmap은 여기서 조금 흔들립니다.

추천 형식:

* **회사 전체 다음 행동**
* **Codex 다음 행동**
* **Claude Code 다음 행동**
* **외부 의존 blocker**

이렇게 4줄로 쪼개면 혼선이 거의 없어집니다.
