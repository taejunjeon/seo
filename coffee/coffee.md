## 한 줄 결론

- `coffee.md`는 **업데이트가 필요한 상태**였다. `0402` 기준으로 Toss/attribution/로컬 캐시 상황이 바뀌었기 때문이다.
- 최근 더클린커피 매출 데이터는 운영 DB뿐 아니라 `seo/backend`의 로컬 Toss 캐시와 live attribution 원장까지 생겼다.
- 다만 현재 확보된 Toss 커피 거래는 여전히 `2026-01-01 ~ 2026-02-23` 구간만 확인된다. **Toss 연동이 생겼다고 해서 바로 2025 커피 고객 분석까지 열린 것은 아니다.**
- 그래서 TJ님이 원하는 `LTR`, `재구매율`, `재구매 횟수`는 **Toss만으로는 아직 계산 불가**하다. 고객 식별 가능한 주문 원장이 계속 필요하다.
- `GA4_PROPERTY_ID=326949178`로 다시 조회해도 현재 서비스 계정은 해당 속성 권한이 없어 `PERMISSION_DENIED`가 난다.
- `seo/backend/.env`에는 PlayAuto/Imweb/Toss 키가 들어 있어 **"자격증명이 아예 없다"는 전제는 더 이상 맞지 않는다.** 대신 coffee 전용 수집 경로와 site_code/secret 구조가 아직 덜 닫힌 상태다.
- 2019~2020의 오래된 커피 Imweb 주문 흔적은 본 문서와 향후 딥리서치 기준에서 계속 제외한다.

## 0. 0402 업데이트 판단

결론부터 적으면, **문서 업데이트가 필요하다.**

이유는 4가지다.

1. `seo/backend/data/crm.sqlite3`에 Toss 로컬 캐시가 생겼다.
   - `toss_transactions` 총 `37,959건`
   - `toss_settlements` 총 `2,942건`
2. 더클린커피 live attribution row가 실제로 들어오기 시작했다.
   - `thecleancoffee_imweb + live` 기준 `3건`
   - 결제금액 합계 `77,425원`
   - `channel_talk / campaign` 유입 실주문 `1건`, `34,825원`
3. 커피 Toss 거래도 0406 기준 과거 범위로 넓어졌다.
   - coffee MID `iw_thecleaz5j`
   - `5,043건` 적재
   - 기간은 `2025-01-01 08:17:06 ~ 2026-02-23 16:21:57`
4. 기존 문서의 `PlayAuto / Imweb 자격증명 부재` 전제는 현재와 맞지 않는다.
   - `seo/backend/.env`와 `/health` 기준으로는 두 API 모두 `ready=true`

즉 지금 문서는 이렇게 읽어야 한다.

- **업데이트된 것**
  - 최근 결제/유입 추적 가능 범위
  - 로컬 Toss 캐시 존재 여부
  - live attribution 존재 여부
  - API 자격증명 준비 상태
- **여전히 안 열린 것**
  - 2025 커피 Toss 백필
  - 최근 커피 고객 식별 주문 원장
  - GA4 property 접근 권한
  - 정식 LTR / 재구매율 / 재구매 횟수

확인 내용:

  - `IMWEB_SITE_CODE` 단일 설정
- `revenue/backend/app/tasks/salesSync/batch.py`
  - `site_code = settings.IMWEB_SITE_CODE`

- 더클린커피 Imweb site_code가 별도 등록/동기화되지 않았으면
- 최근 coffee 고객 주문은 DB에 안 들어오는 구조일 수 있다

## 2. 사이트 / GA4 조사 결과

`https://thecleancoffee.com/`는 현재 Imweb이다.

### 2-2. GA4 조회 여부

조회는 현재 실패했다.

확인 내용:

- `.env` 하단에 `#더클린커피 GA4`
- `GA4_PROPERTY_ID=326949178`
- 기존 SEO 서비스 계정으로 Data API 조회 시:
  - `PERMISSION_DENIED`
  - `"User does not have sufficient permissions for this property"`

정리:

- `GA4_PROPERTY_ID` 값 자체는 들어가 있음
- 하지만 **해당 property에 현재 서비스 계정 권한이 없음**
- 따라서 지금 상태로는 Codex가 더클린커피 GA4 데이터를 읽을 수 없음

또한 `.env`에 `GA4_PROPERTY_ID`가 두 번 선언돼 있다.

- 앞쪽: 바이오컴용 `304759974`
- 뒤쪽: 더클린커피용 `326949178`

현재 dotenv 관점에서는 마지막 값이 적용될 가능성이 높다. 그래서 장기적으로는 아래처럼 분리하는 편이 낫다.

- `GA4_PROPERTY_ID_BIOCOM=304759974`
- `GA4_PROPERTY_ID_COFFEE=326949178`

## 3. 지금 바로 가능한 것 / 불가능한 것

### 3-1. 지금 바로 가능한 것

- 최근 커피 매출 총액, 건수, 취소금액, 결제수단 비중

### 3-2. 지금 불가능한 것

최근 운영 기준의 아래 지표는 지금 계산할 수 없다.

- 고객 LTR
- 재구매율
- 재구매 횟수
- 30/90/180/365일 코호트
- 신규 vs 재구매 매출 분해
- 정기구독 전환율

이유는 단순하다.

- 최근 coffee 데이터는 `tb_sales_toss`에만 있고
- `tb_sales_toss`에는 고객 식별자(`email`, `phone`, `member_id`, `customer_id`)가 없다
- 따라서 결제는 보여도 `같은 고객이 다시 샀는지`를 추적할 수 없다

최소 컬럼:

현재 HTML에 `/정기구독` 메뉴는 보이지만, DB에서 현재 활성 구독자를 식별할 구조는 아직 확인하지 못했다.

### 필수 4. GA4 property 권한

필요 사항:

- 현재 서비스 계정 `seo-656@seo-aeo-487113.iam.gserviceaccount.com`
  - 또는 다른 분석용 서비스 계정
- 를 `property 326949178`에 Reader 이상으로 추가

중요:

- Data API 조회에는 measurement ID보다 property access가 우선이다
- 즉 지금 당장 부족한 건 `measurement ID`보다 **권한**

## 5. PlayAuto API / 아임웹 API로 2025 데이터를 확인할 수 있는가

### 5-1. PlayAuto

결론:

- `운영 DB에 이미 적재된 PlayAuto 데이터`로는 **2025년 더클린커피 데이터 확인 가능**
- `seo/backend/.env` 기준으로는 `PLAYAUTO_API_KEY / EMAIL / PASSWORD`가 들어 있어 **우리 환경에서 재수집을 시도할 준비 자체는 되어 있다**
- 다만 현재 `seo` 프로젝트 안에는 PlayAuto 재수집 전용 라우트/스크립트가 닫혀 있지 않아, **"즉시 클릭 한 번으로 최신화" 상태는 아니다**

코드 구조:

- `revenue/backend/app/tasks/playautoApi/api_client.py`
  - `/orders` API 호출 구현 존재
  - `fetch_orders`, `fetch_orders_all`, `full_backfill_sync` 구현 존재
  - 이론상 `2020-01-01`부터 전체기간 백필도 가능
- 필요 자격증명
  - `PLAYAUTO_API_KEY`
  - `PLAYAUTO_EMAIL`
  - `PLAYAUTO_PASSWORD`

현재 workspace 상태:

- `seo/backend/.env`에는 `PLAYAUTO_API_KEY`, `PLAYAUTO_EMAIL`, `PLAYAUTO_PASSWORD`, `PLAYAUTO_BASE_URL`이 있다
- `GET /health` 기준으로도 `playauto.ready=true`
- 다만 현재 문서의 수치는 **운영 DB `tb_playauto_orders` 기준**이며, 이번 업데이트에서 PlayAuto live API 재호출까지 다시 돌린 것은 아니다

운영 DB에 이미 있는 PlayAuto 2025 coffee-like 데이터:

- 필터 기준
  - `order_name like '%더클린커피%'`
  - 또는 `shop_sale_name like '%더클린커피%'`
  - 또는 `shop_name`이 coffee/클린커피 계열
- 2025-01-01 이후 coffee-like 행 수: `7,644`
- 이 중 2025년만 보면:
  - order-like rows `6,413`
  - `pay_amt` 합계 `21,870,070원`
- distinct phone key 기준:
  - 고객 추정치 `5,304명`
  - 2회 이상 phone key `1,000명`
  - 재구매 proxy `18.85%`
  - 평균 orders-like `1.44회`

중요한 해석:

- PlayAuto는 `order_name / order_htel / to_name / to_htel / shop_sale_name / ord_time / pay_time / pay_method`가 있어
  - **고객 반복 주문 흔적**을 보는 데는 꽤 유용하다
- 하지만 `pay_amt=0`인 행이 많고, 채널 혼합/물류성 행이 섞일 수 있다
  - 따라서 PlayAuto만으로 `정식 LTR`을 계산하면 위험하다
  - 대신 **재구매율 / 재구매 횟수의 보조 proxy**로는 활용 가능하다

현재 판단:

- PlayAuto는 `2025 더클린커피 재구매 빈도`를 대략 보는 1차 보조 원장으로 쓸 수 있다
- 하지만 `공식 LTR/순매출 원장`은 아니다

### 5-2. 아임웹 API

결론:

- 코드 구조상으로는 **2025 더클린커피 주문을 API로 다시 가져오는 것이 가능**
- 현재 `seo/backend/.env`에는 `IMWEB_API_KEY`, `IMWEB_SECRET_KEY`가 있어 **자격증명 자체는 준비된 상태**
- 하지만 coffee 전용 `site_code` 확인과 coffee 주문을 어디로 적재할지에 대한 수집 경로는 아직 안 닫혔다. 그래서 **즉시 정식 coffee 주문 원장 수집 완료**라고 말할 수는 없다

코드 구조:

- `revenue/backend/app/tasks/iamWebApi/openapi_client.py`
  - `get_openapi_token`, `fetch_orders` 구현 존재
- `revenue/backend/app/tasks/salesSync/batch.py`
  - 월 단위 주문 fetch 후 `tb_iamweb_users` 적재 구현 존재

현재 제약:

- `seo/backend`는 `IMWEB_API_KEY`, `IMWEB_SECRET_KEY` 기반 readiness는 확인됐다
- 하지만 coffee 주문을 당기려면 최소한 아래가 더 필요하다
  - 더클린커피용 `site_code`
  - coffee를 별도 store/ledger로 적재할 경로
  - 기존 바이오컴 단일 sync와 충돌하지 않는 수집 규칙
- 특히 `revenue` 코드 관점에서는 `IMWEB_SITE_CODE` 단일 값 구조라
  - 바이오컴과 별도인 더클린커피 site_code를 분리 반영해야 한다

즉, 아임웹 API는 이런 의미다.

- **재구매/LTR의 정본 후보는 아임웹 주문 원장**
- 그런데 지금은 coffee site_code와 API 자격증명이 없어 아직 못 당겨오고 있다

### 5-3. 최종 판단

우선순위는 이렇게 잡는 것이 맞다.

1. 당장 할 수 있는 것
   - 운영 DB의 `tb_playauto_orders`로 2025 더클린커피 재구매 proxy 보기
2. 반드시 필요한 정식 데이터 보강
   - coffee Imweb `site_code` 확보
   - Imweb Open API 자격증명 확보

정리하면:

- `PlayAuto`: 지금도 2025 데이터 확인 가능, 다만 **보조 proxy**
- `아임웹 API`: 정식 해법, 현재 키는 있으나 **coffee site_code와 적재 경로 정리**가 필요

### 8-2-1. 재구매율 숫자 해석 기준

헷갈리기 쉬운 숫자를 먼저 고정하면 아래와 같다.

- `재구매율 21.7%`:
  - **2025-01-01 ~ 2025-12-31 전체 관측창 기준**
  - PlayAuto 고유 고객 `7,035명` 중 `2회 이상 구매 고객 1,528명`
  - 즉 `1년치 고객 전체를 한 번에 본 전체 재구매 고객 비중`이다.
- `1회 구매 고객 78%`:
  - 위와 같은 **2025 전체 관측창 기준**
  - `1회 구매 고객 5,507명 / 전체 고객 7,035명 = 78.3%`
  - 따라서 실무적으로는 `1회 구매 후 이탈 78%`라고 줄여 말할 수 있다.
- `6개월 재구매율 26.5%`:
  - **2025년 1~6월에 첫 구매한 고객 3,005명** 기준
  - 같은 고객을 6개월 관찰했을 때의 재구매 비중이다.

중요:
- 위 `21.7% / 78%`는 `90일 기준`도 `180일 기준`도 아니다.
- 이번 업데이트에서 `90일 재구매율`, `180일 재구매율`은 아래 기준으로 재계산했다.
  - 데이터: `tb_playauto_orders`
  - 필터: `커피/콜롬비아/에티오피아/과테말라/드립백/원두/케냐` 키워드 또는 `order_name like '%커피%'`
  - 고객 키: `order_htel` 숫자 정규화
  - 주문 수: `COUNT(DISTINCT bundle_no)`
  - 관찰 종료일: `2025-12-31`
  - 재구매 판정: 첫 주문일 포함 `90일/180일` 안에 `2회 이상 bundle` 발생
- 월별 코호트 표는 관찰 기간이 서로 달라서 그대로 `90일/180일 기준`으로 읽으면 안 된다.

### 8-2-2. 90일 / 180일 재구매율

### 9-2. CRM 전략: "3주 차 윈백" + "충성 고객 유지"

현재 데이터에서 가장 큰 기회는 **1회 구매 후 이탈하는 78%의 고객**이다.

#### 전략 A. 3주 차 첫 재구매 유도 (가장 파급력 큼)

- **왜**: 재구매 간격 p75 = 21일. 이 시점을 넘기면 이탈 확률이 급증
- **누구에게**: 첫 구매 후 14~21일이 지났는데 2회차 주문이 없는 고객
- **무엇을**: "원두가 떨어질 때쯤이죠? 다음 원두를 골라보세요" + 재주문 할인 쿠폰(5~10%)
- **채널**: 알리고 알림톡 (1순위), 채널톡 (2순위)
- **측정**: 메시지 발송 후 7일 내 재주문 전환율 vs 미발송 대조군
- **기대 효과**: 1회 고객 5,507명 중 5%만 전환해도 275명 × 평균 3만원 = **825만원 추가 매출**

#### 전략 B. 정기구독 전환 유도

- **왜**: 정기구독 고객 128명은 물류 기준이라 실제는 더 많을 수 있지만, 전체 대비 매우 적음
- **누구에게**: 3회 이상 구매 고객 (617명 = 2회 + 274명 = 3회 이상)
- **무엇을**: "매번 주문하는 번거로움 없이 정기배송으로" + 정기구독 첫 달 15% 할인
- **타이밍**: 3회차 구매 직후 또는 4회차 주문 시점
- **측정**: 구독 전환율, 구독 유지율(M+1, M+3, M+6), 구독 고객 LTR vs 비구독 LTR

#### 전략 C. 충성 고객 이탈 방지 (VIP 관리)

- **왜**: 6회+ 고객 383명이 매출의 28.7%를 차지. 이 중 10%만 이탈해도 큰 타격
- **누구에게**: 6회 이상 구매했지만 최근 60일 이상 주문이 없는 고객
- **무엇을**: "오랜 단골 고객님을 위한 특별 원두 추천" + 신규 원두 선행 체험 기회
- **절대 하지 말 것**: 할인 쿠폰으로 VIP를 유인하면 안 됨. VIP는 가격이 아니라 경험/신뢰로 유지

#### 전략 D. 상품 크로스셀

- **왜**: 콜롬비아 편중(고객의 46%). 단일 상품 의존은 이탈 시 충격이 큼
- **누구에게**: 콜롬비아만 3회 이상 구매한 고객
- **무엇을**: "콜롬비아를 좋아하시는 분들이 다음으로 많이 선택한 원두" → 에티오피아/과테말라 추천
- **콘텐츠**: 원두별 풍미 비교 콘텐츠 (산미/바디/로스팅 차이)

### 9-3. 가격 인상/인하 판단 프레임

가격 결정은 "감"이 아니라 **4가지 데이터 신호**로 판단한다.

#### 신호 1. 가격 탄력성 (Price Elasticity)

현재 확인 가능한 것:
- Toss 가격대별 분포: 2~4만원대 303건(46%), 4~6만원대 211건(32%)
- 고가(6만+) 132건도 20% → **고가 수요가 존재**
- PlayAuto 유료 평균 29,000~32,000원 → 최근(2026) 31,000원대로 **이미 소폭 상승 추세**

판단:
- 현재 가격대에서 수요가 줄지 않았으므로 **5~10% 인상 테스트 여지 있음**
- 단, 테스트는 전체 가격이 아니라 **특정 상품(예: 에티오피아)만 먼저**

#### 신호 2. 재구매 탄력성 (Repeat Purchase Sensitivity)

- 재구매 고객의 평균 객단가: 12,418원(2회) → 20,248원(4~5회) → **구매 횟수가 늘수록 객단가도 상승**
- 이는 충성 고객이 **더 많이, 더 비싼 것을 사는 것**이지 할인 때문에 사는 게 아님

판단:
- 충성 고객에게는 **가격보다 품질/경험이 결정 요인** → 프리미엄 라인 출시 가능
- 신규 고객에게는 **진입 가격이 중요** → 첫 구매 할인은 유지하되, 정가를 올리는 전략

#### 신호 3. 가격 인상 실험 방법

```
실험: 에티오피아 예가체프 가격 5% 인상 (A/B)
- A그룹: 기존 가격
- B그룹: 5% 인상 가격
- 측정: 30일 전환율, 재구매율, 순매출
- 기간: 4주
- 대상: 신규 방문자 50% 랜덤 배정
- 판단 기준: B그룹 순매출이 A그룹 대비 5% 이상 감소하면 인상 철회
```

#### 신호 4. 인하가 필요한 경우

아래 3가지 신호가 동시에 나타나면 인하를 고려:
1. 월 신규 고객 수가 3개월 연속 감소
2. 첫 구매 전환율(세션→구매)이 전월 대비 20% 이상 하락
3. 경쟁사(프릳츠, 테라로사 등)가 같은 가격대에서 공격적 프로모션

현재 데이터 기준으로는 **이 3가지 신호가 없으므로 인하 불필요**. 오히려 소폭 인상 테스트가 맞는 시점.

- 단, 일부 고객은 이미 다른 곳에서 샀을 수 있음

## 11. Codex 관점 보완 의견 (0401)

> 아래는 CSO 전략(섹션 9)과 PlayAuto 분석(섹션 8)에 대한 백엔드/데이터 설계 관점 보충이다. 기존 내용을 수정하지 않고 덧붙인다.

### 11-1. PlayAuto 데이터 품질 경고

CSO 전략의 숫자들은 모두 PlayAuto 기반이다. 이 데이터에는 구조적 한계가 있다.

**pay_amt=0 문제가 심각하다**
- 전체 16,180행 중 유료 주문(pay_amt > 0)은 1,630건뿐이다. 나머지 14,550행은 물류/증정/정기구독 배송 행이다.
- 이 때문에 "고객당 평균 매출 5,508원"은 실제보다 크게 과소추정된다.
- 정확한 LTR을 구하려면 `tb_sales_toss.store='coffee'`의 결제 금액과 PlayAuto의 전화번호를 조인해야 한다.

**전화번호 정규화의 불안정성**
- PlayAuto의 `order_htel`과 `to_htel`이 다를 수 있다 (주문자 ≠ 수령자)
- 같은 고객이 다른 전화번호로 주문하면 별개 고객으로 집계된다
- 재구매율 21.7%는 실제보다 낮게 잡혔을 가능성이 높다 (동일 고객 분리)

**bundle_no 기준 주문 수의 함정**
- PlayAuto의 `bundle_no`는 출고 묶음 단위이지 결제 단위가 아니다
- 1건 결제에 상품 3개면 bundle_no가 3개일 수 있다
- 따라서 "주문 횟수"가 실제 구매 행위 횟수보다 부풀려졌을 수 있다

**권장: Toss + PlayAuto 크로스 검증**
```sql
-- Toss 기준 2026-01 커피 객단가: 45,578원
-- PlayAuto 기준 같은 기간 유료 평균: 31,925원
-- 차이 = 13,653원 → 묶음 주문/물류 행 중복 가능성
```

### 11-2. "3주 차 재구매 알림톡" 실행 시 기술 제약

CSO 전략 1순위인 3주 차 알림톡 발송을 실제로 하려면 아래가 먼저 풀려야 한다.

| 항목 | 현재 상태 | 필요한 것 |
|------|----------|----------|
| 대상자 추출 | PlayAuto에서 전화번호 추출 가능 | "21일 전 첫 구매 + 2회차 미구매" 쿼리를 매일 실행하는 배치 |
| 알리고 발송 | **전부 확보됨** (apiKey, userId, senderPhone, senderKey, channelId) | 템플릿 등록 + 카카오 심사만 남음 |
| 템플릿 | 미등록 | 알리고 관리자에서 커피 재구매 유도 템플릿 등록 + 카카오 심사 통과 |
| 대조군 설계 | crm_local SQLite에 실험 생성 가능 | 대상자의 50%를 holdout으로 배정하여 효과 측정 |
| 수신 동의 | 확인 불가 | 더클린커피 Imweb 회원가입 시 마케팅 수신 동의 여부 확인 필요 |

즉 "즉시 실행 가능"은 기술적으로는 맞지만, **알리고 자격증명 + 템플릿 심사 + 수신 동의 확인**이 선행되어야 한다. 이 3가지 없이 발송하면 스팸 리스크가 있다.

### 11-3. 가격 인상 A/B 테스트의 기술적 현실

CSO 전략 2순위인 가격 A/B 테스트는 현재 **아임웹 기반 사이트에서 서버사이드 A/B가 불가능**할 수 있다.

- 아임웹은 SaaS 빌더로, 동일 상품의 가격을 사용자별로 다르게 보여주는 기능이 제한적이다
- 가능한 대안:
  1. **쿠폰 방식**: A그룹에는 할인 쿠폰 미제공, B그룹에는 5% 쿠폰 제공 → 역으로 인상 효과 측정
  2. **시간 기반 A/B**: 1주는 기존가, 다음 1주는 인상가 → 시즌 편향 있지만 빠름
  3. **상품 단위**: 에티오피아만 인상하고 콜롬비아는 유지 → 상품 간 전환 효과 측정

### 11-4. 정기구독 데이터의 사각지대

현재 PlayAuto에서 "정기구독 762건, 128명"으로 보이는데, 이것은 **구독 결제가 아니라 구독 배송 물류 행**일 가능성이 높다.

- PlayAuto는 물류 시스템이므로 결제 시점이 아니라 출고 시점 기준이다
- 실제 구독 전환율을 보려면 아임웹의 구독 관리 데이터가 필요하다
- 구독 유지율(M+1, M+3, M+6 churn)은 현재 어떤 데이터로도 계산할 수 없다

### 11-5. 실행 가능한 즉시 백엔드 작업

Codex가 바로 할 수 있는 것:

1. **커피 전용 API 라우트** (`/api/coffee/*`)
   - PlayAuto에서 커피 데이터만 조회하는 전용 엔드포인트
   - 월별 추이, 재구매율, 코호트, 상품 믹스를 실시간으로 계산
   - 현재 `/coffee` 프론트 페이지가 정적 데이터를 쓰고 있으므로, 이것을 API로 교체하면 자동 갱신됨

2. **Toss+PlayAuto 크로스 조인 API**
   - `tb_sales_toss.order_id`와 `tb_playauto_orders.shop_ord_no`를 매칭
   - 매칭되면 Toss 결제 금액 + PlayAuto 전화번호를 결합 → 정확한 고객별 결제 매출 산출

3. **3주 차 재구매 대상자 추출 배치**
   - PlayAuto에서 "21일 전 첫 구매 + 이후 주문 없는 전화번호" 목록을 매일 생성
   - `/api/crm-local/leads`에 자동 적재
   - `/crm` 후속 관리 탭에서 바로 확인 가능

### 11-6. 종합 판단

CSO 전략의 방향은 맞다. 특히 "3주 차 재구매"에 집중하는 것은 데이터가 뒷받침한다. 다만 실행 순서를 기술적 현실에 맞게 조정하면:

1. **이번 주**: Toss+PlayAuto 크로스 조인으로 정확한 커피 객단가/LTR 확인
2. **다음 주**: 알리고 자격증명 확보 + 알림톡 템플릿 등록 신청
3. **2주 후**: 3주 차 재구매 대상자 추출 배치 + 대조군 설계 + 첫 발송
4. **4주 후**: 발송 후 7일 전환율 확인 → 가격 A/B 테스트 여부 결정

### 14-2. 핵심 해석

1. **PlayAuto LTR 5,508원은 심각한 과소추정이었다**. 실제는 98,112원. 물류 행(pay_amt=0)이 분모를 부풀렸기 때문.

2. **실제 커피 객단가는 48,579원**. 이전 Toss 직접 조회(45,500원)와 거의 일치. PlayAuto 유료 평균(31,000원)과는 차이가 있는데, 이는 PlayAuto가 line-item 단위(상품 1개)이고 Toss가 주문 단위(묶음 결제)이기 때문.

3. **재구매율은 양쪽 모두 ~21%로 일치**. 재구매율 자체는 PlayAuto도 신뢰할 수 있음.

4. **Toss 매칭 고객 517명 vs PlayAuto 전체 7,035명**: 이 수치는 backfill 전 2개월 기준 비교다. 0406 backfill 이후에는 2025~2026 범위로 다시 맞춰 계산해야 한다.

### 14-3. 수정된 커피 전략 시사점

- **쿠폰 15% 할인의 실제 마진 영향**: 객단가 48,579원 × 15% = **₩7,287 할인** (기존 추정 4,650원보다 큼)
- **VIP 1인당 가치 재계산**: Toss LTR 98,112원 기준으로 VIP(6회+)는 평균 **₩588,000+**
- **3주 차 재구매 쿠폰 ROI**: 전환율 5%p 상승 시 1인당 48,579원 × 5% 추가 = **₩2,429 추가 매출** vs 할인 7,287원 → 마진 관점에서 재검토 필요

### 14-4. PlayAuto와 Toss 차이의 근본 원인 분석 (Codex 상세)

#### 원인 1: PlayAuto는 물류 시스템이라 결제 금액이 0원

PlayAuto 커피 데이터 16,179행의 pay_amt 분포:

| pay_amt 구간 | 행 수 | 비율 | 평균 금액 |
|-------------|-------|------|----------|
| **0원 (물류/증정)** | **14,413행** | **89.1%** | **₩0** |
| 1~1만원 | 17행 | 0.1% | ₩8,665 |
| 1~3만원 | 736행 | 4.5% | ₩19,852 |
| 3~5만원 | 983행 | 6.1% | ₩35,624 |
| 5만원+ | 30행 | 0.2% | ₩81,930 |

**핵심**: PlayAuto는 **주문 관리/물류 시스템**이다. 결제는 Toss에서 처리되므로, PlayAuto에는 결제 금액이 기록되지 않는다. PlayAuto의 역할은 "어떤 상품을 어디로 보냈는지"이지, "얼마를 결제했는지"가 아니다.

그래서 PlayAuto 기반 LTR(5,508원)은 전체 행(0원 포함)의 평균이라 과소추정된 것이고, Toss 기반 LTR(98,112원)이 실제 결제 금액이다.

#### 원인 2: PlayAuto 1주문 = N행, Toss 1주문 = 1행

bundle_no당 행 수 분포:

| 행 수/주문 | 주문 수 | 평균 금액 |
|-----------|---------|----------|
| 1행 | 9,244 | ₩4,946 |
| 2행 | 2,096 | ₩2,512 |
| 3행 | 605 | ₩2,062 |
| 4행+ | 218 | ₩0 |

PlayAuto는 **상품 line-item 단위**로 행을 만든다. 1건 주문에 콜롬비아 + 에티오피아 + 드립백을 함께 사면 PlayAuto에는 3행이 생기지만, Toss에는 1행(합산 결제)만 있다.

실제 비교 예시:

| Toss 결제 | PlayAuto 행 수 | PlayAuto 금액 | 상품 |
|----------|---------------|---------------|------|
| ₩56,715 | 3행 | ₩0 | 케냐 + 과테말라 + 추가구매 |
| ₩43,600 | 2행 | ₩0 | 드립백 × 2 |
| ₩66,600 | 3행 | ₩0 | 콜롬비아 × 2 + 추가구매 |
| ₩21,300 | 1행 | ₩0 | 콜롬비아 |

**PlayAuto pay_amt = 0이고 Toss amount = 실 결제**인 것이 명확히 보인다.

#### 원인 3: "주문 횟수" 정의가 다름

- **PlayAuto 기준 주문 횟수**: `COUNT(DISTINCT bundle_no)` — 물류 출고 단위. 같은 결제도 분할 출고되면 여러 bundle
- **Toss 기준 주문 횟수**: `COUNT(DISTINCT order_id)` — 결제 1건 = 주문 1건
- 그래서 PlayAuto 평균 1.73회 > Toss 평균 1.28회

#### 결론: 앞으로의 지표 기준

| 지표 | 사용할 데이터 | 이유 |
|------|-------------|------|
| **LTR / 객단가 / 순매출** | **Toss** | 실제 결제 금액 |
| **재구매율** | PlayAuto 또는 Toss | 양쪽 ~21%로 일치 |
| **고객 식별 (전화번호)** | **PlayAuto** | Toss에는 전화번호 없음 |
| **상품별 분석** | **PlayAuto** | 상품명/옵션(홀빈/분쇄) 있음 |
| **PG 수수료** | **Toss 정산 API** | 건별 수수료 확인 (3.41~3.63%) |
| **크로스 분석** | **Toss × PlayAuto 조인** | orderId 매칭으로 "누가 얼마를 샀는지" |
