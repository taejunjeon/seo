# biocom GTM 오류 정리

작성일: 2026-04-09

## 1. 한 줄 결론

- 지금 문제는 **결제 데이터 전체가 안 들어가는 장애**로 보이지 않는다.
- 대신 **biocom 결제완료 페이지에서 외주 CRM용 GTM 코드가 브라우저 오류를 내고 있는 상태**로 보는 것이 맞다.
- 그래서 우선순위는 아래 순서다.
  1. **결제완료 페이지를 안정화**
  2. **메인 구매 추적은 건드리지 않기**
  3. **외주 CRM 코드가 아직 실제로 필요한지 확인**
  4. 필요 없으면 단계적으로 제거

## 2. 지금 확인된 사실

### 이 문서에서 말하는 "자체 솔루션"이란

- 전태준 대표님이 직접 구축한 **자체 데이터 수집/분석 시스템**을 뜻한다.
- 쉽게 말해:
  - 사이트 결제완료 페이지에서 결제 성공 정보를 별도로 받아서 저장하고
  - 그 데이터를 광고/유입 데이터와 나중에 연결해 보는 내부 솔루션이다.
- 따라서 이 문서에서 말하는
  - `payment-success 적재`
  - `원장`
  - `기록 테이블`
  같은 표현은 모두 **대표님이 구축한 자체 솔루션에 저장되는 내부 데이터**를 뜻한다.

### 메인 구매 적재는 살아 있다

- 전태준 대표님이 구축한 **자체 솔루션의 결제 성공 기록 테이블**에는 지금도 데이터가 계속 들어오고 있다.
- 최근에는 그 기록 안에
  - `ga_session_id`
  - `client_id`
  - `user_pseudo_id`
  같은 웹 추적 식별자도 일부 함께 저장되기 시작했다.
- 즉 "결제 성공 데이터가 아예 다 안 들어가는 상태"라고 보긴 어렵다.

### 문제는 결제완료 페이지의 외주 CRM 보조 코드다

- `shop_payment_complete` 또는 유사 결제완료 페이지에서 콘솔 오류가 관찰됐다.
- 오류 형태는 `Cannot read properties of null (reading 'includes')` 계열이다.
- 가장 유력한 원인은 **외주 CRM용 보조 GTM 컨테이너 `GTM-W7VXS4D8` 안의 Custom HTML 태그**다.
- 구체적으로는 `sessionStorage`에서 값을 읽은 뒤, 값이 비어 있는데도 `.includes(...)`를 호출하는 코드가 의심된다.

### 메인 구매 추적과 외주 CRM 추적은 분리해서 봐야 한다

- 메인 구매 추적:
  - `GTM-W2Z6PHN`
  - 전태준 대표님이 구축한 자체 솔루션으로 결제 성공 정보를 보내는 사이트 스크립트
- 외주 CRM 보조 추적:
  - `GTM-W7VXS4D8`

즉 지금은 **메인 구매 추적을 고칠 문제가 아니라**, 외주 CRM 보조 코드가 결제완료 페이지에서 오류를 내는 문제를 먼저 막아야 한다.

## 3. 무엇이 왜 문제인가

### 문제 1. 결제완료 페이지에서 브라우저 오류가 난다

- 결제완료 페이지는 가장 민감한 페이지다.
- 여기서 오류가 나면:
  - 외주 CRM 이벤트
  - 리타겟팅 이벤트
  - 특정 구매/오픈 이벤트
  가 일부 빠지거나 꼬일 수 있다.

### 문제 2. "어떤 데이터가 정상이고 어떤 데이터가 깨졌는지"가 불명확해진다

- 메인 구매 적재는 살아 있어도,
- 외주 CRM 코드가 따로 잘못 동작하면
- 운영팀은 "숫자가 왜 다르지?"를 계속 겪게 된다.

즉 이 문제는 단순 콘솔 경고가 아니라, **결제완료 페이지의 추적 신뢰도를 떨어뜨리는 문제**다.

### 문제 3. 외주사 코드가 아직 필요한지 불명확하다

- 해당 GTM 컨테이너는 임시로 세팅된 외주 CRM용 보조 컨테이너다.
- 나중에 제거할 가능성이 높다면,
- 지금 해야 할 일은 "정교하게 살리기"보다
- **정말 필요한 기능만 확인하고, 결제완료 페이지에서 안전하게 영향 차단하는 것**에 가깝다.

## 4. 해결 방향

### 기본 원칙

1. **메인 구매 추적은 건드리지 않는다**
2. **결제완료 페이지의 외주 CRM 코드 오류를 먼저 막는다**
3. **외주사가 아직 실제로 쓰는 기능인지 증거로 확인한다**
4. **필요 없으면 payment page에서 먼저 끄고, 나중에 전체 제거를 검토한다**

### 가장 현실적인 해결책

#### A안. 외주사가 아직 필요하다고 하면

- 문제 코드를 `null-safe`하게 수정한다.
- 즉 값이 없을 때 `.includes()`를 바로 호출하지 않도록 고친다.

예:

```js
var utmSource = session && typeof session.utmSource === "string"
  ? session.utmSource
  : "";

if (!utmSource.includes("RETOUS_")) return;
```

#### B안. 외주 기능이 사실상 안 쓰이거나 중요도가 낮다면

- 결제완료 페이지에서만 해당 외주 CRM 태그/컨테이너 발화를 먼저 막는다.
- 이 방식은 빠르고 안전하다.
- 메인 구매 추적은 그대로 두고, 문제 구간만 분리할 수 있다.

## 5. 개발팀에 요청할 사항

아래는 **우리 회사 개발팀이 직접 해야 할 일**이다.

### 1. 결제완료 페이지에서 어떤 GTM이 실제로 실행되는지 다시 확인

**무엇을 해야 하나**

- `shop_payment_complete`, `shop_order_done` 또는 실제 결제완료 URL에서
  - `GTM-W2Z6PHN`
  - `GTM-W7VXS4D8`
  두 컨테이너가 각각 무엇을 실행하는지 확인

**왜 필요한가**

- 지금 문제는 "전체 사이트"가 아니라 "결제완료 페이지"일 가능성이 높다.
- 어느 태그가 여기서만 문제를 내는지 분리해야 한다.

**어떻게 확인하나**

- GTM Preview / Tag Assistant
- 브라우저 콘솔
- Network 탭
- 실제 결제완료 URL 기준 확인

### 2. 메인 구매 추적이 정상인지 먼저 고정 확인

**무엇을 해야 하나**

- 아래 3개가 여전히 정상인지 확인
  - GA4 purchase 이벤트
  - 전태준 대표님이 구축한 자체 솔루션으로 결제 성공 정보가 정상 저장되는지
  - 그 결제 정보가 내부 기록 테이블에 정상 반영되는지

**왜 필요한가**

- 외주 CRM 코드를 끄거나 수정하더라도 메인 구매 추적이 깨지면 안 된다.

**어떻게 확인하나**

- 가상계좌 주문 1건 또는 테스트 주문 1건으로 확인
- 확인 항목:
  - 콘솔 오류 사라졌는지
  - GA4 DebugView purchase 들어오는지
  - 대표님이 구축한 자체 솔루션의 결제 성공 수집 API가 계속 정상 응답하는지
  - 그 결과가 내부 기록 테이블에 실제로 남는지

### 3. 결제완료 페이지에서 외주 CRM 태그를 임시 차단할 준비

**무엇을 해야 하나**

- 외주사 대응이 늦으면, 개발팀이 GTM 관리자와 함께
  - 결제완료 페이지에서만
  - `GTM-W7VXS4D8` 전체 또는 문제 태그 발화를 막는 안을 준비

**왜 필요한가**

- 지금 필요한 것은 "빠른 안정화"다.
- 결제완료 페이지 오류를 방치할 이유가 없다.

**어떻게 해결하나**

- page path 기준 exception trigger
- payment complete 전용 trigger 제외
- 문제 Custom HTML / 문제 이벤트 태그만 pause

### 4. 외주사에 "정상 동작 증거"를 요구

**무엇을 해야 하나**

- 외주사에 "지금 이 태그가 실제로 필요하다"는 말이 아니라
- **정상 동작 증거**를 요청

**왜 필요한가**

- 필요 없는 태그를 유지하면 오류만 남고 운영 복잡도만 커진다.

**어떻게 요청하나**

- 아래 3가지를 요구
  1. GTM Preview fired tags
  2. 콘솔 에러 없는 화면
  3. 실제 CRM/광고/웹훅 전송 로그 또는 수신 로그

## 6. 리인벤팅 CRM 외주사에 개발팀이 요청할 사항

아래는 **우리 개발팀이 리인벤팅 쪽에 직접 요청해야 할 내용**이다.

### 1. 현재 외주 CRM 태그가 실제 운영에서 아직 필요한지 먼저 답변 요청

**질문**

- `GTM-W7VXS4D8` 안의 CRM 관련 태그가 지금도 실제 운영 보고서나 자동화에 쓰이고 있는가?
- 특히 `c_retous_crm_open` 또는 유사 이벤트가 아직 필요하냐?

**왜 필요한가**

- 필요 없으면 고칠 이유보다 제거할 이유가 더 크다.

### 2. 문제 코드 수정 또는 결제완료 페이지 예외 처리 요청

**질문**

- `sessionStorage` 또는 `utmSource`가 비어 있을 때 `.includes()`를 호출하지 않도록 수정 가능한가?
- 바로 수정이 어렵다면 결제완료 페이지에서만 태그 예외 처리 가능한가?

**왜 필요한가**

- 현재 오류는 결제완료 페이지 안정성을 해친다.
- 결제완료 페이지는 매출 데이터가 닫히는 핵심 구간이므로, 보조 추적보다 안정성이 우선이다.

### 3. 수정 후 증거 제출 요청

**요청할 자료**

1. 수정한 GTM 태그 이름 또는 ID
2. 수정 전후 코드 차이
3. 결제완료 페이지 GTM Preview 결과
4. 브라우저 콘솔 에러가 사라진 화면
5. 실제 CRM 수신 또는 광고 이벤트 수신 로그

**왜 필요한가**

- "고쳤다"는 말만으로는 다시 같은 문제가 반복된다.
- 운영팀이 재검증할 수 있는 증거가 필요하다.

### 4. 더 이상 안 쓰는 태그 목록 요청

**질문**

- 지금 실제로 안 쓰는 리타겟팅 태그, CRM open 태그, legacy 태그가 무엇인지 목록으로 줄 수 있는가?

**왜 필요한가**

- 결제완료 페이지뿐 아니라, 나중에 `GTM-W7VXS4D8` 전체를 줄이거나 제거할 때 기준이 된다.

## 7. 개발팀과 외주사 역할을 한 번에 보면

### 우리 개발팀

- 결제완료 페이지 기준 현상 재확인
- 메인 구매 추적 정상 여부 확인
- 외주사 수정 전 임시 차단안 준비
- 수정 후 재검증

### 리인벤팅 외주사

- 현재 태그 필요 여부 답변
- 오류 코드 수정 또는 예외 처리
- 수정 결과 증거 제출
- 더 이상 안 쓰는 태그 정리

## 8. 완료 기준

아래 5개가 만족되면 이번 이슈는 일단 닫아도 된다.

1. 결제완료 페이지 콘솔에서 `includes` 오류가 사라짐
2. 메인 purchase 추적은 그대로 유지됨
3. 전태준 대표님이 구축한 자체 솔루션의 결제 성공 기록이 계속 정상 저장됨
4. 외주 CRM 태그의 필요 여부가 명확해짐
5. 필요 없으면 payment page 차단 또는 전체 제거 계획이 확정됨

## 9. 지금 가장 현실적인 판단

- **지금 바로 필요한 것은 "외주 GTM을 살리기 위한 정밀 수리"가 아니라, 결제완료 페이지를 안정화하는 것**이다.
- 메인 구매 추적은 유지하고,
- 외주 CRM 태그는
  - 정말 필요하면 고치고
  - 아니면 결제완료 페이지에서 먼저 끄고
  - 이후 전체 제거를 검토하는 순서가 맞다.

## 10. 아주 짧은 버전

- **문제**: biocom 결제완료 페이지에서 외주 CRM GTM 코드가 `null.includes(...)` 오류를 내는 것으로 보임
- **영향**: 전태준 대표님이 구축한 자체 솔루션의 메인 결제 기록은 살아 있는 것으로 보이지만, 결제완료 페이지의 CRM/보조 이벤트 신뢰도가 떨어짐
- **우선순위**: 메인 구매 추적은 유지, 외주 CRM 태그는 필요 여부 확인 후 수정 또는 payment page 차단
