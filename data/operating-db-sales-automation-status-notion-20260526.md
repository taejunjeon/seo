## 10초 요약

이 문서는 우리 솔루션이 직접 호출할 API 목록이 아니다.

개발팀이 운영DB 쪽에서 매출 채널을 어떻게 자동 수집하거나 엑셀 업로드로 처리하는지 정리한 기준 문서다. 따라서 더클린커피 Slack 매출보고에서는 “직접 API 가능”의 근거가 아니라, 운영DB에 이미 들어올 수 있는 source와 수동 엑셀 source를 구분하는 참고 문서로 써야 한다.

특히 스마트스토어는 개발팀 기준으로 공식 API 자동 동기화가 있다고 적혀 있지만, Codex/VM Cloud에서 더클린커피 스마트스토어를 직접 조회할 권한이 있다는 뜻은 아니다. 더클린커피는 네이버 통합매니저 권한을 받아야 하며, TJ님이 2026-05-26 고객센터에 문의를 남겼다. 2026-05-27 KST에 재확인한다.

## Codex 해석 메모

### 이 문서가 도움 되는 점

- 운영DB 쪽 매출 채널별 수집 방식이 API인지 엑셀인지 빠르게 구분할 수 있다.
- 스마트스토어 매출은 운영DB에 API 기반으로 들어올 가능성이 있어, PlayAuto 경고를 줄일 다음 조사 대상이다.
- 쿠팡 3P와 쿠팡 로켓그로스는 개발팀 문서 기준으로도 엑셀 업로드 중심이라, 현재 우리가 쿠팡 API와 정산표를 병행 대조하는 판단과 충돌하지 않는다.
- 네이버페이 정산 내역은 API가 아니라 엑셀이라고 되어 있어, NPay actual 매출과 정산 보고를 섞지 말아야 한다는 기존 규칙과 맞는다.

### 주의할 점

- `API 제공`은 “개발팀 운영DB 동기화 가능”이라는 뜻이다. Codex가 보유한 네이버 커머스API 권한, VM IP 허용, 더클린커피 스토어 scope가 열렸다는 뜻이 아니다.
- 운영DB 문서는 원장 위치와 수집 정책의 참고값이다. 실제 Slack 매출보고 숫자는 window, freshness, site 분리를 다시 검증해야 한다.
- 정산일, 결제일, 결제 승인일, 매출 인식일이 채널마다 다르다. 같은 날짜 범위라도 Excel 보고서와 자동 집계가 다를 수 있다.

## 2. API 자동 동기화 채널

### 토스페이먼츠

- 공식 OpenAPI 제공.
- 인증은 HTTP Basic 방식이다. 상점별 인증 키는 운영 환경에서 관리하며 이 로컬 보관본에는 원문 키를 저장하지 않는다.
- 목록 endpoint와 상세 endpoint를 조합한다.
- 상점 3개를 별도 키로 관리한다. 예: 알로스타, 펫, 자회사 커피.
- 결제수단을 자동 분류한다. 카드, 가상계좌, 계좌이체를 각각 구분한다.

### 네이버 스마트스토어

- 공식 OpenAPI 제공.
- 인증은 OAuth2와 bcrypt 전자서명 방식이다.
- 건별 부가세 정산내역 endpoint를 사용한다고 적혀 있다.
- 기준일은 `settleBasisDate`다. 결제일이 아니라 네이버가 정산 기준으로 확정한 날짜다.

## 3. 수동 엑셀 업로드 채널

### 네이버페이 정산 내역

- 정산 API를 제공하지 않는 것으로 정리되어 있다.
- 부가세 API는 자동 동기화되지만, 어떤 주문이 언제 얼마로 정산되었는지 보는 정산 내역 API는 별도로 없다고 되어 있다.
- 회계담당자가 네이버 커머스 셀러센터에서 정산 내역 엑셀을 다운받아 업로드한다.
- `sub_channel`로 바이오컴, 커피, 펫을 구분한다.

### 나이스페이

- 정산 API 없음으로 정리되어 있다.
- 회계담당자가 나이스페이 가맹점 페이지에서 월별 정산보고서를 엑셀로 다운받아 업로드한다.

### 쿠팡 3P

- 쿠팡 셀러용 Open API에 매출/정산 조회 기능이 없다고 정리되어 있다.
- 원본 시스템에 쿠팡 API 클라이언트는 있지만 재고·상품명 조회 전용이고 매출 조회 기능은 없다고 되어 있다.
- 회계담당자가 쿠팡 WING에서 정산내역 엑셀을 다운받아 업로드한다.

### 쿠팡 로켓그로스

- 위탁판매 정산내역도 쿠팡이 위탁자에게 엑셀 형태로 제공한다고 정리되어 있다.
- 3P와 동일하게 WING 엑셀을 쓴다.

### 세금계산서

- 홈택스 공식 API 미제공으로 정리되어 있다.
- 서드파티 솔루션을 쓰면 가능할 수 있으나 추가 계약과 비용이 필요하다.
- 현재는 홈택스 매출세금계산서 발급 내역 엑셀을 업로드한다.

### 여신금융협회

- 매출 채널이 아니라 카드 정산 검증용 데이터다.
- 카드사 매입자료를 협회에서 통합 제공하고, 회계담당자가 다운받아 리커버리랩 카드 정산 검증 용도로 사용한다.
- 매출 집계에는 포함하지 않는다.

### 리커버리랩

- 알로스타 자체 운영 서비스이고 현재 자체 시스템과 매출 API 연동이 구현되어 있지 않다.
- 외부 업체 제약이 아니라 사내 우선순위 이슈로 정리되어 있다.
- 현재는 리커버리랩 관리자에게 엑셀파일을 전달받아 업로드한다.

## 4. 매출 인식 기준

기본 원칙은 결제·정산 완료 기준이다. 다만 채널별로 완료를 정의하는 시점이 외부 시스템 정책에 따라 다르다.

| 채널 | 기준 시점 | 회계 의미 |
|---|---|---|
| 토스 | `approvedAt` 결제 승인 | PG가 카드사로부터 결제 승인을 받은 즉시 |
| 네이버 VAT | `settleBasisDate` 정산기준일 | 네이버가 매출을 확정·정산 처리한 날. 결제 후 며칠 지연 가능 |
| 쿠팡 3P | `sales_recognition_date` 매출 인식일 | 쿠팡이 지정한 매출 인식일. 구매 확정 후 |
| 쿠팡 RG | `payment_complete_date` 결제 완료일 | 결제 완료 시점 |
| 세금계산서 | `issue_date` 발행일 | B2B 회계 표준 |
| 나이스페이 | 정산일 | 결제 후 PG 정산 처리된 날 |
| 리커버리랩 | `payment_date` 결제일 | 결제 시점 |

왜 채널별로 다른가:

1. 외부 시스템이 통보해주는 기준이 다르다. 토스는 결제승인 즉시, 네이버는 정산 확정 후, 쿠팡은 구매확정 후다.
2. 회계상 어느 날 매출로 잡을지는 정산금이 확정된 시점이 가장 안전하다. 결제 직후 잡으면 환불·취소 시 매출 정정이 많아진다.
3. 세금계산서는 발급일이 매출일의 법정 기준이다.

공통 원칙은 구매 완료나 배송 완료 기준이 아니라, 외부 시스템에서 결제·정산이 확정된 시점이다.

## 5. 토스 매출 제외 상태

아래 상태는 매출 인식에서 제외한다.

| 상태 | 의미 |
|---|---|
| `WAITING_FOR_DEPOSIT` | 입금 대기. 가상계좌 발급 후 미입금 |
| `READY` | 결제 페이지 진입 후 미진행 |
| `IN_PROGRESS` | 결제 처리 중. 미확정 |
| `ABORTED` | 사용자 중도 취소 |
| `EXPIRED` | 결제 유효기간 경과 |

즉, 결제가 실제 승인되어 자금이 이동한 건만 매출 인정한다. 원문 기준으로는 `DONE`, `CANCELED`, `PARTIAL_CANCELED` 상태가 대상이라고 정리되어 있다. 단, 취소·부분취소 상태를 매출 보고에서 어떻게 순액 처리할지는 운영DB 컬럼과 실제 회계 기준으로 별도 확인해야 한다.

## 6. 의사결정 한 줄 요약

공식 API가 제공되는 채널만 자동화한다. 토스와 네이버 스마트스토어가 여기에 해당한다.

그 외 채널은 외부 시스템 정책상 엑셀 업로드가 기본이다. 매출 인식 시점은 각 외부 시스템이 통보하는 정산·승인 확정 시점을 따른다.

## 7. 향후 자동화 검토 가능 항목

| 채널 | 자동화 검토 |
|---|---|
| 리커버리랩 | 자체 서비스이므로 사내 우선순위가 정해지면 직접 API 연동 가능 |
| 네이버 정산 | 네이버가 정산 API를 신규 제공하면 즉시 연동 가능. 현재는 미제공으로 정리 |
| 세금계산서 | 홈택스 공식 API 없음. 서드파티 솔루션 경유 가능하나 비용·계약 필요 |
| 나이스페이 | 나이스페이먼츠 공식 API 출시 대기 |
| 쿠팡 | 쿠팡 WING 정산 API 공개 여부 모니터링 |

여신금융협회는 매출 채널이 아니라 검증용 데이터라 자동화 대상에서 제외한다.

## 더클린커피 매출보고 프로젝트에 반영할 규칙

1. 스마트스토어는 당분간 운영DB PlayAuto 기준을 유지하되, 개발팀 운영DB의 네이버 스마트스토어 API 동기화 테이블과 site 분리 가능성을 2026-05-27에 다시 확인한다.
2. 더클린커피 네이버 통합매니저 권한이 열리기 전까지, Codex/VM Cloud의 네이버 커머스API 직접 수집은 primary가 아니다.
3. 쿠팡은 이 Notion 기준으로도 엑셀 정산이 남는 채널이다. 따라서 쿠팡 API 주문·매출인식 값과 회계 정산표를 계속 대조한다.
4. 네이버페이 정산 내역은 엑셀 source로 분리한다. NPay 실제 결제완료 매출과 정산 내역을 같은 지표로 섞지 않는다.
5. Slack 보고서에는 채널별 기준일을 표시한다. 예: 자사몰은 결제완료일, 스마트스토어는 PlayAuto/정산기준일 후보, 쿠팡은 revenue-history 또는 정산표 기준.
