# 공동구매 운영 체크리스트

작성 시각: 2026-04-22 00:45 KST
문서 성격: 운영팀용 공구 개설 절차 체크리스트
원본 출처: [[../../공동구매내역#Phase2-Sprint4]]에서 분리
적용 사이트: biocom (coffee/aibio은 마스터 파일 생성 후 확장)

## 10초 요약

공동구매를 앞으로 **엑셀 없이** 자동 식별하려면 공구 오픈 **전에** 전용 쿠폰·UTM·랜딩을 시스템에 심어두고 `data/coop_campaigns.biocom.json` 마스터에 레코드를 등록해야 한다. 이 체크리스트는 공구 운영자가 매번 놓치지 말아야 할 7단계를 한 페이지로 정리한 것이다.

## 한눈에 보기 (고등학생 비유)

학교 동아리가 공구를 할 때 "우리 동아리 쿠폰", "우리 동아리 링크", "우리 동아리 등록장"을 미리 만들어 두면, 나중에 회계부가 영수증만 봐도 어느 동아리 매출인지 바로 알 수 있다. 이걸 안 하면 지난 분기 아랑 4차처럼 회사 엑셀 없이는 누가 샀는지 확인 못 하는 상태가 된다.

## 왜 필요한가

2026 Q1 아랑 4차는 326건/81.9M 공구였지만, 당시 파트너 전용 쿠폰·UTM·랜딩이 하나도 없어 로컬/운영 DB/GA4/Meta 어느 데이터로도 자동 식별되지 않았다. 회사 엑셀에 기록이 없었다면 이 매출은 영영 일반 주문으로 섞였을 것이다. 다음 공구부터는 이 상황이 반복되지 않도록 체크리스트를 적용한다.

## 새 공구 개설 체크리스트 (7단계)

운영팀이 공구 시작 **전에** 반드시 수행해야 하는 단계다.

### 1. 파트너 전용 쿠폰 코드 생성
- 위치: 아임웹 관리자 → 쿠폰 관리
- 필수 필드: master 이름을 `[파트너명] ... 공동구매 할인` 양식으로 통일
- 예: `[송율공구 5차] 지연성알러지검사 10% 공동구매 할인`
- 주의: 일반 할인 쿠폰 재사용 금지. 공구별 새 코드를 발급한다.

### 2. 쿠폰 classification 태깅
- 위치: 내부 DB `imweb_coupon_masters.classification`
- 필수 값: `classification='coop'`로 즉시 태깅
- 방법: 운영 개발자가 수동 UPDATE 또는 내부 endpoint 사용
- 주의: 태깅 누락 시 쿠폰 쓴 주문이 자동 분류에서 빠짐

### 3. 전용 UTM 세트 발급
- 파트너에게 배포할 링크에 `utm_source`, `utm_medium`, `utm_campaign` 세트 부여
- 예: `utm_source=meta_biocom_songyuul08&utm_medium=meta_biocom_songyuul08&utm_campaign=meta_biocom_songyuul08`
- 동시 등록: `data/meta_campaign_aliases.biocom.json`에도 해당 alias 넣기
- 주의: 파트너 여러 명이면 UTM alias도 여러 개 발급

### 4. 전용 랜딩 경로 (권장)
- 가능하면 `/{partner}{round}` 형태 전용 랜딩 생성
- 예: `/arang4`, `/songyuul08`
- 어려우면 상품 상세 URL 고정 사용 (상품명 접두사 `[파트너명]`으로 식별 가능)

### 5. `data/coop_campaigns.biocom.json` 레코드 추가
- 위치: 공구 오픈 **당일 오전까지** 반드시 등록
- 필수 필드:
  - `partner`, `round`, `product_families`, `start_date`, `end_date`
  - `coupon_codes` 또는 `utm_*` 중 **최소 하나**
- 선택 필드: `landing_path_pattern`, `order_no_allowlist`(과거 소급용)
- `classification_basis`: 기본 `product_name`, allowlist 있으면 `allowlist` 승격
- 예시 레코드는 원본 문서 [[../../공동구매내역#Phase2-Sprint4]] 참조

### 6. 공구 종료 후 `end_date` 확정
- 시점: 종료 후 24시간 내
- 위치: `data/coop_campaigns.biocom.json`의 해당 레코드 업데이트
- 주의: `end_date`가 열려 있으면 그 이후 주문도 공구로 계산되는 위험

### 7. 주 1회 정기 점검
- 대상: `time+product` 매칭으로 잡혔지만 `coupon/utm/landing` 매칭이 없는 주문
- 위치: `/api/ads/coop-order-summary` 응답의 `by_basis.product_name_pattern` 버킷
- 행동: 운영자가 샘플 확인 후 맞으면 마스터의 `order_no_allowlist`에 주문번호 추가
- 도구: `/ads` 상단 공동구매 v2 카드 Top 10 공구 표

## 과거 데이터 보정 전략

과거 공구는 쿠폰·UTM·랜딩이 없는 경우가 대부분이라 아래 순서로 닫는다.

1. 회사 공유 엑셀(예: `data/공동구매_26년 1분기.xlsx`)에서 파트너별 `start_date`, `end_date`, 달성매출을 읽어 마스터 레코드 생성
2. 엑셀에 주문번호가 없으면 로컬 `imweb_orders + imweb_order_items`에서 `product_families`와 `start_date ~ end_date`로 후보 주문 추출
3. 후보 주문 중 회사 엑셀의 주문수/달성매출과 ±5% 이내로 맞는 세트를 1차 allowlist 후보로 삼는다. 차이가 크면 샘플 확인.
4. 1월 초 주문 헤더 누락 구간은 [[../../공동구매내역#Phase1-Sprint3]] 백필이 먼저다. 백필 전에는 `revenue_source='company_xlsx'`로 남긴다.
5. allowlist 확정 후 `classification_basis='allowlist'`로 고정.

## 역할 구분

- **TJ**: 운영 정의 승인(공구 범위·마스터 필드·태깅 규칙), 회사 공유 엑셀 공유, 이 체크리스트 실제 운영 도입 결정
- **Claude Code**: 마스터 파일 편집, 매칭 스크립트 구현, review_queue 추출 쿼리
- **운영팀**: 위 7단계 실제 수행 (쿠폰 생성, UTM 발급, 랜딩 준비, 레코드 등록)

## 완료 기준

- 새 공구가 오픈될 때마다 마스터에 레코드가 먼저 등록된다.
- 공구 종료 후 24시간 내 `end_date`가 확정된다.
- 주 1회 `product_name_pattern` 버킷이 review 되고 정제된다.
- 분기 말 회고에서 "엑셀 없이도 공구 매출이 자동 분류된다"는 말이 나온다.
