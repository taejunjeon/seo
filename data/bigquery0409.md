# BigQuery 0409

작성일: 2026-04-09

## 한 줄 결론

- **AIBIO와 더클린커피는 새 GCP 프로젝트로 그대로 진행**
- **biocom은 새로 연결하기 전에 기존 `hurdlers-naver-pay` export 상태부터 확인**

## 먼저 팀이 같이 알아야 할 사실

- 우리가 확인하려는 것은 **GA4 원본 이벤트가 BigQuery에 실제로 쌓이고 있는지**다.
- 이 설정은 **GCP 콘솔에서 관리하는 것이 아니라, GA4 Admin > BigQuery 링크**에서 관리한다.
- 현재 상황은 아래와 같다.
  - **AIBIO, 더클린커피**: 새 프로젝트 `project-dadba7dd-0229-4ff6-81c`에 연결됨
  - **biocom**: 이미 `hurdlers-naver-pay` 쪽에 링크가 있어서 새 링크를 추가할 수 없음
- 중요한 점:
  - **biocom을 새 프로젝트로 바로 옮겨도 과거 데이터가 자동으로 따라오지 않는다**
  - 그래서 지금은 **이전 데이터가 살아 있는지, 우리가 읽을 수 있는지**를 먼저 확인해야 한다

## 왜 이 확인이 필요한가

- 지금 광고 성과, 구매 데이터, GA4 숫자가 서로 다르게 보이는 구간이 있다
- 이 차이가
  - 실제 추적 누락인지
  - 날짜 기준 차이인지
  - 특정 페이지/이벤트 문제인지
  를 보려면 **GA4 원본 이벤트를 직접 조회할 수 있어야 한다**
- 특히 biocom은 기존 export가 있는 것으로 보여서,
  - 이미 데이터가 잘 쌓이고 있다면 **새로 만들 필요 없이 조회 권한만 받으면 되고**
  - 데이터가 없거나 죽어 있다면 **그때 새 프로젝트로 옮기는 판단**을 해야 한다

## 개발팀용 슬랙 초안

```text
안녕하세요. BigQuery 관련 현재 결론과 확인 요청 공유드립니다.

현재 결론:
- AIBIO와 더클린커피는 새 프로젝트로 그대로 진행합니다.
- biocom은 새로 연결하기 전에 기존 `hurdlers-naver-pay` export 상태를 먼저 확인해야 합니다.

배경:
- GA4 원본 이벤트를 BigQuery에 쌓는 기능은 GCP 콘솔이 아니라 `GA4 Admin > BigQuery 링크`에서 관리됩니다.
- 현재 AIBIO, 더클린커피는 새 프로젝트 `project-dadba7dd-0229-4ff6-81c`에 연결되어 있습니다.
- biocom은 이미 `hurdlers-naver-pay` 프로젝트에 기존 링크가 있어서 새 링크를 추가할 수 없는 상태입니다.
- 이 상태에서 biocom을 새 프로젝트로 바로 옮기면, 과거 raw 데이터가 자동으로 따라오지 않을 수 있습니다.

그래서 개발팀에서 아래 3가지를 우선 확인 부탁드립니다.

1. biocom 기존 export 상태 확인
- `hurdlers-naver-pay` 프로젝트에 biocom GA4 raw export dataset이 실제로 있는지 확인 부탁드립니다.
- 확인 대상 예시: `analytics_304759974`
- 함께 확인할 내용:
  - dataset 존재 여부
  - 최근 `events_YYYYMMDD` 테이블이 계속 생성되고 있는지
  - 최근 3~7일치 데이터가 정상 적재되고 있는지

2. biocom 조회 권한 확보
- 기존 export가 살아 있다면, 우리 쪽에서 조회할 수 있도록 권한을 부탁드립니다.
- 우선 요청 계정: `biocomkr.sns@gmail.com` 또는 운영용 계정
- 필요 권한:
  - `BigQuery Data Viewer`
  - `BigQuery Job User`

3. 이후 운영 구조 정리
- 나중에 backend에서 BigQuery를 직접 읽게 되면, 서비스 계정을 별도로 분리하는 것이 좋습니다.
- 권장 방향:
  - 읽기 전용 서비스 계정 1개 생성
  - raw dataset 권한과 가공용 mart dataset 권한 분리
  - 사람이 직접 보는 계정과 서버가 읽는 계정도 분리

이번 요청의 목적은 “당장 새로 만들자”가 아니라,
1) 기존 biocom 데이터가 살아 있는지 확인하고
2) 살아 있으면 먼저 읽을 수 있게 만들고
3) 그 결과에 따라 이동 여부를 판단하자는 것입니다.

가능하시면 아래 형식으로 답 주시면 가장 빠릅니다.

- biocom dataset 존재 여부: yes / no
- 최근 events 테이블 적재 여부: yes / no
- 조회 권한 제공 가능 여부: yes / no
- 계속 기존 프로젝트 사용 가능 여부: yes / no
- 새 프로젝트로 이동이 더 맞다면 그 이유: 한 줄
```

현재 확인한 내용으로는, biocom GA4의 BigQuery raw export가 `hurdlers-naver-pay` 프로젝트에 이미 연결된 것으로 보입니다.
이 때문에 저희 쪽 새 프로젝트에 biocom을 추가로 연결하지 못하고 있습니다.

저희가 먼저 확인하고 싶은 것은 “새로 옮길지”가 아니라,
현재 hurdlers 쪽에 연결된 export가 실제로 살아 있는지, 그리고 저희가 읽을 수 있는지입니다.

확인 부탁드리는 항목은 아래 4가지입니다.

- 예시 이름: `analytics_304759974`

2. 최근 raw event 테이블이 계속 생성되고 있는지

3. 저희 운영 계정에 조회 권한을 줄 수 있는지
- 우선 후보 계정: `biocomkr.sns@gmail.com`
- 필요 권한:
  - `BigQuery Data Viewer`
  - `BigQuery Job User`

4. 앞으로도 이 프로젝트를 계속 사용하는 것이 맞는지
- 계속 사용 가능하면 그대로 가고 싶고,
- 아니라면 새 프로젝트로 옮기는 시점과 방법을 같이 정리하고 싶습니다.

중요한 점은, biocom 링크를 새 프로젝트로 바로 옮기면 과거 raw 데이터가 자동으로 따라오지 않을 수 있다는 점입니다.
그래서 삭제/이동 전에 현재 데이터가 살아 있는지부터 먼저 확인하려고 합니다.

- dataset 존재 여부: yes / no

- 조회 권한 제공 가능 여부: yes / no
- 계속 기존 프로젝트 사용 가능 여부: yes / no

## 내부 메모: 이 요청에서 꼭 얻어야 하는 답

### 1. biocom 데이터가 살아 있는가

- 살아 있으면:
  - 일단 **기존 프로젝트 유지 + 조회 권한 확보**가 우선
- 죽어 있으면:
  - **새 프로젝트 relink 검토**로 넘어감

### 2. 우리가 직접 조회할 수 있는가

- 조회가 가능하면:
  - raw event 확인
  - 최근 purchase / transaction_id / source 값 sanity check
  - 이후 backend read 설계로 연결
- 조회가 불가능하면:
  - 권한 문제를 먼저 해결해야 함

### 3. 나중에 서버가 읽을 때 구조를 어떻게 나눌 것인가

- 사람이 직접 조회하는 계정과
- backend가 자동 조회하는 서비스 계정은 분리하는 편이 안전함

## 답변이 오면 바로 다음에 할 일

1. biocom dataset 상태 확인
2. 최근 `events_*` 적재 확인
3. 조회 권한 확보
4. raw sanity query 실행
5. 그 결과 보고
   - 기존 프로젝트 유지
   - 새 프로젝트로 이관
   둘 중 하나 결정
