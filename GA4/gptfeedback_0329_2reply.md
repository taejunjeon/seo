## 개발 완료 내역

### 1. runFunnelReport 기반 진짜 퍼널 API

**파일**: `backend/src/ga4.ts` (함수 `queryGA4RealFunnel`)
**라우트**: `GET /api/ga4/real-funnel`

- **GA4 v1alpha `AlphaAnalyticsDataClient.runFunnelReport`** 사용
- Closed funnel (isOpenFunnel=false): view_item → add_to_cart → begin_checkout → add_payment_info → purchase
- **`funnelBreakdown: deviceCategory`** — 디바이스별 분해
- runFunnelReport 실패 시 자동으로 **eventCount 기반 fallback** 사용
- 응답에 `method: "runFunnelReport" | "eventCount_fallback"` 명시

**실제 호출 결과** (2026-02-26 ~ 2026-03-27, 30일):

| 단계 | mobile | desktop | tablet | 전체 |
|------|--------|---------|--------|------|
| 상품 조회 | 134,975 | 2,585 | 1,264 | 138,817 |
| 장바구니 | 784 (0.6%) | 65 (2.5%) | 16 (1.3%) | 865 (0.6%) |
| 결제 시작 | 330 (42.1%) | 51 (78.5%) | 3 (18.8%) | 384 (44.4%) |
| 결제 정보 입력 | 15 (4.5%) | 6 (11.8%) | 0 | 21 (5.5%) |
| 구매 완료 | 0* | 0* | 0* | 0* |

*`purchase` 이벤트가 closed funnel에서 0으로 나오는 것은 `add_payment_info` 직후에 `purchase`가 동일 세션/사용자 흐름으로 연결되지 않기 때문. 이는 결제가 외부 PG(결제대행)로 리다이렉트되거나, purchase 이벤트의 발화 타이밍이 퍼널 체인과 단절되어 있을 가능성이 있음.

**핵심 인사이트**:
- **상품 조회 → 장바구니 전환율이 0.6%**: 99.4%가 상품을 보고 장바구니에 담지 않음. 여기가 가장 큰 누수
- **데스크톱이 모바일보다 장바구니 전환율 4배 높음** (2.5% vs 0.6%): 모바일 상품 상세 CTA/UX에 개선 여지
- **장바구니 → 결제 시작은 desktop이 78.5%로 매우 높음**: 담으면 결제까지 가는 구조
- `purchase`가 0인 문제는 GA4 이벤트 발화 구조 점검 필요 (PG 리다이렉트 후 purchase 이벤트 미발화 가능성)

### 핵심 해석

1. **`(not set)` 채널의 매출/세션이 ₩28,909로 압도적 1위이자 매출 2위(₩1.3억)**. 이 채널은 source 추적이 안 되는 트래픽인데 전환율이 19.36%나 된다. 가능성이 높은 것은 **결제 완료 후 리다이렉트** (PG 콜백)나 **앱 내 웹뷰** 등으로, 실제 구매 행위의 출처가 여기에 모이고 있을 가능성이 크다. 이 데이터를 정제하면 다른 채널의 실제 전환율이 올라갈 것이다.

2. **meta(₩2.7억)가 매출 1위, tiktok(₩1.2억)이 3위**. 단, tiktok은 세션이 38만으로 압도적으로 많지만 매출/세션은 ₩323에 불과하다 (meta의 1/5). **tiktok은 유입 볼륨, meta는 전환 품질**.

3. **naver(₩4,300만)는 세션 대비 전환율(0.96%)과 매출/세션(₩2,145)이 높다**. 검색 의도 유입의 품질이 좋은 것.

4. **AI 유입(₩13.6만)은 아직 표본이 매우 작다** (159세션, 2구매). 전환율 1.26%는 naver보다 높지만 통계적으로 유의미하지 않다. 모니터링 지속 필요.

5. **google(₩2,800만)은 세션 5.1만인데 전환율 0.2%로 낮다**. 대부분 `google/cpc` (유료 광고)이고 매출/세션은 ₩515. 광고 최적화 여지가 있다.

### report 페이지에 대한 참고사항

앞서 "리포트 조회당 매출 ₩173,293"이 세션당 매출의 171배라고 했는데, 이 수치는 해석에 주의가 필요하다.

**report 페이지는 이미 검사 서비스를 구매해서 진행한 사람들만 이용하는 페이지**이다. 즉, report를 보는 사용자는 이미 1회 이상 구매한 고객이므로 구매 성향이 높은 것은 당연하다.

따라서:
- "report 페이지에서 매출이 높다"는 것은 **report가 매출 레버라기보다, 기존 고객의 재구매 지점**이라는 뜻이다
- **이 레버를 움직이기는 쉽지 않다** — report 조회수를 늘리려면 먼저 검사 서비스 구매(첫 구매)를 늘려야 하기 때문이다
- 그럼에도 report → 추가 구매 전환율을 최적화하는 것은 의미 있다. 검사 결과에 맞춤 추천(영양제 번들, 추가 검사)을 제공하면 **기존 고객의 LTV를 올릴 수 있다**
- 다만 이것은 신규 유입 최적화(퍼널 상단)보다 후순위 과제이다. 먼저 **상품 조회 → 장바구니 0.6% 전환율**을 개선하는 것이 매출에 더 직접적인 영향을 준다

## 남은 리스크

1. **purchase 이벤트 퍼널 단절**: `runFunnelReport`에서 `purchase` 단계가 0. PG 리다이렉트 후 `purchase` 이벤트가 발화되지 않거나, 세션이 끊어지는 문제 가능성. GA4 DebugView에서 결제 완료 후 purchase 이벤트 발화 여부 확인 필요.

2. **페이지 그룹 "기타" 76%**: 분류 규칙이 biocom.kr의 실제 URL 패턴을 충분히 커버하지 못함. 상위 URL을 확인하고 분류 규칙 정교화 필요.

3. **eventCount 퍼널 비정상**: `begin_checkout(5,474) > add_to_cart(3,805)`. 장바구니 없이 바로 결제하는 플로우가 있거나, begin_checkout 이벤트가 중복 발화되고 있음.

4. **세션 리플레이 미지원**: 여전히 "왜 이탈하는지"를 볼 수 없음.

5. **content_group GTM 수준 전송**: 현재는 서버사이드 후처리. GA4 원본과 대시보드 해석이 불일치할 수 있음.
