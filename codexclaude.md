# Codex vs Claude Code — 역할 분담 정리 (2026-02-27)

대상: TJ님의 `/Users/vibetj/coding/seo` 레포 운영 방식(Claude Code=프론트, Codex=백엔드) 점검 및 가이드.

---

## 1) “프론트=Claude Code, 백엔드=Codex”에 동의하는가?

**부분 동의**하오. 다만 “항상”은 아니고, **업무 성격/검증 방식/변경 규모**에 따라 뒤집힐 수 있소.

### 왜 이런 인식이 생기나(현장 감각)

- 프론트엔드는 **UX/문장/레이아웃/컴포넌트 구조**처럼 “사람이 느끼는 품질”이 큰 비중을 차지하고,
  Claude Code가 이런 영역에서 **설명/구조화/컴포넌트 분리**를 잘하는 편이라는 평가가 많소.
- 백엔드는 **정합성(타입/에러처리)/성능/보안/테스트/인프라**처럼 “기계적으로 검증 가능한 품질”의 비중이 커서,
  Codex가 **명확한 입출력, 리팩터링, 테스트 통과, 로그/레이트리밋/캐시 같은 운영 요소**를 끝까지 밀어붙이기 좋소.

### 이 레포 기준 “증거(작업 성격)”로 보면

- 프론트(Claude Code)가 강점 발휘한 작업 유형:
  - `page.tsx` 탭 컴포넌트 분리, DataTable 적용, KPI 카드/차트/UX 정리(상태/표현/스타일 연동)
- 백엔드(Codex)가 강점 발휘한 작업 유형:
  - `server.ts` 라우터 분리, rate limit/logging/circuit breaker/Redis 캐시 같은 인프라성 작업
  - 신규 API 설계/표준화(메타, 캐시, 에러 처리), 날짜/기간 계산 로직 + 테스트

즉, “프론트=Claude, 백엔드=Codex”는 이 프로젝트에서도 **합리적인 기본값**이오.

## 2) next1.1.md의 “page.tsx 상태 정리”는 프론트/백엔드 중 어디에 가까운가?

**프론트엔드에 더 가깝소.**

`next1.1.md`의 Sprint 7은 “UI 컴포넌트는 분리됐는데 상태는 page.tsx에 82개 useState로 몰려 있음 → props 드릴링 과다” 문제를 푸는 작업이오.

이 작업은 본질적으로:
- React state/props/effect/useMemo 의존성,
- 탭 단위 상태 캡슐화(커스텀 훅),
- 화면 동작을 유지하면서 구조만 바꾸는 리팩터링,
에 가깝기 때문에 프론트 역량이 핵심이오.

### 누가 더 잘 할 수 있나?

- **Claude Code가 “주도”하는 게 유리**하오.
  - 이유: 탭 UI 동작을 깨지 않으면서 상태를 재배치하는 과정에서 “사용자 경험” 관점의 판단이 자주 필요함
- **Codex는 “보조/리뷰/검증” 역할이 강점**이오.
  - 예: 상태 훅 분리 후 타입 깨짐/런타임 에러/빌드 실패 잡기, API 계약 변화 필요 시 백엔드 조정, 회귀 테스트 보강

## 3) Codex와 Claude Code의 차이(업무 관점)

### A. 강점이 잘 드러나는 작업 유형

**Claude Code가 유리한 편**
- UI 컴포넌트 설계/분리(Props 디자인 포함)
- CSS/디자인 토큰/미세 UX(문구/레이아웃/상태 표현)
- React 상태/훅 리팩터링(“읽기 쉬운 구조”를 만드는 작업)
- 화면 요구사항을 “기획→구현”으로 번역하는 문서화/설명

**Codex가 유리한 편**
- API 설계/표준화(파라미터/응답/에러/메타)
- 캐시/레이트리밋/서킷브레이커/로그/크론/운영 준비도
- 날짜/기간/통계 계산(엣지 케이스, 윤년/월말) + 단위 테스트
- 보안 취약점 조치(`npm audit`, 의존성 업데이트)와 검증 루틴

### B. 동일 작업에서도 접근 방식이 다른 지점

- Claude Code: “가독성/사용자 관점” 최적화에 강함  
  (예: 상태 이름/컴포넌트 책임 분리, 화면 문구/empty state 설계)
- Codex: “정합성/테스트/실행 가능성” 최적화에 강함  
  (예: 타입/런타임 오류 방지, 테스트 추가, 캐시 키/TTL/에러 처리 같은 운영 요소)

## 4) page.tsx 상태 정리(Sprint 7)를 ‘잘’ 하는 방식(권장 흐름)

> next1.1.md의 방향(커스텀 훅 추출)을 그대로 따르되, “회귀 방지”에 초점을 둠.

1) **상태를 탭 도메인별로 묶기**
- 예: `useKeywordState`, `useColumnState`, `useCwvState`, `useBehaviorState` …
- “탭 UI 컴포넌트”가 필요한 상태/로더/핸들러만 받도록 최소화

2) **서로 다른 탭에서 공유하는 상태만 page.tsx에 남기기**
- 예: `activeTab`, “진단 URL 이동” 같은 전역 네비게이션 성격

3) **부수효과(useEffect)와 데이터 로딩을 훅 내부로 이동**
- 훅이 `loadX()`와 `state`를 함께 제공
- `AbortController`/inflight 처리로 중복 요청 최소화(가능하면)

4) **회귀 검증**
- 최소: `npm run lint`, `npm run build`
- 가능하면: 기존에 사용하던 수동 시나리오(탭 이동, 기간 변경, refresh 등) 체크리스트를 문서로 남김

  - Codex가 **빌드/타입/회귀 테스트 관점에서 2차 방어선**을 맡는 협업이 가장 안전하오.
