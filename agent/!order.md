조사 핵심

1) 토스페이먼츠 MCP는 뭐냐
토스페이먼츠 공식 가이드에 LLM/MCP 연동 문서가 실제로 있습니다.

확인한 내용:
- MCP 설정 예시 패키지: @tosspayments/integration-guide-mcp@latest
- MCP 도구 목록:
  - get-v2-documents
  - get-v1-documents
  - document-by-id

즉, 이 MCP는 토스페이먼츠 문서를 AI가 잘 읽고 코드/연동을 더 정확히 하도록 도와주는 용도입니다.

중요:  
제가 확인한 공식 문서 기준으로는, 이 MCP가 결제 실행 자체를 위한 트랜잭션 도구는 아닙니다.  
그래서 “토스페이먼츠 MCP만 붙이면 텔레그램 주문 결제가 된다”는 구조는 아닙니다.

2) 토스페이먼츠 쪽에서 실제로 쓸 만한 기능
공식 가이드에서 확인한 건 다음입니다.

A. 링크페이
공식 문서상:
- no-code로 결제 링크 생성 가능
- 문자/SNS로 공유 가능
- 필요 시 API 연동 가능

이건 텔레그램 에이전트 주문에 아주 잘 맞습니다.  
에이전트가 주문 내용을 정리한 뒤, 결제 링크를 보내고 고객이 결제하게 만들면 됩니다.

B. 자동결제(빌링)
공식 문서상:
- 정기배송/구독형 서비스용
- 리스크 검토 및 추가 계약 후 사용 가능
- 카드/계좌 자동결제 지원
- API 방식은 카드 중심

영양제, 커피는 반복구매가 많아서  
성공하면 이게 성장 레버리지가 될 가능성이 큽니다.

C. 웹훅
공식 문서상 이벤트 예:
- PAYMENT_STATUS_CHANGED
- CANCEL_STATUS_CHANGED
- BILLING_DELETED
- ORDER_PAYMENT_STATUS_CHANGED

즉 결제 성공/실패/취소를 실시간으로 받아서
- 주문 확정
- 알림 발송
- CRM 반영
- 미결제 리마인드
같은 걸 자동화할 수 있습니다.

3) 아임웹 공개 API의 한계
아임웹 공개 OpenAPI도 확인했습니다.

확인한 내용:
- 상품 관련 API: 꽤 있음
- 주문 관련 API: 주문 조회/배송/송장/취소/반품/교환 처리 중심
- 결제 관련 공개 엔드포인트: 제가 확인한 스펙상  
  PATCH /payments/{orderNo}/bank-transfer/confirm
  정도가 핵심이었습니다.

즉, 공개 스펙 기준으로는 아임웹 API만으로  
“에이전트가 신규 카드 주문을 생성하고 체크아웃을 완전히 대체”하는 구조는 쉽지 않아 보입니다.

이 말은 곧:

- 아임웹은 유지 가능
- 하지만 대화형 주문+결제의 핵심 플로우는 별도 서버/토스 연동으로 빼는 게 현실적이라는 뜻입니다.

가장 현실적인 MVP 설계

MVP 플로우
고객이 텔레그램에서:
- “지난번 커피 다시 주문”
- “오메가3 한 달치 보내줘”
- “카페인 적은 커피 추천해줘”

라고 말하면 에이전트가:

1. 고객 식별
2. 주문 이력/선호 확인
3. 상품 추천 or 재주문 제안
4. 수량/배송지 확인
5. 토스 결제 링크 발송
6. 고객이 링크 눌러 결제
7. 웹훅 수신 후 주문 확정
8. 텔레그램으로 “주문 완료” 안내

이 구조면:
- 보안/결제 승인 책임은 PG UI가 담당
- 에이전트는 대화와 주문 오케스트레이션 담당
- 구현 난이도 대비 효과가 큼
