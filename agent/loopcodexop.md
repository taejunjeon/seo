## 10초 결론

TJ님이 지금처럼 직접 지시하고, Codex가 탐색 -> 구현 -> 검증 -> 보고하는 방식은 현재 단계에서 꽤 효율적이다. 특히 AIBIO/GTM/Meta/원장/유입 분석처럼 운영 맥락이 계속 바뀌는 일은 TJ님의 실시간 판단이 품질을 많이 올린다.

다만 이 방식은 TJ님이 계속 병목이 된다. 같은 설명을 반복해야 하고, 결정 이유와 검증 증거가 작업마다 흩어진다. 그래서 loopagent 문서의 방향은 도움이 된다. 단, 전체를 한 번에 도입하면 오히려 느려지고 복잡해진다.

내 판단은 이렇다. **전면 도입은 하지 말고, `작은 운영 루프`만 2주 실험한다.** 실험 대상은 프로덕션 코드를 마음대로 고치는 에이전트가 아니라 `계획 문서`, `검증 증거`, `incident 정리`, `재검증 task`를 자동으로 남기는 운영 방식이어야 한다.

## 현재 방식 평가

현재 방식은 아래 구조다.

```text
TJ님이 맥락/우선순위/운영 화면을 제공
        ↓
Codex가 repo와 문서를 읽음
        ↓
필요한 코드/문서 변경
        ↓
로컬 검증 또는 운영 화면 확인
        ↓
TJ님이 다음 판단 제공
```

이 방식의 장점은 크다.

| 항목 | 평가 |
|---|---|
| 속도 | 단기 속도는 좋다. 설명이 바로 들어오고, 내가 바로 실행할 수 있다. |
| 품질 | 운영 맥락이 정확히 들어오므로 엉뚱한 방향으로 덜 간다. |
| 위험 관리 | GTM, Meta, 운영 데이터 같은 위험 작업에서 TJ님이 자연스럽게 승인 게이트 역할을 한다. |
| 우선순위 | 지금 중요한 일과 나중 일을 TJ님이 바로 정리해준다. |

하지만 약점도 분명하다.

| 문제 | 실제 영향 |
|---|---|
| TJ님 피로 | 계속 화면 캡처, 판단, 우선순위 설명을 해야 한다. |
| 반복 설명 | 같은 기준을 매번 다시 설명하게 된다. |
| 증거 분산 | 네트워크 payload, 원장 row, GTM 결과, 문서 판단이 한 곳에 남지 않는다. |
| 재검증 누락 | "며칠 뒤 다시 봐야 하는 일"이 자동 task로 잘 남지 않는다. |
| 작업 간 학습 부족 | 이전 incident의 승인/거절 이유가 다음 판단에 구조적으로 재사용되지 않는다. |

그래서 현재 방식은 **초기 개발과 모호한 운영 판단에는 좋지만, 반복 운영에는 점점 TJ님이 병목**이 된다.

## loopagent 문서 평가

### loopagent1

loopagent1의 방향은 맞다.

핵심은 `깊은 자율 에이전트`가 아니라 `얕은 재귀개선 루프`다. 이 프로젝트에는 이 판단이 맞다. 광고, CAPI, 원장, CRM, 결제, 테스트 제외 기준이 얽혀 있어서 AI가 독자적으로 진실을 판단하면 위험하다.

좋은 점:
- `Truth Loop`, `Skill Loop`, `Guarded Patch Loop`로 나눈 점
- 숫자는 evaluator가 판단하고 AI는 설명/증거/액션을 맡긴 점
- 1단 fan-out만 허용한 점
- GTM/Meta/DB/운영 데이터는 승인 뒤에 둔 점

주의할 점:
- "재귀개선"이라는 말이 커 보인다. 실제로는 `반복 작업 기록을 남기고 다음 작업 지시를 덜 하게 만드는 운영 습관`부터 시작해야 한다.
- 외부 툴 패턴을 너무 빨리 흡수하면 지금 당장 필요한 CAPI/freshness/원장 정합성보다 도구 세팅이 앞설 수 있다.

### loopagent2

loopagent2는 설계가 좋지만 지금 전부 구현하기에는 크다.

좋은 점:
- Truth Evaluator와 AI Agent를 분리했다.
- 수정 허용 범위와 금지 범위가 명확하다.
- `Incident Precision`, `False Positive Rate`, `Median Triage Time` 같은 평가 지표가 있다.
- `PLANS.md`를 로드맵이 아니라 살아있는 실행 문서로 본 점이 좋다.

위험한 점:
- 디렉터리와 역할이 한 번에 너무 많이 생긴다.
- `AGENTS.md`, `CLAUDE.md`, `PLANS.md`, `.codex`, `.claude`, `agent/programs`, `agent/evals`, `agent/evidence`, `agent/decisions`, `agent/proposals`를 한 번에 만들면 관리 비용이 먼저 온다.
- 아직 `Verification Harness`가 없어서, 루프를 먼저 만들면 "그럴듯한 보고서"만 늘 수 있다.
- 에이전트가 많아지면 내가 실제 구현보다 조정과 정리에 시간을 더 쓸 수 있다.

따라서 loopagent2는 **최종 그림으로는 유효하지만, 지금은 20%만 가져와야 한다.**

## 속도가 빨라질까

단기에는 느려질 가능성이 높다.

이유는 단순하다. 지금은 TJ님이 바로 지시하고 내가 바로 수정한다. loop 방식은 작업 전 계획, 증거 형식, 평가 기준, 재검증 task를 남겨야 한다. 그래서 첫 1~2주는 순수 개발 속도가 느려질 수 있다.

하지만 반복 작업에서는 빨라질 수 있다.

| 작업 유형 | 현재 방식 | loop 방식 예상 |
|---|---|---|
| 새로운 문제 탐색 | 현재 방식이 빠름 | 오히려 느릴 수 있음 |
| 운영 화면 캡처 기반 판단 | 현재 방식이 빠름 | 증거 템플릿이 생기면 비슷하거나 조금 빠름 |
| 원장/freshness 반복 점검 | 매번 확인 필요 | 자동 evidence가 있으면 빨라짐 |
| CAPI/Meta/GTM 재검증 | 매번 맥락 설명 필요 | runbook과 evidence가 있으면 빨라짐 |
| CRM/실험 반복 운영 | 현재는 TJ님 부담 큼 | loop 방식이 유리 |

정리하면 **기능 개발 속도는 바로 빨라지지 않는다. 운영 반복 속도는 빨라질 가능성이 높다.**

## 퀄리티가 좋아질까

조건부로 좋아진다.

퀄리티가 좋아지는 조건:
- 같은 입력이면 같은 결과가 나오는 evaluator가 있다.
- 변경 전후 검증 명령이 있다.
- incident마다 증거가 남는다.
- 사람이 승인/거절한 이유가 저장된다.
- AI가 숫자를 판단하지 않고 설명만 한다.

이 조건이 없으면 퀄리티가 좋아지지 않는다. 오히려 문서와 에이전트가 늘어나면서 `그럴듯한데 틀린 결론`이 많아질 수 있다.

그래서 순서는 아래가 맞다.

```text
검증 기준 고정
    ↓
증거 저장
    ↓
작업 계획/결정 기록
    ↓
작은 반복 루프
    ↓
에이전트 분리
```

에이전트 분리를 먼저 하면 안 된다.

## TJ님이 덜 고생할까

바로 덜 고생하지는 않는다.

초기에는 오히려 TJ님이 아래를 승인해야 한다.
- 어떤 일을 자동으로 맡겨도 되는지
- 어떤 숫자를 정본으로 볼지
- 어떤 incident를 무시해도 되는지
- 어떤 작업은 사람 승인 없이는 금지할지

하지만 이 기준이 한 번 문서와 evaluator로 고정되면 TJ님 부담은 줄어든다.

줄어드는 일:
- 같은 설명 반복
- "이거 테스트야 실고객이야?" 같은 반복 판별
- "어디까지 확인됐지?" 확인
- "나중에 다시 봐야 하는 일" 기억
- 운영 화면/원장/네트워크 증거를 매번 따로 모으는 일

안 줄어드는 일:
- 사업 기준 승인
- 예산/광고/DB/개인정보/배포 승인
- 실제 운영 계정에서만 보이는 화면 확인
- 신규 전략 판단

따라서 loop 방식의 목표는 **TJ님을 빼는 것**이 아니다. 목표는 **TJ님이 반복 설명 대신 승인과 판단에만 집중하게 하는 것**이다.

## 더 나빠질 가능성

가능성이 있다. 특히 아래 경우에는 더 나빠진다.

| 위험 | 설명 |
|---|---|
| 문서 과잉 | 파일은 많아지는데 실제 검증은 그대로일 수 있다. |
| 에이전트 조정 비용 | 여러 에이전트 결과를 합치느라 실제 구현이 느려질 수 있다. |
| 중요도 왜곡 | AI가 쉬운 문서 정리와 UI 문구만 계속 만질 수 있다. |
| 가짜 확신 | evidence 없이 자연어 보고서만 좋아질 수 있다. |
| 평가기 부재 | 개선 여부를 숫자로 못 보면 루프가 자기합리화가 된다. |
| live 위험 | GTM/Meta/DB/운영 데이터까지 자동화하면 사고 반경이 커진다. |

그래서 지금 당장은 `자동 개발 에이전트`가 아니라 `작업 운영 방식`만 바꾸는 게 맞다.

## 지금 당장 도입할 최소 루프

전면 도입 대신 아래 4개만 한다.

### 1. 작업마다 작은 실행 기록

복잡한 작업만 `plans/*.md`에 남긴다.

적용 기준:
- 2시간 이상 걸릴 작업
- 백엔드/프론트/운영 화면이 같이 걸린 작업
- GTM/Meta/CAPI/DB/CRM처럼 재검증이 필요한 작업
- TJ님 승인 포인트가 있는 작업

짧은 문구 수정이나 단일 파일 버그는 계획 문서 없이 바로 처리한다.

### 2. evidence-first 보고

중요 작업은 결과 보고에 항상 아래 4개를 붙인다.

| 항목 | 예시 |
|---|---|
| 무엇을 바꿨나 | 파일, API, 태그, 문서 |
| 무엇으로 검증했나 | `curl`, `tsc`, Playwright, 운영 화면, 원장 row |
| 무엇은 아직 미확인인가 | Meta 이벤트 지연, 운영 계정 화면, 실제 입금 후 confirmed |
| 다음 재검증 시점 | 24시간 뒤, 다음 폼 제출 뒤, 실제 주문 뒤 |

### 3. 금지 범위 고정

자동화 금지:
- GTM/Meta 게시
- DB 스키마 변경
- 운영 데이터 수정
- 프로덕션 env 변경
- 광고 업로드
- 개인정보 수집 확대

이건 TJ님 명시 승인 전에는 건드리지 않는다.

### 4. 재검증 task 만들기

완료 보고 끝에 "다음에 다시 봐야 할 것"을 남긴다.

예:
- AIBIO Meta Lead는 다음 비테스트 폼 제출 후 Events Manager에서 재확인
- coffee 가상계좌는 실제 입금 후 pending -> confirmed -> CAPI 1회 확인
- acquisition 분석은 다음 실고객 전환 5건 쌓이면 test 제외율 재확인

이 4개만 해도 현재 방식보다 TJ님 반복 부담이 줄어든다.

## 아직 하지 말 것

지금은 아래를 하지 않는 게 맞다.

| 하지 말 것 | 이유 |
|---|---|
| `.codex/agents` 대량 생성 | 실제 병렬 에이전트 운영 전에 평가기가 없다. |
| `.claude/agents` 대량 생성 | Claude Code 운영이 실제로 반복되기 전에는 유지비가 크다. |

| 자동 PR/자동 merge | 현재는 운영 맥락이 중요하고 live 위험이 있다. |
| recursive fan-out | 조정 비용이 개발 시간을 먹는다. |
| 프로덕션 write-back 자동화 | 승인/감사/롤백 체계 전에는 위험하다. |

## 비교 실험 방법

이 방법이 현재 수동 지시보다 나은지는 감으로 판단하지 않는다. 2주 동안 일부 작업에만 적용해서 비교한다.

### 실험군

loop 방식으로 처리할 작업:
- `Revenue Integrity Agent` 데이터 계약
- AIBIO/coffee/biocom freshness 점검
- CAPI/Meta/GTM 검증 evidence 정리
- acquisition 분석의 test/debug 제외 검증

### 대조군

현재 방식으로 처리할 작업:
- 긴급 버그
- 신규 전략 판단
- 운영 계정 화면 확인이 필요한 단발 이슈
- TJ님 선호/사업 판단이 큰 UI 결정

### 측정 지표

| 지표 | 의미 | 좋아졌다고 보는 기준 |
|---|---|---|
| 요청 -> 검증 완료 시간 | 실제 끝나기까지 시간 | 20% 이상 단축 |
| TJ님 개입 횟수 | 중간에 TJ님에게 다시 묻는 횟수 | 30% 이상 감소 |
| 첫 결과 채택률 | 수정 없이 바로 쓸 수 있었는지 | 70% 이상 |
| 재작업 횟수 | 방향 오류/누락으로 다시 한 횟수 | 감소 |
| evidence completeness | 검증 증거가 충분히 남았는지 | 90% 이상 |
| 중요 작업 비율 | 실제 사업/운영 병목에 닿았는지 | 80% 이상 |
| false confidence | 근거 없이 확정처럼 말한 사례 | 0건 |

### 도입 판정

도입한다:
- TJ님 개입이 줄고
- 검증 증거가 더 잘 남고
- 중요한 일이 밀리지 않고
- 작업 완료 시간이 비슷하거나 줄어든다.

축소한다:
- 문서만 늘고
- 실제 구현이 느려지고
- 내가 엉뚱한 주변 작업을 하고
- TJ님이 오히려 리뷰할 게 많아진다.

중단한다:
- 운영 위험 작업을 잘못 제안하거나
- evidence 없이 확정 판단을 하거나
- 중요한 live 이슈 대응 속도가 떨어진다.

## 내가 보는 최종 판단

지금 TJ님이 일일이 지시하는 방식은 나쁘지 않다. 오히려 현재 프로젝트 단계에서는 꽤 좋은 방식이다. 이유는 프로젝트가 아직 `운영 기준선`, `정본 전환`, `CAPI/freshness`, `test/debug 제외`, `CRM 실행 기준`을 고정하는 중이기 때문이다. 이 단계에서는 사람의 판단이 많이 들어가야 한다.

loopagent 방식은 지금 방식을 대체하는 것이 아니라 보조해야 한다.

가장 좋은 형태는 아래다.

```text
TJ님: 사업 기준, 승인, 우선순위, 운영 화면 판단
Codex: 탐색, 구현, 검증, evidence 정리, 재검증 task 생성
Loop: 반복 설명을 줄이는 계획/증거/결정 기록
Claude Code: 운영자 UI/copy/runbook 보조
```

이렇게 하면 품질은 좋아질 가능성이 높다. 속도는 단기에는 조금 느려질 수 있지만, 반복 운영에서는 빨라질 수 있다. TJ님 부담은 즉시 크게 줄지는 않지만, 같은 종류의 질문과 검증을 반복하는 부담은 줄어든다.

반대로 loopagent2 전체를 지금 그대로 구현하면 더 안 좋아질 가능성이 높다. 구조가 너무 빨리 커지고, 내가 중요한 CAPI/원장/CRM 병목보다 도구와 문서 구조를 만질 위험이 있다.
