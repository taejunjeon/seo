좋습니다. 이건 **가장 먼저 만들 가치가 큰 에이전트**입니다.
이유는 간단합니다. **광고비 의사결정은 숫자를 믿을 수 있을 때만 맞게 움직이고**, 지금 시장에서 가장 흔한 실패는 “리포트는 화려한데 정작 숫자가 안 맞는 상태”이기 때문입니다.

# GA4 / 광고 / 내부 주문 정합성 에이전트 PRD v0.1

가칭: **Revenue Integrity Agent**

## 10초 요약

이 제품은 **GA4, 광고 플랫폼, 내부 주문 시스템의 숫자가 왜 안 맞는지 자동으로 찾아서**, 원인과 다음 액션까지 보여주는 운영형 에이전트입니다.
핵심은 “또 하나의 대시보드”가 아니라, **잘못된 입찰·예산 판단을 막는 진단 시스템**으로 만드는 것입니다.

---

## 왜 이 제품이 필요한가

### 원인 3가지

**1) 정의가 서로 다름**
무슨 일인지: GA4의 `purchase`, 광고 전환, 내부 주문의 `paid/confirmed/refunded`가 서로 다른 뜻인데 한 줄 숫자로 비교합니다.
왜 문제인지: 사과와 배를 비교하는 셈이라, 버그가 없어도 숫자가 어긋나 보입니다.
결과 영향: 팀마다 “내 숫자가 맞다” 싸움만 하고, 예산 판단이 흔들립니다.

**2) 처리 시간과 집계 방식이 다름**
무슨 일인지: GA4, BigQuery, 광고 플랫폼, 내부 주문 DB는 들어오는 속도와 마감 시간이 다릅니다.
왜 문제인지: 같은 “어제 매출”처럼 보여도 실제로는 아직 덜 들어왔거나 나중에 보정됩니다.
결과 영향: 오전에 본 숫자와 오후 숫자가 달라지고, 같은 날 기준 판단이 틀어집니다.

**3) 추적·업로드가 중간에 깨짐**
무슨 일인지: 이벤트 중복, 누락, 클릭 ID 누락, 오프라인 전환 업로드 실패가 섞입니다.
왜 문제인지: 문제는 실제로는 기술 이슈인데 마케팅 성과 문제처럼 보입니다.
결과 영향: 잘못된 캠페인을 살리고, 잘 되는 캠페인을 끄는 일이 생깁니다.

---

## 설계 전제: 팩트와 추론

### 팩트 1

공식 문서 기준으로 GA4 표준 속성은 intraday 데이터가 보통 **2-6시간**, 전체 처리에는 **24-48시간**이 걸릴 수 있습니다. 즉, 같은 날 또는 T+1 초반 숫자는 바뀔 수 있습니다. ([Google 도움말](https://support.google.com/analytics/answer/11198161?hl=en "https://support.google.com/analytics/answer/11198161?hl=en"))

### 추론 1

그래서 제품은 숫자를 한 덩어리로 보여주면 안 됩니다.
반드시 **실시간 추정 / 잠정 / 확정** 3단계 상태를 나눠 보여줘야 합니다.

---

### 팩트 2

GA4는 BigQuery로 **원시 이벤트(raw events)** 를 내보낼 수 있고, 일별 export와 streaming export가 있습니다. 다만 streaming은 현재일 데이터를 빠르게 주지만 **새 사용자/새 세션의 트래픽 소스 데이터가 빠질 수 있고**, 일별 테이블도 늦게 들어온 이벤트 때문에 **최대 2일 이상 갱신**될 수 있습니다. 표준 속성의 일별 BigQuery export는 **1일 100만 이벤트 제한**도 있습니다. ([Google 도움말](https://support.google.com/analytics/answer/9358801?hl=en "https://support.google.com/analytics/answer/9358801?hl=en"))

### 추론 2

그래서 이 제품의 숫자 진단 엔진은

- **당일 구조적 이상 탐지**에는 streaming/aggregate를 쓰고

- **정밀 원인 분석**에는 BigQuery 이벤트 레벨 조인을 써야 합니다.
    또, 대용량 고객에서는 Data API만으로 버티지 말고 BigQuery를 기본 진단 평면으로 깔아야 합니다.


---

### 팩트 3

GA4와 BigQuery 숫자를 비교할 때는 **보고 ID(Device ID)** 와 **시간대(time zone)** 를 맞춰야 하고, export에서 제외한 데이터 스트림/이벤트가 있으면 차이가 생길 수 있습니다. ([Google 도움말](https://support.google.com/analytics/answer/13578783?hl=en "https://support.google.com/analytics/answer/13578783?hl=en"))

### 추론 3

그래서 제품은 “버그 경고”를 띄우기 전에 먼저
**설정 오류 경고**를 띄워야 합니다.
예: 시간대 불일치, 보고 ID 불일치, export 제외 이벤트, currency 불일치.

---

### 팩트 4

GA4 Data API는 속성 기준 일일/시간당 quota가 있고, 표준 속성의 Core quota는 **일 200,000 토큰, 시간 40,000 토큰**입니다. 요청 복잡도에 따라 소모가 커집니다. ([Google for Developers](https://developers.google.com/analytics/devguides/reporting/data/v1/quotas "https://developers.google.com/analytics/devguides/reporting/data/v1/quotas"))

### 추론 4

그래서 이 제품은

- **대시보드 요약**과 **빠른 점검**에는 Data API

- **백필, 대량 비교, 주문 레벨 증거 추적**에는 BigQuery
    로 역할을 나눠야 합니다.


---

### 팩트 5

Google Ads는 오프라인 전환 업로드를 지원하고, 현재는 **Enhanced Conversions for Leads** 사용을 권장합니다. 이 방식은 클릭 식별자와 사용자 제공 데이터를 함께 써서 매칭 정확도를 높입니다. 또 업로드 데이터 품질을 보는 **offline data diagnostics** 페이지도 제공합니다. 한편 클릭 기준 오프라인 전환은 일반적으로 **90일이 지난 클릭**은 가져올 수 없고, enhanced conversions for leads는 **63일 제한**이 걸릴 수 있습니다. ([Google 도움말](https://support.google.com/google-ads/answer/2998031?hl=en-EN "https://support.google.com/google-ads/answer/2998031?hl=en-EN"))

### 추론 5

그래서 제품은 월말 몰아서 맞추는 구조가 아니라
**매일 또는 최소 주 단위**로 업로드 상태와 누락분을 감시해야 합니다.
그리고 “광고 최적화에 영향 있는 오류”를 별도로 표시해야 합니다.

---

### 팩트 6

GA4 Admin API는 BigQuery 링크, Google Ads 링크, 커스텀 측정항목·차원 같은 설정 데이터를 프로그램으로 다룰 수 있습니다. ([Google for Developers](https://developers.google.com/analytics/devguides/config/admin/v1 "https://developers.google.com/analytics/devguides/config/admin/v1"))

### 추론 6

그래서 이 제품은 단순 숫자 비교만 하지 말고,
**설정 자체를 읽어와서 진단하는 설정 감사 모드**가 있어야 합니다.

# 2. 문제 정의

현재 대부분의 팀은 아래 셋 중 하나에 갇혀 있습니다.

1. **숫자가 안 맞는 걸 알지만 어디서 틀어졌는지 모름**

2. **원인은 알 것 같지만 증거를 찾는 데 반나절 이상 씀**

3. **광고 최적화 신호가 오염됐는데도 방치함**


이 제품이 해결하려는 핵심 문제는 하나입니다.

> **“우리 돈을 쓰는 기준 숫자를 믿을 수 있게 만든다.”**

# 3. 목표와 비목표

## 목표

- GA4 / 광고 / 내부 주문 간 **정본 숫자 차이**를 자동 탐지

- 차이를 **정의 문제 / 설정 문제 / 추적 문제 / 업로드 문제**로 분류

- 원인 후보와 증거를 보여주고 **다음 액션** 제안

- 광고 최적화에 영향 주는 오류를 우선 표시

- 사람이 승인하면 티켓, 체크리스트, 업로드 작업으로 넘김


## 비목표

- 완전 자동 입찰 운영

- 풀스택 마케팅 믹스 모델

- 창의적 광고 소재 추천

- 재무 마감 시스템 대체

- 사람 승인 없는 자동 수정


냉정하게 말하면, **MVP에서 자동 수정까지 욕심내면 망합니다.**
처음은 **읽기 + 진단 + 액션 제안**이 맞습니다.

- 리포트 오류인지, 실제 성과 악화인지 구분하고 싶다


**핵심 베네핏**

- 캠페인 평가 속도 개선

### 페르소나 B. 데이터/그로스 운영 담당

**하는 일**
GA4, GTM, 광고 전환, SQL, 태그 점검

- 같은 이슈가 반복된다


**원하는 것**

- 주문 단위로 matched / unmatched 이유를 보고 싶다

- 어느 룰이 깨졌는지 바로 알고 싶다

- 엔지니어에게 넘길 증거가 자동으로 정리됐으면 좋겠다


**핵심 베네핏**

- 조사 시간 대폭 감소

- 원인 증거 확보

- 반복 이슈 관리 체계화


---

### 페르소나 C. CEO / Head of Growth

**하는 일**
주요 숫자 확인, 예산 배분, 팀 간 기준 정리

**현재 고통**

- 마케팅팀, 데이터팀, 재무팀 숫자가 다르다

- 회의 때마다 “이 숫자 믿어도 되나?”가 먼저 나온다


**원하는 것**

- 한 화면에서 “오늘 숫자는 믿어도 되는지” 알고 싶다

- 심각한 문제만 위로 보고받고 싶다


**핵심 베네핏**

- 의사결정 속도 향상

- 숫자 신뢰 회복

- 팀 간 책임 소재 명확화

# 6. 핵심 개념 정의

이 제품에서 제일 중요한 개념은 **정본 전환(Truth Conversion)** 입니다.

예시:

- `결제완료`

- `입금확정`

- `배송완료`

- `상담확정`

- `최종 confirmed order`


핵심 원칙은 단순합니다.

> 비교를 시작하기 전에,
> 회사마다 **무엇을 “진짜 전환”으로 볼지 먼저 고정**해야 합니다.

없으면 계속 헛도는 이유:

- GA4 `purchase`는 결제 성공일 수 있음

- 내부 주문은 `confirmed`일 수 있음

- 광고 업로드 전환은 `qualified lead`일 수 있음


이 셋이 다르면 숫자는 틀리는 게 정상입니다.

# 9. 주요 기능 요구사항

## 9-1. 데이터 연결 및 설정

- 사용자는 GA4 property, 광고 계정, 주문 데이터 소스를 연결할 수 있어야 한다.

- 사용자는 정본 전환 정의를 선택할 수 있어야 한다.

- 사용자는 시간대, 통화, VAT/배송비/환불 반영 여부를 정할 수 있어야 한다.

- 시스템은 설정 불일치를 먼저 점검해야 한다.


## 9-2. 정합성 검사 엔진

- 날짜별, 채널별, 캠페인별로 주문 수/매출 차이를 계산한다.

- 주문 단위로 matched/unmatched를 판정한다.

- 차이가 허용 범위를 넘으면 incident를 생성한다.

- 데이터 freshness 상태를 반영해 incident를 억제 또는 완화한다.


## 9-3. 원인 진단 엔진

원인 후보는 최소 아래 클래스로 분류한다.

- 정의 불일치

- 시간대 불일치

- currency/세금/배송비 기준 차이

- purchase 이벤트 누락

- purchase 이벤트 중복

- refund/cancel 반영 누락

- click ID / user ID 연결 부족

- 오프라인 전환 업로드 실패/지연

- BigQuery export 누락 또는 설정 이슈

- 광고계정/속성 연결 이슈


## 9-4. 액션 제안 엔진

시스템은 원인별로 아래 액션 중 하나를 제안해야 한다.

- 태그 점검 체크리스트

- GTM 수정 포인트

- 개발 이슈 초안

- 오프라인 업로드 대상 주문 export

- 무시 규칙 추가

- 변환 정의 수정 요청


## 9-5. 검증 및 학습

- 사용자는 incident를 “실제 문제 / 정상 차이 / 무시”로 표시할 수 있어야 한다.

- 시스템은 이 피드백을 바탕으로 다음 incident 우선순위를 조정한다.

- 단, 숫자 계산 규칙 자체를 AI가 마음대로 바꾸면 안 된다.

# 16. MVP에서 특히 조심할 리스크 3가지

### 1) 같은 숫자처럼 보이지만 사실 정의가 다름

무슨 일인지: paid와 confirmed를 섞으면 제품이 계속 가짜 경고를 냅니다.
왜 문제인지: 사용자가 첫 주에 제품을 불신합니다.
결과 영향: “이 제품도 틀린다”로 끝납니다.

### 2) 데이터 freshness를 무시함

무슨 일인지: 아직 잠정치인 값을 확정치처럼 경고합니다.
왜 문제인지: 오전마다 가짜 사고가 납니다.
결과 영향: 운영팀이 알림을 꺼버립니다.

### 3) 자동 수정부터 욕심냄

무슨 일인지: GTM 수정이나 업로드를 자동으로 밀어버립니다.
왜 문제인지: 숫자보다 더 큰 사고를 낼 수 있습니다.
결과 영향: 고객 신뢰를 잃습니다.
