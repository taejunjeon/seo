> **기준일**: 2026-04-04
> **대상 페이지**: `http://localhost:7010/crm`
> **캡처 파일**: `crm/0404/crm_tab{1~4}_*.png`

## 1. 현재 상태 분석

### 캡처에서 확인된 구조

```
헤더: "CRM 관리 허브" + 4개 상태 배지(우측)
  ↓
탭 셀렉터: 후속 관리 | 실험 운영 | 알림톡 발송 (3열)
           결제 추적 (2행 1열 — 레이아웃 불균형)
  ↓
탭별 콘텐츠
```

### 현재 문제점

| # | 문제 | 영향 |
|---|------|------|
| **P1** | **솔루션(사이트) 구분이 전혀 없다** | 바이오컴/더클린커피/AIBIO의 데이터가 한 덩어리로 섞여 있어, "커피 쪽 상담 후속은?" 같은 질문에 즉답 불가 |
| **P2** | **탭이 4개인데 3열 그리드** | 결제 추적이 2행으로 밀려나 시각적 위계가 깨짐 |
| **P3** | **헤더 배지 4개가 운영 주의사항** | 첫 화면에서 "못 하는 것" 목록이 먼저 보임 → 부정적 첫인상 |
| **P4** | **사이트별 KPI가 없다** | 3개 사이트의 회원 수, 매출, 상담 건수가 다른데 합산만 보여줌 |
| **P5** | **알림톡 탭이 "로딩 중..." 상태** | 데이터 없이 빈 화면. 사이트 context 없으면 어떤 템플릿을 쓸지도 판단 불가 |

## 3. 구현 계획

### 변경 범위

| 파일 | 변경 내용 |
|------|----------|
| `frontend/src/app/crm/page.tsx` | 솔루션 필터 state 추가, SiteSelector 컴포넌트, 사이트별 KPI 카드, 탭 그리드 4열 수정 |
| `frontend/src/app/crm/page.module.css` | siteSelector, siteButton, siteButtonActive, siteBadge 스타일 추가, tabSelector 4열 수정 |

### 구현 순서

| # | 작업 | 내용 |
|---|------|------|
| 1 | **솔루션 필터 바 추가** | `site` state (`all` / `biocom` / `thecleancoffee` / `aibio`), URL search param 연동 |
| 2 | **탭 그리드 4열 수정** | `grid-template-columns: repeat(3, ...)` → `repeat(4, ...)` |
| 3 | **사이트별 KPI 카드** | 선택된 사이트의 회원 수, SMS 동의율, Meta 광고비, 마지막 활동 |
| 4 | **헤더 배지 정리** | 4개 운영 주의사항을 접이식 또는 제거 → 사이트 context 정보로 대체 |
| 5 | **하위 탭 데이터 필터링** | 향후 API에 `site` 파라미터 추가 시 연결 가능하도록 구조 준비 |

### 설계 원칙

- 기존 기능을 깨지 않는다. "전체"가 기본값이므로 기존과 동일하게 시작.
- 솔루션 선택은 URL query `?site=biocom&tab=consultation`로 관리 → 링크 공유 가능.
- 사이트별 KPI는 이미 확보된 데이터로 채운다 (consent-stats API, Meta overview API).
- 탭 내 데이터 필터링은 API가 site 파라미터를 지원할 때 점진적으로 연결.

## 7. 핵심 문제 — 사이트별 비즈니스 모델이 다르다 (0404 추가)

> **캡처**: `crm/0404/스크린샷_aibio_mismatch.png`
> AIBIO를 선택해도 "상담 완료 → 미구매 후속" 테이블이 그대로 나옴. 상담 서비스가 없는 사이트에 상담 중심 UI가 나오는 것은 UX 오류.

### 3사이트 비즈니스 모델 비교

| | 바이오컴 | 더클린커피 | AIBIO |
|---|---------|----------|-------|
| **핵심 비즈니스** | 검사키트 → 상담 → 영양제 구매 | 스페셜티 커피 판매 | 바이오해킹 체험(리커버리랩) |
| **상담 서비스** | ✅ 있음 (8,305건) | ❌ 없음 | ❌ 없음 |
| **주요 CRM 시나리오** | 상담 후 미구매 후속, 영양제 재구매 | 재구매 유도, 장바구니 이탈 | 광고 유입 → 방문 전환, 리드 수집 |
| **주요 채널** | 알림톡, 전화 | 알림톡, SMS | Meta 광고, 알림톡 |
| **데이터 규모** | 69,681명 · 상담 8,305건 | 13,236명 · live row 3건 | 100명 · Meta ₩148만/월 |
| **결제 추적** | ✅ Toss live | ✅ Toss live | ⏸ 쇼핑몰 대기 |

### 현재 문제

현재 4개 탭이 **바이오컴의 비즈니스 흐름**(상담 → 후속 → 실험 → 발송 → 결제 추적)에 맞춰져 있다.

| 탭 | 바이오컴 | 더클린커피 | AIBIO |
|----|---------|----------|-------|
| **후속 관리** | ✅ 핵심 (상담 후 미구매 고객) | ❌ 상담이 없으므로 빈 테이블 | ❌ 상담이 없으므로 빈 테이블 |
| **실험 운영** | ✅ 상담 기반 실험 | △ 가능하지만 시나리오 다름 | △ 가능하지만 시나리오 다름 |
| **알림톡 발송** | ✅ 검사 결과 안내 등 | ✅ 재구매 알림 등 | △ 회원 100명으로 제한적 |
| **결제 추적** | ✅ live 가동 중 | ✅ live 가동 중 | ⏸ 쇼핑몰 대기 |

**결론**: 더클린커피와 AIBIO를 선택하면 "후속 관리" 탭이 비어 있거나 무의미한 데이터를 보여줌. 각 사이트에 맞는 "첫 화면"이 달라야 한다.

## 8. 개선 설계 — 사이트별 탭 분기

### 핵심 원칙

1. **사이트를 선택하면, 그 사이트에 맞는 탭만 보여준다.**
2. **각 사이트의 "오늘 할 일"이 다르다.**
3. **공통 기능(실험, 알림톡, 결제 추적)은 유지하되 첫 탭이 달라진다.**

### 사이트별 탭 구성

#### 바이오컴 (상담 중심 — 현재와 동일)

```
[상담 후속]  [실험 운영]  [알림톡 발송]  [결제 추적]
     ↑ 기본 탭
```

- 상담 후속: 기존 그대로 (상담 완료 → 미구매, 부재/변경 → 재연락)
- 바이오컴만의 KPI: 완료 상담 3,947건, 90일 전환율 26.1%, 상담 1건당 가치 ₩57,881

#### 더클린커피 (커머스 중심 — 새로운 구성)

```
[구매 현황]  [재구매 관리]  [알림톡 발송]  [결제 추적]
     ↑ 기본 탭
```

- **구매 현황** (새 탭): 최근 주문 요약, 장바구니 이탈 현황, 베스트셀러
- **재구매 관리** (새 탭): 첫 구매 후 N일 미재구매 고객, 재구매 코호트 요약
- 더클린커피만의 KPI: 회원 13,236명, live row 3건, UTM 추적 활성, 최근 주문 현황

#### AIBIO (광고·리드 중심 — 새로운 구성)

```
[광고 성과]  [리드 관리]  [알림톡 발송]  [결제 추적]
     ↑ 기본 탭
```

- **광고 성과** (새 탭): Meta 노출/클릭/비용/CPC 요약, 일별 추이 차트, 캠페인별 성과 (`/ads` 페이지 데이터 재활용)
- **리드 관리** (새 탭): 유입된 리드 목록, 전환 상태, 방문 쿠폰 발급 현황 (P7-S4 연계)
- AIBIO만의 KPI: 100명, Meta 48만 노출/월, ₩148만/월, 전환율 0.006%

#### 전체 (합산 — 현재와 동일 + 사이트별 비교)

```
[사이트 비교]  [실험 운영]  [알림톡 발송]  [결제 추적]
      ↑ 기본 탭
```

- **사이트 비교** (새 탭): 3사이트의 핵심 지표를 나란히 비교하는 테이블/차트

## 10. 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 새 탭(광고 성과, 구매 현황 등)의 콘텐츠가 아직 없음 | 탭을 누르면 빈 화면 | 1단계에서는 "준비 중" 안내 + 관련 페이지 링크로 대체 |
| 사이트별 API 필터가 아직 없음 | 데이터가 섞여 나올 수 있음 | 프론트에서 가능한 범위로 필터, 백엔드 작업은 3단계 |
| AIBIO 리드 관리 데이터가 아직 없음 | P7-S4 전까지 빈 상태 | "P7에서 구현 예정" 안내 표시 |
| 탭 분기가 복잡해지면 유지보수 부담 | 코드 복잡도 증가 | 사이트별 탭 컴포넌트를 별도 파일로 분리 |
