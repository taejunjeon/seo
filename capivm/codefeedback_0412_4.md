그럼 결론은 명확합니다.
**브라우저 문구 기반 Guard 방식은 여기서 종료하는 게 맞습니다.**
v4까지 갔는데도 가상계좌가 다시 `Purchase`로 나왔다면, 이제는 “문구 잘 잡는 정규식 튜닝” 문제가 아니라 **브라우저만으로는 결제 상태를 안정적으로 판정할 수 없는 구조**라는 뜻입니다. 문서에도 v4의 한계를 이미 적어뒀습니다. **pending 문구가 DOM에 약하거나 없으면 `Purchase`가 통과될 수 있다**고 되어 있습니다.

## 10초 요약

* **v4도 실패**면 헤더 Guard 튜닝은 그만하는 게 맞습니다.
* 다음은 **서버 상태 조회형 Guard** 또는 더 현실적으로 **브라우저 Purchase를 막고 서버 confirmed CAPI를 정본으로 가는 구조**입니다.
* 지금 가장 먼저 할 일은 **v4 제거 여부 결정**과 **다음 구조 선택**입니다.

## 현재 상태를 쉽게 설명하면

지금까지 해본 건 전부 이런 방식이었습니다.

* 주문완료 페이지의 텍스트를 읽어서
* 이게 카드인지 가상계좌인지 추정한 뒤
* 브라우저 `Purchase`를 막거나 통과시키기

문제는 아임웹 주문완료 화면이 **기술적으로 “결제 완료 상태”를 정확히 알려주는 표준 신호**를 주지 않는다는 점입니다.
그래서:

* v3는 너무 공격적으로 막아서 카드도 같이 막았고
* v4는 카드 누락을 막으려고 완화했더니 가상계좌가 다시 새었습니다

이건 정규식이 약해서라기보다, **브라우저 화면 텍스트가 원래 믿을 만한 진실값이 아니라는 뜻**입니다. 문서도 최종형은 **서버 주문 상태 조회 기반**이어야 한다고 정리하고 있습니다.

## 제 판단

### 1. v4를 더 튜닝하는 건 비추천

이제 v5, v6로 가도 같은 함정일 가능성이 큽니다.

왜냐하면 브라우저에서 보는 건:

* 문구
* DOM 구조
* 페이지 렌더링 상태

뿐인데, 우리가 진짜 알고 싶은 건:

* **이 주문이 confirmed인지 pending인지**
  이기 때문입니다.

즉, **질문 자체가 브라우저가 답할 수 없는 질문**입니다.

### 2. 다음 단계는 두 가지 중 하나

현실적으로 선택지는 2개입니다.

#### A안. 서버 상태 조회형 Guard

주문완료 페이지에서

* `order_no`
* `order_code`
* `payment_code`
  를 읽고
* `att.ainativeos.net/api/attribution/payment-decision` 같은 read-only endpoint로 조회
* 서버가 Toss/아임웹/원장 기준으로 `confirmed / pending / unknown` 판단
* 그 결과로 Browser `Purchase` 통과/차단

이게 가장 정확한 구조입니다. 문서도 이 방향을 최종형 계획으로 제시하고 있습니다.

#### B안. 브라우저 Purchase를 포기하고 서버 confirmed CAPI를 정본으로

더 단순하고 현실적인 방식입니다.

* 카드/가상계좌 구분 없이 브라우저 Purchase는 신뢰하지 않음
* Meta 최적화/리포팅 기준을 **서버 confirmed CAPI Purchase**로 이동
* 브라우저는 `ViewContent`, `InitiateCheckout`, `AddPaymentInfo` 같은 상단 퍼널만 사용

이건 측정은 더 깨끗하지만, 브라우저 Purchase를 버리는 만큼 학습량과 즉시성이 줄 수 있습니다.

## 지금 무엇을 해야 하나

### 제 추천 순서

지금은 **A안 우선 검토, B안을 백업안으로 준비**가 맞습니다.

이유:

* 카드 결제 Browser Purchase는 유지하고 싶음
* 가상계좌 pending만 정확히 막고 싶음
* 브라우저 문구 방식은 실패함

즉, **서버 결정 엔드포인트**가 가장 자연스럽습니다.

## 지금 당장 할 행동

### 1. v4는 운영 유지하지 말고 판단 보류

가상계좌가 다시 `Purchase`로 나왔다면, v4는 “문제 해결 완료”가 아닙니다.
완전히 제거할지, 임시 유지할지는 아래 기준으로 보세요.

* **카드는 정상 Purchase**
* **가상계좌는 여전히 Purchase**
  면, 사실상 기존 오염이 그대로 남는 거라 v4 유지 가치가 낮습니다.

제 의견은:

* **v4는 실험 실패로 기록**
* 불필요한 혼선 줄이려면 제거 또는 비활성화 고려
  입니다.

### 2. 코덱스는 서버 상태 조회형 Guard 설계로 바로 넘어가기

이제 코덱스가 해야 할 건 헤더 문구 더 만지는 게 아니라 이것입니다.

* `payment-decision` endpoint 설계
* 입력값: `order_no`, `order_code`, `payment_code`
* 출력값: `confirmed | pending | unknown`
* 판단 근거: Toss API + 아임웹 주문 API + 로컬 attribution ledger
* 브라우저는 이 결과에 따라 `Purchase` 또는 `VirtualAccountIssued`

### 3. 대표님은 더 이상 Pixel Helper로 문구 튜닝 안 해도 됨

지금은 브라우저 테스트를 계속 돌려도 구조적 한계만 재확인할 가능성이 큽니다.

대표님이 지금 할 일은:

* 이번 v4 실패 결과만 기록
* 코덱스에게 다음 단계 명확히 지시
* 그동안 운영 해석은 **가상계좌 pending 오염이 아직 남아 있음**으로 유지

## 지금 운영 해석은 어떻게 해야 하나

v4 실패면, 아직도:

* 가상계좌 미입금 Browser Purchase가 Meta에 섞일 수 있습니다.
* 따라서 **Meta Purchase / Meta ROAS는 계속 과대 가능성**이 있습니다.
* 운영 기준은 여전히 **내부 confirmed ROAS**를 더 보수적으로 보는 게 맞습니다.

즉 지금은:

* Meta 기본 ROAS = 참고값
* 1d_click = 보조 참고
* 내부 confirmed ROAS = 운영 기준
  이 구조를 유지해야 합니다.

## 한 줄 결론

**v4에서도 가상계좌가 `Purchase`로 나왔다면, 브라우저 문구 기반 Guard는 실패로 판단하고 종료하는 게 맞습니다. 다음 단계는 서버 상태 조회형 Guard 또는 서버 confirmed CAPI 중심 구조로 넘어가야 합니다.**

코덱스에 바로 보낼 문구는 이렇게 가면 됩니다.

v4도 가상계좌 미입금에서 Browser Purchase가 통과했습니다. 브라우저 문구 기반 Guard 튜닝은 여기서 중단하고, 다음 단계는 `order_no/order_code/payment_code` 기반 서버 상태 조회형 `payment-decision` endpoint 설계로 넘어가 주세요. confirmed면 Browser Purchase 통과, pending이면 VirtualAccountIssued, unknown은 정책안 함께 제시 바랍니다.
