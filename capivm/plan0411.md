# Google Cloud 임시 배포 계획 - 2026-04-11

## 바로 결론

Google Cloud에 올려도 지금처럼 Claude Code로 빠르게 개발 가능하다.

핵심은 서버를 Google Cloud로 옮기더라도 **소스 수정은 계속 로컬 repo에서 하고, 배포만 명령어/스크립트로 반복 가능하게 만드는 것**이다. 지금처럼 로컬에서 Claude Code로 빠르게 수정하고 `typecheck`, 테스트를 돌린 뒤 Google Cloud 서버에 반영하는 흐름을 유지할 수 있다.

다만 현재 백엔드는 결제/Attribution/CAPI 수집 데이터를 다루므로, 서버에 직접 접속해서 임의 수정하는 방식보다는 **로컬에서 수정 -> Git 또는 rsync/scp 배포 -> 서버 재시작 -> health/API 검증** 순서로 고정하는 것이 안전하다.

## 현재 상태

- 프론트엔드: Next.js, 로컬 포트 `7010`.
- 백엔드: Express + TypeScript Node 서버, 로컬 포트 `7020`.
- 백엔드 실행 구조: `backend/src/server.ts`에서 Express app을 띄우고 `startBackgroundJobs()`를 실행한다.
- 백그라운드 작업: CAPI auto-sync는 30분 주기, Attribution 결제상태 sync는 15분 주기.
- 로컬 영속 데이터: `backend/data/crm.sqlite3`를 사용한다.
- 로컬 로그: `backend/logs/*.jsonl` 형태로 CAPI/알림톡/기타 전송 로그가 쌓인다.
- 운영성 외부 의존: Meta API, Toss API, Imweb API, GA4/GSC, Supabase, 운영 Postgres `DATABASE_URL`, Aligo, ChannelTalk 등이 `.env`로 연결된다.
- 현재 공개 endpoint: `att.ainativeos.net`은 Cloudflare 앞단을 타고 있지만, origin은 아직 TJ님 노트북의 로컬 백엔드에 의존하는 구조로 보인다.

현재 가장 큰 운영 리스크는 코드 자체보다 **노트북 origin 의존성**이다. 노트북 잠자기, 재부팅, 네트워크 끊김, `tsx watch` 프로세스 종료가 발생하면 결제완료 수집, CAPI 전송, 결제상태 동기화가 같이 흔들린다.

## 올리는 범위: CRM 발송 솔루션까지인가, CAPI만인가

결론부터 말하면 **백엔드 서버는 전체를 올리되, 1차 운영 목적은 Attribution/CAPI 안정화로 제한**하는 것이 맞다.

현재 `backend`는 하나의 Express 서버 안에 여러 기능이 같이 들어 있다.

- 결제완료/checkout 식별자 수집.
- Attribution 원장 저장.
- 결제상태 sync.
- Meta CAPI 전송.
- Meta 광고/ROAS 조회.
- 쿠폰/CRM/상담/가격/알림톡/채널톡 관련 API.
- 로컬 SQLite/JSONL 기반 보조 원장과 로그.

따라서 “CAPI만 올린다”는 말은 실제로는 두 가지 방식 중 하나다.

1. 현재 백엔드 전체를 올리고, 외부에서 쓰는 endpoint를 Attribution/CAPI 중심으로만 운영한다.
2. CAPI 관련 코드만 떼어내 별도 미니 서버를 새로 만든다.

현재 단계에서는 1번이 맞다. 이유는 CAPI가 단독 기능이 아니라 결제완료 수집, Attribution 원장, 결제상태 확인, CAPI 로그와 연결되어 있기 때문이다. CAPI만 분리하면 오히려 원장/로그/식별자 흐름을 다시 이어야 해서 데이터 정합성 리스크가 커진다.

1차로 반드시 올려야 하는 범위:

- `/api/attribution/checkout-context`
- `/api/attribution/payment-success`
- `/api/attribution/*` 중 결제상태 sync와 caller coverage 확인 API
- `/api/meta/capi/*`
- `/api/ads/*`, `/api/meta/*` 중 ROAS/광고비 대조에 필요한 조회 API
- `/health`

1차에서 “같이 올라가지만 적극 운영하지 않는” 범위:

- CRM 발송 솔루션 관련 API
- 알림톡/채널톡/상담/가격 관련 API
- 쿠폰 분석 API
- 기타 대시보드 보조 API

이 기능들은 같은 백엔드에 있으므로 서버를 올리면 코드상 같이 배포된다. 다만 1차 목적은 CAPI/Attribution 정합성 안정화이므로, CRM 발송 기능을 새 운영 채널로 확장하거나 자동 발송을 켜는 작업은 별도 승인 후 진행해야 한다.

운영 노출 기준:

- `att.ainativeos.net`에는 결제완료 수집과 CAPI/Attribution 관련 endpoint만 실제로 연결되는 것이 바람직하다.
- 관리자 대시보드용 API는 가능하면 Cloudflare Access, IP 제한, Basic Auth, VPN 중 하나로 보호해야 한다.
- CRM 발송류 endpoint는 오발송 리스크가 있으므로 공개 인터넷에 무방비로 열면 안 된다.

정리하면, 서버 배포 단위는 **backend 전체**이고, 운영 우선순위는 **Attribution/CAPI**다. CRM 발송 솔루션은 “코드는 같이 올라가지만 1차 운영 범위는 아님”으로 본다.

가능하다. 단, 개발과 운영 반영 방식을 분리해야 한다.

권장 흐름:

1. 로컬 `/Users/vibetj/coding/seo`에서 Claude Code로 수정한다.
2. 로컬에서 `npm --prefix backend run typecheck`와 필요한 테스트를 실행한다.

주의할 점:

- 운영 서버에서 직접 파일을 고치면 로컬 repo와 서버 코드가 갈라진다.
- `.env`와 서비스 계정 JSON은 Git에 넣지 말고 Secret Manager 또는 서버 파일로 따로 관리해야 한다.
- 운영 데이터가 있는 `crm.sqlite3`, `logs/*.jsonl`은 배포 때 덮어쓰면 안 된다.

## 배포 선택지

## 선택지 A. Google Compute Engine VM + Cloudflare Tunnel

현재 단계에서 가장 추천한다.

구조:

- Google Compute Engine VM 1대 생성.
- VM에 Node.js, npm, pm2 또는 systemd 설치.
- 현재 `backend` 코드를 거의 그대로 빌드/실행.
- `backend/data/crm.sqlite3`와 `backend/logs`를 VM의 persistent disk에 보관.
- Cloudflare Tunnel 또는 Cloudflare DNS를 VM으로 연결해서 `att.ainativeos.net` 유지.
- 프론트엔드는 당분간 로컬/별도 배포를 유지해도 되고, 필요하면 이후 Cloud Run/Vercel/GCS로 분리.

장점:

- 현재 Express 백엔드를 거의 그대로 올릴 수 있다.
- SQLite와 JSONL 로그를 그대로 유지할 수 있다.
- 15분/30분 `setInterval` 백그라운드 작업이 현재처럼 동작한다.
- 개발팀 인계가 쉽다. “VM + Node 프로세스 + 데이터 파일 위치 + env”로 설명 가능하다.
- Cloudflare는 계속 앞단 도메인/터널 역할을 하면 된다.

단점:

- 서버 패치, PM2/systemd, 디스크 백업, 로그 로테이션을 직접 관리해야 한다.
- 장기적으로는 Cloud SQL/Cloud Run보다 운영 표준성이 낮다.

임시 운영용으로는 이 선택지가 가장 현실적이다.

## 선택지 B. Cloud Run + Cloud SQL/Supabase + Cloud Scheduler

개발팀 인계용 정식 구조로는 이 방향이 더 좋다.

구조:

- 백엔드를 Docker image로 빌드해서 Cloud Run 서비스로 배포.
- 로컬 SQLite 의존을 Cloud SQL Postgres 또는 Supabase/Postgres로 이전.
- JSONL 로그는 Cloud Logging, Cloud Storage, BigQuery, Postgres 테이블 중 하나로 이전.
- CAPI auto-sync와 결제상태 sync는 서버 내부 `setInterval` 대신 Cloud Scheduler가 `/api/.../sync` endpoint를 주기 호출하도록 분리.
- Secret Manager로 Meta/Toss/Imweb/GA4 키를 관리.

장점:

- 서버 관리 부담이 낮다.
- 배포와 롤백이 명확하다.
- Secret Manager, Cloud Logging, Cloud Scheduler와 연결하면 개발팀 인계에 좋다.
- 트래픽 증가 시 확장성이 좋다.

단점:

- 현재 코드를 그대로 올리면 데이터 정합성 문제가 생길 수 있다.
- Cloud Run의 로컬 파일 시스템은 컨테이너 인스턴스 기준이라 영구 저장소로 보면 안 된다.
- Cloud Run 기본 요청 기반 CPU 설정에서는 요청 처리 중이 아닐 때 CPU 할당이 제한될 수 있어, 현재 같은 내부 `setInterval` 잡은 안정성이 떨어진다.
- 따라서 Cloud Run으로 가려면 저장소와 스케줄러 분리가 먼저 필요하다.

정식 운영 구조로는 좋지만, “오늘 노트북 의존 제거” 목적이면 VM보다 작업량이 크다.

## 선택지 C. Cloudflare Workers로 백엔드 이전

현재는 비추천한다.

이유:

- 현재 백엔드는 Express, `better-sqlite3`, 로컬 SQLite, 로컬 JSONL 로그, 백그라운드 주기 작업을 쓴다.
- Cloudflare Workers는 edge/serverless 런타임이라 현재 구조와 맞지 않는다.
- Workers의 파일 시스템은 메모리 기반 virtual file system이고 `/tmp`도 요청 단위 임시 파일 성격이다.
- Workers로 이전하려면 D1, R2, Queues, Cron Triggers 중심으로 재설계가 필요하다.

Cloudflare는 지금처럼 **도메인, Tunnel, 보안 앞단**으로 쓰는 것이 맞다. compute를 Cloudflare Workers로 옮기는 것은 별도 재설계 과제다.

## 백엔드를 현재 그대로 올려도 되는가

VM에는 거의 그대로 올려도 된다.

필수 조치:

- `backend/.env`를 서버에 안전하게 배치한다.
- `backend/data/crm.sqlite3`를 서버 persistent disk로 이동한다.
- `backend/logs`를 서버 persistent disk에 유지한다.
- `npm --prefix backend run build` 후 `npm --prefix backend run start`로 실행한다.
- `pm2` 또는 `systemd`로 프로세스가 죽어도 재시작되게 한다.
- Cloudflare Tunnel 또는 DNS를 서버의 `7020` 백엔드로 연결한다.
- 서버 health endpoint는 `/health` 기준으로 확인한다.

Cloud Run에는 “그대로” 올리면 안 된다.

Cloud Run에서도 HTTP API 자체는 Express 그대로 컨테이너화할 수 있지만, 현재 데이터 정합성 관점에서는 아래가 걸린다.

- `backend/data/crm.sqlite3`: 컨테이너 로컬 파일에 두면 재배포/스케일/인스턴스 교체 시 데이터 지속성이 보장되지 않는다.
- `backend/logs/*.jsonl`: 로컬 파일 로그도 같은 문제가 있다.
- `startBackgroundJobs()`: Cloud Run 기본 요청 기반 CPU에서는 요청이 없을 때 내부 주기 작업이 안정적으로 돈다고 보기 어렵다.
- 여러 인스턴스로 스케일되면 같은 sync가 중복 실행될 수 있다.

따라서 Cloud Run은 다음 수정 후 가는 것이 맞다.

- SQLite 원장을 Postgres 계열로 이전.
- CAPI 로그를 Postgres/Cloud Logging/BigQuery 중 하나로 이전.
- 15분/30분 sync를 Cloud Scheduler 호출 방식으로 변경.
- 중복 실행 방지를 위해 lock을 DB 기반으로 변경.

## Next.js로 백엔드를 바꿀 필요가 있는가

지금은 바꿀 필요 없다.

판단 이유:

- 프론트는 이미 Next.js고, 백엔드는 Express API 서버로 역할이 명확하다.
- 현재 백엔드는 결제완료 수집, Attribution 원장, CAPI 전송, 외부 API 동기화, 주기 작업, 로컬 DB/로그 관리가 핵심이다.
- 이 기능들은 Next.js API Route나 Server Action으로 합치는 것보다 Express 독립 서버로 유지하는 것이 더 단순하고 안전하다.
- 특히 `setInterval` 백그라운드 잡, SQLite, JSONL 로그, 긴 외부 API 호출은 Next.js 서버리스/edge 배포 모델과 잘 맞지 않는다.
- Next.js로 합치면 코드 이동량이 크고, 현재 중요한 ROAS/Attribution 정합성 작업보다 우선순위가 낮다.

Next.js 전환을 검토할 수 있는 경우:

- 백엔드 기능이 단순 BFF 수준으로 줄어든 경우.
- 원장/로그/스케줄러가 전부 외부 DB와 Cloud Scheduler로 분리된 경우.
- 프론트와 API를 하나의 Cloud Run 서비스로 묶어 배포해야 할 강한 이유가 생긴 경우.

현재 결론:

- 백엔드는 Express 유지.
- 프론트는 Next.js 유지.
- 둘을 억지로 합치지 않는다.

## 추천 실행 계획

## 1단계. 오늘 가능한 임시 안정화

목표: 노트북 의존 제거.

작업:

1. Google Compute Engine VM 생성.
2. VM에 Node.js LTS, npm, git, pm2 설치.
3. repo clone 또는 현재 repo를 rsync.
4. `backend/.env`를 서버에 배치.
5. `backend/data/crm.sqlite3`와 `backend/logs`를 서버로 복사.
6. `npm --prefix backend ci` 또는 `cd backend && npm ci`.
7. `npm --prefix backend run build`.
8. `pm2 start dist/server.js --name seo-backend` 또는 systemd service 등록.
9. `curl http://localhost:7020/health`로 서버 내부 확인.
10. Cloudflare Tunnel을 VM에서 실행해 `att.ainativeos.net`을 VM의 `localhost:7020`으로 연결.
11. 외부에서 `https://att.ainativeos.net/health` 확인.
12. 결제완료 테스트 주문 1건으로 `/api/attribution/payment-success` 적재 확인.
13. CAPI sync가 중복 없이 도는지 `/api/meta/capi/log` 확인.

이 단계에서는 코드 변경을 최소화한다.

## 2단계. 빠른 개발/배포 루틴 만들기

목표: Claude Code 수정 속도 유지.

작업:

1. 로컬에서 빌드/테스트 통과.
2. 변경분만 서버에 반영하는 `deploy-backend.sh` 작성.
3. 배포 스크립트에서 다음 순서를 고정.
4. 서버 백업: `crm.sqlite3`, `logs`.
5. 코드 반영.
6. `npm ci` 필요 시 실행.
7. `npm run build`.
8. `pm2 restart seo-backend`.
9. `/health`와 핵심 API smoke test.

이렇게 하면 Claude Code로 로컬 개발을 계속하면서도 운영 반영이 반복 가능해진다.

## 3단계. 개발팀 인계용 정식화

목표: VM 임시 운영을 Cloud Run/Cloud SQL 구조로 넘길 수 있게 준비.

작업:

1. Attribution ledger 저장소를 SQLite에서 Postgres로 이전할지 결정.
2. CAPI 전송 로그를 JSONL에서 Postgres/BigQuery/Cloud Logging 중 하나로 이전.
3. `startBackgroundJobs()`의 `setInterval`을 비활성화할 수 있는 env flag 추가.
4. Cloud Scheduler가 호출할 sync endpoint를 명확히 분리.
5. sync 중복 방지를 DB lock으로 변경.
6. Dockerfile 작성.
7. Secret Manager 변수 목록 정리.
8. Cloud Run 배포 문서 작성.
9. 데이터 마이그레이션 절차 작성.

이 단계가 끝나면 개발팀은 VM을 계속 운영하지 않고 Cloud Run 기반으로 정식 이관할 수 있다.

## 필요한 보안/운영 체크

- `.env` 원문은 문서나 Git에 남기지 않는다.
- Cloudflare Tunnel token은 서버 secret으로만 관리한다.
- Google service account key JSON은 가능하면 파일 업로드보다 Secret Manager 또는 Workload Identity를 쓴다.
- `crm.sqlite3`는 매일 백업한다.
- `logs/*.jsonl`은 크기가 커질 수 있으므로 logrotate 또는 Cloud Logging 전환이 필요하다.
- 운영 서버 timezone은 KST 표시와 UTC 저장 기준을 문서화한다.
- CAPI 중복 전송 방지를 위해 서버가 2대 이상 뜨지 않도록 한다. Cloud Run 전환 전까지는 VM 단일 인스턴스가 더 안전하다.

## 의사결정

현재 권장 의사결정:

- 당장: **Google Compute Engine VM + Cloudflare Tunnel**.
- 백엔드: **Express 유지**.
- 프론트엔드: **Next.js 유지**.
- Cloud Run: **정식화 2단계 이후 검토**.
- Cloudflare Workers: **현재 구조로는 보류**.

이유:

- 지금 가장 중요한 문제는 ROAS/Attribution/CAPI 데이터 정합성이다.
- 백엔드를 Next.js로 바꾸거나 Workers로 재작성하는 것은 데이터 정확도를 바로 올리지 않는다.
- 오히려 지금은 노트북 의존을 제거하고, 현재 검증 중인 수집/전송 로직을 안정적으로 계속 돌리는 것이 우선이다.
