## 1. 결론 3줄

1. **Imweb API는 IP 필드를 제공하지 않음**(공식 확인). 따라서 차단은 IP가 아니라 **전화번호·이름 패턴·행동 패턴·가상계좌 반복 발급 빈도**로 만들어야 함.
2. **현재 구조상 어뷰저 주문은 Meta CAPI/브라우저 픽셀에는 영향이 없음**. Attribution 라우터가 `paymentStatus === 'confirmed'`일 때만 CAPI를 발화하고, 가상계좌 pending은 `block_purchase_virtual_account`로 명시적 차단되어 있음 (`backend/src/routes/attribution.ts:585-588`).
3. **진짜 문제는 2개 영역**: ① Imweb 기반 매출/취소 집계(§6 Imweb CANCEL 금액 과장) ② GA4 purchase 이벤트. 특히 GA4는 `imwebAttributionSnippet.ts`가 결제완료 페이지 접근 시 **결제 상태 무관하게 `purchase` 이벤트를 쏘므로**, 가상계좌 주문도 GA4에 매출로 집계된 후 미입금 만료되어도 GA4 refund가 미구현이라 **영구 과장**됨. ROAS 왜곡의 실제 원인은 여기.

## 3. Imweb API 제약 — IP 필드 없음 (공식 확인)

Imweb 개발자센터 `old-developers.imweb.me/orders/get` 기준으로 주문 조회 응답 구조를 재확인함 (2026-04-15).

### 3-1. 주문 응답에 IP 관련 필드 전무

- `orderer` 객체: `member_code`, `name`, `email`, `call`, `call2` — 끝.
- `device` 객체: `type` (`pc` 또는 `mobile`) — 끝.
- 추가 디바이스 필드: `app` (android/ios), `device_code` — fingerprinting 아님.
- 확인한 "없는 필드": `ip`, `ip_addr`, `client_ip`, `customer_ip`, `orderer_ip`, `remote_ip`, `access_ip`, `user_ip`, `user_agent`, `referer`.

→ **Imweb v2 API로는 주문자의 IP를 알 수 없음**. Imweb이 내부적으로 IP 로그를 가지고 있는지는 고객센터 요청 또는 수사기관 영장으로만 확인 가능.

### 3-2. 대안 — IP를 확보하려면

1. **Imweb 관리자 페이지에서 주문별 접속 IP가 보이는지 직접 확인** (수동). 관리자 UI에 있고 API에 없는 필드일 가능성 — 있으면 사람이 수동으로 조회 가능.
2. **결제완료 페이지 footer snippet(`imwebAttributionSnippet.ts`)에서 IP를 직접 수집**하는 방법. 브라우저에서 IP는 바로 못 읽고, 서버 side에서 요청 헤더(`x-forwarded-for`)로 받아야 함. 현재 `POST /api/attribution/payment-success` 엔드포인트가 이 요청을 받는데, **여기서 `req.ip` 또는 `req.headers['x-forwarded-for']`를 저장하면 됨**. 단 이건 결제완료 페이지 도달한 IP이지 주문 접수 시점 IP가 아님 (가상계좌 발급 시점과 다를 수 있음).
3. **결제창 진입 시점 IP는 Toss 쪽에만 있을 가능성**. Toss API 응답 확인 필요.
4. **장기**: Imweb v3 OpenAPI (OAuth 별도) 사용 시 추가 필드가 있는지 재검증.

결론: **단기에는 IP 기반 차단이 불가능**. 차단 로직은 IP 없이 동작해야 함.

## 4. 현재 구조에서 어뷰저가 미치는 영향 — 레이어별 분석

### 4-1. Meta CAPI — ✅ 안전

`backend/src/metaCapi.ts:260-267` 확인:

```ts
entries.filter(
  (entry) =>
    entry.touchpoint === "payment_success" &&
    entry.captureMode === "live" &&
    entry.paymentStatus === "confirmed",
)
```

→ CAPI는 **`paymentStatus === 'confirmed'`일 때만 Meta `Purchase` 이벤트 발화**. 가상계좌 pending, unknown, canceled는 모두 제외.

**결과**: 어뷰저의 가상계좌 미입금 주문은 **영영 `confirmed`가 되지 않으므로 Meta CAPI에 안 감**. Meta 쪽에서 이 매출을 기억하지 않음. 🎉

### 4-2. 브라우저 픽셀 — ✅ 안전

`backend/src/routes/attribution.ts:585-588` 확인:

```ts
if (status === "pending") {
  ...
  browserAction: "block_purchase_virtual_account",
```

→ 프런트엔드 Meta Pixel도 가상계좌 주문에 대해서는 **`Purchase` 이벤트를 명시적으로 차단**. `datacheck0406.md`의 P0 #2 ("WAITING_FOR_DEPOSIT를 메인 ROAS/CAPI 기준값에 섞지 말 것") 원칙이 코드에 실장되어 있음.

### 4-3. Toss 결제 승인 — ✅ 안전

Toss는 가상계좌 발급 후 입금 안 되면 자동 만료. `toss_transactions.status='WAITING_FOR_DEPOSIT'`에서 만료 후 Toss 쪽에서 별도 상태 전이 없이 그냥 남음. `amount=0`. 재무/정산 기준 매출에 섞이지 않음.

### 4-4. Imweb CANCEL 집계 — 🔴 **왜곡 있음**

§2-2에서 설명한대로 `imweb_orders.imweb_status='CANCEL' + pay_type='virtual'`에 ₩782.7M가 **"실제로는 결제가 없었던 금액"**으로 쌓임. `payment_amount` 컬럼이 "결제 예정 금액"을 그대로 남겼기 때문. 이건 **오늘 새로 만든 `purchase-confirm-stats` 엔드포인트**에서 직접 노출되기 때문에, 이 엔드포인트를 그대로 ads 대시보드에 연결하면 **취소율이 68%로 보임**.

### 4-5. GA4 purchase 이벤트 — 🔴 **가장 큰 왜곡**

`imwebAttributionSnippet.ts`는 결제완료 페이지 방문 시 **결제 상태를 조회하지 않고 `payment_success` 이벤트를 쏨**. 가상계좌 주문이 Imweb의 "주문완료" 페이지에 도달하는 시점에 GA4 `purchase` 이벤트 + CAPI `payment_success` 로그가 발화됨.

- CAPI로 쏘는 payment_success는 attribution ledger에 `pending` 상태로 저장. 나중에 `sync-status/toss` 배치가 `WAITING_FOR_DEPOSIT`을 발견하면 `pending`에 머물거나 canceled로 승격. Meta CAPI에는 안 감 (§4-1). **안전**.
- 그러나 **GA4 purchase 이벤트는 이미 발화됨**. Measurement Protocol로 refund를 쏘지 않는 이상(§7.3 미구현) GA4는 영구 기억. ROAS를 GA4 기반으로 보는 경우 이 매출이 **영구 과장**.

**영향도**:
- `/ads/roas` — 0406에 이미 메인 ROAS를 Attribution 기반으로 통일함. **이 페이지는 안전**.
- `/ads` 기타 GA4 매출 카드, GA4 Data API로 매출을 뽑는 모든 리포트 — **오염**.
- Google Ads OCI(Offline Conversion Import)로 GA4 이벤트를 업로드하는 경로가 있다면 Google Ads 학습에 어뷰저 매출 주입. **확인 필요**.
- BigQuery raw export도 같은 이벤트가 그대로 쌓임. 과거 분석 시 주의 필요.

| 레이어 | 어뷰저 영향 | 이유 |

## 5. 차단 시스템 설계

### 5-1. 사용 가능한 신호 (IP 없이)

| 신호 | 접근 경로 | 품질 |
| ---- | --------- | ---- |
| **전화번호 반복 카운트** | `imweb_orders.orderer_call` 로컬 집계 | 🟢 높음 |
| **가짜 번호 패턴** | `^01(00000000|11111111|22222222|...)$` 정규식 | 🟢 높음 |
| **이름 패턴** (한글 한글자 반복, 특수문자, 무의미 단어) | `orderer_name` 정규식 | 🟡 중간 — false positive 위험 |
| **가상계좌 미입금 이력** | `imweb_status='CANCEL' & pay_type='virtual'` 로컬 | 🟢 높음 (이미 수집됨) |
| **단위 시간당 발급 건수** | `imweb_orders.order_time` | 🟢 높음 |
| **금액 상한** | `payment_amount` | 🟡 중간 — 실제 대량 주문과 구분 어려움 |
| **IP** | ❌ 없음 (Imweb API 미제공) | 🔴 불가 |
| **device_code / user_agent** | Imweb에 없음, 결제완료 페이지에서 수집 가능 | 🟡 결제 후에만 |

### 5-2. 3-layer 차단 전략

```
           [Layer 1: 구매 전 차단 — sync 시점 블랙리스트]
                         │
                         ▼
          [Layer 2: 구매 후 분리 — 집계 시 exclude]
                         │
                         ▼
            [Layer 3: 사후 감사 — 주기적 패턴 탐지]
```

#### Layer 1 — 사전 차단 (장기)

- **어디서**: 결제창 진입 시점 또는 가상계좌 발급 직전. Imweb 체크아웃 페이지에 script 주입으로 구현.
- **무엇을**: 전화번호 블랙리스트 체크 + 가짜 번호 패턴 거절 + 단위 시간당 발급 상한.
- **한계**: Imweb 체크아웃 흐름을 우리가 직접 제어할 수 있는지 확인 필요. script injection 가능 여부가 go/no-go.
- **우선순위**: P2 (검토 후 결정).

#### Layer 2 — 사후 분리 (즉시 가능)

- **어디서**: 우리 `purchase-confirm-stats`, `ads` 대시보드, 모든 매출 리포트.
- **무엇을**: `imweb_orders` 행에 `is_abusive_candidate` 플래그 추가. 플래그 기준:
  - `orderer_call IN (BLACKLIST)`
  - `orderer_call` 가짜 번호 정규식 매칭
  - 동일 `orderer_call`이 최근 90일 내 가상계좌 미입금 3회 이상
  - `orderer_name` 의미없는 한글 1~2글자 반복 또는 특수 키워드 ("사기", "리셀러", "광고" 등)
- **어떻게 쓰나**: 집계 쿼리에서 `AND is_abusive_candidate = 0`로 필터. 원본 데이터는 건드리지 않음.
- **우선순위**: **P0** (즉시 가능).

#### Layer 3 — 주기적 패턴 감사

- **어디서**: 일 1회 또는 6시간 주기 cron.
- **무엇을**:
  - 신규 어뷰저 패턴 자동 감지 (같은 번호 N회 미입금 → 블랙리스트 자동 추가)
  - 기존 블랙리스트 행동 트렌드 (예: `01089744005`가 이번 주에 몇 건 더 시도했나)
  - Slack/메일 경고 (임계값 초과 시)
- **우선순위**: P1.

## 6. 구현 단계

### 6-1. Day 1 (즉시, P0)

1. **`abuser_blacklist` sqlite 테이블 신설** (`crmLocalDb.ts` schema 추가):
   ```sql
   CREATE TABLE IF NOT EXISTS abuser_blacklist (
     phone TEXT PRIMARY KEY,
     reason TEXT NOT NULL,
     attempts INTEGER DEFAULT 0,
     total_vbank_amount INTEGER DEFAULT 0,
     first_seen TEXT,
     last_seen TEXT,
     blocked INTEGER NOT NULL DEFAULT 1,  -- 0이면 경고만
     source TEXT,                           -- 'manual' | 'auto'
     created_at TEXT NOT NULL DEFAULT (datetime('now')),
     updated_at TEXT NOT NULL DEFAULT (datetime('now'))
   );

   CREATE TABLE IF NOT EXISTS abuser_exceptions (
     phone TEXT PRIMARY KEY,
     reason TEXT NOT NULL,
     created_at TEXT NOT NULL DEFAULT (datetime('now'))
   );
   ```
2. **초기 블랙리스트 3건 + 예외 2건 seed 삽입** (§5-3).
3. **`imweb_orders`에 `is_abusive_candidate INTEGER DEFAULT 0` 컬럼 추가**.
4. **라벨링 함수**: 기존 `sync-order-statuses` 완료 후 자동 실행. 조건:
   - phone ∈ `abuser_blacklist.phone` → `is_abusive_candidate=1`
   - 가짜 번호 정규식 (`^01[0-9]{2}(00000000|11111111|22222222|33333333|44444444|55555555|66666666|77777777|88888888|99999999)$` 또는 `^010[0-9]{8}$ AND phone이 반복 문자`) → `1`

## 7. ROAS 왜곡 방지 — 체크리스트

어뷰저 주문이 ROAS를 왜곡할 수 있는 경로를 전부 점검한 결과:

| 경로 | 현재 상태 | 조치 |
| ---- | -------- | ---- |
| Meta CAPI (서버) | ✅ 차단 중 | 유지 |
| Meta Pixel (브라우저) | ✅ 차단 중 | 유지 |
| `/ads/roas` 메인 Attribution ROAS | ✅ 안전 (0406 이후 confirmed만) | 유지 |
| `/ads` 기타 GA4 매출 카드 | 🔴 오염 중 | §6-1 #5의 `is_abusive_candidate` 필터 적용 |
| **GA4 Data API direct query** | 🔴 오염 중 | GA4 refund 구현 전까지 근본 해결 없음 |
| **GA4 BigQuery raw export** | 🔴 오염 중 | 같음. 과거 데이터 분석 시 `transaction_id`별로 Imweb CANCEL 매칭해 사후 보정. |
| Google Ads OCI 업로드 | ❓ 확인 필요 | OCI 경로가 활성화돼 있는지 재확인. 있다면 긴급 차단. |
| Attribution ledger raw (`pending` 포함) | 🟡 raw 보존 | 메인 집계에는 안 쓰이므로 유지. 별도 경고 배지 추천. |
| **`purchase-confirm-stats` 엔드포인트** | 🔴 오염 중 (오늘 새로 만든 것) | §6-1 #5 서브분리 즉시 적용 |

### 7-1. 권장 운영 원칙 (datacheck0415.md와 통합)

1. **매출 기준값 = Toss `confirmed_revenue`** (변동 없음).
2. **취소 기준값 = Toss `CANCELED + PARTIAL_CANCELED` 합계** (Imweb CANCEL 쓰지 말 것).
3. **구매확정 기준값** (playauto 정지 기간):
   - Imweb `imweb_status='PURCHASE_CONFIRMATION' AND is_abusive_candidate=0`
   - 커버리지 80% (쿠팡/네이버/수동 제외) 배지 항상 표시.
4. **어뷰저 플래그 = 참고용 서브 카드**. 메인 수치에 섞지 말고 별도 표시.
5. **GA4 매출 읽을 때** (`/ads` GA4 카드, BigQuery 분석): 오염 경고 배지 필수. 근본 해결은 GA4 refund 구현까지 대기.

### 7-2. CAPI 영향 재확인

**어뷰저가 현재 CAPI에 주는 영향 = 0**. 근거:
- `metaCapi.ts:260-267`에서 `paymentStatus === 'confirmed'` 필터.
- `attribution.ts:585-588`에서 가상계좌 pending `block_purchase_virtual_account` 명시.
- 어뷰저의 가상계좌 미입금은 영영 `confirmed`가 안 되므로 CAPI에 안 감.

**잠재 리스크 1건**: 어뷰저가 실제로 입금 후 즉시 환불하는 케이스. 이건 `confirmed`까지 가므로 CAPI 발화됨. 이후 Toss `CANCELED` 전이가 GA4/CAPI에 refund로 반영되지 않으면 영구 기억. **GA4 refund 미구현과 동일한 문제의 CAPI 버전**. 단 실측 데이터상 이 케이스는 매우 드물음 (어뷰저는 대부분 입금 안 함, 상위 10건 전부 `amount=0`).

→ **단기 조치 필요 없음**. GA4 refund 구현 시 같이 Meta CAPI `Refund` 이벤트(Conversions API의 `event_name=Refund` 또는 `Purchase`의 `value` 음수 보정)도 함께 설계.

## 8. 한계와 리스크

1. **IP 기반 차단 불가**. Imweb API에 IP 필드 없음. 수사기관 영장 없이는 실명 추적 불가.
2. **결제창 사전 차단 구현 가능 여부 불확실**. Imweb 체크아웃 커스텀 스크립트 허용 여부 확인 전까지 Layer 1은 설계만.
3. **이름 패턴 정규식 false positive**. "ㅇㅇ", "ㄱㄱ" 같은 짧은 한글은 실제 고객 별명일 수도 있음. 차단 조건은 복합(이름 + 전화번호 패턴 + 반복 횟수) 필수.
4. **어뷰저가 전화번호를 변주**하면 블랙리스트 회피 가능. 가짜 번호 정규식으로 일부 커버되지만 완벽 차단 불가.
5. **내부 테스트 번호와 정상 반복 구매 고객 혼동 위험**. `abuser_exceptions` 테이블을 반드시 먼저 seed. TJ 본인 `01087418641`은 예외 필수.
6. **Imweb `CANCEL` 집계 자체는 바뀔 수 없음**. 우리 로컬 레이어에서 필터링만 가능. Imweb 원본을 수정할 권한 없음.
7. **소송 실효성 낮음**. 업무방해죄 입증 공수 크고, IP 없이는 실명 추적 어려움. §8-2 참조.

### 8-1. 법적 대응 옵션 (참고)

- **업무방해죄(§314)**: 가장 현실적. 반복성·조직성·업무 저해 증명 필요.
- **사기죄(§347)**: 금원 취득 없어 성립 어려움.
- **증거 수집**: 오늘 로컬 DB에 쌓인 데이터 + Imweb 관리자 IP 로그(있다면)가 전부.
- **가성비**: 기술 차단이 고소보다 빠르고 싸고 효과적. 명백한 악의(예: 01089744005)는 경찰 상담 가치 있으나 단독 실효성은 낮음.

## 9. 참고 증거 (재현 가능한 쿼리)

### 9-1. 반복 전화번호 탐지

```sql
SELECT orderer_call, COUNT(*) AS attempts, SUM(payment_amount) AS sum_amt,
  GROUP_CONCAT(DISTINCT orderer_name) AS names
FROM imweb_orders
WHERE site='biocom' AND imweb_status='CANCEL' AND pay_type='virtual'
GROUP BY orderer_call
HAVING COUNT(*) >= 3
ORDER BY attempts DESC;
```

### 9-2. CANCEL × Toss 교차

```sql
SELECT tt.status, COUNT(*), SUM(io.payment_amount), SUM(tt.amount)
FROM imweb_orders io
LEFT JOIN toss_transactions tt ON tt.order_id LIKE (io.order_no || '%')
WHERE io.site='biocom' AND io.imweb_status='CANCEL'
GROUP BY tt.status;
```
