   - 로컬: 이 맥북 PC. 대표 로컬 DB는 `/Users/vibetj/coding/seo/backend/data/crm.sqlite3`

   나쁜 예시:
   - 성공 기준은 “Google이 구매로 세는 값 중 실제 결제완료가 아닌 후보를 action 단위로 분리”입니다.

   왜 나쁜가:
   - `action 단위`, `후보`, `분리` 같은 내부 작업자 표현만 있고, TJ님이 어떤 화면에서 무엇을 확인해야 하는지 알 수 없다.
   - “그래서 Google ROAS를 믿어도 되는지”, “무엇을 바꾸면 되는지”, “성공하면 어떤 숫자가 어떻게 보여야 하는지”가 빠져 있다.

   좋은 예시:
   - Google Ads가 “구매완료”라고 세는 숫자 안에 실제 결제완료가 아닌 클릭/결제시작/NPay 클릭이 섞였는지 확인한다.
   - 성공 기준은 Google Ads 전환 화면과 우리 내부 주문 장부를 나란히 봤을 때, `실제 결제완료 매출`, `NPay 실제 결제완료 매출`, `구매가 아닌데 구매처럼 잡힌 신호`가 각각 금액과 건수로 따로 보이는 것이다.
   - 이 상태가 되면 TJ님은 Google Ads ROAS를 그대로 예산 판단에 쓸지, 아니면 내부 confirmed ROAS를 우선할지 결정할 수 있다.

4-0. **진척률을 항상 적는다.**
   - 결과보고서에는 `전체 프로젝트/스프린트 기준 진척률 %`, `이번 batch 기준 진척률 %`, `100%까지 남은 단계`, `다음 병목`, `사람이 이해할 수 있는 1문장 설명`을 넣는다.
   - 결과보고 텍스트 출력에는 프로젝트 본연의 `Objective`, `Key Results`, `액션플랜`, `액션플랜별 진척률 %`, `이번 작업이 OKR에 준 영향`을 항상 넣는다.
   - 작업이 작더라도 국소 작업 완료만 쓰지 말고, 전체 프로젝트 목표와 액션플랜에서 이 작업이 어느 위치인지 적는다.
   - 숫자가 추정이면 `약`을 붙이고, 무엇을 끝내면 몇 %가 올라가는지 함께 적는다.
   - Phase 번호순보다 실제 개발 우선순위(P0/P1/P2 실행순)를 먼저 보여준다.

4-1. **대화 출력에서는 넓은 표를 쓰지 않는다.**
   - 최종 답변, 중간 보고, 터미널 결과 공유에서는 `다음 할일`을 5컬럼 이상 Markdown 표로 쓰지 않는다.
   - 표 셀에 문장형 설명, 긴 URL, 긴 파일 경로, 명령어, 성공 기준, 실패 대응이 들어가면 반드시 번호형 액션 카드로 바꾼다.
   - Obsidian 문서나 개발 부록에는 표를 쓸 수 있다. 다만 대화에 그대로 붙여 넣지 말고 `Codex가 할 일`과 `TJ님이 할 일` 카드로 요약한다.
   - 숫자 요약표처럼 2-4컬럼이고 셀이 짧은 경우만 대화 출력 표를 허용한다.
   - 출력 전 자체 체크: `다음 할일이 표인가`, `5컬럼 이상인가`, `한 셀이 한 문장을 넘는가`, `긴 URL/파일 경로가 표 안에 있는가`를 확인하고 하나라도 YES면 카드형으로 바꾼다.

4-2. **다른 에이전트 검증 여부를 표시한다.**
   - 자신감이 낮거나, 운영/돈/광고 플랫폼/DB/배포 영향이 크거나, 판단이 복잡하면 `다른 에이전트 검증 권장`이라고 쓴다.
   - TJ님과 Codex가 충분히 처리 가능한 Green/Yellow 범위라면 `다른 에이전트 검증 불필요`라고 쓰고 진행한다.
   - 다른 에이전트 검증을 권장할 때는 무엇을 검증받을지 한 문장으로 적는다.

5. **옵션이 여러 개면 자신감 %를 붙인다.**
   - 자신감 % 앞에는 반드시 `Codex 추천: 진행 추천 / 진행 비추천 / 보류 추천` 중 하나를 먼저 쓴다.
   - 자신감 %는 사용자가 고른 옵션이 아니라 Codex가 추천한 방안에 대한 확신이다.
   - 예: `Codex 추천: 진행 비추천`, `추천 방향에 대한 자신감: 88%`, `이유: Imweb API rate-limit가 이미 발생했고 5분 주기는 호출량을 3배로 늘린다.`
   - 100%: 특이사항 없이 해야 하는 일
   - 80~95%: 근거가 충분하고 진행 권고
   - 50~79%: 가능하지만 추가 확인 필요
   - 50% 미만: 지금 진행보다 조사 우선

6. **하지 않은 것도 명시한다.**
   - 운영 배포, 운영DB write, 외부 플랫폼 전환 전송처럼 위험한 작업은 “안 했다”가 중요한 정보다.

7. **승인 요청은 별도 문서로 만든다.**
   - VM Cloud 배포, 운영DB write, 외부 플랫폼 전환 전송, GTM Production publish처럼 위험이 있는 작업은 대화 답변만으로 승인 요청하지 않는다.
   - 승인 문서 작성 자체는 승인 없이 진행한다.
   - 별도 승인 문서에 `무엇을`, `왜`, `어떻게`, `허용 범위`, `금지 범위`, `Hard Fail`, `Success Criteria`, `Rollback`, `자신감 %`를 적는다.
   - 최종 답변에는 승인 문서 링크와 핵심 승인 문구만 짧게 제시한다.

8. **Auditor verdict를 남긴다.**
   - `PASS`, `PASS_WITH_NOTES`, `FAIL_BLOCKED`, `NEEDS_HUMAN_APPROVAL` 중 하나로 쓴다.
   - Green / Yellow / Red Lane과 no-send/no-write/no-deploy/no-publish/no-platform-send 확인 결과를 같이 적는다.

9. **피드백은 핵심과 고도화를 분리한다.**
   - 지금 승인·진행 여부를 바꾸는 내용은 `지금 반드시 필요한 핵심 피드백`에 둔다.
   - 품질을 올리지만 지금 결정을 막지 않는 내용은 `나중에 고도화 phase로 넘길 피드백`에 둔다.
   - 같은 위험 항목을 여러 섹션에서 반복하지 않는다.

10. **Green 자율 진행과 Yellow/Red 중단선을 분리한다.**
   - read-only 조사, no-send smoke, local code implementation, fixture test, 문서/정본 업데이트, gptconfirm 패키징, validation, 관련 변경 선별 commit/push는 Green Lane이면 별도 확인 없이 진행 가능하다.
   - 다음 할일은 지금 고친 파일이나 국소 이슈만 보지 않는다. 하네스 Green 영역 안에서 목표 달성을 위해 필요한 조사, 설계, 문서화, 검증, 로컬 구현은 Codex가 자율적으로 진행한다는 가정 하에 구상한다.
   - 본 할일이 끝난 뒤 새로 얻은 인사이트가 있으면, 추가 조사·설계·제안이 필요한지 판단하고 Green 영역이면 기다리지 말고 이어서 진행한다.
   - 단, Green 자율 다음 할일도 운영 배포, 운영DB write, 외부 전송, 광고 예산/캠페인 변경처럼 Yellow/Red 중단선을 넘으면 승인 항목으로 분리한다.
   - 운영 deploy, operational schema migration, GTM Production publish, Imweb production save, platform send, Google Ads conversion action 변경, conversion upload, 운영 저장 canary, 광고 예산/캠페인 변경은 명시 승인 전 금지다.

11. **gptconfirm batch를 한 묶음으로 만든다.**
   - 경로는 `gptconfirm/YYYYMMDD-N/` 또는 기존 호환이 필요한 경우 `gptconfirm/gpt0508-N/` 형식을 쓴다.
   - 각 batch에는 `00-result-report.md`, 승인 대상 문서 1-3개, `99-total-current-copy.md`, `manifest.json`을 둔다.
   - 다음 batch는 기존 batch를 덮어쓰지 않는다.
