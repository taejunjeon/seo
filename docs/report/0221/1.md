# report0221-1 — AI 답변 인용도(0/20) 점검 리포트

작성일: 2026-02-21
대상: “AI 답변 인용 빈도(AI 답변 인용도)”가 0/20으로 표시되는 원인 및 시스템 준비 여부 점검

## 0) 결론(요약)

- 준비 여부: **있음(백엔드 구현 완료).** “AI 답변 인용도”를 **Google AI Overview + ChatGPT(Search) + Perplexity**까지 확장하는 백엔드 측정 모듈을 구현했으며, AEO 점수(`/api/aeo/score`) 산출 과정에서 사용하도록 연결함. (프론트 미구현)
  - 현재 로컬 설정 기준: Google AIO(SerpAPI) + ChatGPT(Search/OpenAI) + Perplexity 모두 측정 가능(키/호출 테스트 완료)
- 0/20이 뜨는 대표 원인 3가지:
  1) (공통) 프로바이더 키 미설정/실패/쿼터 소진(=eligible 감소 → 0점)  
  2) (Google AIO) 표본 키워드에서 AI Overview가 거의 안 뜸(=분모 0에 가까워 0점)  
  3) (공통) AI 답변/출처는 존재하지만 **biocom.kr이 출처 링크에 포함되지 않음**(=인용률 0%로 0점)

## 1) 현재 시스템에 구현된 측정 백엔드(존재 확인)

### 1-1. 핵심 파일/모듈

- 측정 로직: `backend/src/aiCitation.ts`
  - 함수: `measureAiCitation(...)`
  - 결과 타입: `AiCitationResult` (provider/hl/gl/siteHost/sample/인용률 등 포함)
- 멀티 프로바이더 측정(확장): `backend/src/aiCitationMulti.ts`
  - 함수: `measureAiCitationMulti(...)`
  - 포함 프로바이더: Google AIO(SerpAPI) / ChatGPT(Search, OpenAI web_search) / Perplexity
- ChatGPT(Search) 출처 추출: `backend/src/openaiSearch.ts`
- Perplexity 출처 추출: `backend/src/perplexity.ts`
- SerpAPI 호출: `backend/src/serpapi.ts`
  - 함수: `fetchSerpApiGoogleAiOverview(...)`
  - 엔드포인트: `https://serpapi.com/search.json?engine=google&q=...`
  - AI Overview 판정: `payload.ai_overview.references` 배열 길이로 판단
- AEO/GEO 점수에 반영:
  - AEO 엔드포인트: `backend/src/server.ts`의 `GET /api/aeo/score`
  - GEO 엔드포인트: `backend/src/server.ts`의 `GET /api/geo/score`
  - 점수 계산: `backend/src/scoring.ts`의 `calculateAeoScore` / `calculateGeoScore`

### 1-2. 실제 데이터 흐름(페이지 진단/오버뷰에서 보이는 0/20의 출처)

1) `/api/aeo/score` 호출 시, GSC에서 최근 구간 키워드(최대 500개)를 가져옴  
2) 그 키워드를 `measureAiCitationMulti`에 전달하여 표본(기본 5개)을 선정하고,
   - SerpAPI(Google AI Overview) references
   - OpenAI(ChatGPT Search) 출처 URL
   - Perplexity citations(설정 시)
   를 각각 조회/매칭하여 종합 인용률을 계산  
3) 각 프로바이더에서 **도메인이 `biocom.kr`인 링크가 존재하면 ‘인용’으로 카운트**  
4) `calculateAeoScore`가 “AI 답변 인용 빈도(20점)” 항목을 계산해 breakdown에 넣음

## 2) 측정 알고리즘(현재 구현 상세)

### 2-1. 표본 키워드 선택 방식

- 입력: `{ query, impressions?, clicks? }[]`
- 선택:
  - `query` trim 후 빈 값 제거
  - **impressions 내림차순 → clicks 내림차순** 정렬
  - 상위 `sampleSize`개를 표본으로 선택 (기본 5, 최소 1 ~ 최대 20)

### 2-2. AI Overview 및 인용 판정 방식

- 각 표본 query에 대해 SerpAPI 요청:
  - `engine=google`
  - `q=<query>`
  - `hl` 기본 `ko`, `gl` 기본 `kr`
  - `num` 기본 10
- 응답에서:
  - `hasAiOverview = (ai_overview.references.length > 0)`
  - `references`의 `link` URL hostname이 `siteHost`와 **동일하거나 하위 도메인**이면 매칭으로 판단
    - `www.` 제거 + 소문자 정규화
    - 예: `www.biocom.kr`, `m.biocom.kr` 모두 매칭
- 산출:
  - `aiOverviewPresent`: hasAiOverview=true인 표본 수
  - `citedQueries`: cited=true인 표본 수(매칭 링크 1개 이상)
  - `citedReferences`: 매칭된 reference 링크 개수 합
  - `citationRateAmongAiOverview = citedQueries / aiOverviewPresent` (aiOverviewPresent=0이면 0)

### 2-3. 캐시/중복호출 방지(중요)

- `backend/src/aiCitation.ts`에서 **6시간 TTL 메모리 캐시** 사용
- 캐시 키: `siteHost|hl|gl|sampleSize|fingerprint(표본 쿼리)`
- 동시성: query별 SerpAPI 호출은 최대 2개 동시 실행
- 멀티 프로바이더 캐시: `backend/src/aiCitationMulti.ts`에서 동일 TTL로 캐시(표본 fingerprint 포함)

## 3) AEO 점수에서 0/20이 계산되는 방식(스코어 정의)

- 위치: `backend/src/scoring.ts` (AEO breakdown의 `aiCitation`)
- 점수:
  - `score = min(20, round(citationRateAmongAiOverview * 200))`
  - 즉, **AI Overview가 뜬 표본 중 10% 이상에서 biocom.kr이 인용되면 20점 만점**
  - AI Overview가 0개이거나, 인용이 0개면 → 0점

## 5) 0/20의 원인 분류와 “구체적 대응” (Runbook)

### 5-1. 원인 A: 측정 자체가 안 되는 경우(=unavailable로 0점)

가능 원인:
- `SERP_API_KEY` 미설정
- 키 만료/오류/쿼터 소진
- SerpAPI 응답 포맷 변경(`ai_overview` 필드 미제공 등)

확인 방법:
- 백엔드 `/health`의 `apis.serpapi`가 `true`인지 확인
- `/api/serpapi/account`(서버에 구현되어 있음)으로 키 상태 확인
- AEO 스코어 breakdown에서 “AI 답변 인용 빈도”의 detail이 “설정 필요”류 문구인지 확인

대응:
- SerpAPI 키/플랜/쿼터 확인
- 측정 실패 사유를 UI에 노출하도록 백엔드에서 에러를 삼키지 말고(현재는 catch 후 null) “측정 실패(사유)”를 detail로 내려주도록 개선 권장

### 5-2. 원인 B: AI Overview가 표본에서 안 뜨는 경우(=분모 0 → 0점)

가능 원인:
- 표본이 브랜드/상업성 키워드 위주라 AI Overview가 잘 안 뜸
- 지역/언어(hl=ko, gl=kr)에서 AI Overview 노출이 낮음(키워드 성격 영향)

확인 방법:
- breakdown detail에 `AI Overview 0개`로 표시되는지 확인

대응(시스템 측면):
- 표본 추출 전략 개선(권장)
  - “Q&A 키워드(정보성)” subset에서 표본을 뽑기
  - 브랜드/네비게이션성 키워드 제외 룰 추가
  - sampleSize를 5 → 20으로 늘리고(비용 고려) 변동성 감소
- (옵션) hl/gl을 UI에서 설정 가능하게 만들어, KR/KO 외 시장도 테스트하도록 개선
- “AI Overview 노출(분모)”과 “인용률(분자)”을 분리 지표로 표시(둘 다 0이면 원인 파악이 쉬움)

### 5-3. 원인 C: AI Overview는 뜨는데 biocom.kr이 references에 없음(=인용률 0% → 0점)

이 경우는 “측정은 정상인데 인용이 안 됨” 상태이오.

근거(샘플 1건 확인):
- SerpAPI로 `프로바이오틱스 효능`(hl=ko, gl=kr) 조회 시:
  - AI Overview references 12개 감지
  - 그 중 `biocom.kr` 매칭 0개

대응(시스템 측면: ‘무엇을 고쳐야 하는지’가 나오게 만들기):
- AEO 스코어 breakdown에는 “프로바이더별 cited/eligible”까지만 요약되므로,
  **상세 reference(어떤 사이트가 인용되는지 / biocom 매칭 링크가 무엇인지)**는 디버그 API로 확인하는 구조가 적합함.
- 개선 제안(우선순위 높음)
  1) 디버그 API(`/api/ai/citation`)에서 표본별 references/matchedReferences를 내려줘서,
     어떤 사이트들이 인용되는지(=목표 벤치마크)를 바로 볼 수 있게 하기
  2) 결과를 Supabase 등에 주간/일간으로 적재하여 추세를 볼 수 있게 하기(현재는 6h 메모리 캐시만 존재, DB 스키마 변경은 사전 승인 필요)

대응(콘텐츠 측면: 인용될 확률을 올리는 액션 예시)
- Answer-first 요약(1~2문장) + 근거(연구/가이드) + FAQ 섹션 구성
- Article/Author/Organization 스키마(이미 일부 페이지에서 감지됨) + 건강 주제 신뢰도 스키마 보강
- 동일 주제의 Q&A 커버리지 확장(“People Also Ask” 질문군을 H2/H3/FAQ로 흡수)

## 7) 백엔드 개발 계획(프론트 제외)

목표:
- 한국/한국어(hl=ko, gl=kr) 기준으로 “AI 답변 인용도”를 **Google AIO + ChatGPT(Search) + Perplexity**에서 일관된 방식으로 측정하고,
  “왜 0점인지 / 무엇을 해야 인용되는지”가 백엔드 레벨에서 바로 추적 가능하도록 만들기.

Phase 1 — 안정화/관측(1~2일)
1) 디버그 API 고도화
   - `/api/ai/citation`에 `providers=google_ai_overview,chatgpt_search,perplexity` 옵션 추가(선택 실행)
   - `queries=` 파라미터(쉼표/배열) 지원하여 특정 키워드만 즉시 재측정 가능
   - 응답에 provider별 실패 사유/레이트리밋/타임아웃 구분 필드 추가
2) 비용/성능 가드
   - provider별 호출량(표본 수 * 프로바이더 수) 상한 고정, `refresh=1`은 개발 환경/관리자만 허용(보안 정책은 사전 합의 필요)
   - OpenAI/Perplexity 호출은 동시성 1로 유지(현재 구현), SerpAPI는 2로 유지(현재 구현)
3) 로깅/메트릭
   - provider별 요청 성공률, 평균 latency(ms), citations 수 분포, biocom 매칭률을 서버 로그/간단 메트릭으로 남김

Phase 2 — 측정 신뢰도 개선(2~4일)
1) 표본 키워드 선택 개선
   - “정보성(Q&A) 의도” 키워드 우선 표본화(현재는 impressions/clicks 상위)
   - 브랜드/네비게이션성 키워드 제외 룰 추가(분모 0 방지)
2) ChatGPT(Search) 출처 추출 품질 개선
   - 모델이 URL을 ‘생성’하지 않도록: URL 유효성 검사(HTTP 2xx/3xx) 통과한 것만 citations로 인정
   - URL 0개일 때는 eligible=false로 처리하여 지표 왜곡 방지(현재는 eligible=true)
3) Perplexity citations 파서 보강
   - citations가 응답 최상단에 없거나 포맷이 달라도(버전 변경) 안전하게 추출되도록 방어 로직 추가

Phase 3 — 지속 측정/히스토리(승인 필요, 3~7일)
1) 결과 적재(승인 필요: DB 스키마 변경)
   - Supabase 테이블 신설: 날짜/프로바이더/표본/인용률/예시/매칭 링크 저장
   - 주간 리포트(변화량) 생성용 뷰/쿼리 준비
2) 스케줄링
   - Cron endpoint로 매일 1회 측정(표본 20), 급격한 변화 감지 시 알림(슬랙 연동)
3) 스코어 정책 재정의
   - 프로바이더 가중치(예: Google AIO 50%, Perplexity 30%, ChatGPT 20%) 또는 “최대값/평균값” 등 정책 결정 후 반영
