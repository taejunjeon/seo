## 1) 피드백 요약(핵심만)

`report0221-feedback2.md`에서 강조한 핵심은 아래 3가지였소.

1) “0점(0/20)”을 **측정 실패 / 노출(분모) 0 / 인용(분자) 0**으로 즉시 분해해서 보여줘야 함  
2) OpenAI/Perplexity의 출처는 **모델 텍스트 파싱이 아니라 API 메타데이터(annotations / search_results)** 기준으로 고정해야 함  
3) SerpAPI AIO는 케이스에 따라 **page_token follow-up**이 필요할 수 있으니, 판정 로직을 보강해야 함

### 2-2. 멀티 프로바이더 인용도(SerpAPI + ChatGPT(Search) + Perplexity) 관측 강화

## 6) Phase 2 이후 목표를 그렇게 세운 “이유”

### Phase 2(측정 신뢰도 개선)

1) 표본(키워드 샘플) 전략 개선이 필요한 이유
- 지금은 impressions/clicks 상위로만 뽑기 때문에, “AIO/출처가 거의 안 붙는 키워드”가 섞이면 eligible이 줄고 지표가 쉽게 0으로 흔들릴 수 있음
- “제품 지표”로 쓰려면, 정보성(Q&A) 중심 표본으로 바꿔 **분모를 안정화**해야 함

2) OpenAI/Perplexity 추출 안정화(메타데이터 고정)를 최우선으로 둔 이유
- 텍스트 파싱은 포맷 변경에 매우 취약해 “인용 0” 오탐을 반복해서 만들기 쉬움
- 메타데이터(annotations/search_results) 기반으로 고정하면, 지표가 “프롬프트/표현”이 아니라 “출처 데이터”에 묶여 제품 지표로써 신뢰도가 올라감

### Phase 3(지속 측정/히스토리)

1) 히스토리 적재를 별도 Phase로 둔 이유
- “인용률이 올랐는지/내렸는지”는 단발 측정으로 판단이 불가능하고, 반드시 시간축으로 추적해야 함
- 다만 DB 스키마/운영/보안 이슈가 동반되므로, 승인 단계로 분리하는 것이 안전함

2) 스케줄링/알림이 필요한 이유
- 인용도는 “한 번 개선”보다 “변화 감지 → 원인 추적 → 액션”의 루프가 더 중요함
- 자동 측정 + 알림이 있어야 운영 비용이 줄고, 실제 개선 활동이 지속될 수 있음

3) 스코어 정책(가중치/합산/최대값 등)을 Phase 3로 둔 이유
- 현재는 빠른 원인 분해/관측이 우선이라 단순 합산(eligibleTotal 기반)으로 시작
- 히스토리가 쌓이고 “어떤 프로바이더가 더 중요한가”가 정해져야, 가중치 정책이 제품 지표로 의미를 가짐
