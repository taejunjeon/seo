# Frontend Report Writing Rules

작성 목적: 보고서형 프론트엔드 화면을 만들 때 사용자가 내부 용어, 원본 키, 숫자 ID를 갑자기 보고 멈추지 않도록 한다. 화면은 개발자가 이해하는 데이터 구조가 아니라, 운영자가 판단해야 하는 질문과 액션을 중심으로 작성한다.

## 1. 사람 말이 먼저, 원본값은 나중

화면의 제목, 카드명, 표의 첫 열에는 사람이 이해하는 이름을 먼저 쓴다.

나쁜 예:

```text
alias
meta_biocom_cellcleanerreel_igg
```

좋은 예:

```text
사람이 읽는 이름
cellcleanerreel로 들어온 음식물 과민증 검사 주문

원본 내부 추적값
meta_biocom_cellcleanerreel_igg
```

원본값은 숨기지 않는다. 다만 원본값만 먼저 보여주지 않는다. 원본값은 "검증용", "추적용", "개발/데이터 확인용"이라는 역할을 붙여 표시한다.

## 2. 내부 용어는 처음 나오는 자리에서 정의

`alias`, `utm_campaign`, `campaign_id`, `adset_id`, `quarantine`, `manual_verified` 같은 말은 운영자에게 낯설 수 있다. 처음 등장하는 카드나 섹션에서 한 문장으로 정의한다.

예:

```text
내부 추적값이란 광고 링크에 붙어 주문 원장에 남은 값입니다. 사람이 보는 캠페인명이 아니라 시스템이 유입을 구분하려고 저장한 별명입니다.
```

용어 정의 없이 raw key만 제목으로 쓰지 않는다.

## 3. 첫 화면은 "지금 무엇을 판단해야 하는가"를 말한다

첫 화면에는 데이터 구조 요약보다 운영 질문을 먼저 쓴다.

좋은 구조:

```text
이 화면의 질문
이 주문 매출을 어느 Meta 캠페인에 붙일지 결정합니다.

왜 중요한가
잘못 붙이면 ROAS가 부풀고, 안 붙이면 미매핑 매출로 남습니다.
```

`오늘의 결론`처럼 문맥이 부족한 작은 요약 박스는 피한다. 결론 카드를 쓸 때는 반드시 "결정", "근거", "영향", "다음 액션"을 같이 넣는다.

## 4. 숫자는 기준과 출처를 같이 보여준다

보고서 숫자에는 최소한 아래 5가지를 붙인다.

- 기준 시각
- 데이터 출처
- 집계 기간
- 적용 필터
- 신뢰도 또는 주의점

예:

```text
기준: 2026-04-25 로컬 원장
출처: attribution_ledger + imweb_orders
주의: audit 파일은 생성 시각이 오래되어 현재 원장보다 작게 잡힐 수 있음
```

숫자만 크게 보여주고 기준을 각주로 밀어두지 않는다.

## 5. 판단 카드에는 결론보다 근거를 더 쉽게 쓴다

운영자가 보는 카드는 아래 순서를 따른다.

1. 사람이 읽는 이름
2. 판정
3. 원본 추적값 또는 원본 ID
4. 이 값이 무엇인지 설명
5. 근거 3-5개
6. 다음 액션

근거는 데이터 필드명이 아니라 자연어로 쓴다.

나쁜 예:

```text
utm_term=120213362391830396, parent=120213362391690396
```

좋은 예:

```text
주문에 남은 광고세트 ID가 음식물 과민증 검사 전환캠페인 하위 광고세트로 확인됩니다.
```

필요하면 그 아래에 원본 ID를 코드 스타일로 작게 표시한다.

## 6. 원본 ID와 사람이 읽는 이름을 항상 짝지어라

숫자 ID는 단독으로 두지 않는다.

나쁜 예:

```text
대상: 120213362391690396
```

좋은 예:

```text
대상 캠페인
[바이오컴] 음식물 과민증 검사 전환캠페인(10/14~)
원본 campaign id: 120213362391690396
```

테이블에서도 ID 열을 먼저 두지 말고, 이름 열 아래 보조 텍스트로 둔다.

## 7. "확정", "임시", "추정"을 시각적으로 구분

데이터 판단 상태는 색상과 문구를 같이 써서 구분한다.

- 확정: 사람이 승인했거나 직접 ID 증거가 있는 값
- 임시 분류: 운영상 먼저 반영하지만 추후 사람 컨펌이 남은 값
- 추정: 보조 증거만 있고 자동 반영하면 위험한 값
- 제외: ROAS에 붙이지 않아야 하는 값

`자신감 93%`만 쓰지 않는다. 왜 93%인지, 운영에서 지금 써도 되는지 같이 쓴다.

## 8. 액션 문구는 동사로 끝낸다

보고서가 "그래서 뭘 하라는지" 애매하면 실패다. 카드마다 다음 액션을 명령형 또는 완료형으로 쓴다.

예:

```text
음식물 과민증 검사 전환캠페인으로 임시 분류했습니다. 추후 TJ님 컨펌 후 최종 확정합니다.
```

예:

```text
Meta 캠페인 ROAS에는 넣지 않고 non-meta IGG bucket으로 분리합니다.
```

## 9. 표보다 설명 카드가 먼저다

낯선 데이터는 표로 바로 시작하지 않는다. 먼저 2-3개의 카드로 맥락을 만들고, 그 다음 표를 둔다.

추천 순서:

1. 이 화면의 질문
2. 현재 상태 요약
3. 가장 중요한 판단 카드
4. 상세 표
5. 원본 증거 펼쳐보기


## 10. 상세 표 및 판단 카드 작성 규칙

- 표의 첫 열은 사람이 읽는 설명이어야 한다.
- 판단 카드 작성 시는 아래 예시 내용을 참고로 한다. (필수는 아님)
- 판단 카드 구성  예시 (필요에 따라 변경 가능)
  
1. 사람이 읽는 제목  
예: 음식물 과민증 검사 주문 유입값  
  
2. 상태 배지  
예: 확정 / 임시 분류 / 추정 / 제외 / 확인 필요  
  
3. 추천 결정  
예: Meta 음식물 과민증 검사 전환캠페인에 붙입니다.  
  
4. 운영 영향  
예: 이 매출 ₩245,000이 해당 캠페인 ROAS에 포함됩니다.  
  
5. 쉬운 근거  
- 주문에 남은 광고세트 ID가 해당 Meta 캠페인 하위 값과 일치합니다.  
- 같은 값으로 들어온 주문이 최근 42건 있습니다.  
- Meta evidence에서 campaign_id가 확인됩니다.  
  
6. 원본 추적값  
alias: meta_biocom_cellcleanerreel_igg  
campaign_id: 120213362391690396  
adset_id: 120213362391830396  
  
7. 다음 액션  
예: 이 값은 자동 확정하고 ROAS 집계에 반영합니다.


## 11. 시각 위계는 텍스트 순서가 아니라 시선 순서로 정한다

1번 규칙이 "무엇을 먼저 쓸지"였다면, 이 규칙은 "무엇이 먼저 눈에 들어올지"다. 운영자가 화면을 열면 0.5초 안에 "내가 지금 액션이 필요한가 아닌가"가 보여야 한다.

폰트 크기 3단계, 색 강도 3단계로 위계를 정한다.

- **가장 강함(액션 필요)**: 24px 이상, 강한 색(빨강, 주황), 화면 상단
- **중간(결론과 근거)**: 16-18px, 검정 또는 진한 회색, 본문 영역
- **약함(원본값과 메타데이터)**: 13-14px, 옅은 회색, 카드 하단 또는 펼치기 안

같은 톤의 회색 글씨로 제목, 결론, 라벨이 전부 표시되면 운영자는 어디서부터 읽을지 결정하지 못한다. 화면에서 가장 큰 글씨는 항상 "지금 해야 할 액션" 또는 "이 화면의 질문"이어야 한다.

## 12. 페이지 종류별 표준 레이아웃을 강제한다

같은 종류의 페이지는 항상 같은 섹션 순서를 갖는다. 페이지마다 구조가 바뀌면 운영자는 두 번째 페이지부터 다시 학습해야 한다.

**판단형 페이지**(매핑 결정, 이상치 검토, 승인 대기)는 다음 5개 섹션을 이 순서로 갖는다.

1. 이 화면의 질문(1문장)
2. 액션이 필요한 항목 수와 우선순위
3. 결론 카드(케이스별, 5번 규칙 형식)
4. 보조 설명 카드 또는 상세 표
5. 원본 증거 펼쳐보기

**대시보드형 페이지**(KPI 모니터링, 트렌드 확인)는 다음 4개 섹션을 갖는다.

1. 이 화면의 질문
2. KPI 카드(기준 시각과 출처 포함, 4번 규칙)
3. 차트 또는 표
4. 이상 신호와 보조 설명

**카테고리(혼합)형 페이지**(같은 운영 질문에 KPI 모니터링과 판단·진단이 같이 필요한 경우)는 다음 5개 섹션을 이 순서로 갖는다.

1. 이 화면의 질문(카테고리가 답하는 운영 질문 1문장)
2. 핵심 KPI 카드(3~6개로 압축. 자세한 모니터링은 대시보드형 화면으로 링크)
3. 결론·판단 카드(5번 규칙 형식)
4. 보조 표 / 차트 / 진단
5. 원본 증거 펼쳐보기

카테고리형은 "한 운영 질문 안에서 KPI와 판단이 분리되면 페이지를 왕복해야 하는" 경우에만 사용한다. 판단 카드 없이 KPI만 보고 싶으면 대시보드형, KPI 없이 판단만 필요하면 판단형을 쓴다. 카테고리형 페이지에서 KPI 카드를 7개 이상 늘어놓지 말 것 — KPI가 많아지면 카테고리를 더 좁히거나 모니터링 화면으로 분리한다.

페이지를 새로 만들 때는 세 템플릿 중 어느 쪽인지 먼저 정한다. 셋 중 어느 쪽도 아닌 페이지는 만들지 않는다.

## 13. 결론 박스는 결정, 액션 여부, 액션 내용을 모두 담는다

3번 규칙은 작은 요약 박스를 피하라고 했지만, 박스를 둘 거라면 다음 셋이 모두 있어야 한다.

- 무엇을 결정했는가 또는 결정해야 하는가
- 사람이 지금 액션을 해야 하는가 아닌가
- 액션이 필요하다면 정확히 무엇인가

나쁜 예:

```text
cellcleanerreel
음식물 과민증 전환캠페인으로 매핑. 자신감 93%.
```

좋은 예:

```text
검토 필요 1건
cellcleanerreel로 들어온 4건(1,060,200원)을 음식물 과민증 전환캠페인에 붙일지 결정해야 합니다.
[승인하고 ROAS에 반영] 버튼
```

셋 중 하나라도 빠지면 박스를 빼고 본문 카드로 흡수한다.

## 14. 신뢰도는 퍼센트가 아니라 근거의 종류와 운영 사용 가능 여부로 표시한다

"자신감 93%"는 운영자에게 두 질문을 남긴다. "93%가 어디서 나왔는가", "93%면 지금 써도 되는가". 첫 줄에는 다음 셋 중 하나를 쓴다.

**근거의 종류로**

```text
강한 증거: 직접 ID 일치 2건, 이름 일치 1건
보조 증거: 상품군 일치만 있음
```

**운영 사용 가능 여부로**

```text
바로 ROAS에 반영 가능
사람 검토 후 반영
ROAS에 반영하지 않음
```

**위험 등급으로**

```text
안전: ID 직접 증거가 있어 자동 반영함
주의: 보조 증거뿐이므로 사람 컨펌 후 반영
위험: 증거 부족, 별도 bucket으로 분리
```

퍼센트는 보조 표시로 카드 우상단에 둘 수 있지만, 첫 줄에 와서는 안 된다.

## 15. 시스템이 처리한 결과와 사람이 할 액션을 시각적으로 분리한다

8번 규칙은 "동사로 끝낸다"였지만, 운영자가 헷갈리는 지점은 "이건 시스템이 이미 한 거고 나는 보면 되는가, 내가 클릭을 해야 하는가"다. 셋을 다른 시각 처리로 분리한다.

- **이미 처리된 결과(통보)**: 회색 또는 무채색 카드, "처리됨" 라벨, 완료형 동사("매핑했습니다")
- **사람 액션 필요(요청)**: 강조 색 카드, 명령형 동사("승인하세요", "검토하세요"), 액션 버튼 동봉
- **자동 처리 예정(예약)**: 노란색 또는 옅은 강조, "12시간 후 자동 반영" 같은 시간 표시

같은 톤으로 두 종류가 섞이면 운영자는 매번 카드 본문을 끝까지 읽어야 행동을 결정할 수 있다.

## 16. 점진적 공개로 카드 높이를 일정하게 유지한다

근거 5개, 원본 ID 3개, 메타데이터까지 모두 펼쳐 두면 카드 하나가 화면 절반을 차지한다. 다음 기준으로 접고 펼친다.

**항상 펼침**: 사람이 읽는 이름, 판정, 다음 액션, 가장 강한 근거 1개

**기본 접힘**: 추가 근거(2번째 이후), 원본 ID, 디버깅용 메타데이터, 과거 변경 이력

펼치기 토글 문구는 "근거 자세히 보기", "원본 데이터 확인", "변경 이력"처럼 동사형으로 쓴다. "+", "더보기" 같은 모호한 표현은 쓰지 않는다.

한 카드를 펼친 상태가 다른 카드에 전파되지 않게 한다. 운영자가 의도적으로 연 카드만 열려 있어야 한다.

## 17. 빈 상태와 에러 상태도 일관된 형식으로 보여준다

운영자가 가장 헷갈리는 순간은 "정상 케이스가 없을 때"다. "데이터 없음"만 적고 끝내지 않는다. 무엇 때문에 비었는지, 그래서 어떻게 해야 하는지를 같이 쓴다.

**후보 0개**

```text
Meta 후보가 없습니다.
가능한 원인 3가지:
- 광고가 삭제됨
- 비Meta 유입(예: 인플루언서, 자연 검색)
- UTM 규칙 누락
다음 액션: 강제 매핑하지 말고 non-meta IGG bucket으로 분리합니다.
```

**숫자 충돌**

```text
audit 합계 245,000원, 로컬 원장 합계 250,000원.
차이 5,000원이 발생했습니다.
원인 추정: audit 생성 후 추가된 1건이 원장에 들어옴.
어느 값을 쓸지: 최신 결정에는 로컬 원장, 과거 보고서 비교에는 audit을 사용합니다.
```

**API 응답 실패**

```text
Meta API 응답 없음(마지막 시도: 2026-04-28 10:32).
캐시된 직전 데이터를 사용합니다(기준: 2026-04-27 23:00).
새로 시도: [재시도] 버튼
```

## 18. 상태 구분은 색 외에 텍스트 또는 모양도 함께 쓴다

7번 규칙의 "확정, 임시, 추정, 제외" 구분이 색에만 의존하면 색맹 사용자, 흑백 인쇄, 어두운 화면에서 사라진다. 색, 텍스트 라벨, 모양 중 두 가지 이상을 항상 동시에 쓴다.

- **색**: 초록(확정), 노랑(임시), 주황(추정), 회색(제외)
- **텍스트 라벨**: "확정", "임시", "추정", "제외"
- **모양 또는 아이콘**: ●(확정), ◐(임시), ◯(추정), ✕(제외)

좋은 예:

```text
● 확정    cellcleanerreel
◐ 임시    inpork
✕ 제외    songyuul08
```

배지를 작게 만들수록 색만으로 구분하기 어려우므로, 작은 배지일수록 텍스트 라벨을 함께 쓴다.

## 19. 상세 근거를 보기 전에는 결정을 요구하지 않는다

상단 카드가 운영자에게 첫 번째로 보이는 영역이라면 질문형 승인 문구를 먼저 두지 않는다. 사용자가 아직 근거, 삽입 위치, 영향 범위, 롤백 기준을 보지 못한 상태에서 `YES/NO`, `진행할까요?`, `결정합니다` 같은 문구를 보면 압박으로 읽힌다.

상단은 다음 형식으로 쓴다.

```text
상태
승인안 D 실행 패키지 준비 완료.

안내
상세 구성품과 삽입 방법은 아래 섹션에 정리했습니다.

[아래 상세 정보 보기]
```

결정 요청 버튼이나 답변 복사 버튼은 상세 근거를 본 뒤 나오는 섹션에 둔다. 단, 이미 사용자가 이전 화면에서 근거를 모두 확인했고 이 화면이 순수 승인 게이트라면 상단에 결정 CTA를 둘 수 있다. 그때도 `왜 지금 결정이 필요한지`, `YES/NO 영향`, `게시 전 마지막 확인 여부`를 함께 적는다.

## 19. 숫자, 날짜, 통화 포매팅을 한 화면에서 통일한다

같은 화면에 "245,000원"과 "₩245,000"이 섞이면 보고서의 신뢰도가 무너진다. 다음 규칙을 따른다.

- **통화**: 항상 "245,000원"(원 표기 뒤). "₩"와 "원"을 섞어 쓰지 않는다.
- **천단위 구분**: 쉼표(245,000). 1000 미만은 구분자 없음.
- **날짜**: 2026-04-25 형식. "2026. 04. 11.", "10/14", "26/4/25" 같은 형식을 같이 쓰지 않는다.
- **시각**: 2026-04-25 14:30 (24시간제). "오후 2시 30분"은 사용하지 않는다.
- **비율**: 93%(소수점 없이) 또는 93.5%(소수 한 자리). 한 화면에서 자릿수를 통일한다.
- **기간**: "2026-03-11부터 2026-04-11까지"처럼 "부터-까지"를 명시한다. 물결표나 단순 대시로 기간을 줄여 쓰지 않는다.

날짜 캐싱 시각, 보고 기준 시각, 갱신 시각이 같은 화면에 나올 때는 셋 다 같은 형식을 쓴다.

## 20. 화면의 청중은 처음 보는 운영자 1명이다

화면을 만들 때 마음속의 청중은 "내가 만든 시스템을 처음 보는 운영자"다. 이 사람은 alias가 무엇인지 모르고, IGG가 무엇의 약자인지 모르며, 왜 ROAS가 부풀면 안 되는지 모른다. 화면이 이 사람에게 작동하면 베테랑에게도 작동한다. 반대는 성립하지 않는다.

정의 없는 내부 약어 두 개 이상이 한 화면에 등장하면 그 화면은 실패다. 정의는 부록 용어 사전을 참조하되, 화면 안에서 처음 등장할 때는 한 문장 정의를 인라인으로 붙인다.

---

## 부록 A. 운영 용어 사전

화면이나 본문에서 처음 등장할 때 한 줄 정의를 인라인으로 붙이고, 자세한 정의는 이 부록을 참조한다. 새 용어가 코드나 화면에 추가되면 이 부록에 함께 추가한다.

- **alias**: 광고 링크에 붙어 주문 원장에 남는 식별자. 사람이 만든 별명.
- **utm_campaign**: URL에 들어가는 캠페인 식별 파라미터. alias의 표준화된 위치.
- **campaign_id**: Meta가 캠페인에 부여하는 숫자 ID. 절대 식별자.
- **adset_id**: Meta가 광고세트에 부여하는 숫자 ID. campaign_id의 하위.
- **ROAS**: Return On Ad Spend. 광고비 대비 매출. 캠페인별 효율 측정 지표.
- **attribution_ledger**: 주문이 어느 캠페인에 귀속되는지 기록하는 내부 원장.
- **quarantine**: 자동 분류가 위험한 항목을 일단 격리해두는 상태. 사람 컨펌 후 풀린다.
- **manual_verified**: 사람이 직접 확인하고 확정한 분류 상태.
- **seed**: 자동 매핑 규칙이 닿지 않을 때 수동으로 채워 넣는 기준값 모음.
- **IGG**: 면역글로불린 G(Immunoglobulin G) 기반 음식물 과민증 검사 상품군 약어.
- **non-meta IGG bucket**: Meta 광고가 아닌 경로(인플루언서, 자연 유입)로 들어온 IGG 매출을 분리해 담는 카테고리.
- **audit 파일**: 정기적으로 스냅샷을 떠 두는 검증용 파일. 로컬 원장보다 시점이 과거일 수 있다.

## 부록B. 구현 전 체크리스트

- raw key, alias, 숫자 ID가 제목으로 먼저 나오지 않는가?
- 처음 보는 사람이 10초 안에 "이 화면이 뭘 판단하는지" 말할 수 있는가?
- 모든 주요 숫자에 기준 시각과 출처가 있는가?
- 결론 카드마다 근거와 다음 액션이 있는가?
- "확정", "임시", "추정", "제외"가 구분되는가?
- 개발자용 원본값은 보이되, 사람이 읽는 설명 아래에 배치했는가?
- 모바일에서 긴 원본값 때문에 가로 스크롤이 생기지 않는가?
-  화면 상단의 가장 큰 글씨가 "지금 해야 할 액션 또는 화면의 질문"인가?
- 페이지가 판단형 / 대시보드형 / 카테고리(혼합)형 표준 레이아웃 중 어느 쪽인지 명확한가?
- 카테고리형이라면 KPI 카드를 3~6개로 압축했고, 자세한 모니터링은 별도 대시보드 화면으로 링크했는가?
- 시스템이 처리한 항목과 사람이 액션해야 할 항목이 시각적으로 구분되는가?
- 결론 박스에 "결정, 액션 여부, 액션 내용" 셋이 모두 있는가?
- 상세 근거를 보기 전에 상단에서 YES/NO 결정을 요구하고 있지는 않은가?
- 신뢰도를 퍼센트로만 표시하지 않았는가?
- 상태 배지가 색 외에 텍스트 또는 모양으로도 구분되는가?
- 빈 상태와 에러 상태에 원인과 다음 액션이 적혀 있는가?
- 통화, 날짜, 시각, 비율 형식이 한 화면에서 통일되는가?
- 카드 높이가 펼침 토글로 조절 가능한가?
- 정의 없는 내부 약어가 한 화면에 두 개 이상 있지는 않은가?

이 문서는 보고서형 페이지를 새로 만들거나 고칠 때 먼저 확인한다.
