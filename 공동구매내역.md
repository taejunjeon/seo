## 10초 요약

2026년 1~3월 바이오컴 공동구매성 주문은 **2026-04-21 23:05 KST 아랑 4차 후보 allowlist 88건 추가 후 1,923건, 593,316,365원**이다. 같은 기간 전체 주문 8,058건, 2,280,617,219원 중 주문수 23.9%, 매출 26.0%를 차지한다. 회사 공유본 `data/공동구매_26년 1분기.xlsx` 기준(2,253건, 645,941,680원)과 비교하면 **주문수 85.3%, 매출 91.9%**를 잡았다(백필 전 79.4% → 1월 헤더 백필 후 87.5% → 아랑 4차 allowlist 후 91.9%).

이 작업의 대부분이 닫혔다. `/ads` 공동구매 캠페인명 분리 v1 live, GA4 가상계좌 미입금 purchase 오염 차단(GTM v137) 완료, 1월 1-7일 주문 헤더 복구 713건 완료, 아랑 4차 후보 allowlist 88건 `data/coop_campaigns.biocom.json`에 반영 완료. 남은 작업은 (1) 공동구매 운영 마스터 3 → 25개 전체 확장, (2) 세트 상품 언분해 실물 파일, (3) `/ads` 주문 단위 분리 v2 엔드포인트, (4) 아랑 4차 회사 기준 326건 중 나머지 238건 확인(회사 엑셀에서 주문번호 공유 필요).

현재 로컬 원천으로는 상품명 `[파트너/공구/차수]` 패턴 외에 공동구매를 자동 식별할 신호가 없다. 쿠폰, UTM, 랜딩, 주문 건수 급증 어느 것도 Q1 공동구매 주문을 단 1건도 자체 식별하지 못한다. 앞으로 엑셀 없이 자동 식별하려면 공구 오픈 **전에** 전용 쿠폰 · UTM · 랜딩을 심는 운영 규칙이 필수다.

## 한눈에 보기 (고등학생 비유)

공동구매는 학교 동아리가 외부 판매자와 단체로 한 번 싸게 사는 것과 같다. 회사 영수증(아임웹 주문 원장)에는 "누가 얼마에 샀다"만 있고, 어느 동아리 단체구매였는지는 영수증만 봐서는 모른다. 지금까지는 상품 이름에 `[아무 동아리]`라고 적혀 있으면 그 동아리 구매로 쳤는데, 어떤 동아리는 이름을 안 적고 일반 상품으로 그냥 팔아버렸다. 그러면 영수증만 보고는 단체구매였는지 알 길이 없다.

해결책은 두 가지다. 앞으로는 동아리마다 "우리 동아리 쿠폰" 또는 "우리 동아리 링크"를 써야 하고, 동아리장이 운영 마스터에 동아리 이름/기간/상품/쿠폰 정보를 먼저 등록해야 한다. 과거 데이터는 이미 늦었으니 회사가 가진 동아리 회비 장부(엑셀)와 주문번호를 수동으로 맞춘다.

## 문서 목적

이 문서는 2026년 1~3월 바이오컴 공동구매가 실제 주문과 매출에서 어느 정도 비중이었는지 확인하고, 앞으로 엑셀 없이 자동으로 공동구매를 식별할 체계를 설계하는 기준 문서다. 고객명, 전화번호, 이메일 같은 개인정보는 다루지 않는다.

## 이 작업이 하는 일

아임웹 주문 헤더(`imweb_orders`)와 PlayAuto 상품 라인(`imweb_order_items`)을 연결해서 공동구매성 주문을 월별·파트너별·차수별로 정리한다. 이 정리가 끝나면 `/ads` 화면에서 일반 광고 ROAS와 공동구매 매출이 섞이지 않게 분리해서 보여줄 수 있다.

## 왜 필요한가

공동구매는 바이오컴 매출의 22.7%, 24.6%를 차지하는 큰 축이지만 현재 `/ads` 화면은 Meta 광고 ROAS에 공동구매 매출이 섞여 들어가 일반 광고 효율을 과대평가한다. 또 GA4 `(not set)` 구매의 상당 부분도 공동구매 외부 유입이 섞여 있어 해석이 흔들린다. 공동구매를 분리해야 광고비 판단 기준이 바닥부터 흔들리지 않는다.

## 집계 기준

원천은 로컬 SQLite 파일 `backend/data/crm.sqlite3`이다.

사용 테이블:

| 테이블 | 사용 이유 |
|---|---|
| `imweb_orders` | 주문 헤더, 주문일, 주문 결제금액 확인 |
| `imweb_order_items` | 상품명(item_name), 수량, 공동구매 상품명 확인 |

집계 기준:

| 항목 | 기준 |
|---|---|
| 사이트 | `site='biocom'` |
| 기간 | `2026-01-01 <= 주문일 < 2026-04-01` |
| 금액 | `imweb_orders.payment_amount` |
| 상품명 | `imweb_order_items.item_name` |
| 넓은 공동구매 기준 | 상품명이 `[`로 시작하고 `정기구독`, `비밀링크`, `KOLAS 인정 획득 기념`을 제외한 파트너/공구성 상품 |
| 좁은 공동구매 기준 | 상품명에 `공구` 또는 `공동구매`가 직접 들어간 상품 |

금액은 상품 라인아이템 금액이 아니라 주문 헤더 금액이다. `imweb_order_items.pay_amt`는 모두 0원으로 들어와 있어 상품 라인별 순매출은 바로 계산할 수 없다. 파트너/차수 단위 집계에서는 같은 주문에 여러 공동구매 상품이 들어 있어도 주문을 한 번만 센다.

## 현재 상태

### 확인된 것

- 2026-01-07 이후 주문 헤더와 상품 라인은 대부분 조인된다.
- 2026년 1~3월 공동구매성 주문은 최소 1,668건, 매출은 최소 512,906,191원이다.
- 전체 주문 대비 공동구매성 주문 매출 비중은 24.6%다.
- 회사 공유본 기준 공동구매 매출 645,941,680원 중 로컬은 79.4%(512,906,191원)를 포착했다.
- `/ads` 화면에는 공동구매 캠페인명 기준 분리(v1)가 live 운영 중이다.
- GA4 가상계좌 미입금 purchase 오염 차단은 2026-04-21 01:40 KST GTM v137 publish로 완료됐다.
- 공동구매 운영 마스터 파일 `data/coop_campaigns.biocom.json` 초안(3개 레코드)이 생성됐다.

### 아직 안 된 것

- 2026-01-01 ~ 2026-01-07 주문 헤더는 로컬 `imweb_orders`에 없다. 운영 DB `tb_iamweb_users`에서 복구해야 한다.
- 라인아이템별 실제 결제금액은 없다. 주문 전체 결제금액을 붙인 값이다.
- 회사 공유본의 아랑 4차 326건/81,923,940원은 로컬 상품명 기준으로 자동 분류되지 않는다.
- 공동구매 운영 마스터 3개 레코드의 `order_no_allowlist`가 비어 있어 과거 주문번호 역매핑이 남아 있다.
- 세트 상품 언분해 마스터(`data/product_sets.biocom.json`)는 설계만 있고 실물 파일이 없다.
- `/ads` 주문 단위 분리 v2 엔드포인트(`/api/ads/coop-order-summary`)는 미구현이다.
- `imweb_issue_coupons` 테이블의 `use_date` 싱크 범위가 2025-10-24까지이고 2026년 이후가 비어 있어 쿠폰 classification 기반 조회가 막혀 있다.

### 지금 가장 큰 병목

첫째, 1월 1-7일 주문 헤더 누락이 Q1 공동구매 대사의 가장 큰 금액 격차를 만든다. 운영 DB 컬럼명 확인 후 백필을 실행하면 약 133,035,489원의 격차 중 67,722,100원이 닫힌다.

둘째, 아랑 4차 326건/81,923,940원은 상품명/쿠폰/UTM 어느 것도 자동 신호가 없어 운영 마스터 `order_no_allowlist` 역매핑 외에는 잡히지 않는다.

셋째, 공구 오픈 전에 전용 쿠폰/UTM/랜딩을 심는 운영 규칙이 아직 도입되지 않았다. 이 규칙 없이는 4월 이후 공구도 과거와 같은 문제를 반복한다.

## 역할 구분 (문서 전체)

현재 Codex가 불안정하여 이 문서에 등장하는 모든 코드·스크립트·문서 작업은 Claude Code가 수행한다.

- **Claude Code**: 운영 DB `tb_iamweb_users` read-only 쿼리, 로컬 SQLite 집계, 마스터 JSON 생성/편집, 백엔드 helper 함수, 프론트 UI 수정, 드라이런 스크립트 작성, 문서 작성·검증.

#### 자동 분류 규칙 (OR 조건, 가장 강한 신호 채택)

```text
for each order in imweb_orders (site='biocom', order_time in range):
  for each campaign in coop_campaigns.biocom.json:
    if order.order_no in campaign.order_no_allowlist:
      match(order, campaign, basis='allowlist', confidence=1.0)

    else if order.use_issue_coupon_codes ∩ issue_codes(campaign.coupon_codes) != ∅:
      match(order, campaign, basis='coupon', confidence=0.95)

    else if attribution_ledger(order).utm matches campaign.utm_*:
      match(order, campaign, basis='utm', confidence=0.9)

    else if attribution_ledger(order).landing_path matches campaign.landing_path_pattern:
      match(order, campaign, basis='landing', confidence=0.85)

    else if order.order_time in [start_date, end_date]
         and order.item_name in campaign.product_families
         and (campaign.has_coupon == false):
      match(order, campaign, basis='time+product', confidence=0.6, needs_review=true)

unmatched orders with '[' prefix product name → review_queue
```

| 신호 | 신뢰도 | 이유 |
|---|---:|---|

#### 분해 로직 의사코드

`imweb_order_items` 원장은 건드리지 않는다. 분석 뷰에서 세트 마스터를 JOIN해서 구성 단품으로 펼친다.

```text
for each row in imweb_order_items:
  set_master = match_product_set(row.item_name)   -- item_name 정확 일치 또는 alias_patterns LIKE 매칭
  if set_master is None:
    yield row as-is  (단품 그대로)
  else:
    for component in set_master.components:
      yield exploded_row(
        order_no = row.order_no,
        line_no  = row.line_no + ':' + component.sku,
        item_name_original = row.item_name,
        sku       = component.sku,
        item_name = component.name,
        sale_cnt  = row.sale_cnt * component.qty_per_set,
        source    = 'set_expanded',
        revenue_share_hint = pro_rata_by_standalone_price(component, set_master)
      )
```

매출 배분 방식 비교:

| 방식 | 장점 | 단점 |
|---|---|---|
| `qty` 기반 집계만 유지 | 왜곡 없음 | 상품별 매출 합계 계산 불가 |
| 세트 정가의 구성별 정가 비율로 배분 | 상품별 매출 추정 가능 | 세트 할인율만큼 구성 매출 합이 세트 판매가와 다르게 배분됨. 단품 정가 필요 |
| 주문 헤더 `payment_amount`를 세트 수량에 비례 배분 | 전체 매출 보존 | 같은 주문에 단품+세트가 섞이면 부정확 |

운영 기본안 제안: `qty` 기반 집계는 즉시 도입, `revenue_share`는 단품 정가 수집 후 2차 도입.

### 월별 전체 대비 공동구매성 비중 쿼리

```sql
WITH all_orders AS (
  SELECT
    substr(order_time,1,7) AS month,
    COUNT(DISTINCT order_no) AS all_orders,
    SUM(payment_amount) AS all_payment
  FROM imweb_orders
  WHERE site='biocom'
    AND order_time >= '2026-01-01'
    AND order_time < '2026-04-01'
  GROUP BY month
),
offer_items AS (
  SELECT DISTINCT order_no
  FROM imweb_order_items
  WHERE site='biocom'
    AND ord_time >= '2026-01-01'
    AND ord_time < '2026-04-01'
    AND item_name LIKE '[%'
    AND item_name NOT LIKE '[정기구독]%'
    AND item_name NOT LIKE '[비밀링크]%'
    AND item_name NOT LIKE '[KOLAS%'
),
offer_orders AS (
  SELECT
    substr(o.order_time,1,7) AS month,
    COUNT(DISTINCT o.order_no) AS offer_orders,
    SUM(o.payment_amount) AS offer_payment
  FROM offer_items oi
  JOIN imweb_orders o
    ON o.site='biocom'
   AND o.order_no=oi.order_no
  WHERE o.order_time >= '2026-01-01'
    AND o.order_time < '2026-04-01'
  GROUP BY month
)
SELECT
  a.month,
  a.all_orders,
  a.all_payment,
  COALESCE(g.offer_orders,0) AS groupbuy_orders,
  COALESCE(g.offer_payment,0) AS groupbuy_payment,
  ROUND(100.0*COALESCE(g.offer_orders,0)/a.all_orders,1) AS order_share_pct,
  ROUND(100.0*COALESCE(g.offer_payment,0)/a.all_payment,1) AS revenue_share_pct
FROM all_orders a
LEFT JOIN offer_orders g
  ON g.month=a.month
ORDER BY a.month;
```

### 문서 작성 판단

이 문서는 `공동구매`를 회계 계정으로 확정하는 문서가 아니다. 현재 주문 원장에서 관찰되는 공동구매성 판매 흐름을 정리한 운영 기준 문서다. 회계 정산용으로 쓰려면 [[#Phase1-Sprint3]] 백필과 상품별 실결제금액 확보가 먼저 필요하다.
