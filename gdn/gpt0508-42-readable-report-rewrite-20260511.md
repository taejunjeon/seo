# gpt0508-42 결과보고 리라이트 (gpt0508-43 작업1, 사람 말 우선)

작성 시각: 2026-05-11 17:10:00 KST
원본: `gptconfirm/gpt0508-42/00-result-report.md` (개발 용어 중심)
본 문서: 동일 sprint 결과를 사람 말 5 필드로 다시 적음.

## 1. 이번에 가능해진 것

고객이 우리 사이트 (biocom.kr) 에 들어와서 광고 클릭으로 시작했든, 결제 단계에 진입했든, 결제를 완료했든 — **그 유입 흔적이 자동으로 고객 유입 장부 (site_landing_ledger) 에 한꺼번에 적히는 연결** 이 backend 안에 끝났다. 또 그 장부의 분포 (organic / paid / direct / referral 비율 등) 를 화면이 읽을 수 있는 조회 기능 (`GET /api/attribution/site-landing/summary`) 으로 만들었고, 그 결과를 보여주는 간단한 페이지 (`/ads/site-landing`, 68 줄) 도 추가했다.

## 2. 왜 필요했는지

직전 sprint (gpt0508-41) 까지는 "유입 장부" 라는 구조만 만들어 두고 실제로는 빈 상태였다. 이번 sprint 의 핵심은 그 빈 장부에 **실제 트래픽이 흘러 들어오는 연결점** 4 곳 — 광고/마케팅 유입 신호 (marketing-intent) / 결제 단계 진입 신호 (checkout-context) / 결제완료 신호 (payment-success) / 유료 광고 클릭 신호 (paid-click-intent) — 을 박는 것. 이게 끝나야 다음 sprint 에서 "실제로 organic 이 몇 %, paid 가 몇 %, direct 가 몇 % 들어오는지" 사업적으로 의미 있는 비율을 우리 자체 DB 로 측정할 수 있다.

## 3. 어떻게 작동하는지 (비개발자용)

지금까지는 광고 클릭 / 결제 완료 같은 시점에 서버가 알아도 그 정보를 흘려보내고, 광고 ROI 계산할 때만 따로 끄집어내야 했다. 이번 변경으로 그 4 곳의 서버 입구 (backend handler) 가 받은 신호를 **본래 가던 길로도 보내고, 동시에 새 고객 유입 장부에도 한 줄씩 기록** 하도록 연결됐다. 광고 클릭 ID 처럼 민감한 값은 평문 그대로 저장하지 않고 단방향 hash 로 변환해 저장한다. 이 장부는 외부 광고 플랫폼에 절대 전송되지 않고, 우리 분석 용도로만 쓴다.

## 4. 실제로 확인된 결과

- 테스트 데이터 14 개 시나리오에서 모두 정상 동작 — 광고 UTM 이 들어왔을 때 paid 채널로 분류 / 네이버 검색 referrer 가 들어왔을 때 organic_search 로 분류 / 자기 도메인 referrer 일 때 직접 방문 (direct) 로 분류 / 광고 click ID 가 들어왔을 때 hash 만 저장되고 평문은 어디에도 안 남음 / 같은 세션 + 같은 URL + 10 분 안에 두 번째 들어오면 중복 처리.
- 화면 페이지의 코드 줄 수 68 줄로 권장 한도 (80) 안.
- 채팅 텍스트 보고 양식이 다음 sprint (gpt0508-43) 부터 "사람 말 우선" 으로 바뀌도록 규칙 정본도 같은 sprint 에서 갱신.

## 5. 아직 안 된 것

- **실제 운영 트래픽 배포는 안 됨** — backend 코드 변경은 main 에 커밋 (`8121bfd`) 까지 갔지만, VM (`34.64.104.94`) 에서 pull + pm2 재시작이 아직 안 일어남. 이 단계가 끝나야 실측 분포가 쌓인다.
- 광고가 아닌 일반 페이지 방문 (organic page_view) 은 아직 우리 backend 가 받는 신호가 없어서 장부에 안 들어옴. 이건 다음 sprint 의 GTM Preview 작업 후보.
- naver 광고 클릭 ID (`nclick_id`) 캡쳐는 footer 코드에서 아직 안 잡힘.

## 6. 사업적으로 의미

다음 단계 deploy 가 끝나면, 우리는 처음으로 "이번 주 유입 중 광고가 몇 %, 자연 검색이 몇 %, 직접 방문이 몇 %" 같은 비율을 **외부 분석 도구 (GA4 등) 의존 없이 우리 자체 DB 로** 산출할 수 있게 된다. 그리고 그 결과를 같은 sessionKey 로 주문 / 결제 데이터와 join 가능해, "이 광고 캠페인이 실제로 결제까지 이어졌는가" 를 우리 측 데이터로 직접 검증할 기반이 생긴다. campaign_id exact 같은 더 정밀한 분석은 그 다음 sprint 의 2차 목표.
