네, **Google도 Meta CAPI처럼 “최종 구매 전 단계 신호를 함께 학습/최적화에 활용하는 방법”이 있습니다.**
다만 Meta CAPI와 **완전히 같은 제품명/구조는 아니고**, Google에서는 여러 기능을 조합해서 구현합니다.

한 줄로 말하면:

> **Google도 중간 전환을 학습 신호로 쓸 수 있습니다.**
> 하지만 `begin_checkout`, `add_payment_info`, `NPay click`, `scroll`, `engaged_view` 같은 앞단계 이벤트를 무작정 Primary 전환으로 넣으면 Google이 “구매자”가 아니라 “클릭/탐색/결제시작자”를 학습할 수 있습니다.
> 그래서 **실제 구매는 Primary**, **중간 퍼널은 Secondary/관찰용 또는 별도 실험용 goal**로 시작하는 게 안전합니다.

## 1. Google에서 Meta CAPI와 비슷한 기능은 뭐가 있나?

Meta CAPI와 1:1로 같은 건 아니지만, 목적별로 보면 아래가 대응됩니다.

|목적|Meta 쪽|Google 쪽 대응|
|---|---|---|
|서버에서 실제 구매 전송|Meta CAPI Purchase|Google Ads Offline Conversion Import / Data Manager / Google Ads API conversion upload|
|브라우저+서버 이벤트 매칭|Pixel + CAPI dedup|Google tag + Enhanced Conversions / Offline conversions / server-side tagging|
|구매 전 퍼널 이벤트 수집|ViewContent, AddToCart, InitiateCheckout, AddPaymentInfo|GA4 events / Google Ads conversion actions / key events|
|측정 품질 보강|CAPI + Advanced Matching|Enhanced Conversions, Google tag gateway, server-side tagging|
|입찰 최적화|Meta event optimization|Google Ads conversion goals + Primary conversion actions + Smart Bidding|

Google Ads의 Smart Bidding은 Google AI를 사용해 전환 또는 전환 가치를 기준으로 입찰을 최적화합니다. Target CPA, Target ROAS, Maximize conversions, Maximize conversion value가 모두 Smart Bidding 전략입니다. ([구글 도움말](https://support.google.com/google-ads/answer/7065882?hl=en&utm_source=chatgpt.com "About Smart Bidding - Google Ads Help"))

---

## 2. Google도 중간전환을 머신러닝에 쓸 수 있나?

**가능합니다.**
Google Ads에서는 전환 액션을 만들고, 그 액션을 **Primary**로 둘지 **Secondary**로 둘지 정할 수 있습니다.

중요한 차이는 이겁니다.

|구분|의미|
|---|---|
|**Primary conversion action**|`Conversions` 컬럼에 잡히고, 해당 goal을 쓰는 캠페인에서 입찰 최적화에 사용될 수 있음|
|**Secondary conversion action**|관찰용. 보통 `All conversions`에만 잡히고 입찰에는 쓰이지 않음|

Google 공식 문서도 Primary action은 `Conversions` 컬럼에 보고되고 bidding에 사용될 수 있으며, Secondary action은 관찰용이고 `All conversions`에 보고되지만 bidding에는 쓰이지 않는다고 설명합니다. ([구글 도움말](https://support.google.com/google-ads/answer/11461796?hl=en&utm_source=chatgpt.com "About primary and secondary conversion actions"))

즉, Google에서도 아래 이벤트를 전환 액션으로 만들 수 있습니다.

```text
view_item
add_to_cart
begin_checkout
add_payment_info
npay_click
qualified_product_view
engaged_view
```

하지만 이걸 **Primary**로 넣으면 Google이 그 행동을 최적화 대상으로 학습합니다.

그래서 조심해야 합니다.

---

## 3. 그러면 중간전환을 Primary로 넣으면 좋은가?

대부분의 경우 **바로 Primary로 넣는 건 비추천**입니다.

예를 들어 `NPay click`을 Primary로 두면 Google은 “실제 구매자”가 아니라 “NPay 버튼을 누르는 사람”을 좋은 전환으로 학습할 수 있습니다. 지금 우리가 겪은 Google Ads ROAS 오염 문제가 바로 이 계열입니다.

마찬가지로:

|이벤트|Primary로 바로 넣을 때 위험|
|---|---|
|`scroll_50`|오래 읽는 사람을 학습하지만 구매자는 아닐 수 있음|
|`engaged_view`|콘텐츠 소비자를 학습할 수 있음|
|`add_to_cart`|장바구니만 담고 구매 안 하는 사람을 학습할 수 있음|
|`begin_checkout`|결제 시작만 하고 이탈하는 사람을 학습할 수 있음|
|`NPay click`|실제 결제가 아니라 버튼 클릭자를 구매자로 학습할 수 있음|

Google Ads에서 Maximize conversion value나 Target ROAS는 설정한 conversion value를 기준으로 최적화합니다. Google 공식 문서도 Maximize conversion value는 전환 가치를 최대화하도록 입찰하고, Target ROAS는 목표 ROAS에 맞춰 전환 가치를 최대화한다고 설명합니다. ([구글 도움말](https://support.google.com/google-ads/answer/7684216?hl=en&utm_source=chatgpt.com "About Maximize conversion value bidding"))

따라서 중간 이벤트에 높은 value를 주거나 Primary로 넣으면, Google이 그 신호를 진짜 사업 목표처럼 학습할 수 있습니다.

## 4. 그럼 어떻게 쓰는 게 안전한가?

추천 구조는 이렇습니다.

### 1단계: 실제 구매만 Primary

Google Ads의 Primary purchase는 아래만 써야 합니다.

```text
홈페이지 실제 결제완료
NPay 실제 결제완료
구독 실제 결제완료
가상계좌 실제 입금확정
```

즉, 내부 결제완료 원장 기준 `BI confirmed_purchase`를 만드는 게 핵심입니다.

Google Ads API의 offline conversion upload는 conversion action ID, conversion value, conversion time과 함께 `GCLID`, `GBRAID`, `WBRAID` 중 하나를 제공해야 한다고 설명합니다. ([Google for Developers](https://developers.google.com/google-ads/api/samples/upload-offline-conversion?utm_source=chatgpt.com "Upload offline conversion | Google Ads API"))

그래서 지금 하고 있는 `paid_click_intent ledger write`는 이 구조의 선행 작업입니다. 실제 구매를 Google에 보내려면 먼저 Google click id가 주문까지 살아남아야 하니까요.

---

### 2단계: 중간 퍼널은 Secondary / 관찰용

아래는 처음엔 Secondary로 두는 게 안전합니다.

```text
add_to_cart
begin_checkout
add_payment_info
NPay click
qualified_product_view
engaged_view
```

이렇게 하면 Google Ads 화면에서 관찰은 가능하지만, 기본적으로 입찰이 바로 그 이벤트를 최적화하지는 않습니다. Google 문서도 Secondary action은 관찰용이고 `All conversions`에 보고되지만 bidding에는 쓰이지 않는다고 설명합니다. ([구글 도움말](https://support.google.com/google-ads/answer/3419678?hl=en&utm_source=chatgpt.com "About \"All conversions\" - Google Ads Help"))

---

### 3단계: 구매와 상관관계가 검증된 중간 이벤트만 실험

예를 들어 BigQuery로 분석해보니:

```text
begin_checkout 후 24시간 내 실제 구매율이 높다
add_payment_info 후 실제 구매 전환율이 높다
qualified_product_view가 실제 구매와 강한 상관이 있다
```

이런 결과가 나오면, 그때 일부 캠페인에서 custom goal 또는 실험용 conversion goal로 테스트할 수 있습니다.

하지만 Google 공식 문서는 새 conversion action을 기존 goal에 추가할 때 바로 Primary나 account default로 설정하지 말고, 먼저 올바르게 설정되고 값이 정상 보고되는지 확인하라고 권장합니다. ([구글 도움말](https://support.google.com/google-ads/answer/14593208?hl=en&utm_source=chatgpt.com "Change Smart Bidding strategies and change conversion ..."))

즉, 순서는:

```text
Secondary로 관찰
→ BigQuery로 실제 구매와 상관관계 확인
→ 작은 캠페인에서 실험
→ Primary 또는 custom goal 검토
```

입니다.

## 5. “Google CAPI”라고 부를 만한 건 뭐냐?

Google에는 Meta의 “Conversions API”와 같은 이름의 단일 제품은 없지만, 비슷한 역할은 아래 조합으로 합니다.

### A. Offline Conversion Import / Google Ads API

서버에서 실제 구매를 Google Ads에 업로드합니다.
이게 **Google Ads용 confirmed purchase**에 가장 가깝습니다.

사용 예:

```text
주문 원장에 실제 결제완료 발생
→ gclid/gbraid/wbraid 연결
→ conversion time/value/order_id 생성
→ Google Ads API 또는 Data Manager로 upload
```

### B. Enhanced Conversions

전환 페이지에서 수집한 first-party 고객 데이터를 SHA256으로 해시해 Google에 보내 매칭 정확도를 높입니다. Google 문서도 Enhanced conversions for web이 이메일, 이름, 주소, 전화번호 같은 first-party 데이터를 conversion tag에서 캡처하고 해시해 Google 계정과 매칭한다고 설명합니다. ([구글 도움말](https://support.google.com/google-ads/answer/15712870?hl=en&utm_source=chatgpt.com "About enhanced conversions for web - Google Ads Help"))

다만 개인정보/동의 이슈가 있으므로, 우리 프로젝트에서는 바로 켜기보다는 별도 법무/동의 검토가 필요합니다.

### C. Server-side Tagging

Google Tag Manager server-side container로 브라우저에서 여러 수집 서버로 직접 보내던 데이터를 서버 측에서 처리하고 라우팅할 수 있습니다. Google 문서도 server-side tagging이 서버 컨테이너를 통해 사용자 활동을 측정하고, page container가 여러 서버로 직접 보내던 구성을 바꾸는 방식이라고 설명합니다. ([Google for Developers](https://developers.google.com/tag-platform/tag-manager/server-side/intro?utm_source=chatgpt.com "An introduction to server-side tagging"))

### D. Google Tag Gateway

Google tag gateway는 Google tag를 first-party infrastructure, 예를 들어 CDN, load balancer, web server를 통해 배포하는 기능입니다. Google 문서도 이 기능이 Google tag를 웹사이트 도메인의 first-party infrastructure에서 배포하게 해주며, 기존 Google tag 또는 GTM container와 CDN/load balancer가 필요하다고 설명합니다. ([Google for Developers](https://developers.google.com/tag-platform/tag-manager/gateway/setup-guide?utm_source=chatgpt.com "Set up Google tag gateway for advertisers"))

다만 Google tag gateway는 **측정 품질 보강**이지, confirmed purchase 원장 전송이나 NPay click/count 오염 해결을 직접 대신하지 않습니다.
