- **의존성**: 다른 작업과 병행 가능. 단, 3번 승인이 없으면 임계값 상수 못 넣음.

#### 2. [TJ] 광고 효과 실험(iROAS live)에 쓸 캠페인 선정 + 홀드아웃 허용 범위 결정

- **의존성**: iROAS 실험 설계의 선행 필수. (실험 설계 자체는 Codex가 이어서 문서화)

### Sprint 1. CAPI/Purchase Guard clean baseline

무엇:

브라우저 Pixel, 서버 CAPI, Purchase Guard가 같은 주문을 같은 의미로 기록하게 만드는 작업이다.
쉽게 말하면, 한 주문에 대해 "아임웹 결제완료", "Meta Pixel Purchase", "서버 CAPI Purchase", "우리 Attribution 원장"이 서로 다른 말을 하지 않게 맞추는 것이다.

비유:

택배 송장과 같다. 물건은 하나인데 택배사 송장, 쇼핑몰 주문번호, 고객센터 주문번호가 서로 다르면 나중에 누가 샀는지, 배송됐는지, 반품됐는지 맞출 수 없다. 지금의 `event_id`, `order_code`, `order_no`, `paymentKey`가 바로 송장번호 역할을 한다. 이 번호들이 같은 주문을 같은 주문으로 가리켜야 Meta ROAS와 내부 ROAS를 비교할 수 있다.

왜:

가상계좌 미입금 주문이 Purchase로 잡히거나, 같은 주문이 서로 다른 `event_id`로 여러 번 서버 전송되거나, biocom 주문이 coffee source로 섞이면 Meta는 매출이 있었다고 보는데 내부 원장은 없다고 볼 수 있다. 그러면 "광고가 좋은가"가 아니라 "데이터가 서로 다른 주문을 보고 있는가"를 비교하게 된다.

어떻게:

1. VM CAPI 로그에서 최근 운영 결제만 본다.
   - `scope=recent_operational`
   - Meta 2xx 성공률
   - duplicate event_id
   - duplicate order/event group
   - 같은 주문+이벤트에 event_id가 여러 개인지 확인

2. Meta Events Manager에서 브라우저 Pixel과 서버 CAPI가 dedup되는지 본다.
   - 같은 `event_id`로 browser/server가 들어오는지
   - Event Match Quality가 너무 낮지 않은지
   - Purchase와 VirtualAccountIssued가 의도대로 분리되는지

3. clean baseline 날짜를 분리한다.
   - 바이오컴: 2026-04-13부터 교정된 POST 창으로 본다.
   - 더클린커피: 2026-04-15 CAPI/footer v3 배포, 2026-04-15 20:53 KST source-label 교정 이후부터 안정화 구간으로 본다.
   - 더클린커피의 운영 판단 기준일은 2026-04-16 이후 닫힌 날짜다.

Codex가 할 일:

- VM CAPI 로그를 조회해 `success/failure`, duplicate event/order group, 위험 주문 샘플을 정리한다.
- `/ads`에 clean baseline 기준일과 현재 해석을 표시한다.
- 위험 이벤트가 나오면 주문번호, event_id, pixel_id, source, send_path를 추적한다.

TJ님이 할 일:

- Meta Events Manager 화면에서 browser/server dedup 상태와 Event Match Quality를 확인한다.
- 운영 중인 광고비를 갑자기 크게 바꾸지 않고, clean baseline이 쌓일 때까지 유지한다.
- Events Manager에 경고 문구가 있으면 캡처를 공유한다.

### Sprint 2. Attribution identity coverage

무엇:

결제 시작과 결제완료가 같은 사용자/세션/주문으로 이어지도록 `ga_session_id`, `client_id`, `user_pseudo_id`를 자체 원장에 저장하는 작업이다.

비유:

매장에 손님이 들어올 때 입구에서 번호표를 줬는데, 계산대에서 번호표를 회수하지 못하면 "이 계산이 어느 손님이 한 것인지" 모른다. `checkout_started`는 입구 번호표를 받은 상태이고, `payment_success`는 계산대다. 지금 문제는 입구에서는 번호표가 잘 찍히는데, 계산대에서 번호표가 절반 정도 사라진다는 것이다.

현재 수치:

| 구분 | payment_success all-three | checkout_started all-three | 해석 |
|---|---:|---:|---|
| 전체 VM 최신 | 50.26% | 93.06% | 결제 시작은 식별자가 잘 붙고, 결제완료에서 많이 떨어진다 |
| 더클린커피 | 50.79% | 93.33% | 더클린커피도 같은 패턴이다 |

여기서 all-three는 `ga_session_id`, `client_id`, `user_pseudo_id`가 모두 있는 비율이다.
목표는 payment_success all-three 80% 이상이다.

왜:

Meta ROAS와 내부 Att ROAS 차이를 줄이려면 "광고 클릭 → 랜딩 → 결제 시작 → 결제완료 → CAPI 전송"이 한 줄로 이어져야 한다. payment_success에서 식별자가 빠지면 주문은 존재하지만, 그 주문이 어떤 광고 클릭에서 왔는지 확정하기 어렵다. 그러면 Att ROAS가 실제보다 낮게 보일 수 있다.

어떻게:

1. 결제 시작 시점에 식별자를 먼저 저장한다.
   - `ga_session_id`
   - `client_id`
   - `user_pseudo_id`
   - `fbclid`, `_fbp`, `_fbc`
   - UTM
   - 임시 checkout/session id

2. 결제완료 시점에 주문 키와 식별자를 다시 묶는다.
   - `order_code`
   - `order_no`
   - `payment_code`
   - Toss `paymentKey`
   - Toss `orderId`

3. 결제완료 서버 호출이 들어오면 기존 checkout row와 조인한다.
   - 같은 브라우저 저장소 값이 있으면 직접 저장
   - 브라우저 값이 비어 있으면 `order_no/order_code/paymentKey/orderId`로 checkout_started row에서 carry-over
   - 최종적으로 payment_success row의 `ga_session_id/client_id/user_pseudo_id`를 채운다

4. coverage를 매일 확인한다.
   - 전체 payment_success all-three
   - 더클린커피 payment_success all-three
   - checkout_started 대비 payment_success 하락 폭
   - 최근 누락 샘플 URL과 missingFields

Codex가 할 일:

- 원장 저장 로직을 보강한다.
- 결제완료 페이지 스크립트와 서버 수신 API에서 `ga_session_id/client_id/user_pseudo_id`를 빠짐없이 받도록 수정한다.
- `checkout_started -> payment_success` carry-over 로직을 만든다.
- `/ads`에서 전체/사이트별 coverage를 계속 표시한다.

TJ님이 할 일:

- 아임웹 관리자에서 footer/header/custom code 교체가 필요한 경우 배포를 승인하고 반영한다.

### Sprint 2-1. 아임웹/토스/내부 개발 가능 범위 판단

결론:

아임웹은 외부 솔루션이므로 아임웹 내부 결제 엔진 자체를 우리가 수정할 수는 없다.
하지만 지금처럼 아임웹 사이트의 header/footer/custom code, GTM, 외부 서버 API, 아임웹/토스 조회 API를 조합해서 "겉에서 감싸는 방식"의 개발은 가능하다. 이 방식은 지금까지 footer CAPI, Purchase Guard, attribution snippet에서 이미 사용한 방식이다.

즉 "아임웹 코어를 고친다"가 아니라 "아임웹 페이지 위에 추적 레이어를 얹고, 우리 서버 원장과 연결한다"가 현실적인 방식이다.

비유:

아임웹은 임대 매장이다. 벽을 허물거나 계산대 내부 프로그램을 뜯어고칠 수는 없다. 대신 매장 입구에 방문 기록 카메라를 달고, 계산대 앞에 영수증 스캐너를 두고, 우리 사무실 서버로 기록을 보내는 것은 가능하다. 지금 우리가 하는 footer 코드와 서버 원장이 그 카메라와 스캐너 역할이다.

우리가 직접 개발 가능한 것:

| 영역 | 가능 여부 | 설명 |
|---|---|---|
| 아임웹 header/footer/custom code 삽입 | 가능 | 지금처럼 footer 코드로 UTM, GA 식별자, fbq event_id, CAPI mirror를 넣을 수 있다 |
| GTM/Clarity/Meta Pixel 보강 | 가능 | 태그와 브라우저 이벤트 수집은 우리가 제어 가능하다 |
| 자체 서버 원장 API | 가능 | `att.ainativeos.net` 또는 로컬 백엔드가 payment_success/checkout_started를 받을 수 있다 |
| checkout_started와 payment_success carry-over | 가능 | 브라우저 저장소 + 서버 원장 + 주문 키 조인으로 구현 가능하다 |
| Toss API 조회/검증 | 가능 | `paymentKey`, `orderId`, `DONE` 상태 조회로 confirmed 결제를 검증할 수 있다 |
| 아임웹 주문 API 조회/검증 | 가능 | 현재 레거시 v2 주문 API는 사용 중이고, Open API는 권한 확인이 더 필요하다 |

우리가 직접 고칠 수 없는 것:

| 영역 | 불가능/제한 | 설명 |
|---|---|---|
| 아임웹 결제 엔진 내부 로직 | 제한 | 아임웹 SaaS 내부 코드라 직접 수정 불가 |
| 결제완료 페이지가 렌더링되기 전 서버 내부 흐름 | 제한 | 아임웹이 처리하는 서버 사이드 흐름은 외부에서 직접 패치 불가 |
| Toss 결제창 내부 동작 | 제한 | Toss는 결제 승인/리다이렉트/웹훅을 제공하는 PG이지, GA 식별자를 보관하는 CRM이 아니다 |
| Toss에 GA 식별자 저장 | 비권장 | 개인/광고 식별자는 우리 first-party 원장에 저장하는 것이 맞다 |

- `order_code`, `order_no`, `payment_code`를 프론트에서 안정적으로 읽을 수 있는지
- 결제완료 페이지/주문완료 페이지별 script 삽입 가능 범위

- `successUrl` 리다이렉트에서 `paymentKey`, `orderId`, `amount`가 어떤 조건으로 붙는지

### Sprint 3. 캠페인 매핑/LTV ROAS

무엇:

캠페인별로 Meta ROAS, 내부 confirmed Att ROAS, 180일 후속 구매를 포함한 LTV ROAS를 나란히 보는 작업이다.

비유:

전체 매출만 보는 것은 "학교 평균 점수"만 보는 것과 같다. 평균은 괜찮아도 어떤 학생이 잘하고 어떤 학생이 못하는지 모르면 보충수업을 누구에게 해야 할지 모른다. 캠페인 매핑은 각 학생 이름표를 붙이는 작업이다.

왜:

전체 광고 ROAS가 좋아도 어떤 캠페인이 진짜 confirmed 매출을 만드는지 모르면 예산을 어디에 늘리고 어디에서 줄일지 결정할 수 없다. 특히 바이오컴은 상담·영양제 후속 전환이 붙는 캠페인이 장기적으로 더 중요하다.

어떻게:

1. `utm_id`를 Meta campaign id 1순위로 본다.
2. `utm_term`을 adset id로 보고 부모 campaign을 찾는다.
3. 이름만 비슷한 `utm_campaign`은 fallback으로만 쓴다.
4. `fbclid` only는 Meta 유입 가능성은 높지만 캠페인 확정으로 강제 배분하지 않는다.

### Sprint 6. Incrementality OS loop

무엇:

왜:

어떻게:

1. 7일 clean baseline을 먼저 확보한다.

   - 지역/세그먼트 holdout
   - CRM 메시지 holdout
4. 실험 기간, 성공 지표, 중단 조건을 정한다.
5. 실험 후 광고비를 유지/증액/감액한다.

Codex가 할 일:

- 실험 설계안을 문서화한다.
- iROAS 계산 엔진과 대시보드를 연결한다.
- holdout 전후 매출, 신규 고객, 재구매, 상담·영양제 후속 전환을 계산한다.

TJ님이 할 일:

- holdout을 허용할 수 있는 비율을 정한다. 예: 5%, 10%, 15%.
- 절대 끄면 안 되는 캠페인과 실험 가능한 캠페인을 구분한다.
- 실험 중 매출 감소를 어느 정도까지 허용할지 결정한다.

상담·영양제 전환 캠페인만 발라내는 방법:

1. **캠페인 확정 게이트**

2. **결제 확정 게이트**
   - `payment_success + confirmed` 또는 Toss confirmed 주문만 본다.
   - 가상계좌 `pending`, 테스트 이벤트, CAPI retry/duplicate 의심 이벤트는 예산 판단에서 제외한다.

3. **상담·영양제 후속 게이트**
   - 주문의 고객 식별자(`order_member`, phone hash, customer key, orderId)를 상담 원장과 연결한다.
   - 연결된 고객이 `상담 완료` 상태이거나, 90/180일 안에 `productCategory=supplement` 주문을 만든 경우만 “상담·영양제 전환 확인”으로 판정한다.
   - 1차 구매 상품이 검사여도 후속 영양제 구매가 붙으면 증액 후보가 될 수 있다.

4. **증액 테스트 게이트**
   - clean window에서 캠페인별 confirmed 주문 2건 이상 또는 confirmed 매출 ₩500,000 이상을 최소 조건으로 둔다.

   - 증액 후 48~72시간은 전체 예산을 유지하고, campaign-level Attribution ROAS와 상담·영양제 후속 전환만 본다.

### E. vbank_expired 10억원이 `/ads` net 계산에 포함되는지 검증

**결론 먼저**: **`/ads` 메인 ROAS net에는 포함되지 않는다** ✅. vbank_expired는 attribution_ledger에서 `payment_status='pending'`으로 기록되고, `/ads`는 `paymentStatus==='confirmed'`만 분자로 쓰기 때문.

**코드 경로 분석** (`backend/src/routes/ads.ts`):

1. `loadLedgerOrders()` (1647행) → attribution_ledger 전체 조회
2. `buildNormalizedLedgerOrders()` (895행) → `touchpoint='payment_success'` 이벤트만 추림, `paymentStatus` 결정
3. 한 주문에 여러 이벤트가 있으면 `PAYMENT_STATUS_PRIORITY = { pending:1, confirmed:2, canceled:3 }` 순위로 **가장 높은 우선순위 값 선택**. 즉 `canceled`가 하나라도 있으면 canceled로 확정, 그게 없으면 confirmed > pending.
4. `completed: paymentStatus === "confirmed"` (958행) 플래그 설정.
5. `buildCampaignRoasRows()` (1155행) → `.filter((c) => c.completed).filter(isMetaAttributedOrder)` 통과한 주문만 revenue에 누적.
6. 즉 `paymentStatus==='pending'` 또는 `'canceled'`는 분자에서 **자동 제외**.

**실제 샘플 검증**: 바이오컴 vbank_expired 상위 5건을 `attribution_ledger`에서 조회 (2026-04-20):

**주의사항 — 별도 리스크 2건**:

### G. 바이오컴 공동구매 매출을 일반 매출과 분리할 수 있는가?

**배경**: 바이오컴은 인플루언서·카페·맘카페 등과 **공동구매**를 운영. 고객이 인플루언서 전용 URL 또는 공동구매용 쿠폰 코드로 유입해 일반가보다 10~20% 할인된 가격에 구매. 더클린커피·aibio에는 공동구매 없음. 광고비 판단에서는 공동구매 매출을 **일반 매출과 분리**해야 함 (공동구매 ROAS는 광고가 아닌 파트너 채널의 성과이기 때문).

**공동구매 식별 가능한 신호 3가지**:

#### 신호 1. 쿠폰 이름 패턴 (현재 가장 강함)

`imweb_coupon_masters` 실측:

| 키워드 | 쿠폰 종류 수 | 누적 use_count | 비고 |
|---|---:|---:|---|
| `공동구매` 또는 `공구` 포함 | **115종** | **2,055건** | 주요 공동구매 통로 |
| `인플루언서` 포함 | 포함(위 115에 일부) | — | 별도 브랜드 컬래버 |

샘플:
- `[하림 공동구매] 영양중금속검사 10% 할인쿠폰` → 419건 사용
- `[우주맘 키슬] 영양중금속 10% 공동구매 할인` → 176건

- `inpork_biocom_igg`: 7건

UTM만으로는 공동구매 전체를 잡을 수 없음. 쿠폰이 주 식별 경로.

**분리 구현의 기술적 장벽**:

공동구매 쿠폰 마스터와 주문을 조인하려면 중간에 **issue_coupon** 매핑 테이블이 필요하다 — 한 쿠폰 마스터가 N명의 고객에게 개별 발행(issue)되고, 주문의 `use_issue_coupon_codes`는 그 **발행 인스턴스 코드** (예: `i20260107038438f95331b`)를 참조하기 때문.

**현재 로컬 DB 상태** (실측):

### L. Sprint1 #2 "freshness 응답에 severity·action 필드 추가 검토" — 🟢 설계 완료 (구현은 TJ 승인 대기)

**현 로직 분석** (`backend/src/sourceFreshness.ts`):
- 전체 원천에 **단일 임계값** 적용 (`warnHours: 36`, `staleHours: 72`).
- 즉 Toss/PlayAuto에도 GA4 BigQuery에도 로컬 미러에도 똑같이 적용됨.
- 문제: 실시간성이 다른데 같은 기준으로 판정 → GA4가 48h 되어도 `fresh`, Toss가 48h 되면 위험해야 하는데 `fresh`.

**제안 1: 원천별 임계값 테이블** (TJ 승인 필요)

```ts
const PER_SOURCE_THRESHOLDS: Record<string, { warnHours: number; staleHours: number; actionHint: string }> = {
  // 결제·주문 원장 (실시간성 높음)
  toss_operational:      { warnHours: 24, staleHours: 36, actionHint: "Toss sync 스케줄 확인 · POST /api/toss/sync 재실행" },
  playauto_operational:  { warnHours: 24, staleHours: 36, actionHint: "PlayAuto 주문 싱크 점검" },
  imweb_local_orders:    { warnHours: 24, staleHours: 48, actionHint: "POST /api/crm-local/imweb/sync-orders 재실행" },
  // Attribution (거의 실시간이어야 함)
  attribution_ledger:    { warnHours: 12, staleHours: 24, actionHint: "VM att.ainativeos.net 헬스 점검 · webhook 확인" },
  // GA4 daily export (T+24~36h 관행)
  ga4_bigquery_thecleancoffee: { warnHours: 48, staleHours: 72, actionHint: "GA4 export 허들러스에 확인" },
  // 로컬 미러 (스케줄 느슨해도 OK)
  toss_local_transactions:  { warnHours: 48, staleHours: 72, actionHint: "Toss 로컬 미러 스케줄 점검" },
  toss_local_settlements:   { warnHours: 48, staleHours: 72, actionHint: "Toss 정산 미러 스케줄 점검" },
};
```

**제안 2: 응답 필드 추가**

현재 `SourceFreshnessResult`:
```ts
{ source, storage, table, status, totalRows, freshnessAt, ageHours, note }
```

추가할 필드:
```ts
severity: "ok" | "watch" | "alert" | "critical"   // status를 운영자 친화 용어로 매핑
action:   string | null                            // 운영자가 지금 할 일 자연어 문구
```

매핑 규칙:
- `status=fresh`  → severity=`ok`,       action=null
- `status=empty`  → severity=`watch`,    action=`"데이터 비어 있음 — 수집 경로 확인"`
- `status=warn`   → severity=`watch`,    action=`"추이 모니터링"` (심각하지 않음)
- `status=stale`  → severity=`alert`,    action=해당 원천의 `actionHint` 그대로
- `status=missing`→ severity=`critical`, action=`"스키마/연결 복구 필요 — 운영팀 에스컬레이션"`
- `status=error`  → severity=`critical`, action=`"조회 실패 — 에러 메시지 확인"`

**제안 3: 상수 위치**

- `PER_SOURCE_THRESHOLDS`를 `backend/src/sourceFreshness.ts`에 상수 추가.
- 기존 `getDefaultSourceFreshnessOptions`는 per-source 맵이 없을 때 fallback으로만 유지.
- CLI `check-source-freshness.ts`도 자동 반영됨.

**블로커**: TJ가 위 시간 숫자를 승인해야 구현 시작. 승인 전에는 설계 문서로만 보관.

**현 원천별 실측 age (2026-04-20 기준)**:

| 원천 | 현재 age | 현 규칙 status | 제안 규칙 status |
|---|---:|---|---|
| toss_local_transactions | 297.4h | stale | stale (alert) |
| toss_local_settlements | 297.4h | stale | stale (alert) |
| imweb_local_orders | 109.9h | stale | stale (alert) |
| attribution_ledger | 190.1h | stale | **critical 수준** — 12h 기준 넘은지 오래 |
| ga4_bigquery_thecleancoffee | 26.5h | fresh | fresh (ok) |
| toss_operational | 29.3h | fresh | **warn**(watch) — 24h 기준 넘음 |
| playauto_operational | 30.3h | fresh | **warn**(watch) — 24h 기준 넘음 |

**시사점**: 현재 단일 임계값 로직으로 "fresh"로 분류되고 있는 `toss_operational`, `playauto_operational`이 제안 규칙에서는 **watch 단계로 올라와야** 한다. 운영 쪽에서는 이미 확인이 필요한 수준일 가능성이 높음.
