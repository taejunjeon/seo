# 구매확정시점 분석

작성 시각: 2026-04-22 14:00 KST
기준일: 2026-04-22

## 결론

현재 SEO/ROAS 시스템에서 `pending -> confirmed`는 아임웹의 배송 후 `PURCHASE_CONFIRMATION`이 아니라 **Toss Payments 결제 상태가 `DONE`이 되는 시점**을 기준으로 봐야 한다.

정확히는 다음과 같다.

| 구분 | pending으로 보는 시점 | confirmed로 보는 시점 | 현재 시스템 반영 시점 |
|---|---|---|---|
| 카드/간편결제/계좌이체 | 결제 승인 전 또는 원장에 상태가 아직 없을 때 | Toss `status=DONE`, `approvedAt` 생성 | 결제완료 페이지의 `payment-decision` 즉시 조회 또는 15분 주기 status sync |
| 가상계좌 | 가상계좌 발급 직후. Toss `WAITING_FOR_DEPOSIT` | 구매자가 실제 입금해서 Toss `status=DONE`이 된 시점 | 현재는 웹훅이 아니라 15분 주기 polling/sync로 반영 |
| 아임웹 `PAY_COMPLETE` | 아임웹 주문이 입금대기/미결제 상태일 때 | 아임웹 주문 상태가 결제완료로 넘어간 시점 | 현재 ROAS confirmed 정본으로 직접 사용하지 않음 |
| 아임웹 `PURCHASE_CONFIRMATION` | 배송/구매확정 전 | 배송 완료 이후 구매확정 처리된 주문관리 상태 | ROAS 결제 확정 기준으로 쓰면 너무 늦음. 현재 attribution confirmed 기준 아님 |

따라서 광고 ROAS 정합성에서 말하는 `confirmed purchase`는 “배송 후 구매확정”이 아니라 “PG 결제 확정”이다. 현 시스템의 안전한 기준은 **Toss `DONE`**이다.

추가 결론: **PG사 정산 기준과 주문 기준은 목적이 다르다.** 광고 ROAS는 주문/결제 확정 기준으로 보고, 재무 입금 대사는 PG 정산 기준으로 봐야 한다. 정산 기준을 광고 ROAS 분자로 쓰면 수수료 차감과 지급일 지연 때문에 광고 성과가 실제보다 낮고 늦게 보인다.

가상계좌 미입금 자동 취소 기간은 **바이오컴 운영 기준 24시간**으로 확인됐다. 즉 가상계좌 주문 생성 후 24시간 이내 입금이 없으면 취소된다. Toss 문서상 API 직접 발급 기본값은 168시간이지만, 바이오컴의 아임웹/결제 설정은 24시간으로 봐야 한다.

## 용어 분리

이 이슈에서 가장 위험한 혼선은 `구매 확정`이라는 말이 두 가지 의미로 쓰인다는 점이다.

1. **PG 결제 확정**
   - Toss Payments에서 결제가 실제 승인된 상태다.
   - Toss `DONE`이 기준이다.
   - 광고 ROAS, Meta/TikTok Purchase, CAPI Purchase의 기준으로 써야 한다.

2. **아임웹 주문관리 구매확정**
   - 아임웹 주문 상태 `PURCHASE_CONFIRMATION`이다.
   - 배송 이후 고객/관리자/자동 정책에 의해 닫히는 주문관리 상태에 가깝다.
   - 광고 매출 인식 기준으로 쓰면 결제일보다 늦고, 당일 ROAS가 과소 집계된다.

3. **PG 정산**
   - 고객이 결제한 금액을 PG사가 정산 주기, 수수료율, 취소/부분취소 등을 반영해 상점에 지급하는 회계/입금 기준이다.
   - Toss Settlement 객체의 핵심 값은 `amount`, `fees`, `fee`, `payOutAmount`, `soldDate`, `paidOutDate`다.
   - 광고 전환 시점 판단이 아니라 정산 대사, 현금흐름, PG 수수료 검증에 사용한다.

## Toss Payments 기준

Toss 공식 문서 기준으로 Payment 객체의 `status`는 다음 의미를 가진다.

- `WAITING_FOR_DEPOSIT`: 가상계좌 결제 흐름에서만 있는 상태이며, 발급된 계좌에 구매자가 아직 입금하지 않은 상태다.
- `DONE`: 인증된 결제수단으로 요청한 결제가 승인된 상태다.
- `approvedAt`: 결제 승인이 일어난 시각이다.

가상계좌는 특히 중요하다. Toss 문서상 가상계좌는 결제 승인 API를 호출해도 “계좌가 발급된 것”이지 결제가 완료된 것이 아니다. 구매자가 입금한 시점에 Payment 객체 상태가 `DONE`으로 바뀐다. Toss는 이런 비동기 상태 변경을 웹훅으로 받는 것을 권장한다.

현재 우리 운영 VM에는 Toss 웹훅 수신 경로가 없다. 대신 다음 방식으로 확인한다.

1. 결제완료 페이지 또는 guard가 `payment-decision` endpoint를 호출한다.
2. endpoint가 Toss direct API를 paymentKey 또는 orderId로 조회한다.
3. Toss 상태가 `DONE`이면 `confirmed` / `allow_purchase`로 판단한다.
4. Toss 상태가 `WAITING_FOR_DEPOSIT`이면 `pending` / `block_purchase_virtual_account`로 판단한다.
5. 이후 `sync-status/toss`가 pending 원장을 주기적으로 다시 조회해 `DONE`이면 ledger를 `confirmed`로 승격한다.

현재 운영 헬스 기준:

| 항목 | 값 |
|---|---:|
| Attribution status sync | enabled |
| Attribution status sync interval | 900,000ms = 15분 |
| Attribution status sync limit | 100 |
| Toss auto sync | enabled |
| Toss auto sync interval | 15분 |
| Toss sync window | 최근 6시간 |
| Imweb order sync | enabled |
| Imweb order sync interval | 15분 |

즉 결론은 이렇다.

- 카드 결제는 Toss `DONE`이 결제완료 시점에 바로 나오므로 거의 즉시 `confirmed`가 될 수 있다.
- 가상계좌는 주문완료/계좌발급 시점에는 `pending`이다.
- 가상계좌는 실제 입금 시 Toss `DONE`으로 바뀐다.
- 우리 원장 반영은 실시간 웹훅이 아니라 polling이므로 보통 **입금 후 다음 status sync 실행 시점**, 운영 설정상 최대 약 15분 지연될 수 있다.

## 가상계좌 미입금 자동 취소 기간

결론부터 보면, 바이오컴 운영 기준은 **주문 후 24시간 이내 미입금 시 취소**다. 이 값은 TJ 확인값이며, ROAS 분석에서는 운영 확정 규칙으로 사용한다.

| 확인 지점 | 문서상 내용 | 판단 |
|---|---|---|
| Toss 가상계좌 기본 원리 | 고객은 설정된 기한 안에 정확한 금액을 입금해야 한다 | 기한은 존재하지만 공통 N일이 아니다 |
| Toss API 직접 발급 | `validHours` 또는 `dueDate` 중 하나로 입금기한을 설정한다 | 상점/요청 단위 설정값이다 |
| Toss API 직접 발급 기본값 | `validHours`를 넣지 않으면 168시간, 최대 2160시간이다 | API 직접발급 기본값은 7일 |
| Toss 결제위젯/어드민 | 상점관리자에서 가상계좌 입금기한을 설정할 수 있다 | 바이오컴 운영값은 24시간으로 확인 |
| Toss 만료 웹훅 | 가상계좌 만료 자체는 별도 웹훅을 보내지 않는다 | 만료 후 주문 취소는 상점/플랫폼 처리 영역이다 |
| Imweb 주문 API | `PAY_WAIT`, `PAY_COMPLETE`, `CANCEL`, `PURCHASE_CONFIRMATION` 상태를 제공한다 | 공개 API 문서상 N일은 없지만 운영값은 24시간 |

따라서 “가상계좌 주문 생성 후 며칠 지나야 취소되는가”의 안전한 답은 다음과 같다.

1. **Toss 단독 기준**
   - API 직접 발급이면 기본값은 7일이다.
   - 하지만 요청에서 `validHours` 또는 `dueDate`를 지정하면 그 값이 우선이다.
   - 발급된 Payment 객체의 `virtualAccount.dueDate`가 실제 입금기한이다.

2. **아임웹 경유 기준**
   - 공개 Imweb 주문 API 문서에서는 `입금대기 -> 취소완료` 상태값은 확인된다.
   - 하지만 “가상계좌 발급 후 N일 뒤 자동취소”라는 고정 일수는 확인되지 않았다.
   - 바이오컴 운영값은 TJ 확인 기준 **24시간**이다.

3. **바이오컴 데이터 기준**
   - 로컬 `toss_transactions`에는 `dueDate` 또는 `validHours` 컬럼이 없다.
   - 로컬 `imweb_orders`에는 입금기한/취소시각 컬럼이 없다.
   - 따라서 현재 로컬 캐시만으로 취소 발생 시각을 초 단위로 재계산할 수는 없다.
   - 다만 로컬 스냅샷에서 `imweb_status='CANCEL' AND pay_type='virtual'` 주문 339건은 확인된다. 최신 예시는 주문 생성 후 약 28.2시간 뒤 로컬 sync 시점에 이미 `CANCEL`로 관측됐다. 이는 24시간 운영 규칙과 모순되지 않는다.

운영에서 정확히 닫으려면 둘 중 하나가 필요하다.

| 방법 | 할 일 | 장점 |
|---|---|---|
| 관리자 설정 확인 | 아임웹 관리자 또는 Toss 상점관리자의 가상계좌 입금기한 24시간 설정 확인 | 가장 빠름. 2026-04-22 TJ 확인 완료 |
| 데이터 저장 보강 | Toss Payment 조회 결과의 `virtualAccount.dueDate`를 원장에 저장 | 주문별 실제 기한을 장기 분석 가능 |

ROAS 프로젝트 기준으로는 자동취소까지 기다릴 필요 없이, 입금 전 `WAITING_FOR_DEPOSIT` 상태는 계속 `pending`으로 두는 것이 맞다. 실제 입금되어 Toss `DONE`이 되면 confirmed로 승격하고, 24시간이 지나 취소되면 `canceled/vbank_expired`로 분류한다.

## PG 정산 기준 vs 주문 기준

정산 기준과 주문 기준은 같은 결제를 다른 목적으로 보는 두 장부다.

| 기준 | 대표 날짜 | 대표 금액 | 목적 | 광고 ROAS 사용 여부 |
|---|---|---:|---|---|
| 주문 기준 | 아임웹 `order_time`, `payment_time` | 상품 주문금액, 결제 요청/완료 금액 | 주문·상품·고객 분석 | 보조. 결제 상태 확인이 필요 |
| PG 결제 확정 기준 | Toss `approvedAt`, `status=DONE` | 고객이 실제 승인한 결제 gross amount | 광고 전환, confirmed revenue | 메인 기준 |
| PG 정산 기준 | Toss `soldDate`, `paidOutDate` | `payOutAmount = amount - fee` | 입금 대사, 수수료 검증, 현금흐름 | ROAS 메인 기준으로 쓰지 않음 |
| 주문관리 구매확정 기준 | 아임웹 `complete_time`, `PURCHASE_CONFIRMATION` | 배송/구매확정된 주문 금액 | CS/배송 완료, 구매확정 지연 분석 | 광고 Purchase 기준으로 쓰지 않음 |

### 왜 숫자가 달라지는가

1. **시점 차이**
   - 주문은 고객이 주문서를 만들거나 결제창을 완료한 시점에 생긴다.
   - Toss 결제 확정은 `DONE`과 `approvedAt` 기준이다.
   - Toss 정산은 결제 다음 날부터 조회되고, `soldDate`와 실제 지급일 `paidOutDate`가 따로 있다.
   - 아임웹 구매확정은 배송 이후라 결제일보다 훨씬 늦다.

2. **금액 차이**
   - 주문/결제 기준은 고객 결제 gross 금액을 본다.
   - 정산 기준은 PG 수수료와 수수료 부가세 등이 차감된 `payOutAmount`를 본다.
   - 따라서 정산 입금액은 광고 전환 매출보다 작다.

3. **범위 차이**
   - 아임웹은 상품, 쿠폰, 회원, 배송 상태를 안다.
   - Toss는 결제 승인, 취소, 정산을 안다. 상품 라인아이템이나 광고 유입은 알지 못한다.
   - 광고 원장은 유입 식별자와 결제 상태를 연결한다. 상품/정산 정보는 별도 조인이 필요하다.

4. **취소/부분취소 처리 차이**
   - 주문 기준에서는 `PAY_COMPLETE` 이후 `CANCEL`, `RETURN`, `PARTIAL_CANCELED` 같은 상태가 뒤늦게 생길 수 있다.
   - Toss 정산 기준에서는 취소·부분취소가 정산 금액 또는 후속 정산에 반영된다.
   - 광고 ROAS는 먼저 `DONE` gross를 confirmed로 잡고, 취소/환불은 별도 refund 또는 net revenue 지표로 차감하는 방식이 안전하다.

### Toss 정산 필드 해석

Toss 공식 문서 기준:

- `amount`: 결제한 금액이다.
- `fees`: 결제 수수료 상세 배열이다.
- `fee`: 수수료 금액이다.
- `payOutAmount`: 지급 금액이다. 결제 금액 `amount`에서 수수료 `fee`를 제외한 금액이다.
- `soldDate`: 지급 금액의 정산 기준이 되는 정산 매출일이다. 상점 정산 주기에 따라 달라진다.
- `paidOutDate`: 상점에 지급할 정산 지급일이다.
- `/v1/settlements`는 기본적으로 `soldDate` 기준이며, `dateType=paidOutDate`를 쓰면 지급일 기준으로 조회한다.
- 정산 기록은 결제가 일어난 다음 날부터 조회된다.

## 실무 판단 기준

| 질문 | 써야 할 기준 |
|---|---|
| TikTok/Meta 광고가 실제 매출을 만들었나? | Toss `DONE` 주문/결제 확정 gross |
| 오늘 광고 ROAS가 얼마인가? | confirmed gross revenue / ad spend |
| 회사 통장에 얼마 들어오나? | PG 정산 `payOutAmount` / `paidOutDate` |
| PG 수수료율이 맞나? | Settlement `fee`, `fees`, `payOutAmount` |
| 배송/CS까지 완전히 닫힌 주문인가? | 아임웹 `PURCHASE_CONFIRMATION` 또는 `complete_time` |
| 순매출/net profit을 보고 싶은가? | confirmed gross - refund - PG fee - coupon - COGS. 별도 지표로 분리 |

TikTok ROAS 정합성 프로젝트에는 다음 원칙을 적용한다.

- 플랫폼 구매값과 비교할 내부 매출은 **Toss `DONE` gross confirmed**다.
- `WAITING_FOR_DEPOSIT`은 pending이며, 입금 전에는 Purchase가 아니다.
- PG 정산 payout은 광고 성과 검증용 분자가 아니라 재무 대사용이다.
- 다만 장기 수익성 분석에서는 PG fee를 차감한 net revenue도 별도 보조 지표로 만들어야 한다.

## 아임웹 기준

아임웹 주문 API는 주문 상태 필터로 다음 값을 제공한다.

- `PAY_WAIT`: 입금대기
- `PAY_COMPLETE`: 결제완료
- `STANDBY`: 배송준비중/배송대기
- `DELIVERING`: 배송중
- `COMPLETE`: 배송완료
- `CANCEL`: 취소완료
- `RETURN`: 반품완료
- `EXCHANGE`: 교환완료
- `PURCHASE_CONFIRMATION`: 구매확정, 주문관리 v2 전용

아임웹 API는 주문 상태를 조회할 수는 있지만, 현재 우리 구현에서 광고 ROAS의 `confirmed` 정본으로 직접 쓰지는 않는다. 이유는 세 가지다.

1. 아임웹 `PURCHASE_CONFIRMATION`은 결제 확정이 아니라 주문관리 구매확정이다.
2. 아임웹 API 응답에는 주문 상태 필터는 있지만, 우리 코드상 상태 전이 이력을 안정적으로 저장하는 구조가 아니다.
3. 운영 VM의 `/api/crm-local/imweb/purchase-confirm-stats?site=biocom` 현재 응답은 `imweb_status`가 아직 채워지지 않은 상태다. 즉 운영 기준 아임웹 상태 캐시는 ROAS confirmed 판단에 쓰기 어렵다.

운영 VM 확인 결과:

| 항목 | 값 |
|---|---:|
| biocom imweb_orders total | 9,161 |
| `imweb_status=_null_` | 9,161 |
| latestStatusSyncAt | null |
| `PURCHASE_CONFIRMATION` count | 0 |

반면 로컬 SQLite 스냅샷에는 과거 수동/로컬 sync 결과가 남아 있다.

| 로컬 상태 | 건수 | 금액 |
|---|---:|---:|
| `PURCHASE_CONFIRMATION` | 6,690 | 1,555,714,670원 |
| `CANCEL` | 726 | 1,068,343,161원 |
| `COMPLETE` | 585 | 134,050,823원 |
| `PAY_WAIT` | 3 | 468,000원 |

이 로컬 값은 아임웹 주문 상태 분포를 보는 데는 도움이 되지만, 현재 운영 attribution confirmed 전환 시점의 정본은 아니다. 현재 운영 confirmed 판단은 Toss direct/status sync가 더 정확하다.

## 현재 코드 기준 전환 로직

관련 구현:

- `backend/src/attribution.ts`
- `backend/src/routes/attribution.ts`
- `backend/src/bootstrap/startBackgroundJobs.ts`
- `backend/src/routes/crmLocal.ts`

상태 매핑:

| 원천 상태 | 내부 상태 |
|---|---|
| `WAITING_FOR_DEPOSIT`, `PENDING`, `WAITING`, `READY`, `DEPOSIT`, `REQUESTED`, `IN_PROGRESS` | `pending` |
| `DONE`, `PAID`, `APPROVED`, `SUCCESS`, `SUCCEEDED`, `CONFIRMED`, `COMPLETED`, `COMPLETE` | `confirmed` |
| `CANCEL`, `FAIL`, `ABORT`, `REFUND`, `EXPIRE`, `VOID` | `canceled` |

전환 흐름:

1. 결제완료 페이지가 `/api/attribution/payment-success`를 호출한다.
2. payload 또는 metadata에 상태가 없으면 pending으로 남을 수 있다.
3. `/api/attribution/payment-decision`은 Toss direct API를 먼저 본다.
4. Toss direct 결과가 `DONE`이면 브라우저/서버 Purchase를 허용한다.
5. Toss direct 결과가 `WAITING_FOR_DEPOSIT`이면 Purchase를 막고 가상계좌 발급 이벤트로 낮춘다.
6. `/api/attribution/sync-status/toss`는 pending 원장을 다시 조회해 `DONE`이면 `confirmed`로 업데이트한다.
7. background job은 이 sync를 15분 주기로 실행하도록 설정되어 있다.

해석:

- auto sync가 켜져 있으므로 정상이라면 다음 배치에서 일부가 반영되어야 한다.
- 다만 dry-run 시점에 43건이 여전히 업데이트 후보로 보였으므로, 백그라운드 job 로그 또는 write-run 결과를 한 번 확인해야 한다.

## TikTok pending 상위 20건과의 관계

이전 TikTok ROAS 분석에서 pending 상위 20건은 별도로 paymentKey 기준 Toss direct read-only 확인을 했다.

| 항목 | 값 |
|---|---:|
| 대상 | TikTok pending 상위 20건 |
| 합계 | 536,729,000원 |
| Toss direct `confirmed` | 0 |
| Toss direct `pending` | 20 |

즉 이 20건은 “동기화가 늦어서 pending으로 보이는 카드 결제”가 아니다. 현재 Toss 기준으로도 아직 입금 완료 전 상태다. ROAS confirmed 매출에 넣으면 안 된다.

## 최종 판단

1. **ROAS 확정매출 기준은 Toss `DONE`으로 유지한다.**
   - 카드/간편결제는 승인 직후 `DONE`이면 즉시 confirmed.
   - 가상계좌는 입금 전 `WAITING_FOR_DEPOSIT`이면 pending.
   - 실제 입금 후 `DONE`이 되면 confirmed.

2. **아임웹 `PURCHASE_CONFIRMATION`은 ROAS 기준이 아니다.**
   - 배송 이후 주문관리 완료 상태로 봐야 한다.
   - 광고 Purchase 기준으로 쓰면 너무 늦다.

3. **현재 운영 반영 지연은 최대 15분이 설계값이다.**
   - 단, dry-run에서 43건 업데이트 후보가 보여 실제 job 로그 확인이 필요하다.
   - 지금 당장 write-run을 실행하면 production ledger 변경이므로 TJ님 승인 후 진행한다.

4. **가상계좌 후속 입금에 대한 TikTok/Meta Purchase 전송은 별도 설계가 필요하다.**
   - 브라우저는 입금 시점에 결제완료 페이지를 다시 열지 않는다.
   - 따라서 입금 후 `DONE` 전환을 광고 플랫폼에 보내려면 Toss webhook 또는 polling 기반 서버 Events API/CAPI가 필요하다.
   - 현재는 오염 방지가 우선이므로, 미입금 가상계좌는 Purchase에서 제외하는 정책이 맞다.

5. **PG 정산 기준은 광고 ROAS 기준과 분리한다.**
   - 정산 기준은 지급액·수수료·입금일 확인용이다.
   - 광고 ROAS 기준은 결제 확정 gross다.
   - 두 기준을 섞으면 “광고 성과”와 “현금 입금”이 섞여 의사결정이 흐려진다.
