## 10초 요약

미니 디지털 트윈은 가능하다.
단, 1차는 매출을 맞히는 예측기가 아니라 "최근 기준선을 읽고 AI가 몇 가지 가능한 운영 시나리오를 자동 제안하는 what-if 시뮬레이터"로 제한한다.

사람이 방문자 수, 광고비, 구매 전환율, 객단가를 직접 바꾸는 슬라이더 방식으로 시작하지 않는다.
에이전트가 source matrix를 보고 기준값을 읽은 뒤, 현재 유지 / 결제 이탈 개선 / 콘텐츠 몰입 개선 / 트래픽 확대 전 점검 / 나쁜 경우 방어선 같은 시나리오를 자동으로 제안한다.

출력값은 5개만 둔다.
예상 주문 수, 예상 매출, 내부 confirmed ROAS, 결제 이탈 영향, 우선 개선 지표다.

## 왜 필요한가

선행지표는 좋은 숫자를 찾는 데서 끝나면 약하다.
운영자가 실제로 궁금한 것은 아래 질문이다.

- 현재 흐름이 그대로 가면 예상 주문 수가 어느 정도인가.
- 결제 이탈률을 1%p 또는 3%p 줄이면 추가 주문이 몇 건 생길 수 있는가.
- 광고비를 늘리기 전에 랜딩/결제 UX를 고치는 것이 더 큰 효과가 있을 수 있는가.
- 어떤 선행지표를 먼저 개선하면 매출에 민감하게 반응할 가능성이 큰가.

미니 디지털 트윈은 이 질문을 빠르게 보는 운영판이다.

## 1차 기준선 4개

기준선은 사람이 손으로 입력하지 않는다.
에이전트가 `project/leading-indicator-source-matrix-20260525.md` 기준으로 읽어온다.

### 1. 방문자 수

뜻:
사이트에 들어온 방문 수 또는 안전한 session 수다.

권장 source:

- VM Cloud `site_landing_ledger` 또는 leading indicator aggregate
- GA4 BigQuery session 집계는 cross-check로 사용 가능

금지:

- 광고 플랫폼 클릭 수를 방문자 수처럼 직접 사용하지 않는다.
- `all_sites` 값을 site별 판단에 자동 혼합하지 않는다.

### 2. 광고비

뜻:
같은 기간에 사용한 광고비다.

권장 source:

- 플랫폼 API/CSV spend
- site/channel/window가 맞는 값만 사용

금지:

- 이 값은 계산용 숫자다.
- 예산 변경 명령이 아니다.
- 광고비 source가 닫히지 않으면 ROAS 시나리오는 HOLD로 둔다.

### 3. 구매 전환율

뜻:
방문자 중 실제 결제완료 주문으로 닫힌 비율이다.

계산:

```text
구매 전환율 = 실제 결제완료 주문 수 / 방문자 수
```

금지:

- 클릭을 구매로 세지 않는다.
- 결제 시작을 구매로 세지 않는다.
- NPay 클릭이나 NPay count를 구매로 세지 않는다.
- add_payment_info를 구매로 세지 않는다.

### 4. 객단가

뜻:
실제 결제완료 주문 1건당 평균 매출이다.

계산:

```text
객단가 = 실제 결제완료 매출 / 실제 결제완료 주문 수
```

주의:

- biocom과 thecleancoffee를 분리한다.
- NPay, 카드, 가상계좌는 결제완료 시점 차이를 주석으로 남긴다.
- 매출 source가 닫히지 않으면 예상 매출과 ROAS는 HOLD로 남긴다.

## AI 시나리오 자동 제안 방식

에이전트는 실행 때마다 source freshness와 행동 병목을 보고 3-5개 시나리오를 자동으로 고른다.
사람은 숫자를 직접 만지는 것이 아니라, "왜 이 시나리오가 나왔는지"와 "이 시나리오를 운영 판단에 쓸 수 있는지"를 본다.

### 1. 현재 유지 기준선

무엇:
최근 7일 또는 28일 흐름이 그대로 이어진다고 보는 기준선이다.

왜:
다른 시나리오가 얼마나 나아지는지 비교하려면 기준점이 필요하다.

계산:
현재 방문자 수, 전환율, 객단가, 광고비를 그대로 쓴다.

HOLD 조건:
방문자 수 또는 결제완료 주문 수 source가 없으면 시뮬레이션 전체를 HOLD로 둔다.

### 2. 결제 이탈 개선

무엇:
결제 시작은 많지만 결제완료가 낮을 때 자동 제안한다.

왜:
이 경우 랜딩보다 결제 단계 UX가 더 큰 병목일 수 있다.

계산:
결제 시작 대비 결제완료율을 1%p, 3%p, 5%p 개선한 주문 수 범위를 보여준다.

HOLD 조건:
결제 시작 source와 실제 결제완료 source의 기간이 다르면 HOLD다.

### 3. 콘텐츠 몰입 개선

무엇:
구매자와 이탈자의 체류시간, 스크롤, 상세페이지 행동 차이가 클 때 자동 제안한다.

왜:
상세페이지/리뷰/콘텐츠 구성이 구매 전환에 영향을 줄 수 있기 때문이다.

계산:
방문자 수는 유지하고 구매 전환율을 보수적으로 올린 low/base/high 범위를 보여준다.

HOLD 조건:
GA4 behavior source가 stale이거나 구매 후 행동이 섞이면 HOLD다.

### 4. 트래픽 확대 전 점검

무엇:
광고비 또는 방문자 수를 늘릴 때 전환율이 유지되지 않을 가능성을 같이 보는 시나리오다.

왜:
예산을 늘리면 방문자는 늘 수 있지만 유입 품질이 떨어질 수 있다.

계산:
방문자 수는 늘리고, 전환율은 유지 case와 하락 case를 함께 보여준다.

HOLD 조건:
광고비 source가 없으면 ROAS는 HOLD로 두고 주문 수 방향성만 표시한다.

### 5. 나쁜 경우 방어선

무엇:
전환율이나 객단가가 낮아지는 경우를 일부러 보여준다.

왜:
좋은 경우만 보면 예산/운영 판단이 과감해져 위험하다.

계산:
전환율과 객단가를 보수적으로 낮춰 주문/매출 하락 범위를 보여준다.

HOLD 조건:
매출 source가 없으면 객단가 하락 시나리오는 비활성화한다.

## 1차 출력값 5개

### 1. 예상 주문 수

계산:

```text
예상 주문 수 = 방문자 수 * 구매 전환율
```

표시:
단일 숫자가 아니라 low/base/high 범위로 표시한다.

### 2. 예상 매출

계산:

```text
예상 매출 = 예상 주문 수 * 객단가
```

표시:
내부 결제완료 기준 예상 매출로 표시한다.
플랫폼 주장 매출과 섞지 않는다.

### 3. 내부 confirmed ROAS

뜻:
실제 결제완료 기준 예상 매출을 광고비로 나눈 값이다.

계산:

```text
내부 confirmed ROAS = 예상 매출 / 광고비
```

주의:
Google Ads ROAS, Meta ROAS처럼 플랫폼이 주장하는 값과 분리한다.

### 4. 결제 이탈 영향

뜻:
결제 시작 후 실제 결제완료로 닫히지 않은 비율을 줄였을 때의 예상 추가 주문/매출이다.

예시:

- 결제 이탈률이 1%p 개선되면 추가 주문 n건
- 결제 이탈률이 3%p 개선되면 추가 매출 n원

주의:
결제 시작은 구매가 아니다.
결제완료 주문으로 닫힌 경우만 매출로 계산한다.

### 5. 우선 개선 지표

뜻:
매출에 민감하고 사람이 바꿀 수 있는 선행지표 후보다.

예시:

- 상세페이지 체류시간
- 90% 스크롤 도달률
- 장바구니 진입률
- 결제 시작 후 이탈률
- 결제수단 선택 단계 이탈률

## 기준선

기준선은 2개를 같이 둔다.

| 기준선 | 역할 |
|---|---|
| 최근 7일 | 현재 변화에 민감한 기준 |
| 최근 28일 | 안정적인 비교 기준 |

모든 숫자는 아래를 같이 기록한다.

- source
- window
- freshness
- site
- confidence

## 계산 방식 초안

```text
base_visitors = 최근 7일 또는 28일 방문자 수
base_orders = 같은 기간 실제 결제완료 주문 수
base_revenue = 같은 기간 실제 결제완료 매출
base_cvr = base_orders / base_visitors
base_aov = base_revenue / base_orders

ai_scenario = 에이전트가 자동 제안한 시나리오
scenario_visitors = base_visitors * ai_scenario.visitor_multiplier
scenario_cvr = base_cvr * ai_scenario.cvr_multiplier
scenario_orders = scenario_visitors * scenario_cvr
scenario_revenue = scenario_orders * base_aov
scenario_roas = scenario_revenue / base_ad_spend
```

low/base/high 범위:

```text
low = base * 0.9
base = base
high = base * 1.1
```

주의:
이 범위는 1차 임시값이다.
실제 오차 검증 후 조정한다.
매출 또는 광고비 source가 없으면 주문 수만 표시하고 매출/ROAS는 HOLD로 둔다.

## 오차 검증 방법

1차 검증은 과거 window 재현으로 한다.

1. 직전 7일을 기준선으로 잡는다.
2. 다음 7일 예상 주문/매출을 계산한다.
3. 실제 결제완료 주문/매출과 비교한다.
4. 주문 수 오차율, 매출 오차율, ROAS 오차율을 따로 기록한다.
5. 실제값이 low/base/high 범위 안에 들어왔는지 확인한다.

HOLD 기준:

- confirmed order가 30건 미만
- source freshness가 stale 또는 unavailable
- GA4 purchase와 내부 결제완료 label 충돌
- 결제수단별 완료 시점이 섞임
- 방문자 source와 매출 source의 window 불일치

## 화면 구성 초안

첫 화면:

1. 현재 기준선: 방문자, 주문, 매출, 전환율, 객단가
2. AI가 자동 제안한 시나리오 카드 3-5개
3. 시나리오별 예상 주문, 예상 매출, 내부 confirmed ROAS

5. 결제 이탈 영향 카드
6. 우선 개선 지표 카드
7. 품질 판정과 금지선

표시 문구:

```text
이 화면은 매출 예측 정답지가 아닙니다.
AI가 최근 기준선을 바탕으로 몇 가지 가능성을 제안하고,
각 가능성의 주문/매출 범위가 어떻게 달라질 수 있는지 보는 운영판입니다.
```
