# docurule.md

기준일: 2026-03-29

이 문서는 `결과보고서`, `로드맵`, `Phase 상세 메모`, `개발 진행 보고`를 쓸 때의 공통 문서 규칙이다.
목표는 하나다.

**대표가 30초 안에 이해할 수 있고, 개발팀은 바로 실행할 수 있는 문서만 남긴다.**

---

## 0. 이 문서가 필요한 이유

현재 문서들은 방향은 좋지만, 종종 아래 문제가 생긴다.

- 기술 용어가 먼저 나와서 문서의 의도와 사업 의미가 늦게 보인다.
- `무엇을 만들었는가`는 보이는데, `그래서 우리 프로젝트에 무슨 도움이 되는가`가 약하다.
- 로컬 검증 완료와 운영 반영 완료가 섞여 보여, 실제 완성도가 헷갈린다.
- 결과보고서, 로드맵, 작업 메모의 톤이 서로 달라 읽는 사람이 문서마다 해석을 다시 해야 한다.

그래서 앞으로 문서는 아래 원칙을 지킨다.

---

## 1. 최상위 원칙

### 원칙 1. 결론이 먼저다
문서는 항상 **결론 -> 이유 -> 근거 -> 다음 행동** 순서로 쓴다.

나쁜 예:
- API, 테이블, 스크립트, 엔드포인트를 길게 설명한 뒤 마지막에 의미를 적는다.

좋은 예:
- 이 작업의 목적은 `메시지가 실제 추가 매출을 만들었는지 확인하는 것`이다.
- 그래서 `실험 장부`, `결제 블랙박스`, `운영자 화면`을 만든다.
- 현재는 장치는 대부분 준비됐고, 실제 서비스 연결만 남았다.

### 원칙 2. 기술보다 의도와 목적이 먼저다
각 문단은 아래 질문 중 최소 3개에 답해야 한다.

1. 이 작업은 **무엇을 하는가**
2. **왜 필요한가**
3. **산출물이 무엇인가**
4. **우리 프로젝트에 어떤 도움이 되는가**
5. **지금 막힌 것이 무엇인가**
6. **다음에 무엇을 해야 하는가**

### 원칙 3. 쉬운 한국어를 쓴다
고등학생도 이해할 수 있어야 한다.

- 어려운 한자말, 번역투, 컨설턴트식 문구를 줄인다.
- 약어는 꼭 필요한 것만 쓴다.
- 처음 나올 때는 쉬운 풀어쓰기를 붙인다.

예시:
- `ledger` → `원장(기록 장부)`
- `shadow mode` → `운영 반영 전 로컬 검증 모드`
- `receiver` → `서버 수신점`
- `cutover` → `운영에 정식 반영`
- `deterministic assignment` → `같은 기준이면 항상 같은 그룹에 들어가게 하는 배정 방식`
- `ITT` → `배정 기준 분석(실제로 보냈는지와 별개로, 처음 나눈 그룹 기준으로 보는 방식)`

### 원칙 4. 과장하지 않는다
문서는 기대가 아니라 **확인된 사실** 기준으로 쓴다.

- “끝났다”는 말은 운영 반영까지 닫혔을 때만 쓴다.
- 로컬에서만 검증됐으면 `우리 기준 완료`, `운영 기준 미완료`로 분리한다.
- 원인이 아직 가설이면 `유력 가설`, `확정 아님`이라고 쓴다.

### 원칙 5. 문서마다 독자가 다름을 인정한다
같은 프로젝트라도 독자가 다르면 문서 구조를 다르게 쓴다.

- 대표/의사결정자: 사업 의미, 위험, 우선순위, 승인 포인트
- 운영자: 지금 무엇을 봐야 하고 무엇을 해야 하는가
- 개발자: 구현 위치, API, 스키마, 테스트, handoff

따라서 **대표용 본문 + 개발 부록** 구조를 기본으로 한다.

## 2. 공통 톤 규칙

### 2-1. 문체
- 짧고 단단하게 쓴다.
- 감정 표현보다 판단과 근거를 우선한다.
- 모호한 낙관 대신 현재 상태를 분명히 쓴다.
- 설명은 친절하게 하되, 어조는 건조하고 명확하게 유지한다.

### 2-2. 문장 길이
- 한 문장에 주장 하나만 넣는다.
- 문장이 길어지면 문단을 나눈다.
- 쉼표로 문장을 질질 끌지 않는다.

### 2-3. 표현 방식
- “해야 할 것 같습니다”보다 “해야 한다”를 쓴다.
- “가능할 것으로 보입니다”보다 “현재 조건에서는 가능하다 / 아직 어렵다”를 쓴다.
- “검토 필요”만 쓰지 말고, 누가 무엇을 결정해야 하는지 쓴다.

### 2-4. 숫자와 날짜
- 상대 표현보다 날짜를 쓴다.
- 숫자는 가능하면 실제 값을 쓴다.
- 퍼센트만 적지 말고 분모도 같이 적는다.

좋은 예:
- `2026-03-29 기준 live ledger row 0건`
- `120명 배정, 78건 conversion 적재`

나쁜 예:
- `최근 많이 좋아짐`
- `거의 완료`

## 3. 모든 문서에 공통으로 들어가야 하는 기본 구조

모든 결과보고서와 로드맵은 아래 순서를 기본으로 한다.

### 3-1. 상단 10초 요약
2-4문장 안에 아래를 넣는다.

- 이 문서가 다루는 주제
- 지금 결론
- 현재 가장 큰 병목
- 다음 행동

예시:
> Phase 1의 목적은 CRM 메시지가 실제 추가 매출을 만들었는지 확인하는 것이다. 현재 실험 장부와 진단 화면은 대부분 준비됐지만, 실제 결제 흐름에서 live 신호가 아직 안 들어와 측정 신뢰는 완전히 닫히지 않았다. 다음 액션은 실제 사이트에서 checkout/payment success 수신점을 연결하는 것이다.

### 3-2. 문서 목적
한 문장으로 쓴다.

예시:
- 이 문서는 Phase 1의 현재 상태, 막힌 점, 다음 순서를 대표와 개발팀이 같은 언어로 이해하도록 정리한다.

### 3-3. 이 작업이 하는 일
기술 용어 없이 먼저 설명한다.

예시:
- 이 단계는 `누구에게 메시지를 보냈고, 그 사람이 실제로 샀는지`를 연결해 보는 작업이다.

### 3-4. 왜 필요한가
프로젝트 전체와 연결한다.

예시:
- 이게 없으면 채널톡, 알림톡, Meta 비용, iROAS 숫자를 믿기 어렵다.

### 3-5. 현재 상태
아래 4가지를 분리한다.

- 확인된 것
- 아직 안 된 것
- 지금 막힌 이유
- 현재 주체

### 3-6. 산출물
파일명이 아니라 **사람이 이해할 결과물 이름**으로 먼저 적는다.

예시:
- 실험 장부
- 결제 블랙박스
- 운영자 화면
- 승인 패키지

필요하면 그 아래에 실제 API, DB, 화면 파일을 붙인다.

### 3-7. 우리 프로젝트에 주는 도움
이 항목은 반드시 넣는다.

질문 형태로 쓰면 좋다.
- 이 산출물이 생기면 무엇을 더 잘 결정할 수 있는가?
- 어떤 리스크를 줄이는가?
- 다음 Phase에서 무엇이 가능해지는가?

### 3-8. 다음 액션
반드시 시간 순서로 적는다.

- 지금 당장
- 이번 주
- 운영 승인 후

## 4. 결과보고서 규칙

결과보고서는 `무엇을 만들었는가`보다 **무엇이 검증됐는가**가 중심이다.

### 4-1. 결과보고서의 목표
- 이번 턴에서 무엇이 실제로 바뀌었는지 보여준다.
- 무엇은 아직 가설인지 구분한다.
- 다음 사람이 바로 이어받아도 되게 만든다.

### 4-2. 결과보고서 기본 구조

#### 1. 이번 턴 한 줄 결론
예시:
- 이번 턴은 `코호트 API를 실사용 가능한 수준으로 보강`했고, `실험 장부 승인 패키지`를 코드로 고정했다.

#### 2. 이번 턴의 목표
딱 1-3개만 쓴다.

#### 3. 실제로 바뀐 것
- 추가된 API
- 바뀐 로직
- 바뀐 화면
- 바뀐 문서

#### 4. 실측 결과
반드시 숫자로 쓴다.

#### 5. 무엇이 증명됐는가
예시:
- local shadow mode에서 `실험 생성 -> 배정 -> 주문 매핑 -> 결과 집계`가 닫혔다.

#### 6. 아직 증명되지 않은 것
예시:
- 실제 서비스에서는 payment success row가 아직 0건이다.

#### 7. 이 결과가 프로젝트에 주는 도움
예시:
- 이제 첫 라이브 실험 전에 코호트 기준 윈백 시점을 데이터로 잡을 수 있다.

#### 8. 다음 액션
한 개 또는 세 단계로 끝낸다.

### 4-3. 결과보고서에서 금지할 표현
- `완료`만 쓰고 기준을 안 적는 것
- `잘 됐다`, `의미 있었다`처럼 애매한 표현
- `운영에서도 될 것 같다`처럼 검증 없이 낙관하는 표현

### 4-4. 결과보고서에서 권장하는 표현
- `우리 기준 90% / 운영 기준 30%`
- `구현은 완료, live 연결 미완료`
- `유력 가설이지만 확정 아님`
- `이번 숫자는 uplift가 아니라 plumbing 검증 수치`

## 5. 로드맵 문서 규칙

로드맵은 할 일 목록이 아니라 **왜 이 순서여야 하는지 설명하는 문서**다.

### 5-1. 로드맵의 목표
- 무엇을 먼저 해야 하는지 정한다.
- 왜 그것이 먼저인지 설명한다.
- 무엇이 막히면 다음 단계가 의미 없어지는지 보여준다.

### 5-2. 로드맵에서 Phase를 쓰는 방식
각 Phase는 아래 5가지를 반드시 가져야 한다.

1. **목표**
2. **왜 지금 해야 하는가**
3. **산출물**
4. **완료 기준**
5. **다음 Phase에 주는 가치**

좋은 예:
- Phase 1은 CRM 메시지가 진짜 추가 매출을 만드는지 확인하는 측정 장치를 만든다.
- 그래서 실험 장부, 결제 블랙박스, 운영자 화면이 필요하다.
- 이게 끝나야 Phase 5의 Meta 비용과 Phase 7의 iROAS가 숫자로 연결된다.

### 5-3. Sprint 설명 규칙
Sprint는 기술명보다 역할명으로 먼저 설명한다.

좋은 예:
- `P1-S1`: 실험 장부
- `P1-S1A`: 결제 블랙박스
- `P1-S2`: 운영자 관제판

그 아래에 실제 코드/API를 붙인다.

### 5-4. 완료율 규칙
로드맵에는 가능하면 완료율을 한 개만 쓰지 말고 아래 중 하나를 쓴다.

- `우리 기준 / 운영 기준`
- 또는 `코드 구현 / 운영 반영`

좋은 예:
- `P1-S1 = 100% / 40%`

나쁜 예:
- `P1-S1 = 70%`

왜냐하면 70%가 로컬 검증인지, 운영 배포인지 알 수 없기 때문이다.

### 5-5. 의존성 규칙
로드맵은 반드시 `왜 이 순서인가`를 적는다.

예시:
- Meta 비용은 중요하지만, 실험 원장과 식별키가 먼저 없으면 단순 리포트로 끝난다.
- 카카오는 중요한 채널이지만, 원장보다 먼저 붙이면 자동화만 생기고 측정 정확도는 안 생긴다.

### 5-6. 승인 포인트를 숨기지 않는다
대표가 승인해야 하는 항목은 문서 아래가 아니라 눈에 잘 띄게 적는다.

예시:
- DB 스키마 추가 승인
- 외부 API 자격증명 연결 승인
- 자동 발송 활성화 승인

## 6. Phase 상세 메모 규칙

Phase 상세 메모는 가장 기술적일 수 있다. 하지만 그래도 **의도 설명이 앞에 있어야 한다.**

### 6-1. 상세 메모의 필수 순서

1. 상단 요약
2. 이 Phase를 쉬운 비유로 설명
3. 각 Sprint의 역할
4. 현재 상태
5. 왜 아직 100%가 아닌가
6. 다음 순서
7. 개발 부록

### 6-2. 쉬운 비유를 적극 사용한다
단, 감상적인 비유 말고 **구조를 쉽게 만드는 비유**만 쓴다.

좋은 예:
- 실험 장부 = 시험 채점표
- 결제 블랙박스 = 사고 원인 기록 장치
- 운영자 화면 = 관제실 모니터

### 6-3. 상세 메모에서 꼭 넣어야 하는 질문
각 Sprint마다 아래를 넣는다.

- 무엇을 하는가
- 왜 필요한가
- 지금 어디까지 왔는가
- 아직 안 된 이유는 무엇인가
- 산출물은 무엇인가
- 우리 프로젝트에 어떻게 도움이 되는가

### 6-4. 상세 메모에서 코드 위치는 맨 뒤로 보낸다
코드 경로는 중요하지만, 본문 첫머리에서 길게 나오면 독해를 깨뜨린다.

권장 순서:
- 먼저 쉬운 설명
- 그다음 상태/의미
- 마지막에 코드 위치

## 7. 용어 사용 규칙

### 7-1. 처음 나오는 기술 용어는 반드시 풀어쓴다
예시:
- `holdout(아무 메시지도 보내지 않는 비교 그룹)`
- `control(기본 비교 그룹)`
- `treatment(메시지를 보내는 실험 그룹)`
- `receiver(서버 수신점)`
- `ledger(기록 원장)`

### 7-2. business term과 tech term을 구분한다
- business term: 대표와 운영자가 이해해야 하는 말
- tech term: 개발 부록에서 필요한 말

예시:
- 본문: `결제 블랙박스`
- 부록: `PG attribution ledger`

### 7-3. 이름이 오해를 부르면 즉시 바꾼다
예시:
- `first_purchase_channel`이 실제로는 결제 수단이면 `first_payment_channel`로 바꾼다.

## 8. 산출물 쓰는 규칙

문서에는 “파일이 생겼다”가 아니라 **운영상 무엇이 생겼는지**를 써야 한다.

좋은 산출물 표현:
- 실험 장부
- 승인 패키지
- 결제 비교표
- 코호트 입력값 API
- CRM 운영 허브

보조로 실제 구현물을 붙인다.

예시:
- 실험 장부
  - `crm_experiments`
  - `crm_assignment_log`
  - `crm_message_log`
  - `crm_conversion_log`

## 9. 우리 프로젝트에 도움이 되는지 쓰는 규칙

이 항목은 빠지면 안 된다.
문서의 모든 큰 섹션은 아래 중 최소 하나에 답해야 한다.

- 이 작업이 끝나면 어떤 판단을 더 잘할 수 있는가?
- 어떤 비용 낭비를 줄이는가?
- 어떤 숫자를 이제 믿을 수 있게 되는가?
- 다음 Phase를 왜 더 안전하게 여는가?

예시:
- 코호트 API가 생기면 `21일`, `30일`, `90일` 윈백 시점을 감이 아니라 데이터로 정할 수 있다.
- 결제 블랙박스가 생기면 `(not set)` 문제를 추측이 아니라 실제 결제 흐름으로 좁힐 수 있다.

---

## 10. 사실, 판단, 추정 쓰는 규칙

문서는 아래 3가지를 구분해서 쓴다.

### 사실
확인된 값, 구현된 것, 테스트 통과, API 응답

예시:
- `live ledger row 0건`
- `pytest 6 passed`

### 판단
사실을 보고 내린 현재 해석

예시:
- 현재 가장 큰 병목은 화면 부족이 아니라 live signal 부족이다.

### 추정
확정 전 가설

예시:
- `(not set) = PG 리다이렉트`는 유력 가설이지만 아직 확정 사실은 아니다.

권장 표기:
- `사실:`
- `현재 판단:`
- `유력 가설:`

## 11. 금지 표현 목록

아래 표현은 되도록 쓰지 않는다.

- `가시화`
- `제고`
- `고도화`만 단독으로 쓰기
- `인사이트 도출`
- `정합성 확보`만 쓰고 무엇이 맞는지 안 쓰기
- `완료 예정`
- `추후 검토`
- `효율화`
- `리소스`
- `유관`
- `시사점`

대체 표현:
- `가시화` → `화면으로 보여줌`
- `제고` → `높임`
- `고도화` → `더 정확하게 만듦` 또는 `자동화 범위를 넓힘`
- `인사이트 도출` → `무엇이 문제인지 찾음`
- `정합성 확보` → `숫자가 서로 왜 다른지 설명 가능하게 만듦`
- `완료 예정` → `다음 단계에서 진행`
- `추후 검토` → `언제, 누가, 무엇을 볼지 적기`

## 13. 최종 체크리스트

문서를 저장하기 전에 아래를 확인한다.

### 공통
- [ ] 상단 10초 요약이 있다
- [ ] 문서 목적이 한 문장으로 있다
- [ ] 쉬운 한국어가 먼저 나오고, 기술 용어는 나중에 나온다
- [ ] 산출물이 무엇인지 적었다
- [ ] 우리 프로젝트에 주는 도움이 적혀 있다
- [ ] 다음 액션이 시간 순서로 적혀 있다
- [ ] 사실과 가설을 구분했다

### 결과보고서
- [ ] 무엇이 바뀌었는지 적었다
- [ ] 무엇이 아직 안 닫혔는지 적었다
- [ ] 실측 숫자가 있다
- [ ] 운영 기준 완료 여부를 분리했다

### 로드맵
- [ ] 왜 이 순서인지 적었다
- [ ] 승인 포인트가 보인다
- [ ] 의존성이 설명돼 있다
- [ ] 완료율 기준이 무엇인지 보인다

### 상세 메모
- [ ] Phase를 쉬운 비유로 설명했다
- [ ] 각 Sprint의 목적과 사업 의미를 적었다
- [ ] 코드 경로는 본문 뒤에 넣었다
