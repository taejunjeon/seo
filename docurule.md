## 0. 이 문서가 필요한 이유

현재 문서들은 방향은 좋지만, 종종 아래 문제가 생긴다.

- 기술 용어가 먼저 나와서 문서의 의도와 사업 의미가 늦게 보인다.
- `무엇을 만들었는가`는 보이는데, `그래서 우리 프로젝트에 무슨 도움이 되는가`가 약하다.
- 로컬 검증 완료와 운영 반영 완료가 섞여 보여, 실제 완성도가 헷갈린다.
- 결과보고서, 로드맵, 작업 메모의 톤이 서로 달라 읽는 사람이 문서마다 해석을 다시 해야 한다.

그래서 앞으로 문서는 아래 원칙을 지킨다.

## 1. 최상위 원칙

### 원칙 1. 결론이 먼저다
문서는 항상 **결론 -> 이유 -> 근거 -> 다음 행동** 순서로 쓴다.

나쁜 예:
- API, 테이블, 스크립트, 엔드포인트를 길게 설명한 뒤 마지막에 의미를 적는다.

좋은 예:
- 이 작업의 목적은 `메시지가 실제 추가 매출을 만들었는지 확인하는 것`이다.
- 그래서 `실험 장부`, `결제 블랙박스`, `운영자 화면`을 만든다.
- 현재는 장치는 대부분 준비됐고, 실제 서비스 연결만 남았다.

### 원칙 2. 기술보다 의도와 목적이 먼저다
각 문단은 아래 질문 중 최소 3개에 답해야 한다.

1. 이 작업은 **무엇을 하는가**
2. **왜 필요한가**
3. **산출물이 무엇인가**
4. **우리 프로젝트에 어떤 도움이 되는가**
5. **지금 막힌 것이 무엇인가**
6. **다음에 무엇을 해야 하는가**

### 원칙 3. 쉬운 한국어를 쓴다
고등학생도 이해할 수 있어야 한다.

- 어려운 한자말, 번역투, 컨설턴트식 문구를 줄인다.
- 약어는 꼭 필요한 것만 쓴다.
- 처음 나올 때는 쉬운 풀어쓰기를 붙인다.

예시:
- `ledger` → `원장(기록 장부)`
- `shadow mode` → `운영 반영 전 로컬 검증 모드`
- `receiver` → `서버 수신점`
- `cutover` → `운영에 정식 반영`
- `deterministic assignment` → `같은 기준이면 항상 같은 그룹에 들어가게 하는 배정 방식`
- `ITT` → `배정 기준 분석(실제로 보냈는지와 별개로, 처음 나눈 그룹 기준으로 보는 방식)`

### 원칙 4. 과장하지 않는다
문서는 기대가 아니라 **확인된 사실** 기준으로 쓴다.

- “끝났다”는 말은 운영 반영까지 닫혔을 때만 쓴다.
- 로컬에서만 검증됐으면 `우리 기준 완료`, `운영 기준 미완료`로 분리한다.
- 원인이 아직 가설이면 `유력 가설`, `확정 아님`이라고 쓴다.

### 원칙 5. 문서마다 독자가 다름을 인정한다
같은 프로젝트라도 독자가 다르면 문서 구조를 다르게 쓴다.

- 대표/의사결정자: 사업 의미, 위험, 우선순위, 승인 포인트
- 운영자: 지금 무엇을 봐야 하고 무엇을 해야 하는가
- 개발자: 구현 위치, API, 스키마, 테스트, handoff

따라서 **대표용 본문 + 개발 부록** 구조를 기본으로 한다.

## 2. 공통 톤 규칙

### 2-2. 문장 길이
- 한 문장에 주장 하나만 넣는다.
- 문장이 길어지면 문단을 나눈다.
- 쉼표로 문장을 질질 끌지 않는다.

### 2-3. 표현 방식
- “해야 할 것 같습니다”보다 “해야 한다”를 쓴다.
- “가능할 것으로 보입니다”보다 “현재 조건에서는 가능하다 / 아직 어렵다”를 쓴다.
- “검토 필요”만 쓰지 말고, 누가 무엇을 결정해야 하는지 쓴다.

### 2-4. 숫자와 날짜
- 상대 표현보다 날짜를 쓴다.
- 숫자는 가능하면 실제 값을 쓴다.
- 퍼센트만 적지 말고 분모도 같이 적는다.
- 문서 최상단에는 `작성 시각: YYYY-MM-DD HH:MM KST` 형식으로 시분과 KST를 반드시 표기한다.

좋은 예:
- `작성 시각: 2026-04-17 14:30 KST`
- `2026-03-29 기준 live ledger row 0건`
- `120명 배정, 78건 conversion 적재`

나쁜 예:
- `최근 많이 좋아짐`
- `거의 완료`

### 2-5. 앵커 네비게이션 (Obsidian wiki 링크)
문서가 길어지면 대표는 요약표만 보고 필요한 곳만 골라 읽는다. 그래서 요약표와 본문을 양방향으로 연결한다.

이 프로젝트의 정본 리더는 **Obsidian**이다. Obsidian은 GitHub 스타일 앵커(`[text](#소문자-하이픈)`)와 `<a id>` HTML 태그를 **인식하지 않는다.** 따라서 같은 문서 내 이동은 Obsidian wiki 링크 `[[#헤딩]]` 형식으로 고정한다.

#### 금지 패턴 (쓰면 반드시 깨진다)

| 금지 | 이유 |
|---|---|
| `[Phase1-Sprint1](#phase1-sprint1)` | GitHub 스타일 소문자-하이픈 앵커 — Obsidian에서 클릭해도 이동 안 함 |
| `<a id="phase1-sprint1"></a>` | HTML 앵커 태그 — Obsidian 에디팅/리딩 뷰 모두 무시 |
| `[[#phase1-sprint1]]` (소문자) | 실제 헤딩은 `Phase1-Sprint1` — 대소문자 틀려서 미매치 |
| `#### Phase1-Sprint1 — 이름 텍스트` + `[[#Phase1-Sprint1]]` | 헤딩이 길면 링크 타겟과 안 맞음. 이름은 헤딩 아래 줄로 분리 |
| 테이블 셀 안 `[[#X\|이동]]`에서 `\|` 이스케이프 빠뜨림 | Obsidian 테이블 구분자와 충돌해 링크 파싱 실패 |

#### 필수 패턴

1. **같은 문서 내 이동 링크는 `[[#헤딩]]` 또는 `[[#헤딩|표시텍스트]]` 형식만 쓴다.**
2. **Sprint 헤딩은 식별자만 남긴다.** `#### Phase1-Sprint1`까지만 쓰고, 이름은 헤딩 아래 `**이름**: ...` 줄로 분리한다.
3. **링크 타겟은 헤딩 텍스트와 공백·하이픈·대소문자까지 바이트 단위로 정확히 일치해야 한다.**
4. **테이블 셀 안의 alias는 `\|`로 이스케이프한다.**
   예: `| Phase1 | [[#Phase1-Sprint1]] | ... | [[#Phase1-Sprint1\|이동]] |`
5. **예시 표·코드 샘플은 fenced code block(```` ```md ```` )으로 감싼다.** 그러지 않으면 설명용 예시가 실제 링크로 렌더링되어 클릭 시 "없는 헤딩" 에러가 뜬다.

#### 문서 저장 전 필수 검증

문서를 저장하기 전에 반드시 아래 스크립트(§A 부록 참조)를 돌려 모든 wiki 링크가 실제 헤딩과 매치되는지 확인한다. 미매치가 하나라도 있으면 저장하지 않는다.

```bash
python3 scripts/validate_wiki_links.py <문서경로>
# 또는 아래 §A의 인라인 스니펫을 직접 실행
```

#### 외부 렌더러 호환이 필요한 경우

GitHub/기타 마크다운 뷰어와 **같이** 쓰는 문서라면 이중 표기를 쓴다:
`[Phase1-Sprint1](#Phase1-Sprint1) [[#Phase1-Sprint1]]`
하지만 **기본은 Obsidian wiki 링크**다. 외부 호환이 불필요한 경우 이중 표기하지 않는다.

## 3. 모든 문서에 공통으로 들어가야 하는 기본 구조

모든 결과보고서와 로드맵은 아래 순서를 기본으로 한다.

### 3-1. 상단 10초 요약
2-4문장 안에 아래를 넣는다.

- 이 문서가 다루는 주제
- 지금 결론
- 현재 가장 큰 병목
- 다음 행동

예시:
> Phase 1의 목적은 CRM 메시지가 실제 추가 매출을 만들었는지 확인하는 것이다. 현재 실험 장부와 진단 화면은 대부분 준비됐지만, 실제 결제 흐름에서 live 신호가 아직 안 들어와 측정 신뢰는 완전히 닫히지 않았다. 다음 액션은 실제 사이트에서 checkout/payment success 수신점을 연결하는 것이다.

### 3-1B. Phase-Sprint 요약표
10초 요약 바로 다음에 **Phase-Sprint 요약표**를 반드시 넣는다. 이 표는 문서의 목차이자 대시보드다. 대표는 여기서 전체 그림을 보고, 필요한 Sprint만 링크로 내려가서 읽는다.

필수 컬럼 (아래는 **예시**다. 실제 문서에 옮겨 쓸 때만 링크가 작동한다):

```md
| Phase | Sprint | 이름 | 담당 | 상태(우리/운영) | 상세 |
|---|---|---|---|---|---|
| Phase1 | [[#Phase1-Sprint1]] | 실험 장부 | Codex + Claude | 100% / 40% | [[#Phase1-Sprint1\|이동]] |
| Phase1 | [[#Phase1-Sprint2]] | 결제 블랙박스 | TJ + Codex | 70% / 0% | [[#Phase1-Sprint2\|이동]] |
| Phase1 | [[#Phase1-Sprint3]] | 운영자 관제판 | Claude | 50% / 0% | [[#Phase1-Sprint3\|이동]] |
| Phase2 | [[#Phase2-Sprint4]] | 코호트 윈백 | Codex | 대기 | [[#Phase2-Sprint4\|이동]] |
```

규칙:
- 표 바로 위에는 `## Phase-Sprint 요약표` 헤딩을 둔다. 이 헤딩이 모든 `▲ 요약표로` 링크의 목적지다.
- 담당 컬럼은 `TJ`, `Codex`, `Claude`(= Claude Code) 중 하나 또는 조합을 쓴다.
- 상태 컬럼은 `우리 기준 % / 운영 기준 %` 형식을 지킨다. 단일 수치 금지.
- 대기/시작 전이면 `대기`, 완전히 닫힌 Sprint는 `100% / 100%`로 적는다.

### 3-2. 문서 목적
한 문장으로 쓴다.

예시:
- 이 문서는 Phase 1의 현재 상태, 막힌 점, 다음 순서를 대표와 개발팀이 같은 언어로 이해하도록 정리한다.

### 3-3. 이 작업이 하는 일
기술 용어 없이 먼저 설명한다.

예시:
- 이 단계는 `누구에게 메시지를 보냈고, 그 사람이 실제로 샀는지`를 연결해 보는 작업이다.

### 3-4. 왜 필요한가
프로젝트 전체와 연결한다.

예시:
- 이게 없으면 채널톡, 알림톡, Meta 비용, iROAS 숫자를 믿기 어렵다.

### 3-5. 현재 상태
아래 4가지를 분리한다.

- 확인된 것
- 아직 안 된 것
- 지금 막힌 이유
- 현재 주체

### 3-6. 산출물
파일명이 아니라 **사람이 이해할 결과물 이름**으로 먼저 적는다.

예시:
- 실험 장부
- 결제 블랙박스
- 운영자 화면
- 승인 패키지

필요하면 그 아래에 실제 API, DB, 화면 파일을 붙인다.

### 3-6B. 역할 구분 (TJ / Codex / Claude Code)
Phase나 Sprint를 설명할 때 각 작업이 **누가 실행하는가**를 분리해서 적는다. 이게 빠지면 "누가 다음에 움직여야 하는가"가 흐려진다.

세 주체:

- **TJ가 할 일**: 의사결정, 승인, 외부 계정·API 키 발급, 실데이터 확인, 운영 반영 판단
- **Codex가 할 일**: 백엔드, DB 설계, 측정 알고리즘, 데이터 적재, 분석 로직
- **Claude Code가 할 일**: 프론트엔드, UI 문구, 문서 작성, 시각화, UX 미세 조정

작성 규칙:
- 각 Sprint 상세에 `### 역할 구분` 소제목을 반드시 둔다.
- 세 주체 모두 해당 없으면 생략하지 말고 `해당 없음`으로 명시한다.
- 경계가 애매하면 `공동`으로 적고 어느 부분을 어떻게 나눌지 한 줄로 설명한다.

예시:
```
### 역할 구분
- TJ: 아임웹 관리자 화면에서 결제 성공 redirect URL을 실제로 눌러 재현
- Codex: receiver 엔드포인트 추가, `crm_payment_attribution` 테이블 적재
- Claude Code: 운영자 대시보드에 결제 블랙박스 탭 추가, 실패 사유 문구 정리
```

### 3-6B-1. TJ 위임 전 사전 시도 원칙

`[TJ]` 태그를 붙이기 전에 Codex나 Claude Code가 먼저 시도할 수 있는지 반드시 검토한다. TJ의 시간은 **사업 판단, 승인, 2FA가 필요한 외부 계정 작업, 실결제처럼 사람만 할 수 있는 일**에 써야 한다. 자동화 가능한 확인 작업까지 TJ에게 넘기면 TJ가 병목이 되고, 같은 질문에 같은 답을 반복해야 한다.

#### Codex / Claude Code가 먼저 시도할 것

이 목록에 해당하면 `[TJ]`를 바로 붙이지 않고, Codex 또는 Claude Code가 한 번 시도한 뒤 결과를 보고한다.

- 운영 DB read-only 쿼리 (Postgres, Imweb, Toss, PlayAuto, attribution ledger)
- 로컬 SQLite, 파일 기반 데이터 조회
- 이미 발급된 토큰으로 API 호출 (GA4 Data API, Meta Graph API, Toss Payments API, Imweb v2 API, BigQuery)
- `curl`, `git log`, `git diff`, `tsc --noEmit`, `npm test`, Playwright 헤드리스 실행
- 공개 문서, 공개 dataset, 웹 리서치, GitHub 이슈·PR 조회
- 환경 변수, 설정 파일, PM2 health 로그 열람
- 운영 화면 중 로그인 없이 접근 가능한 페이지 캡처 (`curl`, Playwright)
- 이미 권한이 있는 BigQuery project/dataset의 `events_*` sanity 쿼리

#### 진짜 TJ만 가능한 것

이 목록은 자동화 불가. `[TJ]` 태그를 붙이되 이유를 한 줄로 명시한다.

- 사업 의사결정, 예산 승인, 운영 반영 판단
- 2FA가 필요한 외부 계정 로그인 (Meta Ads Manager, Google Ads, 아임웹 admin, Toss 관리자)
- 신규 API 키, OAuth credential, 서비스 계정 키 발급
- 실결제 테스트 (카드, 가상계좌 입금, 네이버페이)
- 회사 내부 팀 문의 (마케팅, revenue, 개발 운영팀)
- 로그인 필요한 외부 dashboard 스크린샷
- 운영 DB write, 운영 배포, 광고 업로드, GTM 게시
- 개인정보/보안/법무 승인

#### 애매한 것 (먼저 시도 후 에스컬레이션)

다음은 `[Codex]`/`[Claude Code]`가 1회 시도 후 실패 시 `[TJ]`로 에스컬레이션한다.

- 서비스 계정 BigQuery dataset 권한 추가 — 먼저 현재 권한으로 시도, `permission denied` 확인 후 TJ 요청
- 특정 Meta Events Manager 화면 확인 — Graph API로 같은 데이터 조회 먼저 시도
- 외부 dashboard 데이터 — 공개 API나 export 경로가 있는지 먼저 확인
- 계정 권한 범위 확인 — `GET /me/permissions` 같은 자체 조회 먼저

#### 작성 절차

Sprint 실행 단계를 쓸 때 아래 순서로 자문한다.

1. 이 단계를 Codex 또는 Claude Code가 현재 권한으로 시도할 수 있는가?
2. 시도 가능하면 `[Codex]` 또는 `[Claude Code]` 단계로 먼저 배치하고, 실패 시 TJ 에스컬레이션 단계를 뒤에 둔다.
3. 순수 TJ-only 작업이면 그 이유를 단계 뒤에 한 줄로 적는다. 예: `— 2FA 필요`, `— 실결제 발생`, `— 예산 승인 필요`
4. 같은 에스컬레이션 사유가 Sprint 여러 곳에서 반복되면 사유 해소를 별도 Sprint나 `### 남은 블로커`로 분리한다. 예: 서비스 계정 권한 부족이 반복되면 권한 확보를 독립 작업으로 승격.

#### 예시

나쁜 예:
```
1. [TJ] thecleancoffee BigQuery raw export 적재 여부 확인
2. [TJ] Meta API로 최근 7일 purchase 수 확인
3. [TJ] 운영 Postgres에서 tb_playauto_orders 최신 sync 시각 확인
```
— 세 개 다 Codex가 기존 권한으로 할 수 있다.

좋은 예:
```
1. [Codex] GA4 Data API로 coffee property(`326949178`) 최근 7일 `purchase` 이벤트 조회. 실패 시 2번으로 에스컬레이션
2. [Codex] BigQuery `analytics_326949178.events_*` sanity 쿼리 시도. `permission denied`면 3번으로 에스컬레이션
3. [TJ] 서비스 계정 `seo-656@seo-aeo-487113.iam.gserviceaccount.com`에 `analytics_326949178` dataset read 권한 추가 요청 — 이유: GCP 콘솔 IAM 변경은 프로젝트 소유자 로그인 필요
4. [Codex] 운영 Postgres `tb_playauto_orders`의 `MAX(synced_at)` 조회
5. [Codex] Meta Graph API `act_*/insights`로 최근 7일 purchase count 조회
```

#### 이 규칙과 §5-6 승인 포인트의 차이

`§5-6 승인 포인트를 숨기지 않는다`는 **TJ가 반드시 판단해야 하는 지점을 문서에 드러내는 규칙**이다. 이 규칙(§3-6B-1)은 **판단 이전의 확인 작업을 TJ에게 넘기지 않는 규칙**이다. 두 규칙은 보완 관계다. 확인은 자동화로 줄이고, 승인은 명시해서 드러낸다.

### 3-6B-2. TJ 승인 요청은 선택 가능한 안건으로 만든다

TJ에게 `검토`, `승인`, `확정`을 요청할 때는 절대 후보만 던지지 않는다. Codex 또는 Claude Code가 먼저 추천안을 만들고, TJ가 `YES` 또는 `NO + 수정사항`으로 답할 수 있게 작성한다.

#### 필수 구성

승인 요청 문서나 Sprint 실행 단계에는 아래 7가지를 반드시 넣는다.

| 항목 | 작성 규칙 |
|---|---|
| 추천안 | `추천안 A`처럼 기본안을 하나 제시한다. 복수 대안이 있으면 `A/B/C`로 나누되, 하나를 명확히 추천한다. |
| 답변 형식 | TJ가 그대로 답할 수 있게 `YES`, `NO: 기존안 유지`, `NO: 수정사항` 같은 문장을 제공한다. |
| 추천 이유 | 왜 이 안이 사업/운영/개발 관점에서 좋은지 쉬운 말로 설명한다. |
| Codex 추천 | `제 추천: YES`처럼 결론을 명시한다. |
| 자신감 점수 | `추천 자신감 76%`처럼 숫자로 적고, 낮은 이유를 함께 쓴다. |
| 부족 데이터 | 자신감이 낮다면 어떤 데이터가 더 필요한지 적는다. 예: 최근 30일 export, 상담원 메모 20건, 운영 로그. |
| 승인 후 작업 | TJ가 YES/NO로 답한 뒤 Codex/Claude Code가 무엇을 바꾸는지 적는다. |

#### 프론트엔드가 필요한 경우

문서만으로 이해하기 어려운 승인 건이면 MD만 쓰지 않는다. 아래 중 하나라도 해당하면 프론트엔드 또는 시각 자료를 만든다.

- 화면 배치, 버튼 위치, 폼 단계, 대시보드 정보 구조처럼 실제로 봐야 판단 가능한 경우
- 선택지가 3개 이상이고, 각 선택지가 사용자 흐름을 크게 바꾸는 경우
- 대표/운영자가 텍스트만 보고는 결과를 상상하기 어려운 경우
- 모바일/PC 차이가 중요한 경우

반대로 상태명 8개, 운영 규칙, API 토큰 방식처럼 텍스트 표로 충분히 이해되는 사안은 MD가 우선이다. 이때도 왜 MD로 충분한지 한 줄로 적는다.

#### 금지 예시

나쁜 예:

```md
[TJ] 리드 상태값을 승인한다 — 신규, 연락시도, 연락완료, 예약완료, 방문완료, 결제완료, 노쇼, 불량/중복 중 실제 상담원이 쓰는 표현을 확정한다.
```

왜 나쁜가:

- TJ가 무엇을 기준으로 고쳐야 하는지 알기 어렵다.
- Codex의 추천이 없다.
- YES/NO로 답할 수 없다.
- 자신감과 부족 데이터가 없다.

#### 좋은 예시

```md
[TJ] 리드 상태값 추천안 A에 답한다.
추천안 A: 신규, 연락중, 상담완료, 예약확정, 방문완료, 결제완료, 노쇼, 제외
제 추천: YES
자신감: 76%
이유: 내부 상태 코드는 유지하고 화면 표시명만 바꾸면 되므로 위험이 낮다. 다만 실제 상담원 메모가 없어 현장 용어 일치 자신감은 58%다.
답변 형식: `YES` 또는 `NO: 연락중은 연락시도로 유지`
YES 이후 Codex 작업: admin 드롭다운 표시명, 문서 상태값 정본, Playwright 테스트를 업데이트한다.
부족 데이터: 최근 30일 아임웹 입력폼 export, 상담원 처리 메모 20건.
```

### 3-6C. 실행 단계 (진행 중 Sprint 필수)

역할 구분은 "누가 무엇을 담당하는가"의 **카테고리 요약**이다. 이것만으로는 "지금 다음에 누가 뭘 해야 하는가"가 흐려진다. 그래서 진행 중 Sprint는 **기본 진행 순서와 실제 차단 관계**를 실행 단계에 별도로 적는다.

#### 언제 필수인가

| 상태 | 실행 단계 섹션 |
|---|---|
| 100% / 100% (완료) | **쓰지 않는다.** 잔여 작업이 있으면 완료가 아니거나, 다른 Sprint/문서 레벨 "다음 액션"으로 옮겨야 한다 |
| 진행 중 (0% < X < 100%) | **필수** — 지금부터 완료까지 전 단계를 기본 진행 순서로 적고, 병렬 가능 여부를 함께 적는다 |
| 대기 (0% / 0%) | 진입 조건만 있어도 되지만, 이미 알려진 초안 단계가 있으면 적는다 |

#### 형식 규칙

- 번호 리스트를 쓴다. 번호 순서는 기본 진행 순서를 표현하고, 실제 차단 관계는 `의존성:` 문장으로 확정한다.
- 각 단계는 반드시 **담당 태그**로 시작한다: `[TJ]`, `[Codex]`, `[Claude Code]`, 또는 조합 `[TJ+Codex]`.
- 각 진행 중 Sprint 상세에는 `### 완료한 것`과 `### 남은 것`을 분리해 적는다.
- `### 완료한 것`에는 이미 끝난 단계를 체크박스(`- [x]`)로 남긴다. 무엇을 끝냈는지와 검증 근거를 함께 적는다.
- `### 남은 것`에는 100%까지 남은 단계를 체크박스(`- [ ]`)로 남긴다. 진행률이 55%라면 "나머지 45%가 무엇인지"가 이 섹션만 보고 보여야 한다.
- 한 단계에 반드시 `무엇`, `왜`, `어떻게`, `산출물`, `검증`이 보여야 한다.
- `무엇`: 실제로 만질 대상과 작업 범위를 쓴다. 예: `AIBIO 첫 실험 랜딩 후보 5개 URL을 1개로 좁힌다`.
- `왜`: 이 작업이 사업 판단, 운영 안정성, 데이터 정확성 중 무엇을 해결하는지 쓴다.
- `어떻게`: 사용할 파일, 화면, API, 쿼리, 스크립트, 명령어, 비교 기준을 구체적으로 쓴다.
- `산출물`: 문서 표, API, 화면, 테스트 결과, 승인 결과처럼 끝났을 때 남는 것을 쓴다.
- `검증`: 어떤 명령, 화면 확인, 수치 기준, 승인 조건으로 완료 여부를 볼지 쓴다.
- 한 단계가 길어지면 `무엇: ... 왜: ... 어떻게: ... 산출물: ... 검증: ...` 형식으로 쪼개 쓴다. 그래도 길면 하위 줄에 같은 라벨을 유지한다.
- 병렬 가능한 단계는 같은 번호 아래 `a/b/c`로 분기한다.
- `[TJ]` 단계 다음에 `[Codex]` 또는 `[Claude Code]` 단계가 오면, 후속 단계가 TJ 작업 완료를 기다려야 하는지 반드시 적는다.
  - 표기값은 `의존성: 선행필수`, `의존성: 병렬가능`, `의존성: 부분병렬` 중 하나를 쓴다.
  - `선행필수`는 TJ의 승인, 권한, 계정 접근, 원본 파일, 외부 답변이 없으면 후속 산출물이 확정되지 않는 경우다.
  - `병렬가능`은 후속 단계가 다른 원천, 로컬 코드, 운영 DB read-only 확인처럼 TJ 작업과 독립적으로 진행되는 경우다.
  - `부분병렬`은 스크립트, 화면 자리, 공통 쿼리는 먼저 만들 수 있지만, 최종 수치나 특정 source 항목은 TJ 결과가 들어와야 닫히는 경우다.
- 완료된 단계는 지우지 말고 체크박스(`- [x]`)로 표시한다. 취소선은 더 이상 권장하지 않는다. 문서 독자가 완료된 내용을 빠르게 스캔하기 어렵기 때문이다.

#### 금지

- 담당 태그 없는 단계: `1. 스키마 설계` ✗
- 모호한 단계: `적절히 처리`, `필요하면 작업`, `검토 후 진행` ✗
- 파일명이나 URL만 던지는 단계: `기존 문서와 route 후보 대조 — /shop_view, /main, /59` ✗
- 완료 기준 없는 단계: 최소한 `산출물`과 `검증`이 보여야 한다

#### 예시

나쁜 예:

```
1. [Codex] 기존 로컬 문서와 route 후보 대조 — `/shop_view`, `/main`, `/bio_pulse_M`, `/56`, `/59`
```

왜 나쁜가:
- 무엇을 대조하는지 불분명하다.
- 왜 이 URL들을 보는지 알 수 없다.
- 어떤 문서, 화면, 코드로 확인하는지 없다.
- 대조가 끝났을 때 무엇이 남는지 없다.
- 완료됐는지 검증할 방법이 없다.

좋은 예:

```
1. [Codex] AIBIO 첫 실험 랜딩 후보를 5개 URL에서 1개로 좁힌다 — 무엇: `/shop_view?idx=25`, `/59`, `/main`, `/bio_pulse_M`, `/56`의 역할, 유입, 폼 연결 여부를 표로 비교한다. 왜: 전체 사이트 이관 전에 리드 원장 실험을 한 URL에서 시작해 운영 리스크를 줄이기 위해서다. 어떻게: `imweb/aibio-redirect-map.md`, `aibio/아임웹폼제출API검토.md`, `frontend/src/app/aibio-funnel/page.tsx`, 첨부 아임웹 통계 화면의 인기 페이지 순서를 대조한다. 산출물: `imweb/!imwebplan.md`의 첫 실험 랜딩 결정 표와 redirect map 업데이트. 검증: 각 URL에 source, 기준 시각, confidence가 기록되고 TJ 승인 항목이 분리되어야 한다.
```

실행 단계 전체 예시:

```
### 실행 단계

1. [Codex] TikTok 광고 CSV 적재 스키마를 만든다 — 무엇: `tiktok_ads_daily` 테이블 컬럼을 date, campaign_id, spend, purchase_value, source_file 기준으로 정의한다. 왜: 광고비와 구매액을 같은 날짜·캠페인 단위로 비교하기 위해서다. 어떻게: 기존 `ads` 테이블 네이밍과 `data/!datacheckplan.md` 백필 절차를 맞춘다. 산출물: schema draft와 dry-run SQL. 검증: 샘플 CSV 1건이 중복 없이 dry-run 통과.
2. [TJ] TikTok Ads Manager에서 캠페인 CSV를 export한다 — 무엇: 2026-04-01부터 2026-04-17까지 캠페인별 spend/purchase_value CSV를 내려받는다. 왜: 외부 광고 관리자 원본은 2FA 로그인 권한이 필요하기 때문이다. 어떻게: TikTok Ads Manager > Reports > campaign daily export를 사용한다. 산출물: 원본 CSV. 검증: 기간, 통화, timezone, 계정 ID가 파일명 또는 메모에 기록된다. 의존성: 선행필수.
3. [Codex] CSV 파싱·적재 배치 스크립트를 만든다 — 무엇: 2번 CSV를 로컬 DB에 dry-run 후 apply하는 스크립트다. 왜: 수작업 복사로 숫자가 틀어지는 것을 막기 위해서다. 어떻게: 백업 -> dry-run -> apply -> 중복/금액 검증 순서를 따른다. 산출물: import script와 dry-run 로그. 검증: 총 spend가 CSV 원본 합계와 1원 단위로 일치. 의존성: 부분병렬.
```

## 4. 결과보고서 규칙

결과보고서는 `무엇을 만들었는가`보다 **무엇이 검증됐는가**가 중심이다.

### 4-1. 결과보고서의 목표
- 이번 턴에서 무엇이 실제로 바뀌었는지 보여준다.
- 무엇은 아직 가설인지 구분한다.
- 다음 사람이 바로 이어받아도 되게 만든다.

### 4-2. 결과보고서 기본 구조

> **중요**: 결과보고서에서는 `한 줄 결론` 섹션을 별도로 쓰지 않는다. 10초 요약의 두 번째 문장이 결론 역할을 한다. 두 번 쓰면 독자가 같은 말을 두 번 읽게 된다.

#### 1. 10초 요약 + 고등학생 비유
10초 요약(§3-1) 바로 아래에 **고등학생도 이해할 비유 1-2문장**을 반드시 붙인다.

예시:
- 이번 작업은 학교 시험 채점표에 실제로 누가 몇 점 받았는지 써 넣는 일이다. 예전에는 `누가 시험을 봤다`까지만 기록됐고, 점수는 눈대중이었다.
- 이번 작업은 CCTV는 이미 붙어 있는데, 녹화 버튼이 꺼져 있던 것을 켜는 일이다.

#### 2. 이번 턴의 목표
딱 1-3개만 쓴다.

#### 3. 실제로 바뀐 것
- 추가된 API
- 바뀐 로직
- 바뀐 화면
- 바뀐 문서

#### 4. 실측 결과
반드시 숫자로 쓴다.

#### 5. 무엇이 증명됐는가
예시:
- local shadow mode에서 `실험 생성 -> 배정 -> 주문 매핑 -> 결과 집계`가 닫혔다.

#### 6. 아직 증명되지 않은 것
예시:
- 실제 서비스에서는 payment success row가 아직 0건이다.

#### 7. 이 결과가 프로젝트에 주는 도움
예시:
- 이제 첫 라이브 실험 전에 코호트 기준 윈백 시점을 데이터로 잡을 수 있다.

#### 8. 다음 액션
한 개 또는 세 단계로 끝낸다.

### 4-3. 결과보고서에서 금지할 표현
- `완료`만 쓰고 기준을 안 적는 것
- `잘 됐다`, `의미 있었다`처럼 애매한 표현
- `운영에서도 될 것 같다`처럼 검증 없이 낙관하는 표현

### 4-4. 결과보고서에서 권장하는 표현
- `우리 기준 90% / 운영 기준 30%`
- `구현은 완료, live 연결 미완료`
- `유력 가설이지만 확정 아님`
- `이번 숫자는 uplift가 아니라 plumbing 검증 수치`

## 5. 로드맵 문서 규칙

로드맵은 할 일 목록이 아니라 **왜 이 순서여야 하는지 설명하는 문서**다.

### 5-1. 로드맵의 목표
- 무엇을 먼저 해야 하는지 정한다.
- 왜 그것이 먼저인지 설명한다.
- 무엇이 막히면 다음 단계가 의미 없어지는지 보여준다.

### 5-2. 로드맵에서 Phase를 쓰는 방식
각 Phase는 아래 5가지를 반드시 가져야 한다.

1. **목표**
2. **왜 지금 해야 하는가**
3. **산출물**
4. **완료 기준**
5. **다음 Phase에 주는 가치**

좋은 예:
- Phase 1은 CRM 메시지가 진짜 추가 매출을 만드는지 확인하는 측정 장치를 만든다.
- 그래서 실험 장부, 결제 블랙박스, 운영자 화면이 필요하다.
- 이게 끝나야 Phase 5의 Meta 비용과 Phase 7의 iROAS가 숫자로 연결된다.

### 5-3. Sprint 설명 규칙
Sprint는 기술명보다 역할명으로 먼저 설명한다.

**번호 체계는 `Phase{n}-Sprint{m}` 형식을 쓴다.** Sprint 번호는 Phase가 바뀌어도 **1부터 다시 시작하지 않고 이어서 올라간다.** 이렇게 하면 프로젝트 어디에서 Sprint 번호만 들어도 어느 단계인지 바로 감이 온다.

좋은 예:
- `Phase1-Sprint1`: 실험 장부
- `Phase1-Sprint2`: 결제 블랙박스
- `Phase1-Sprint3`: 운영자 관제판
- `Phase2-Sprint4`: 코호트 윈백
- `Phase2-Sprint5`: 알림톡 자동 발송
- `Phase2-Sprint6`: Meta 비용 연결

나쁜 예:
- `P1-S1`, `P1-S1A`, `P1-S2` (약어와 하위 분기는 금지)
- `Phase2-Sprint1` (Phase가 바뀌었는데 번호를 다시 시작)

하위 분기 금지 원칙:
- 기존 Sprint가 너무 커서 쪼개야 한다면 `S1A/S1B`로 나누지 말고 새 Sprint 번호를 할당한다.
- 번호는 소진형이다. 버려진 번호가 생겨도 다시 쓰지 않는다.

그 아래에 실제 코드/API를 붙인다.

### 5-4. 완료율 규칙
로드맵에는 가능하면 완료율을 한 개만 쓰지 말고 아래 중 하나를 쓴다.

- `우리 기준 / 운영 기준`
- 또는 `코드 구현 / 운영 반영`

좋은 예:
- `Phase1-Sprint1 = 100% / 40%`

나쁜 예:
- `Phase1-Sprint1 = 70%`

왜냐하면 70%가 로컬 검증인지, 운영 배포인지 알 수 없기 때문이다.

#### 100% / 100% Sprint에 "남은 후속" 금지

완료율이 `100% / 100%`인 Sprint는 **"남은 후속", "추가 작업", "To-do" 류 섹션을 두지 않는다.** 완료 기준을 다 맞췄는데 잔여 작업이 있다는 것은 둘 중 하나다.

1. 사실은 100%가 아니다 → 완료율을 내려 쓴다 (예: `95% / 100%`)
2. 잔여 작업이 이 Sprint의 완료 기준과 **무관**하다 → 그러면 이 Sprint가 아니라 **다른 위치**에 있어야 한다

잔여 작업을 옮길 수 있는 후보:

- 별도 Sprint로 분리 (새 Sprint 번호 할당)
- 문서 레벨의 `## 다음 액션` 섹션 (지금 당장 / 이번 주 / 승인 후)
- 별도 이슈 트래커나 follow-up 문서 링크

검증: 완료된 Sprint 본문을 보고 "여기 뭐 더 해야 할 일이 보이는가?"라고 물었을 때 **"아니오"**가 나와야 한다. 그래야 대표·운영자가 "이 Sprint는 끝났다"를 믿을 수 있다.

#### 진행 중 Sprint는 `§3-6C 실행 단계` 필수

`X < 100%`인 Sprint는 §3-6C에 따라 **실행 단계(담당 태그 포함 번호 리스트)**를 반드시 가진다. 실행 단계 없이 "지금 상태: 30%" 같은 수치만 있으면 "앞으로 무엇을"이 보이지 않아 아무도 움직이지 않는다.

### 5-5. 의존성 규칙
로드맵은 반드시 `왜 이 순서인가`를 적는다.

예시:
- Meta 비용은 중요하지만, 실험 원장과 식별키가 먼저 없으면 단순 리포트로 끝난다.
- 카카오는 중요한 채널이지만, 원장보다 먼저 붙이면 자동화만 생기고 측정 정확도는 안 생긴다.

### 5-6. 승인 포인트를 숨기지 않는다
대표가 승인해야 하는 항목은 문서 아래가 아니라 눈에 잘 띄게 적는다.

예시:
- DB 스키마 추가 승인
- 외부 API 자격증명 연결 승인
- 자동 발송 활성화 승인

## 6. Phase 상세 메모 규칙

Phase 상세 메모는 가장 기술적일 수 있다. 하지만 그래도 **의도 설명이 앞에 있어야 한다.**

### 6-1. 상세 메모의 필수 순서

1. 상단 요약
2. 이 Phase를 쉬운 비유로 설명
3. 각 Sprint의 역할
4. 현재 상태
5. 왜 아직 100%가 아닌가
6. 다음 순서
7. 개발 부록

### 6-2. 쉬운 비유를 적극 사용한다
단, 감상적인 비유 말고 **구조를 쉽게 만드는 비유**만 쓴다.

좋은 예:
- 실험 장부 = 시험 채점표
- 결제 블랙박스 = 사고 원인 기록 장치
- 운영자 화면 = 관제실 모니터

### 6-3. 상세 메모에서 꼭 넣어야 하는 질문
각 Sprint마다 아래를 넣는다.

- 무엇을 하는가
- 왜 필요한가
- 지금 어디까지 왔는가
- 아직 안 된 이유는 무엇인가
- 산출물은 무엇인가
- 우리 프로젝트에 어떻게 도움이 되는가

### 6-4. 상세 메모에서 코드 위치는 맨 뒤로 보낸다
코드 경로는 중요하지만, 본문 첫머리에서 길게 나오면 독해를 깨뜨린다.

권장 순서:
- 먼저 쉬운 설명
- 그다음 상태/의미
- 마지막에 코드 위치

## 7. 용어 사용 규칙

### 7-1. 처음 나오는 기술 용어는 반드시 풀어쓴다
예시:
- `holdout(아무 메시지도 보내지 않는 비교 그룹)`
- `control(기본 비교 그룹)`
- `treatment(메시지를 보내는 실험 그룹)`
- `receiver(서버 수신점)`
- `ledger(기록 원장)`

### 7-2. business term과 tech term을 구분한다
- business term: 대표와 운영자가 이해해야 하는 말
- tech term: 개발 부록에서 필요한 말

예시:
- 본문: `결제 블랙박스`
- 부록: `PG attribution ledger`

### 7-3. 이름이 오해를 부르면 즉시 바꾼다
예시:
- `first_purchase_channel`이 실제로는 결제 수단이면 `first_payment_channel`로 바꾼다.

## 8. 산출물 쓰는 규칙

문서에는 “파일이 생겼다”가 아니라 **운영상 무엇이 생겼는지**를 써야 한다.

좋은 산출물 표현:
- 실험 장부
- 승인 패키지
- 결제 비교표
- 코호트 입력값 API
- CRM 운영 허브

보조로 실제 구현물을 붙인다.

예시:
- 실험 장부
  - `crm_experiments`
  - `crm_assignment_log`
  - `crm_message_log`
  - `crm_conversion_log`

## 9. 우리 프로젝트에 도움이 되는지 쓰는 규칙

이 항목은 빠지면 안 된다.
문서의 모든 큰 섹션은 아래 중 최소 하나에 답해야 한다.

- 이 작업이 끝나면 어떤 판단을 더 잘할 수 있는가?
- 어떤 비용 낭비를 줄이는가?
- 어떤 숫자를 이제 믿을 수 있게 되는가?
- 다음 Phase를 왜 더 안전하게 여는가?

예시:
- 코호트 API가 생기면 `21일`, `30일`, `90일` 윈백 시점을 감이 아니라 데이터로 정할 수 있다.
- 결제 블랙박스가 생기면 `(not set)` 문제를 추측이 아니라 실제 결제 흐름으로 좁힐 수 있다.

## 10. 사실, 판단, 추정 쓰는 규칙

문서는 아래 3가지를 구분해서 쓴다.

### 사실
확인된 값, 구현된 것, 테스트 통과, API 응답

예시:
- `live ledger row 0건`
- `pytest 6 passed`

### 판단
사실을 보고 내린 현재 해석

예시:
- 현재 가장 큰 병목은 화면 부족이 아니라 live signal 부족이다.

### 추정
확정 전 가설

예시:
- `(not set) = PG 리다이렉트`는 유력 가설이지만 아직 확정 사실은 아니다.

권장 표기:
- `사실:`
- `현재 판단:`
- `유력 가설:`

## 11. 금지 표현 목록

아래 표현은 되도록 쓰지 않는다.

컨설턴트식 표현:
- `가시화`
- `제고`
- `고도화`만 단독으로 쓰기
- `인사이트 도출`
- `정합성 확보`만 쓰고 무엇이 맞는지 안 쓰기
- `완료 예정`
- `추후 검토`
- `효율화`
- `리소스`
- `유관`
- `시사점`

컴퓨터 전문용어(대표·운영자 대상 본문에서 단독 사용 금지):
- `파이프라인` 단독
- `인프라` 단독
- `스택` 단독
- `레이어` 단독
- `러닝`, `트레이닝` (학습/훈련으로 풀어쓰기)
- `프로덕션`만 쓰기 (운영이라고 쓴다)
- `배포` 뒤에 대상이 빠진 것 (무엇을 어디에 배포하는지 적는다)

대체 표현:
- `가시화` → `화면으로 보여줌`
- `제고` → `높임`
- `고도화` → `더 정확하게 만듦` 또는 `자동화 범위를 넓힘`
- `인사이트 도출` → `무엇이 문제인지 찾음`
- `정합성 확보` → `숫자가 서로 왜 다른지 설명 가능하게 만듦`
- `완료 예정` → `다음 단계에서 진행`
- `추후 검토` → `언제, 누가, 무엇을 볼지 적기`
- `파이프라인` → `데이터가 A에서 B로 흐르는 길` 또는 실제 이름 (`결제 적재 흐름` 등)
- `인프라` → `서버/DB/외부 API 묶음`
- `프로덕션` → `운영`

필요할 때 한 번 풀어써주면 그 뒤에 기술 용어를 써도 된다.

## 13. 최종 체크리스트

문서를 저장하기 전에 아래를 확인한다.

### 공통
- [ ] 최상단에 `작성 시각: YYYY-MM-DD HH:MM KST`가 있다
- [ ] `기준일`이 작성 시각과 맞춰 적혀 있다
- [ ] 상단 10초 요약이 있다
- [ ] 10초 요약 바로 뒤에 **Phase-Sprint 요약표**가 있다
- [ ] 요약표의 Sprint 번호는 전부 상세 섹션으로 가는 Obsidian wiki 링크 `[[#Sprint명]]`이다
- [ ] 각 Phase·Sprint 상세에 `[[#Phase-Sprint 요약표|▲ 요약표로]]` 링크가 있다
- [ ] 테이블 셀 안의 alias는 `\|`로 이스케이프했다 (예: `[[#Phase1-Sprint1\|이동]]`)
- [ ] Sprint 헤딩은 단순화됐다 (`#### Phase1-Sprint1`까지만, 이름은 헤딩 아래)
- [ ] Sprint 번호는 `Phase{n}-Sprint{m}` 형식이고 Phase가 바뀌어도 이어진다 (약어·하위 분기 없음)
- [ ] 각 Sprint에 `TJ / Codex / Claude Code` 역할 구분이 적혀 있다
- [ ] **100% / 100% Sprint에 "남은 후속", "추가 작업" 류 섹션이 없다** (잔여 작업은 다른 Sprint 또는 문서 레벨 "다음 액션"으로 이동)
- [ ] **진행 중(100% 미만) Sprint에 `실행 단계` 섹션이 있다** (번호 리스트, 각 단계에 `[TJ]/[Codex]/[Claude Code]` 담당 태그)
- [ ] `[TJ]` 단계 다음의 `[Codex]`/`[Claude Code]` 단계에는 `의존성: 선행필수/병렬가능/부분병렬` 중 하나와 이유가 적혀 있다
- [ ] **모든 `[TJ]` 태그는 §3-6B-1 기준으로 Codex/Claude Code가 자동화할 수 없는 일인지 검증했다** (2FA, 실결제, 신규 credential 발급, 사업 승인 등)
- [ ] **자동화 가능한 확인 작업(운영 DB read-only 쿼리, 기존 토큰 API 호출, `curl`/`tsc`/`git log`)은 `[Codex]` 또는 `[Claude Code]`로 먼저 할당했다**
- [ ] **애매한 권한/접근 작업은 `[Codex]`/`[Claude Code]` 1회 시도 뒤 `[TJ]` 에스컬레이션 순서로 배치했다**
- [ ] 실행 단계마다 `무엇`, `왜`, `어떻게`, `산출물`, `검증`이 보인다
- [ ] URL, 파일명, 테이블명만 나열한 단계가 없고, 사용 이유와 확인 방법이 같이 적혀 있다
- [ ] 실행 단계에 모호한 표현(`적절히`, `필요시`, `검토 후`)이 없다
- [ ] GitHub 스타일 앵커(`](#소문자-하이픈)`)와 `<a id>` 태그가 **문서에 한 개도 없다**
- [ ] 예시 표·샘플 마크다운은 fenced code block으로 감쌌다 (실제 링크로 렌더링되지 않도록)
- [ ] **링크 검증 스크립트(§15)를 돌려 미매치 0개 확인했다** — 이게 통과하지 않으면 저장 금지
- [ ] 문서 목적이 한 문장으로 있다
- [ ] 쉬운 한국어가 먼저 나오고, 기술 용어는 나중에 나온다
- [ ] 금지된 컴퓨터 전문용어(§11)를 본문에서 단독 사용하지 않았다
- [ ] 산출물이 무엇인지 적었다
- [ ] 우리 프로젝트에 주는 도움이 적혀 있다
- [ ] 다음 액션이 시간 순서로 적혀 있다
- [ ] 사실과 가설을 구분했다

## 15. 링크 검증 스크립트

문서를 저장하기 전 반드시 이 스크립트를 돌린다. 미매치가 하나라도 있으면 저장하지 않는다.

### 15-1. 인라인 실행 (bash + python3)

```bash
python3 - <<'PY' <문서경로>
import re, sys
from pathlib import Path

path = Path(sys.argv[1])
text = path.read_text()

# fenced code block과 inline code 제외 (예시·샘플은 검증 대상 아님)
body = re.sub(r'```[a-zA-Z]*\n.*?\n```', '', text, flags=re.DOTALL)
body = re.sub(r'`[^`\n]*`', '', body)

headings = {m.group(1).strip() for m in re.finditer(r'^#{1,6}\s+(.+?)\s*$', body, re.MULTILINE)}
# [[#헤딩]] 또는 [[#헤딩|alias]] (테이블 셀에서는 \| 이스케이프)
wiki_links = [m.group(1).strip() for m in re.finditer(r'\[\[#([^\]\\|]+)(?:\\?\|[^\]]*)?\]\]', body)]
gh_anchors = re.findall(r'\]\(#([a-z0-9\-_]+)\)', body)
html_ids = re.findall(r'<a\s+id\s*=\s*["\']([^"\']+)["\']', body)

missing = [l for l in wiki_links if l not in headings]
errors = 0
print(f"문서: {path}")
print(f"  헤딩 {len(headings)}개, wiki 링크 {len(wiki_links)}개")
if missing:
    print(f"  ✗ 미매치 wiki 링크 {len(missing)}개: {missing}")
    errors += 1
else:
    print(f"  ✓ wiki 링크 전부 헤딩과 매치")
if gh_anchors:
    print(f"  ✗ GitHub 스타일 앵커 {len(gh_anchors)}개 발견 (금지): {gh_anchors}")
    errors += 1
else:
    print(f"  ✓ GitHub 스타일 앵커 없음")
if html_ids:
    print(f"  ✗ <a id> 태그 {len(html_ids)}개 발견 (금지): {html_ids}")
    errors += 1
else:
    print(f"  ✓ <a id> 태그 없음")
sys.exit(errors)
PY
```

### 15-2. 재사용 가능한 스크립트 파일 만들기 (선택)

프로젝트 루트에 `scripts/validate_wiki_links.py`로 저장해두면 `python3 scripts/validate_wiki_links.py <문서>`로 짧게 부를 수 있다.

### 15-3. 스크립트가 잡는 3가지 오류

1. **wiki 링크 미매치**: `[[#존재하지않는헤딩]]` → Obsidian에서 클릭 시 "없는 헤딩" 에러
2. **GitHub 스타일 앵커 잔존**: `[text](#소문자-하이픈)` → Obsidian에서 클릭해도 이동 안 함
3. **`<a id>` 태그 잔존**: Obsidian에서 무시되므로 의미 없는 markup

세 가지 전부 0이어야 저장 가능하다.

### 15-4. 한계

- 다른 파일을 가리키는 `[[파일명#헤딩]]` 링크의 타겟 파일 존재 여부
- 실제 Obsidian 렌더링 결과 (최종 확인은 Obsidian에서 클릭 테스트)
