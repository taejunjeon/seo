# 전체 회원 Consent 동기화 완료 — 결과보고서

작성일: 2026-04-02

## 10초 요약

아임웹 API v2의 `offset`은 page 번호로 동작하며 (row offset 아님), `orderBy=jointime`과 결합하면 전체 순회가 가능하다. **69,681명 전원 동기화 완료.** SMS 동의율 **45.4%** (31,636명), 전화번호 보유 **99.9%**. consent 데이터 소스 문제가 **진짜로 닫혔다.**

---

## 1. Offset 실측 결과

### 핵심 발견

| 항목 | 결과 |
|------|------|
| `offset`의 의미 | **page 번호** (row offset 아님) |
| `orderBy=jointime` | ✅ 전체 69,980명을 가입일순 정렬 |
| `total_page` | 700 (limit=100 기준) |
| offset 0 | page 1 ✅ |
| offset 1~3 | ❌ 불안정 (빈 결과 또는 page 1 중복) |
| offset 4~698 | ✅ **안정적으로 순차 반환** |
| offset 699 | ❌ 간헐적 빈 결과 |
| offset 700 | ✅ 마지막 페이지 (80건) |
| `join_time_start/end` 필터 | ❌ v2에서 미작동 (0건 반환) |

### 결론

`join_time` 기간 분할은 v2 API에서 불가. 대신 **`orderBy=jointime` + offset을 page로 사용 + 빈 페이지 skip** 방식이 작동한다.

---

## 2. 새 Sync 알고리즘

```
1. offset=0 (page 1) 조회 → totalPage 확인
2. offset=4부터 시작 (1~3은 버그)
3. 각 페이지에서 100건씩 upsert
4. 빈 페이지 → skip, 연속 5회 빈 페이지 → 종료
5. 200페이지마다 토큰 갱신
6. 200ms 간격으로 rate limit 대응
```

### 구현 위치

`backend/src/routes/crmLocal.ts` — `POST /api/crm-local/imweb/sync-members`

## 4. 운영 시 주의점

### 증분 Sync

전체 sync는 ~6분 소요. 매일 돌릴 필요 없다. 증분 방법:
- `orderBy=jointime` + 마지막 sync 시점 이후 가입자만 조회 (`join_time` 기간 필터가 v2에서 안 되므로, 마지막 페이지부터 역순으로 새 회원만 upsert)
- 또는 주 1회 전체 sync로 충분 (6분이면 운영 부담 적음)

### 사이트 구분

현재 `imweb_members` 테이블에 사이트 구분 필드가 없다. 이 API 키가 바이오컴/더클린커피/aibio 전체의 회원을 반환하는지, 바이오컴만인지 확인 필요.

**TJ님 확인 필요**: 더클린커피와 aibio의 아임웹 API 키가 별도인지 동일인지.

### 토큰 만료

아임웹 토큰은 일정 시간 후 만료. sync 중 200페이지마다 자동 갱신 구현됨.

### Rate Limit

200ms 간격으로 호출. 아임웹 공식 rate limit이 명시되어 있지 않으므로, 문제 발생 시 간격 늘리기.
