# AIBIO 리드 상태값 승인안

작성 시각: 2026-04-26 14:30 KST
기준일: 2026-04-26
대상: AIBIO 자체 리드/예약 원장
목적: TJ님이 `YES` 또는 `NO + 수정사항`으로 답할 수 있도록 리드 상태값 추천안을 제시한다.

## 10초 요약

제가 추천하는 상태값은 `신규 -> 연락중 -> 상담완료 -> 예약확정 -> 방문완료 -> 결제완료`를 기본 흐름으로 두고, 예외 상태로 `노쇼`, `제외`를 두는 안이다. 현재 코드의 내부 상태 8개는 그대로 쓸 수 있고, 화면 표시 이름만 일부 바꾸면 된다. 추천 자신감은 **76%**다. 실제 상담원 처리 로그와 아임웹 입력폼 export가 아직 없어서 90% 이상으로 보기는 어렵다.

## TJ님 답변 방법

아래 셋 중 하나로 답하면 된다.

| 답변 | 의미 | Codex 다음 작업 |
|---|---|---|
| `YES` | 추천안 A를 그대로 승인 | 화면 표시명, 문서, 테스트를 추천안 A 기준으로 반영 |
| `NO: 기존명 유지` | 현재 후보명 그대로 사용 | 코드 변경 없이 문서만 승인 상태로 갱신 |
| `NO: 수정` | 특정 이름만 바꿈 | TJ님 수정안을 반영해 화면 표시명과 문서 업데이트 |

수정 예시:

```text
NO: 연락중은 연락시도로 유지, 상담완료는 연락완료로 유지, 제외는 제외로 변경
```

## 추천안 A

**제 추천: YES**

이 안은 상태 개수를 늘리지 않고, 상담원이 실제로 다음 행동을 판단하기 쉬운 이름으로 바꾸는 방식이다. DB 구조 변경 없이 화면 표시명 중심으로 바꿀 수 있다.

| 순서 | 내부 코드 | 현재 표시 | 추천 표시 | 상담원이 이 상태로 바꾸는 시점 | 다음 행동 | 추천 자신감 |
|---:|---|---|---|---|---|---:|
| 1 | `new` | 신규 | 신규 | 폼이 막 들어왔고 아직 상담원이 처리하지 않음 | 가능한 빨리 연락 시작 | 95% |
| 2 | `contact_attempted` | 연락시도 | 연락중 | 전화, 카카오, 문자 중 하나를 시도했지만 아직 상담이 끝나지 않음 | 재연락 또는 답변 대기 | 78% |
| 3 | `contacted` | 연락완료 | 상담완료 | 고객과 연결되어 상담 목적, 방문 가능성, 다음 단계를 확인함 | 예약 확정 또는 제외 판단 | 72% |
| 4 | `reserved` | 예약완료 | 예약확정 | 방문 날짜와 시간이 확정됨 | 방문 전 리마인드 | 88% |
| 5 | `visited` | 방문완료 | 방문완료 | 고객이 실제 센터에 방문함 | 결제 여부 확인 | 86% |
| 6 | `paid` | 결제완료 | 결제완료 | 체험권, 예약금, 센터 결제 중 하나가 확인됨 | 매출 귀속 확인 | 76% |
| 7 | `no_show` | 노쇼 | 노쇼 | 예약했지만 예약 시간에 방문하지 않음 | 재예약 유도 또는 제외 | 80% |
| 8 | `invalid_duplicate` | 불량/중복 | 제외 | 테스트, 중복, 잘못된 연락처, 대상 아님, 장기 무응답 등 퍼널에서 빼야 함 | 메모에 제외 사유 입력 | 68% |

## 왜 이렇게 추천하는가

`연락시도`는 상담원이 여러 번 연락하는 동안 상태가 애매하다. 화면에는 `연락중`이 더 낫다. 한 번 전화했는지, 세 번 전화했는지는 메모나 추후 `연락횟수` 필드로 다루는 편이 좋다.

`연락완료`는 너무 넓다. 단순히 벨이 울렸거나 문자 답변만 온 것도 연락완료처럼 보일 수 있다. AIBIO의 목적은 상담과 방문 전환이므로, 실제 상담이 된 상태는 `상담완료`가 더 분명하다.

`예약완료`는 결제 완료처럼 들릴 수 있다. 오프라인 센터 운영에서는 날짜와 시간이 잡힌 상태이므로 `예약확정`이 더 정확하다.

`불량/중복`은 고객을 낮춰 부르는 표현처럼 보일 수 있고, 운영자가 쓰기에도 거칠다. 화면에는 `제외`라고 두고, 실제 이유는 메모에 `중복`, `테스트`, `연락처오류`, `대상아님`, `무응답3회`, `기타`처럼 남기는 것을 추천한다.

## 운영 규칙 v1

기본 흐름:

```text
신규 -> 연락중 -> 상담완료 -> 예약확정 -> 방문완료 -> 결제완료
```

예외 흐름:

```text
예약확정 -> 노쇼
신규/연락중/상담완료 -> 제외
노쇼 -> 예약확정
```

운영 입력 규칙:

| 상황 | 선택할 상태 | 메모 예시 |
|---|---|---|
| 폼 제출 직후 | 신규 | 자동 생성, 상담원 입력 필요 없음 |
| 전화했지만 안 받음 | 연락중 | 1차 부재, 16시 재연락 |
| 카카오로 답변이 왔지만 예약 전 | 상담완료 | 식욕 조절 관심, 평일 오후 가능 |
| 날짜와 시간이 잡힘 | 예약확정 | 2026-04-28 15:00 방문 예정 |
| 실제 센터 방문 | 방문완료 | 인바디 측정 완료 |
| 결제 확인 | 결제완료 | 현장 결제 또는 체험권 결제 확인 |
| 예약 시간 미방문 | 노쇼 | 당일 미방문, 재예약 필요 |
| 중복/테스트/연락처 오류 | 제외 | 중복 접수, 잘못된 번호, 테스트 |

## YES 이후 Codex가 할 일

추천안 A가 승인되면 Codex가 바로 할 수 있는 작업은 아래다.

| 순서 | 작업 | 무엇을 바꾸는가 | 위험도 |
|---:|---|---|---|
| 1 | 화면 표시명 변경 | `연락시도 -> 연락중`, `연락완료 -> 상담완료`, `예약완료 -> 예약확정`, `불량/중복 -> 제외` | 낮음 |
| 2 | 문서 상태값 정본 갱신 | `imweb/!imwebplan.md`, `imweb/aibio-tj-next-actions.md` | 낮음 |
| 3 | Playwright 테스트 갱신 | admin 드롭다운 라벨이 바뀌어도 테스트가 통과하게 수정 | 낮음 |
| 4 | 운영자 메모 가이드 추가 | `제외` 상태 선택 시 메모에 사유를 남기도록 문구 보강 | 낮음 |

DB 스키마 변경은 필요 없다. 내부 코드는 그대로 두고 화면 표시명만 바꾸면 된다.

## 추천 자신감

| 항목 | 점수 |
|---|---:|
| 추천안 A 전체 자신감 | 76% |
| DB 구조 변경 없이 적용 가능성 | 92% |
| 상담원이 이해하기 쉬울 가능성 | 78% |
| 퍼널 리포트에 바로 쓸 수 있을 가능성 | 82% |
| 실제 현장 용어와 완전히 일치할 가능성 | 58% |

자신감을 76%로 둔 이유:

- 현재 코드와 문서 기준으로는 AIBIO가 `광고 -> 상담 -> 예약 -> 방문 -> 결제` 흐름이라는 점이 분명하다.
- 다만 실제 상담원이 어떤 단어를 쓰는지, 연락 실패를 몇 회까지 추적하는지, 노쇼 후 재예약을 어떻게 처리하는지 실제 운영 로그가 없다.
- 그래서 구조 자체는 자신 있지만, 화면 표시명은 현장 용어 확인 전까지 중간 신뢰도로 둔다.

## 자신감을 올리려면 필요한 데이터

| 필요한 데이터 | 왜 필요한가 | 있으면 바뀔 수 있는 결정 |
|---|---|---|
| 최근 30일 아임웹 입력폼 export | 실제 리드 수, 중복, 연락 가능성 기준 확인 | `제외` 사유 분류를 더 정확히 만들 수 있음 |
| 상담원이 실제 쓰는 처리 메모 20건 | 현장 용어 확인 | `상담완료`, `예약확정` 표시명을 바꿀 수 있음 |
| 전화/카카오 연락 시도 횟수 기준 | 몇 회 무응답이면 제외할지 결정 | `연락중`과 `제외` 사이 규칙 확정 |
| 예약 후 노쇼/재예약 사례 | 노쇼 후 다시 예약되는 비율 확인 | `노쇼 -> 예약확정` 허용 여부 확정 |

## 근거와 데이터 정확성

이번 추천은 실제 상담원 로그가 아니라 현재 코드, 기존 문서, AIBIO 사업 성격을 기준으로 만든 운영 설계안이다.

| 근거 | source | 기준 시각 | window | site | freshness | confidence |
|---|---|---|---|---|---|---:|
| 현재 구현된 내부 상태값 8개 | `backend/src/aibioNativeLeadLedger.ts`, `frontend/src/lib/aibio-native.ts` | 2026-04-26 14:30 KST | 현재 코드 | AIBIO | 높음 | 95% |
| AIBIO가 상담/예약/방문 중심이라는 판단 | `imweb/!imwebplan.md`, AIBIO native lead 구현 문서 | 2026-04-26 14:30 KST | 현재 프로젝트 문서 | AIBIO | 중간 | 82% |
| 실제 상담원 용어 | 아직 없음 | 2026-04-26 14:30 KST | 없음 | AIBIO | 낮음 | 0% |
| 최근 30일 아임웹 폼 처리 결과 | 아직 export 없음 | 2026-04-26 14:30 KST | 없음 | AIBIO | 낮음 | 0% |

그래서 이 문서의 성격은 “확정된 현장 룰”이 아니라 “TJ님이 바로 YES/NO로 승인할 수 있는 추천안”이다.

## 프론트엔드를 만들지 않은 이유

이번 결정은 버튼을 눌러 비교해야 하는 복잡한 화면 문제가 아니다. 상태명 8개와 운영 규칙을 이해하고 승인하는 문제다. 그래서 MD가 더 빠르고 명확하다고 판단했다.

다만 TJ님이 `YES`를 주면 Codex가 다음 단계에서 admin 화면의 드롭다운 표시명을 실제로 바꾸고, 필요하면 `제외 사유` 선택 UI까지 추가할 수 있다.

## 제가 최종 추천하는 답

TJ님에게 제가 추천하는 답은 아래다.

```text
YES
```

단, `제외` 상태를 쓸 때는 메모에 사유를 반드시 남기는 운영 규칙을 같이 적용해야 한다. `제외` 하나로만 끝내면 나중에 광고 품질 분석에서 `중복`, `연락처 오류`, `무응답`, `대상 아님`이 섞여 보이기 때문이다.
